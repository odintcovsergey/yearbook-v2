## ✅ DONE — Авто-зеркало мастеров `page-any` на правой странице

**Проблема.** Мастера общего раздела и сеток нарисованы со смещением контента к
корешку (внутреннее поле ≠ внешнее). Левая и правая страницы зеркальны, но
движок клал один и тот же `page-any` мастер одинаково на обе стороны — значит
смещение к корешку верно только слева, справа контент уезжал к внешнему краю.

**Решение.** На ПРАВОЙ странице отражаем геометрию мастера по вертикальной оси —
но только если у мастера `page_type === 'page-any'`. Дизайнер рисует один
мастер (смещённый к корешку), движок зеркалит для правой стороны. Явные
`-Left/-Right` мастера (E-*, J-Quarter) имеют `page_type` `page-left/page-right`
и под условие не попадают — приоритет «явный правый > авто-зеркало > левый»
получается сам собой.

### Архитектурное решение (отличие от исходного ТЗ)

ТЗ предлагал зеркалить при сборке (`layout-to-buildresult.ts`). Это НЕ подошло,
т.к. в реальности:
- у страницы НЕ хранится своя копия плейсхолдеров — `SpreadInstance` хранит
  только `template_id` + `data`, а координаты рамок берутся из общего мастера
  при каждом рендере. Один `page-any` мастер слева и справа ссылается на ОДИН
  чертёж — отзеркалить чертёж при сборке нельзя (перевернёт и левую);
- сборка сохраняется заморозкой в `album_layouts.spreads`, а сторона (left/right)
  вычисляется на лету при показе (`segmentToSpreads`). После ручных правок
  (замена мастера, перестановка) build-time зеркало рассинхронизировалось бы.

**Поэтому зеркалим в момент рендера**, одной чистой функцией, которую дёргают
оба рендера. Это «одна точка» по духу ТЗ (одна трансформация), просто вызывается
там, где каждый рендер впервые материализует плейсхолдеры мастера.

### Что сделано (файлы)

- **`lib/album-builder/mirror-placeholders.ts`** (новый) — чистые функции:
  - `mirrorPlaceholders(placeholders, pageWidthMm)`: x→W−x−w; y/w/h без изменений;
    rotation→−rotation; text align left↔right (center/justify как есть);
    декор: пересчёт `offset_x_mm` от отражённой базы (foreground attached_to=''
    не трогаем); пиксели картинок НЕ переворачиваются. Вход не мутирует.
  - `resolvePlaceholdersForSide(...)`: применяет зеркало ТОЛЬКО при
    `side==='right' && page_type==='page-any'`, иначе возвращает вход как есть.
- **`lib/album-builder/types.ts`** — добавлен тип `PageType` и поле
  `SpreadTemplate.page_type?: PageType | null`.
- **`app/api/layout/route.ts`** — `page_type` добавлен в `SPREAD_TEMPLATE_FIELDS`
  (редактору). PDF грузит мастера через `select('*')` — там уже было.
- **`lib/pdf-export/pipeline.ts`** — в `renderSpread` зеркало применяется как
  ФИНАЛЬНАЯ трансформация после `applyBalanceFromData` + `orderPlaceholdersForRender`.
  Сторона берётся из уже существующего `ctx.sideByIndex` (тот же `segmentToSpreads`,
  что и для фонов).
- **`app/app/_components/AlbumSpreadCanvas.tsx`** — единый список
  `renderPlaceholders` (с зеркалом) используется ВСЕМИ потребителями координат:
  Konva-слоты, DropZone'ы (drag&drop), PhotoCropOverlay (кроп). Иначе оверлеи
  разъехались бы со слотами на правой странице. `SpreadOrderStrip` использует
  `AlbumSpreadCanvas` с `pageSide` → миниатюры зеркалятся автоматически.

### Почему зеркало ПОСЛЕ балансировки (важно для будущих правок)

`__pos__<label>` (центрирование неполных рядов) пишет ТОЛЬКО rule engine
(`lib/rule-engine/sections/shared.ts` `centerLastRowSlots`) в ИСХОДНЫХ
координатах мастера и mirror-неосознанно. Партнёр слоты НЕ перетаскивает (только
назначает фото, кропает, меняет стиль текста). Поэтому правильный порядок:
сначала балансировка в исходном пространстве → потом отражаем всё целиком.
Партнёрский `__halign__` рендер применяет ПОВЕРХ (в уже показанном, отражённом
пространстве) — его НЕ трогаем; зеркалится только базовый `align` мастера.

### ⚠️ Открытые вопросы / ограничения

1. **§4 ТЗ — стык с InDesign-автовёрсткой (НЕ реализован, продуктовое решение).**
   Авто-зеркало чинит ВНУТРЕННИЙ рендер OkeyBook (редактор + PDF). Если финальная
   типографская вёрстка пойдёт через CSV → InDesign autoflow, InDesign разложит
   контент по фреймам мастера из IDML и так же НЕ отзеркалит — проблема повторится
   там. Варианты: (а) CSV-экспорт отдаёт InDesign уже зеркальные координаты для
   правых страниц; (б) дизайнер рисует реальные `-Left/-Right` мастера в IDML
   (путь B). Решать когда появится автовёрстка. **CSV-экспорта пока нет в коде.**

2. **Soft-альбомы (мягкая обложка) — известное ограничение.** Сторона (left/right)
   для зеркала берётся из `segmentToSpreads`. Редактор зовёт его с
   `{softShift: isSoftAlbum}`, а PDF — без `softShift` (как и для фонов сейчас).
   Для layflat (дефолт) обе стороны совпадают → зеркало консистентно редактор↔PDF.
   Для soft-альбомов возможен off-by-one parity рассинхрон — НО это ПРЕД-существующее
   поведение (фоны в PDF для soft уже зависят от того же `sideByIndex`); зеркало
   лишь наследует его, нового класса багов не вносит. Полное выравнивание требует
   прокинуть effective_print_type в PDF-pipeline (export-роут грузит у альбома
   только id/title/tenant_id) — отложено как отдельная задача.

3. **`J-Collage-6`** из списка ТЗ в `family-mapping.ts` ещё нет (есть только
   `J-Collage-4`). Когда добавят — получит `page-any` и зеркало подхватит само.

### Откат

Фича чисто визуальная, в памяти на этапе рендера. Без миграций БД. Откат —
revert коммитов. Существующие тестовые альбомы: на правых страницах
page-any-мастеров контент сдвинется в зеркало (это и есть цель). Ручные
кропы/назначения фото не пострадают (ключи по label, не по координатам).
