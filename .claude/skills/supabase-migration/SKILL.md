# Миграция Supabase (yearbook-v2)

## Зачем

У нас нет автоматических миграций. Любое изменение схемы — это:
Claude пишет SQL → Сергей применяет руками через Supabase Studio → сообщает
результат. Этот навык держит процесс безопасным и воспроизводимым.

## Когда применять

- Новая таблица / колонка / индекс / enum / RLS-политика
- Изменение типа или ограничения существующей колонки
- Любой `ALTER` / `CREATE` / `DROP`, затрагивающий прод-схему

## Процесс

### 1. Понять, что меняем и зачем

Одна фраза: какое поведение продукта это включает? Какие таблицы трогаем?
Прочитать текущую структуру (`schema.sql` и/или последние файлы в
`migrations/`) ПЕРЕД написанием — не выдумывать колонки.

### 2. Написать датированный SQL-файл

Путь и имя: `migrations/2026-MM-DD-краткое-описание.sql` (формат как у
существующих файлов в папке).

Правила SQL:
- **Префиксы таблиц в JOIN и WHERE всегда** — `albums.id`,
  `spread_templates.template_set_id`, даже если колонка уникальна.
  Иначе при первом join'е стрельнёт `column reference is ambiguous`.
- Идемпотентность где можно: `IF NOT EXISTS`, `IF EXISTS`.
- Индексы на больших/горячих таблицах — `CREATE INDEX CONCURRENTLY`
  (вне транзакции), чтобы не залочить.
- Комментарий сверху файла: что и зачем.

### 3. Выдать текст в чат

Показать готовый SQL прямо в чат (через `cat` файла), чтобы Сергей
скопировал в Supabase Studio → SQL Editor. НЕ пытаться применить самому —
прямого доступа к проду нет.

### 4. План отката

Сразу написать обратный SQL (или явно «как восстановить»):
- добавили колонку → `ALTER TABLE ... DROP COLUMN ...`
- добавили таблицу → `DROP TABLE ...`
- данные мигрировали → как вернуть / идемпотентно ли

### 5. Дождаться подтверждения

Сергей применяет и сообщает результат. Код, который ЖДЁТ эту схему,
выкатывается ТОЛЬКО после подтверждения, что миграция применена.
Иначе прод упадёт на отсутствующей колонке.

### 6. Отразить в schema.sql (когда нужно)

`schema.sql` — стартовая схема для нового проекта Supabase с нуля.
Дублировать туда изменение, если важно, чтобы новый деплой поднялся
корректно. Историю старых миграций не переписываем.

## Грабли проекта

- `column reference is ambiguous` — забыли префикс таблицы. Самая частая.
- Вторая FK на таблицу ломает PostgREST embed → запрос `foo(...)` становится
  неоднозначным, могут «пропасть» данные. Указывать связь явно:
  `target!fk_column(...)`.
- `.env.local` должен содержать НАСТОЯЩИЕ значения перед любыми операциями
  с БД (не заглушки от CI).

## Анти-паттерны

- SQL без префиксов таблиц «потому что поле и так уникальное»
- Выкатка кода до применения миграции
- Миграция без плана отката
- Попытка применить миграцию самому (нет доступа к проду)
- `DROP`/`TRUNCATE` без явного подтверждения Сергея (необратимо)
