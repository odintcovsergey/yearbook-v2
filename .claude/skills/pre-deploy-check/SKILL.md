# Чеклист перед выкаткой (yearbook-v2)

## Зачем

Последняя проверка перед тем, как изменение увидят живые пользователи.
Ловит «забыл применить миграцию», «.env.local остался с заглушкой»,
«сломал привязку альбомов к мастерам», «выкатил в пятницу вечером».

У нас **нет staging и нет автоматического отката** — Vercel катит на прод
сразу при push в `main`. Поэтому проверка идёт ДО мерджа в main.

## Когда применять

- Перед мерджем feature-ветки в `main`
- Перед коммитом в `main`, меняющим видимое пользователю поведение
- Перед миграцией БД, которую применит Сергей

## Когда НЕ применять

- Правка текста/доков/комментариев
- Изменение, которое глазами не проверишь и пользователя не трогает
  (тогда хватает 3 проверок ниже)

## Блок 1. Три обязательные проверки (всегда)

```bash
npx vitest run        # все зелёные (на сейчас ~758/758)
npx tsc --noEmit      # пусто
npx next build        # зелёный
```

- [ ] vitest — зелёные
- [ ] tsc — пусто
- [ ] next build — зелёный

Красный tsc НЕ компенсируется зелёными vitest/build — тесты не
типизируются строго. Все три должны быть зелёными ОДНОВРЕМЕННО.

## Блок 2. Глазами (для видимых изменений)

- [ ] Открыл **preview-ссылку Vercel** на feature-ветке и посмотрел вживую
- [ ] Прокликал именно тот сценарий, что менял (канвас / кабинет партнёра /
      родительский вход — что относится)
- [ ] На preview ничего в БОЕВОЙ базе не сломал (preview смотрит в прод-данные!
      записывающие действия там опасны — см. стратегию тестовой среды)

## Блок 3. База данных / миграции

- [ ] Если менялась схема — датированный SQL лежит в `migrations/`
- [ ] SQL с **префиксами таблиц** в JOIN/WHERE (`albums.id`, не просто `id`) —
      иначе `column reference is ambiguous`
- [ ] Дал Сергею текст миграции для Supabase Studio → SQL Editor
- [ ] Есть план отката миграции (обратный SQL или «как восстановить»)
- [ ] Сергей подтвердил, что миграция применена ДО выкатки кода, который её ждёт
- [ ] Если новый деплой с нуля важен — продублировал изменение в `schema.sql`

## Блок 4. Секреты и окружение

- [ ] `.env.local` содержит НАСТОЯЩИЕ значения (не заглушки от CI/тестов сборки)
- [ ] Никаких секретов/токенов в коде — только env
- [ ] Если добавлялась env-переменная — она задана в Vercel (проверить, не угадывать)

## Блок 5. Специфичные грабли проекта

- [ ] **IDML force=true upload** не делался на проде без нужды — он меняет
      `template_id` (DELETE+INSERT) и ломает привязку альбомов к мастерам.
      В проде — версионирование через новые slug, не перезалив.
- [ ] Если менялась структура мастера — обновлён `family-mapping.ts`
      (`page_role`/`slot_capacity`/`applies_to_configs` берутся ТОЛЬКО оттуда)
- [ ] Если добавлялся шрифт — синхронизированы все 4 места
      (`public/fonts/*.ttf`, `globals.css` @font-face, `font-loader.ts`,
      `AVAILABLE_FONTS`)
- [ ] Не трогал `lib/balance-overrides/` без причины (рабочий движок)

## Блок 6. Риски и откат (объяснить Сергею простым языком)

- [ ] Что может сломаться — простыми словами, без жаргона
- [ ] Кого затронет — партнёр / родитель / фотограф
- [ ] Можно ли откатить и как:
  - код → `git revert` + Vercel пересоберёт сам
  - схема БД → обратный SQL применяет Сергей вручную
  - загруженные файлы / S3 / письма → отдельный план
- [ ] **Не пятница вечер** (если не срочно) — некому будет тушить

## Блок 7. После выкатки

- [ ] Vercel-деплой зелёный (сборка прошла)
- [ ] Прокликал критичный сценарий уже на проде
- [ ] Если что-то странное в первые минуты — готов сделать revert

## Анти-паттерны (так не делаем)

- «Локально работает — катим» без взгляда на preview
- Выкатка кода, который ждёт миграцию, ДО применения миграции
- `.env.local` с заглушкой ушёл в операции с реальной БД
- force=true перезалив IDML на проде «по-быстрому»
- Зелёные тесты вместо реальной проверки в браузере
- Молчаливый коммит без объяснения рисков Сергею
