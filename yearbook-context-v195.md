## Что чинили

Баг с живого прогона: на ВСЕХ портретных обложках показывалось одно имя
«Фёдорова Варвара» (у Смирнова, Морозовой и т.д.), хотя это разные ученики.

### Разбор (важно для будущего)
- Сначала ошибочно подумали, что это запечённый текст в декор-картинке
  `cover_student_name__under`. Скачали PNG — там просто **зелёная лента-плашка
  БЕЗ текста** (корректный декор). Не она.
- Реальная причина: текст-слот `cover_student_name` редактор сохранял на уровне
  ТИПА обложки (scope='type', cover_type='portrait_photo'). Подтверждено
  запросом: в cover_edits лежала строка `portrait_photo / NULL /
  cover_student_name='Фёдорова Варвара'` — общая правка перекрыла авто-имя у
  всех портретных обложек. Имя и класс — ЛИЧНЫЕ поля ученика, их нельзя на тип.

## ✅ DONE (этот инкремент, миграции НЕТ)

1. **lib/cover/editor-merge.ts**: `PER_STUDENT_COVER_LABELS = ['cover_student_name',
   'cover_class']` + единая `mergeCoverData(base, typePatch, studentPatch)` —
   личные метки-КОНТЕНТ из type-слоя выкидываются (база/ученик побеждают).
   Служебные ключи стиля (`__fontSize__cover_student_name` и т.п.) остаются
   общими на тип (все имена выглядят одинаково). mergeCoverEditsInto теперь
   зовёт mergeCoverData → чинит и существующие данные при сборке/превью.
2. **Редактор обложек** (page.tsx):
   - все слияния base⊕type⊕student переведены на mergeCoverData (data-memo,
     coverThumb, превью «Вид»);
   - правка ИМЕНИ/КЛАССА (onTextSubmit) пишется в область УЧЕНИКА
     (scope='student'), остальные тексты — на тип;
   - при загрузке личные метки выкидываются из typePatches (чтобы залипшая
     общая «Фёдорова Варвара» не вернулась при следующем сохранении типа);
   - подсказка под холстом обновлена («Имя и класс — индивидуальны…»).
3. Тесты editor-merge.test.ts (+3, личные метки).
