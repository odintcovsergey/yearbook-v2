# Фаза РЭ.25 — Галка покупки альбома ребёнком

**Статус**: ✅ Закрыта
**Сроки**: 21.05.2026 (одна сессия от РЭ.25.0 до РЭ.25.7)
**Главная цель**: дать партнёру и родителю возможность отметить, что конкретный ребёнок **не заказывает** выпускной альбом. Engine не создаёт для таких детей персональную страницу, но они остаются в общих фото класса и виньетках. Это исправляет фактическое несоответствие между списком учеников класса и списком заказчиков альбома.

## Что было до фазы

После РЭ.24 (готовые шаблоны для партнёра) система работала на двух движках сборки:
- Семантический engine с `section_structure` (новый путь, РЭ.22)
- Legacy через `config_type + print_type` (старый путь)

В обоих случаях секция `students` собиралась по **всем** ученикам из таблицы `children`. Если в классе из 25 человек 3 не заказывали альбом — фотограф получал 25 личных страниц вместо 22. Это:
- увеличивало себестоимость печати,
- ломало соответствие «оплачено = выдано»,
- противоречило реальной практике школьной фотографии.

При этом удалить ребёнка из таблицы было неправильно — он физически на общих фото класса и в виньетке, нужно его там оставить.

## Что сделано

### 1. Миграция БД (РЭ.25.1, commit `b8f671f`)

Файл `migrations/2026-05-21-children-is-purchased-and-album-include-non-purchasers.sql`:
- `children.is_purchased BOOLEAN NOT NULL DEFAULT true` — заказывает ли ребёнок альбом.
- `albums.include_non_purchasers BOOLEAN NOT NULL DEFAULT false` — переопределение для всего альбома: включить ли не-заказчиков в личный раздел независимо от их `is_purchased`.
- Индекс `idx_children_is_purchased_album (album_id, is_purchased)` для агрегаций.

Аддитивная zero-downtime миграция, существующие записи получают дефолты автоматически. Старый код, не видящий новых колонок, продолжает работать. Применена в Supabase 21.05.2026.

### 2. API endpoints (РЭ.25.2, commit `b0d76ca`)

**POST `/api/tenant`:**
- Новый action `update_child` — патч-апдейт ученика: `{ child_id, full_name?, class?, is_purchased? }`. Tenant-aware через `assertChildAccess`, аудит-лог `child.update`. Если все patch-поля пусты → 400.
- `create_album` insert расширен: `include_non_purchasers` (default false).
- `update_album` `allowedFields` расширен: `'include_non_purchasers'`.

**GET `/api/tenant`:**
- `action='children'` возвращает `is_purchased` в каждом элементе.

**PUT `/api/select`:**
- Новый action `set_purchased` — родитель меняет `children.is_purchased`. Token-аутентификация. **Доступен ВСЕГДА**, даже при `submitted_at !== null` — родитель может передумать после отбора фото.

**GET `/api/child`:**
- SELECT ребёнка расширен полем `is_purchased` для UI родителя.

Типы `Album.include_non_purchasers?` и `Child.is_purchased?` в `types/index.ts` и в локальном типе `app/app/page.tsx` (оба опциональны для бэк-совместимости).

### 3. Engine-фильтр (РЭ.25.3, commit `07cac78`)

**Архитектурное решение**: фильтр живёт **не в engine**, а в `lib/smart-fill/build-album-input.ts`. Применяется ДО формирования `AlbumInput.students[]`. Engine (`lib/album-builder`) остаётся **чистым** — работает со списком учеников как с непрозрачной коллекцией, не знает про `is_purchased`.

**Чистая функция** `filterChildrenByPurchase<C>(children, includeAll)` вынесена в отдельный модуль `lib/smart-fill/filter-by-purchase.ts` — без зависимости от Supabase, тестируется напрямую в vitest без env-переменных.

Логика:
- `includeAll=true` → все ученики (мягкий режим).
- `includeAll=false` → отсекаются `is_purchased===false` (строгий режим, default).
- `undefined/null` трактуются как `true` (бэк-совместимость с не-мигрированной БД и тестовыми фикстурами).

**Warning `non_purchasers_filtered`** — для наблюдаемости в build-логах. Эмитится только в строгом режиме при ненулевом количестве отсечённых.

**12 unit-тестов** в `lib/smart-fill/__tests__/filter-children-by-purchase.test.ts`:
- бэк-совместимость (`true` / undefined / null → попадают),
- строгий vs мягкий режим,
- mixed 22+3 в обе стороны,
- пустой массив,
- стабильность порядка,
- идемпотентность,
- сохранение generic типа.

### 4. UI фотографа в `/app` (РЭ.25.4, commits `f3cdbf4` + `30aece9`)

**Форма альбома (`AlbumFormModal`)**:
- Чекбокс «Включить в личный раздел всех учеников, даже не заказавших альбом» в отдельном блоке (рамка + подсказка) между блоком шаблона и legacy-пресетом. По умолчанию выключен (строгое поведение для новых альбомов).
- Подсказка объясняет когда стоит включить: «если не-заказчиков мало и не хочется дырок в альбоме».
- `FormData` расширен полем `include_non_purchasers: boolean`. Загружается из БД при редактировании, отправляется в `create_album`/`update_album`.

**Таблица учеников (вкладка «Ученики»)**:
- **Колонка «Заказ»** с чекбоксами на каждой строке (между «Класс» и «Статус»). Изначально размещение было в expanded-секции — это оказалось неудобно (нужно кликать каждого ученика). Хотфикс `30aece9` вынес чекбокс прямо в таблицу.
- Клик по чекбоксу **не раскрывает** строку — `stopPropagation()`.
- title-атрибут на hover.
- Сохранение через `update_child` action.
- Для read-only роли показывается `✓` / `—` вместо чекбокса.
- **Бейдж «не заказывает»** рядом с ФИО + приглушение строки (`opacity-60`) — визуальные маркеры остаются даже при прокрутке.

### 5. UI родителя в `/[token]` (РЭ.25.5, commit `de4182a`)

Защищённая страница, правка **строго аддитивная** — логика отбора фото / draft / submit / lock / unlock не тронута.

**Чекбокс «Мы заказываем альбом»** показывается в двух местах:
- На основном flow — постоянный блок в верху страницы, виден на всех шагах (от портрета до готово). Родитель может передумать в любой момент.
- На экране «Выбор уже сделан» (`alreadySubmitted`) — родитель может передумать и после submit.

**Optimistic update** с откатом при ошибке. Локальная ошибка показывается inline под чекбоксом, не блокирует остальной UI. Отправка через `PUT /api/select { action: 'set_purchased', value }` — endpoint не блокируется `submitted_at`.

Подсказка: «Если вы не покупаете альбом — снимите галочку. Личной страницы для ребёнка в альбоме не будет, но он останется на общих классных фото и в виньетке класса.»

### 6. CSV-экспорт (РЭ.25.5, тот же commit)

В `action=export_csv` добавлена колонка **«Заказ»** между «Ученик» и «Портрет_страница»:
- `да` для `is_purchased!==false` (default)
- `нет` для `is_purchased=false`
- `—` для учительских строк (колонка не применима)

Колонка нужна для скрипта автовёрстки InDesign (связанный проект) — там тоже теперь можно фильтровать строки. META-строка ниже автодополняется пустыми колонками — изменений в формате не требуется.

### 7. Счётчик «Заказали: N из M» (РЭ.25.6, commit `5d87079`)

**API**:
- `dashboard` action: SELECT children расширен `is_purchased`, `statsMap` получил новое поле `purchased`.
- `album_stats` action: то же для детальной страницы одного альбома.
- Тип `Album.stats` и `AlbumStats` расширены `purchased?: number`.

**UI** в двух местах:
- В компактной карточке альбома (`AlbumCard`) — строка «Заказали альбом: N из M (K без личной страницы)».
- В детальной странице альбома — отдельный info-блок с подсказкой «снять/поставить галки — на вкладке Ученики».

**Условия показа** (одинаковые в обоих местах) — минимизация визуального шума:
- `!album.include_non_purchasers` — не показываем в мягком режиме (там все ученики в альбоме, цифра «N из M» бессмысленна).
- `stats.total > 0` — не показываем для пустых альбомов.
- `stats.purchased < stats.total` — не показываем когда все заказали.

## Архитектура: фильтр в smart-fill, не в engine

Это ключевое решение фазы, зафиксированное в spec §4 и подтверждённое в практике.

```
БД ─→ buildAlbumInput ──┬─→ filterChildrenByPurchase ─→ AlbumInput.students[]
                        │                                       ↓
                        └─→ album.include_non_purchasers   buildAlbum (engine)
                                                                ↓
                                                          spreads / pages
```

Engine получает уже отфильтрованный список учеников. Он не знает про `is_purchased`, про `include_non_purchasers`, про РЭ.25 в принципе. Это даёт:
- **Тестируемость** — фильтр это чистая функция (12 unit-тестов без мока Supabase).
- **Простоту отладки** — один источник истины. Если ученик не попал в личный раздел — смотришь только в одно место.
- **Свободу для будущих фаз** — РЭ.27 (`print_type` в альбом) и РЭ.28 (партнёрские дизайны) не пересекаются с этим фильтром.

## Ключевые решения (4 архитектурные развилки)

Все обсуждены и утверждены с Сергеем перед написанием spec'а 21.05.2026.

| # | Развилка | Решение |
|---|---|---|
| a | Кто проставляет галку — фотограф или родитель? | **Оба**, last-write-wins. Одно поле `children.is_purchased`, обе стороны пишут в одно место. |
| b | Не-покупатели в общих фото и виньетках? | **Попадают**. Это состав класса, не список заказчиков. Фильтр работает только в секции `students`. |
| c | Не-покупатели в личных страницах? | **Не попадают** по умолчанию (строгий режим). Партнёр может включить мягкий режим флагом альбома `include_non_purchasers`. Per-child переопределение НЕ вводим — поднимай `is_purchased` обратно. |
| d | Дефолт миграции? | `children.is_purchased DEFAULT true` (бэк-совместимость), `albums.include_non_purchasers DEFAULT false` (строгие новые альбомы; для старых эффекта нет, потому что у всех `is_purchased=true`). |

## Что НЕ сделано (зафиксировано как долг)

- **Уведомление фотографу о смене статуса родителем после submit.** Сейчас если родитель снимает галку после `submitted_at`, фотограф узнаёт об этом только при следующем заходе в `/app`. Возможное расширение через `parent_messages` — открываем по запросу, когда станет реальной болью. Spec §8 G.
- **История изменений `is_purchased`** (audit log на уровне поля «кто и когда менял значение»). Стандартный аудит-лог `child.update` фиксирует факт, но не значение до/после. Не требуется в РЭ.25.
- **Связь со скидками / биллингом.** Биллинг живёт у Сергея отдельно через менеджера. `is_purchased` — только про вёрстку, не про деньги.

## Что следующее

Зафиксированный в spec РЭ.24 §13/§14 порядок остаётся:

1. **РЭ.27** — `print_type` в альбом + двойная модель редактора (разворотная для layflat, постраничная для soft). Большая фаза ~10-15 коммитов.
2. **РЭ.28** — партнёрские дизайны сценарий A (копирование глобального template_set с переопределением размеров под типографию партнёра).
3. **РЭ.29+** — партнёрский IDML сценарий B, без даты.

Никаких пересечений с РЭ.25: фильтр не-заказчиков живёт в smart-fill и не зависит от типа листов / дизайна / редактора.
