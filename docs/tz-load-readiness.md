## Сквозной принцип: независимость от провайдера БД

Supabase в перспективе переедет на российский managed Postgres (Yandex Managed PostgreSQL / Selectel). Поэтому всё новое в этом ТЗ проектировать **переносимым**:
- Очередь — **чистый Postgres** (таблица + `SELECT … FOR UPDATE SKIP LOCKED`). НЕ использовать Supabase-специфику (Realtime, Edge Functions, обёртки pg_cron Supabase, `functions.invoke`). Проверено: сейчас этой специфики в коде нет — не вводить её и здесь.
- Хранилище — только **S3-совместимый** доступ (YC уже через `@aws-sdk`). Не привязываться к Supabase Storage.
- Доступ к БД абстрагировать так, чтобы при переезде менялись строка подключения и тонкий слой, а не бизнес-логика.

Итог: после переезда этот функционал работает без изменений логики — меняется только конфиг подключения.

> **Связка с переездом в РФ.** Воркер рендера всё равно должен жить на Яндексе (см. №1, Фаза 2), и переезд на YC всё равно предстоит (см. `docs/security-migration-checklist.md` и план миграции). Поэтому **сильный кандидат — строить очередь+воркер сразу как часть переезда на YC**, а не городить временный мост на Vercel-Cron, который потом выбрасывать. Решение про последовательность — см. п.2 в «Решениях для Сергея».

## Узкое место №1 — рендер PDF в request-цикле (ГЛАВНОЕ, остаётся приоритетом)

**Сейчас:** сборка макета идёт синхронно внутри HTTP-запроса (`app/api/layout/route.ts`, actions `export` / `build_album` / `build_album_test`). `sharp` пережимает сотни фото + `pdf-lib` собирает PDF — это минуты CPU/RAM. Лимит Vercel 300с (Pro) для большого альбома — впритык, несколько одновременных рендеров копят таймауты и дорогой compute.

**Цель:** вынести рендер из запроса в фоновую очередь с отдельным воркером.

### Фаза 1 — Очередь задач (Postgres, переносимая)
- Таблица `render_jobs`: `id, album_id, tenant_id, type ('album'|'spread'|'originals_zip'), status ('queued'|'processing'|'done'|'failed'), priority, payload jsonb, attempts, max_attempts (по умолч. 3), result_path (ключ в YC), error, created_at, started_at, finished_at, worker_id`.
- Поставка задачи: actions `export`/`build_album` больше НЕ рендерят синхронно — они вставляют строку в `render_jobs` (status=queued) и сразу возвращают `{ job_id }`.
- **Идемпотентность (уточнено):** не плодить дубль, если для (album_id, type) уже есть **незавершённая** задача (`queued`/`processing`) — вернуть существующий job_id. Завершённая (`done`) задача НЕ блокирует новый рендер — партнёр отредактировал макет → нужен свежий PDF.
- Забор задачи воркером — атомарно, без двойной обработки:
  ```sql
  UPDATE render_jobs SET status='processing', started_at=now(), worker_id=$1
  WHERE id = (
    SELECT id FROM render_jobs WHERE status='queued'
    ORDER BY priority DESC, created_at ASC
    FOR UPDATE SKIP LOCKED LIMIT 1
  ) RETURNING *;
  ```
  (`FOR UPDATE SKIP LOCKED` — стандартный Postgres, переносим на любой провайдер.)
- Ретраи: при падении `attempts++`; если `< max_attempts` → обратно в `queued`, иначе `failed` + текст ошибки. Зависшие (`processing` дольше N минут) — возвращать в `queued` (watchdog).
- Статус-эндпоинт для UI: `GET ?action=render_status&job_id=…` → `{status, progress?, download_url?}`.
  - **Безопасность (добавлено):** проверять, что job принадлежит партнёру запрашивающего (по `tenant_id`, как проверки «своё/чужое» из аудита). Иначе по чужому job_id можно вытащить ссылку на чужой готовый PDF.
  - **Прогресс:** PDF-рендер прогрессирует грубо — отдавать не проценты, а стадии («готовлю фото», «собираю PDF»).

### Фаза 2 — Воркер рендера (вынести с Vercel)
- Тяжёлый рендер вынести на **Yandex Cloud Serverless Containers** (нет жёсткого лимита 300с; ближе к будущей РФ-инфре; не жжёт Vercel-compute). Vercel-Cron как мост — паллиатив (тот же 300с), по возможности пропустить и целиться сразу в контейнер (см. связку с переездом выше).
- **Окружение воркера (добавлено):** контейнер должен везти с собой всё для рендера — шрифты `public/fonts` (12 TTF), нативный `sharp`, `pdf-lib`/`fontkit`. Иначе PDF соберётся без шрифтов.
- Воркер в цикле: берёт job (SKIP LOCKED) → рендерит (существующий код `lib/pdf-export/`) → кладёт PDF в **приватный** YC-бакет → отдаёт через signed URL (приватность фото уже закрыта, см. `project_private_photos_d1`) → `status=done, result_path` → уведомление.
- **Параллелизм ограничить** (concurrency-лимит, напр. 2–4 одновременных рендера на воркер; масштабировать числом воркеров, а не бесконтрольной памятью). Это защита от «10 макетов разом положили память». Учесть: `sharp` тяжёл по CPU+RAM — лимит подбирать под память контейнера.
- Scale-to-zero у контейнера = экономия, но холодный старт добавит секунды к первой задаче — для минутных рендеров приемлемо.
- UI: вместо ожидания ответа — кнопка «Собрать макет» ставит задачу, экран показывает «Макет готовится…» с поллингом статуса (раз в 3–5 с) и ссылкой по готовности.

## Узкое место №3 — пропускная способность и защита от перегрузки
- **Раздача фото родителям** уже идёт напрямую с YC (сервер генерит signed URL, байты качаются браузером из YC, минуя Vercel) — сохранить этот принцип, ничего не проксировать через функции. ✅ Уже так.
- **Лимит одновременных рендеров** — через число воркеров и concurrency-лимит (Фаза 2), а не через автоскейл функций.
- **Backpressure:** при длинной очереди UI честно показывает позицию/«в очереди N задач», а не висит.
- **Идемпотентность** всех тяжёлых операций (Фаза 1) — повторный клик не плодит работу.
- **Мониторинг:** метрики глубины очереди, времени рендера, доли `failed`; алерт при росте очереди и при приближении к лимитам Vercel (bandwidth/compute) и расходам YC.

## Что НЕ делать
- Не вводить Supabase-специфику (Realtime/Edge/их cron) — сломает переносимость под переезд.
- Не паковать большие ZIP в памяти API-функции (для оригиналов уже не пакуем).
- Не рендерить PDF синхронно в HTTP-запросе после внедрения очереди.
- Не проксировать сами фото через Vercel-функции.
- Родительские страницы и логику отбора/submit/draft — не трогать по сути.
