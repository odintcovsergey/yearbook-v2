# Инвентаризация структуры альбома

**Версия:** v0.1 — стартовая инвентаризация на основе пользовательского скрипта `build_album.jsx`
**Дата:** 18.05.2026
**Цель:** зафиксировать в одном месте все секции, мастера, слоты и переходы которые встречаются в реальной сборке альбомов OkeyBook. Это **картина текущего состояния**, не дизайн будущего. Сергей разъяснил, что скрипт сырой и не является конечной правдой — это отправная точка для проектирования архитектуры пресета через UI редактор.

---

## 1. Глобальная структура альбома

Альбом состоит из секций, идущих в строгом порядке:

| Позиция | Секция | Когда есть |
|---|---|---|
| 1 | **Soft-Intro** | только для `sheet_type=soft` (мягкие листы), одна правая страница |
| 2 | **Учительский разворот** | всегда, 1 или 2 страницы |
| 3 | **Портреты учеников** | всегда, длина зависит от комплектации и числа учеников |
| 4 | **Общий раздел (обязательный)** | если есть фотоматериал |
| 5 | **Общий раздел (дополнительный)** | если осталось фото после обязательного |
| 6 | **Виньетка из детских фото** | опционально (отдельная секция, на сейчас не реализована) |
| 7 | **Soft-Final** | только для `sheet_type=soft`, одна левая страница |

Главное правило: **каждая страница занимает один слот**. Развороты — это два соседних слота. Логика никогда не работает «парой сразу», только постранично с учётом позиции (left/right).

## 3. Логика выбора учительского разворота

Зависит от **числа предметников**. Левая страница (классрук) и правая страница (предметники или общее фото) выбираются вместе.

| Число предметников | Левая страница | Правая страница |
|---|---|---|
| 0 | `F-Head-WithPhoto` | `G-HalfClass` если 2+ фото полкласса, иначе `G-FullClass` если 1+ общее, иначе пусто |
| 1-4 | `F-Head-SmallGrid` | то же правило |
| 5-8 | `F-Head-LargeGrid` | то же правило |
| 9 | `F-Head-WithPhoto` | `G-Teachers-3x3` (9 предметников) |
| 10-12 | `F-Head-WithPhoto` | `G-Teachers-4x3` (12 слотов) |
| 13-16 | `F-Head-WithPhoto` | `G-Teachers-4x4` (16 слотов) |
| 17+ | `F-Head-LargeGrid` (8 предметников слева) | `G-Teachers-4x4` (остаток справа) |

**Ключевая идея:** правая страница учительского разворота — это **место перехода в общий раздел**. Часть общего фото уже потрачено здесь, а не в общий раздел.

## 4. Логика секции портретов

### Комплектации 1-3 (Стандарт / Универсал / Максимум) — 1 портрет = 1 страница (или разворот)

Партнёр получает столько страниц, сколько учеников × 1 (или 2 для Максимума). Без сеток.

### Комплектации 4-7 (Медиум / Лайт / Мини / Индивидуальный) — сетки

Engine выбирает сетку под текущий остаток учеников. Например, для Лайт:

- Если осталось 6 учеников → `L-6-*`
- Если осталось 4 → `L-4-*`
- Если осталось 3 → `L-3-*`
- Если осталось 2 → `L-2-*`

Тот же принцип для Мини (`N-*-*`).

**Хвост-комбо**: если в самом конце осталось мало портретов (например, 2 ученика после двух полных сеток по 12), и страница «нечётная» → engine может использовать **последний комбо-мастер** который ставит остаток портретов сверху и общее фото снизу. Сейчас в скрипте это есть только для Медиума (`D-Medium-Last-WithPhoto`); для Лайт/Мини аналога нет, обходится через `flex_C` слот в общем разделе.

## 5. Логика общего раздела

Общий раздел = **массив слотов**, обходимый последовательно. Каждый слот = одна страница.

### Типы слотов

| Слот | Описание | Цепочка попыток (priority chain) |
|---|---|---|
| `H` | страница с полклассом | `J-Half` + 2 фото half_class |
| `Q` | страница с четвертью | `J-Quarter` + 2 фото quarter |
| `FULL` | страница с общим | `J-ClassPhoto` (или `-Right` если правая) + 1 фото full_class |
| `flex_A` | гибкий слот «крупный приоритет» | `J-Collage` (6 sixth) → `J-Half` (2 half) → `J-ClassPhoto` (1 full) |
| `flex_B` | гибкий слот «всё попробовать» | `J-Quarter` (2 quarter) → `J-Collage` (6 sixth) → `J-Half` (2 half) → `J-ClassPhoto` (1 full) |
| `flex_C` | правая нечётная страница (мост от портретов к общему) | `J-Half` (2 half) → `J-Collage` (6 sixth) → `J-ClassPhoto-Right` (1 full) |

**Цепочка попыток** означает: engine пробует первый вариант в цепочке. Если фотоматериала достаточно — используется этот мастер и фото потребляются. Если нет — engine пробует следующий. Если все варианты не подошли — слот пропускается (страница пустая).

### Структура для каждой комплектации (из скрипта)

Сергей подчеркнул что **это его дефолты, не закон**. Партнёры могут собирать свои структуры.

| Комплектация | Mandatory | Additional |
|---|---|---|
| Мини, ≤24 уч., layflat | `H, flex_A` | `flex_A, flex_A, flex_B, flex_B` |
| Мини, ≤24 уч., soft | `H, flex_A, FULL` | `flex_A, flex_B, flex_B, flex_A` |
| Мини, >24 уч., layflat | (пусто) | `flex_A, flex_A, flex_B, flex_B` |
| Мини, >24 уч., soft | `H` | `flex_A, flex_A, flex_B, flex_B` |
| Индивидуальный, layflat | `H, flex_A` | `flex_A, flex_A, flex_B, flex_B` |
| Индивидуальный, soft | `H, flex_A, FULL` | `flex_A, flex_B, flex_B, flex_A` |
| Лайт/Медиум/Стандарт/Универсал/Максимум, чётно, layflat | `Q, Q, H, flex_A, flex_A, flex_A` | `flex_A, flex_A, flex_B, flex_B` |
| Лайт/Медиум/Стандарт/Универсал/Максимум, нечётно (flex_C занял правую) | `Q, Q, H, flex_A` | `flex_A, flex_A, flex_B, flex_B` |
| Лайт/Медиум/Стандарт/Универсал/Максимум, чётно с overflow на soft | `Q, Q, H, flex_A` | `flex_A, flex_A, flex_B, flex_B` |

## 6. Переходные / комбо-мастера

Мастера которые **объединяют две секции** на одном развороте или одной странице:

| Мастер | Между чем переход |
|---|---|
| `F-Head-* + G-HalfClass/G-FullClass` | учительский разворот, правая страница = первое общее фото |
| `D-Medium-Last-WithPhoto` | хвост портретов Медиум + 1 общее фото снизу |
| `flex_C` слот | нечётная правая страница портретов → начало общего раздела с правой |
| Адаптивная сетка `L-2/L-3/L-4` или `N-4/N-6/N-9/N-12` | хвост портретов с подгонкой под остаток |

**Концептуально**: комбо-мастер — это **отдельный мастер в палитре**, не «составление из других». В IDML это одна отдельная master-страница со своей разметкой.

## 7. Параметры пресета которые влияют на сборку

- `print_type` — `layflat` / `soft` / `tryumo` (трюмо — отдельно)
- `density` — `standard` / `universal` / `medium` / `light` / `mini` / `maximum` / `individual`
- `total_pages` — фиксированное число страниц альбома
- `students_count` — приходит из данных альбома
- `subjects_count` — приходит из данных альбома
- Наличие/количество фото каждой категории: `full_class`, `half_class`, `quarter`, `sixth` (+ виньетки в будущем)

## 8. Что меняется по сравнению с текущим rule engine

| Сейчас | Должно стать |
|---|---|
| Правила = JSON-файлы per-page-pattern, поведение зашито в `when`/`produces`/`consumes` | Структура пресета = БД-поле, **редактируется в UI**. Правила генерятся на лету или унифицированы |
| `priority` правил решает что применить | **Цепочка попыток внутри слота** — engine пробует мастера по списку, останавливается на первом доступном |
| Mandatory vs additional = разные правила | Один и тот же тип слота, разные позиции в массиве `section_structure` |
| Альтернативы `либо X, либо Y` в JSON-матрице | Цепочка `flex_A/B/C` — это **именованный шаблон цепочек**, переиспользуемый |

## 9. Открытые вопросы (на обсуждение)

1. **Адаптивные сетки** — есть `L-6/4/3/2` и `N-12/9/6/4`. Должны ли быть и **L-6-X** где X = «3+3+2» для нечётных остатков (Сергей упомянул в обсуждении 18.05)? Или это просто L-8-Left с разметкой 3+3+2?

2. **Виньетка из детских фото** — это самостоятельная секция или специальный мастер из палитры `N-*-*` с другими данными? Где в порядке секций?

3. **Кастомные слоты партнёра** — должен ли партнёр уметь добавлять свои типы слотов (не только H/Q/FULL/flex_A/B/C), или ему достаточно перетасовать имеющиеся в нужном порядке?

4. **flex_C** = «правая нечётная страница» — это слот **общего раздела** или **последний слот секции портретов** (как мост)? В скрипте он добавляется в `commonPages`, но логически — про портреты.

5. **`sheet_type` на уровне альбома vs пресета** — подтверждено ранее, что на уровне альбома. Это означает что один пресет работает для обоих типов листов, а `Soft-Intro/Final` секции добавляются или нет в зависимости от выбора партнёра.

## 10. Что НЕ покрыто этой инвентаризацией

- **Обложка** (`build_cover.jsx`) — отдельная подсистема.
- **Трюмо** (`build_tryumo.jsx`) — отдельная фаза, отложено.
- **Доплаты** — отдельная вкладка в UI, не структура пресета.
- **Личный разворот ученика** — уже в проде, не часть автогенерации.
- **Конкретные параметры мастеров** (размеры, координаты labels) — это в IDML и в `spread_templates`.
