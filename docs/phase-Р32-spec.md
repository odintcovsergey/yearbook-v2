# Фаза РЭ.32 — Конструктор общего раздела

**Статус**: ✅ Утверждён 22.05.2026 (8/8 вопросов решены). Готов к старту этапа А.
**Дата создания**: 22.05.2026
**Базовое состояние**: после revert РЭ.31 (commits 025ae19/58d843a/6c3907a). Legacy density-таблица в `transition.ts` и `common-required.ts` восстановлена. На Тест2 при сборке выпадают 4 warning'а — это нормально, фаза стартует именно отсюда.

## Стратегический контекст

**Решение Сергея 22.05.2026**: НЕ хотим залатать дыру быстро (мапить `student_layout_mode` → density). Хотим переписать архитектуру общего раздела на **конструктор**: партнёр в редакторе шаблона сам собирает список мастеров общего раздела из доступных в template_set'е. Engine только исполняет.

Это правильное решение долгосрочно:
- Партнёры с любыми template_set'ами работают единообразно (нет привязки к таблице OkeyBook).
- Логика общего раздела живёт в **данных** (массив в пресете), не в коде.
- Удаляется legacy-таблица OkeyBook — больше нет двух движков.
- Партнёр получает контроль и предсказуемость.

## Что уже есть в системе (на момент старта фазы)

**На уровне заказа** (`app/app/_components/SpreadOrderStrip.tsx` + `TemplatePickerModal.tsx`):
- drag-n-drop переупорядочивания разворотов
- добавление разворота (выбор мастера из template_set'а)
- удаление разворота
- замена мастера

**На уровне шаблона** (`PresetEditorModal.tsx`):
- список секций (teachers / students / transition / common_required / etc.) с порядком
- параметры каждой секции (для students — режим, grid_size, friend_photos, quote)
- НО для `common_required` параметров нет: это «маркер» места, конкретные мастера определяет engine

**Существующая `template_sets` структура**:
- мастера хранятся в `spread_templates` с `page_role` (student / teacher_left / teacher_right / common / intro / cover)
- общие мастера имеют `page_role='common'` (в okeybook-default это J-Full, J-Half, J-Quarter-Left, J-Quarter-Right, J-Collage-4, J-Collage-6, J-Spread)
- `slot_capacity` у J-мастеров пустой (`{}`) — определять возможности приходится по placeholders

## Главная идея

В пресете появляется новое поле — список страниц общего раздела:

```typescript
type CommonRequiredPage = {
  master_name: string  // имя J-мастера из template_set
  // фото-категория определяется автоматически по placeholders мастера
  // (classphotoframe → full, halfphoto_* → half, и т.д.)
}

preset.common_required_pages: CommonRequiredPage[]  // упорядоченный список
```

Engine при сборке альбома проходит по этому списку и кладёт страницы. Если фото в категории не хватает — пропускает страницу с warning.

Параллельно появляется опциональное поле для переходной страницы:

```typescript
preset.transition_master_name: string | null
// null → engine использует встроенное правило по умолчанию (см. ниже)
// строка → engine использует указанный мастер
```

## Правило переходной страницы по умолчанию

Когда `students` закончился на левой странице и `transition_master_name=null`:

- Для **Медиум**: «до 2 портретов сверху + 1 общая снизу» — engine ищет combined-tail мастер с `students<=2, photos_full=1`. Это уже работает в `students.ts` (combined-tail logic).
- Для **Лайт**: «до 3 портретов + 1 общая» — combined-tail с students<=3, photos_full=1.
- Для **Мини**: «до 4 портретов + 1 общая» — combined-tail с students<=4, photos_full=1.
- Для **Стандарт/Универсал/Максимум/Индивидуал**: переходной нет — students всегда даёт чётное количество (page = чётно, spread = чётно).

Правило **захардкожено в engine** как baseline. Партнёр может переопределить через `transition_master_name`.

## Решения по 8 вопросам (Сергей, 22.05.2026)

1. **Какие мастера в пикере общего раздела** — гибрид: `page_role='common'` + анализ placeholders. Универсальный фильтр «подходит ли мастер под общий раздел». Тот же принцип применяется к другим разделам: учительский пикер показывает только учительские мастера, ученический — только ученические.

2. **Двухуровневая структура (заказ vs шаблон)** — пока только одно направление: шаблон → заказ. Layout живёт в `albums.layout`, правки партнёра в заказе остаются в заказе. Кнопку «сохранить изменения обратно в шаблон» добавляем только если по фидбэку окажется нужным. Сейчас не делаем.

3. **Default `common_required_pages` для нового шаблона** — наследование от `parent_template_set` (для клонов). Глобальные пресеты OkeyBook (7 штук) Сергей заполняет один раз вручную через UI на этапе Г. Партнёрский «с нуля» (РЭ.29 в будущем) — пусто, партнёр сам собирает.

4. **Существующие альбомы на проде (12)** — не трогаем. Продолжают работать через legacy `buildAlbum`. Когда их станет 0 — отдельной фазой удалим legacy движок.

5. **Где хранить массив страниц общего раздела** — внутри маркера секции в `section_structure`:
   ```
   { type: 'common_required', pages: [ { master_name: 'J-Quarter' }, ... ] }
   ```
   Старые сохранённые пресеты получают `pages: []` (engine выдаёт warning «общий раздел пуст», партнёр заполняет).

6. **`cover_preview_url` (долг РЭ.28)** — отдельной микро-фазой когда дойдёт очередь, не в РЭ.32.

7. **Левая/правая позиция мастеров** — партнёр кладёт **имя без суффикса** (`J-Quarter`). Engine при сборке смотрит позицию страницы:
   - Если правая позиция И в template_set есть `<name>-Right` — берёт его
   - Иначе — берёт исходный (универсальный мастер)
   - Дизайнер решает: один универсальный (симметричный layout) или пара Left/Right (асимметрия под дизайн)

   В UI пикере общего раздела показываются только мастера **без суффикса `-Right`** (зеркальные правые скрыты — они подбираются автоматически).

8. **J-Spread** — одна запись в массиве, engine учитывает `spread_templates.is_spread=true` и выделяет 2 позиции (left+right). Следующая страница в списке идёт со следующего разворота.

## Отложенные идеи из фидбэка Сергея 22.05.2026

Записываю чтобы не забыть, делаются после РЭ.32.

### РЭ.33 (предв.) — Чистка терминологии
В коде/UI путаница:
- `presets` (БД) ≈ «шаблон» (то что собирается партнёром в редакторе)
- `template_sets` (БД) ≈ «дизайн» (набор InDesign-макетов)
- `spread_templates` (БД) ≈ «мастер» (один разворот в дизайне)
- В UI кнопка зовётся «Пресеты», в коде «preset», описание «пресет вёрстки»

Привести к единым терминам:
- `Дизайн` = template_set
- `Шаблон` = preset (наш конструктор)
- `Мастер` = spread_template
- `Заказ` = album

Это **большой проход** по UI всего проекта. Делается отдельной фазой.

### РЭ.34 (предв.) — Quick-start пресетов
Если у партнёра 20 дизайнов × 7-8 шаблонов = 140 пресетов вручную создавать. Идея — генератор-конструктор:

1. Партнёр выбирает дизайн
2. Видит карточки готовых вариантов: «Мини (до 12)», «Лайт (13-24)», «Медиум (25-32)», «Стандарт» и т.д.
3. Кликает → создаётся **пресет с уже заполненными** common_required_pages, transition, students-параметрами под выбранную комплектацию
4. Партнёр оставляет как есть или дорабатывает

Эталонные шаблоны для генератора — у OkeyBook 7 глобальных пресетов в xlsx (Структура_альбомов_для_автоверстки). При выборе дизайна партнёром эти шаблоны мапятся на J-мастера партнёрского template_set'а.

Может быть отдельным этапом РЭ.34, может частью РЭ.32 если решим что фотографы без quick-start будут страдать.
