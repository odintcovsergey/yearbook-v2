# ТЗ: Привязанный декор к слотам + текстовые эффекты

## Контекст и проблема

Система делает выпускные альбомы. Сейчас работает простой дизайн «Белый»
(строгая геометрия, без декора). Теперь подключаются художественные дизайны
для ДЕТСКИХ САДОВ и 4 классов — акварельные и сказочные, с богатым декором:

- Художественные обрамления портретов (теремки, овальные рамки с орнаментом,
  растушёванные обводки фреймов).
- Декоративные ленточки/баннеры под именами.
- Текст имён с обводкой и свечением (чтобы читался на пёстром фоне).

ПРОБЛЕМА: сейчас парсер ВЫКИДЫВАЕТ весь декор без меток
(extract-geometry.ts: «unlabeled frames skipped (decorative)»). И главное —
декор должен вести себя ДИНАМИЧЕСКИ: при симметризации/изменении числа
портретов (воспитателей бывает 1-2-5, учеников переменное число) декор
должен ДВИГАТЬСЯ и СКРЫВАТЬСЯ вместе со своим слотом.

Без этого механизма художественные дизайны невозможны — упираемся в потолок.

ПРИОРИТЕТ: высокий, нужно в ближайший месяц (загрузка всех дизайнов в работу).
Школьные дизайны (9-11 класс) этот механизм не используют — он для садов/4кл.

## Часть 1 — Привязанный декор к слоту (ядро)

### Соглашение по меткам (Script Label в IDML)

Базовый слот имеет метку как сейчас. Прикреплённый декор — метку базового
слота + суффикс слоя:

```
teacherphoto_1           — слот фото (как сейчас)
teacherphoto_1__under    — декор-картинка ПОД фото (подложка/фон слота)
teacherphoto_1__over     — декор-картинка ПОВЕРХ фото (рамка-обрамление, теремок)
teachername_1            — слот текста имени
teachername_1__under     — ленточка-баннер под именем
teachername_1__over      — декор поверх имени (редко)
```

Суффиксы: `__under` (z ниже базового слота), `__over` (z выше базового слота).
Базовый слот (фото/текст) рисуется между ними.

### Что декор хранит

Декор — это статичная КАРТИНКА из IDML (PNG/embedded). У него нет
подстановки данных (в отличие от слота фото). Хранить:
- attached_to: label базового слота (например 'teacherphoto_1')
- layer: 'under' | 'over'
- url: картинка декора (embedded image из IDML → сохранить в storage)
- геометрия: x_mm, y_mm, width_mm, height_mm, rotation_deg (как у слота)
- offset относительно базового слота (см. динамику ниже)

### Парсер (lib/idml-converter/extract-geometry.ts)

Сейчас фреймы без метки выкидываются. Изменить:
- Фрейм с меткой вида `<base>__under` / `<base>__over` → это декор.
- Извлечь embedded-картинку фрейма, сохранить в storage (как фон).
- Сохранить запись декора с attached_to=<base>, layer, url, геометрия.
- Вычислить offset декора относительно базового слота:
  offset_x = deco.x_mm - base.x_mm, offset_y = deco.y_mm - base.y_mm.
  (Чтобы при перемещении слота декор сохранял относительное положение.)
- Фреймы без метки и без суффикса-привязки — по-прежнему skip (глобальный
  декор разворота идёт через фон, см. Часть 4).

Декор хранить либо в placeholders мастера (новый type: 'decoration'), либо
отдельным массивом decorations в spread_template. Реши по месту — placeholders
с type:'decoration' проще (единый список).

### Динамика (КЛЮЧЕВОЕ — связь с симметризацией)

В системе уже есть механизмы:
- `__hidden__<label>='1'` — слот скрыт (нет фото/воспитателя).
- `__pos__<label>='<x>,<y>'` — переопределённая позиция слота (центрирование
  при симметризации, см. shared.ts).

Декор должен СЛЕДОВАТЬ за этими механизмами:
1. Если базовый слот СКРЫТ (`__hidden__teacherphoto_5`) → весь привязанный
   декор `teacherphoto_5__under` и `teacherphoto_5__over` ТОЖЕ скрывается.
2. Если базовый слот ПЕРЕМЕЩ�ён (`__pos__teacherphoto_5='x,y'`) → декор
   смещается на ту же дельту, сохраняя offset:
   deco_pos = new_base_pos + saved_offset.
3. Если слот на месте без правок → декор на своей исходной позиции.

Это надо встроить в builder (lib/rule-engine/sections/) и в рендер.
При выдаче layout для каждого видимого декора вычислять его финальную позицию
на основе позиции базового слота.

### Рендер декора

**Канвас (AlbumSpreadCanvas.tsx):**
Стек отрисовки на каждом слоте:
  [фон разворота] → [слот__under] → [фото/текст слота] → [слот__over]
Konva: декор — это Image-нода с нужным z-index (under до базового, over после).
Скрытый декор не рисуется. Позиция с учётом __pos__ дельты.

**PDF (lib/pdf-export/pipeline.ts):**
Тот же порядок отрисовки. Декор-картинка вставляется через pdf-lib
(embedPng/embedJpg) на вычисленной позиции до/после базового изображения.

## Часть 2 — Обводка фреймов (растушёванная рамка)

Для простых обводок (зелёная/синяя растушёванная рамка вокруг фото, как в
акварельном) — можно НЕ делать отдельным декором, а читать как СВОЙСТВО фрейма.

НО: Сергей уточнил, что в его дизайнах обводка — это часто ОТДЕЛЬНЫЙ
дизайнерский элемент на слое выше фрейма, а не свойство рамки. Поэтому:
- Если обводка — отдельный элемент-картинка → идёт через Часть 1 как
  `<base>__over` (рамка поверх фото).
- Если обводка — простое свойство фрейма в IDML (stroke + corner_radius +
  feather) → опционально читать как атрибуты слота: stroke_color,
  stroke_width_mm, corner_radius_mm. Растушёвку (feather) рендерить
  ПРИБЛИЗИТЕЛЬНО (Сергей подтвердил — пиксель-в-пиксель не нужно, достаточно
  похоже). В Konva — blur/shadow на рамке, в PDF — упрощённо.

Приоритет: сначала Часть 1 (декор-картинкой покрывает обводки-элементы),
свойства фрейма — если останется время или по необходимости.

## Часть 3 — Текстовые эффекты (обводка + свечение)

Имена на пёстром фоне нужно читать → обводка букв (stroke) и свечение (glow).

Расширить текстовый placeholder (поля в дополнение к font_family/size/color):
- text_stroke_color: string | null   — цвет обводки букв
- text_stroke_width_pt: number | null — толщина обводки
- text_glow_color: string | null     — цвет свечения/тени
- text_glow_blur_pt: number | null    — размытие свечения

Парсер: читать из IDML свойства текста (если заданы эффекты) ИЛИ задавать
по соглашению (label-based, как auto_fit). Уточнить в IDML как Сергей задаёт
эти эффекты — через стили InDesign (Outer Glow, Stroke) в Effects.

Рендер:
- Канвас (Konva Text): stroke + strokeWidth для обводки; shadowColor +
  shadowBlur для свечения. Konva это поддерживает нативно.
- PDF (text-shaping.ts): обводка через рисование текста с обводкой
  (pdf-lib: рисовать текст несколько раз со смещением, или lineWidth для
  обводки глифов если доступно); свечение — упрощённо (тень смещением) или
  пропустить если сложно. Приоритет — канвас (видит родитель/партнёр),
  PDF приблизительно.

## Часть 4 — Z-order переднего плана разворота

Сложные дизайны (сказочный: ветки лешего поверх рамок) требуют декора
ПОВЕРХ всего разворота, не привязанного к слоту.

Решение: глобальный декор разворота с меткой `__fg_<n>` (foreground) —
рисуется ПОСЛЕ всех слотов и их декора, поверх. Аналогично фоновому, но
сверху. Хранить per-spread-template (как фон, но layer=foreground).

Это для сказочного дизайна (ближайший месяц). Можно вынести в отдельный
последний этап.

## Часть 5 — Декоративные шрифты дизайна

Сказочные/акварельные заголовки используют декоративные шрифты (Lemon
Tuesday, BebasNeue и т.д. из папки дизайна).

- Добавить шрифты в public/fonts/ + AVAILABLE_FONTS (lib/text-style/fonts.ts).
- font-loader.ts (PDF) должен их подгружать.
- ВАЖНО: проверить лицензии! Minion Pro платный (Adobe) — заменить или купить.
  Сергей проверяет лицензии до добавления.

## Этапы реализации (порядок)

1. **БД + типы:** новый type 'decoration' в placeholder (или массив
   decorations), поля attached_to/layer/offset; текстовые эффекты в text
   placeholder. Миграции если нужны.
2. **Парсер:** обработка `__under`/`__over` меток, извлечение embedded
   картинок декора в storage, вычисление offset. Юнит-тесты.
3. **Builder:** динамика — декор следует за __hidden__ и __pos__ базового
   слота. Юнит-тесты на скрытие и смещение.
4. **Канвас:** рендер декора в правильном z-order, текстовые эффекты Konva.
5. **PDF:** рендер декора, текстовые эффекты приблизительно.
6. **Обводки фреймов** (Часть 2) — если идут свойством, иначе покрыто декором.
7. **Z-order переднего плана** (Часть 4) — для сказочного.
8. **Шрифты** (Часть 5).
9. Тестирование на акварельном, потом сказочном.

## Что НЕ трогать
- Школьные мастера (9-11 класс) — механизм декора их не касается.
- Родительские страницы (только аддитивно).
- Логику отбора/submit.
- Существующую систему фонов (декор работает ПОВЕРХ неё).

## Pre-commit
npx tsc --noEmit && npx next build && npx vitest run — зелёное перед каждым
коммитом. Каждый этап = отдельный коммит. Перед коммитом объяснять Сергею
простыми словами + спрашивать подтверждение.
