# ТЗ: Управляемые реферальные программы

## Контекст

Сейчас реферальная механика ЗАШИТА (фиксированный текст «скидка 50%»):
- Родитель на странице «Спасибо» (app/[token] финальный экран) получает
  реф-ссылку /ref/<token>.
- По ссылке открывается лендинг-заявка (app/ref/[token]/page.tsx) с текстом
  «Вас рекомендует <имя>».
- Заявки сохраняются в таблицу referral_leads (поля: referrer_child_id, name,
  phone, city, school, class_name, status). Видны в модалке «Заявки» в админке.
- Награды (50%) — просто текст, нигде не настраиваются.

ЦЕЛЬ: превратить в УПРАВЛЯЕМЫЕ реферальные программы — конструктор кампаний
с настраиваемыми наградами для обеих сторон, двумя картинками, привязкой к
заказу, глобальностью для партнёров, и аналитикой.

ВАЖНЫЕ РЕШЕНИЯ СЕРГЕЯ:
- Награды НЕ автоматические — Сергей применяет скидки вручную. Система только
  ПОКАЗЫВАЕТ обещание наград и ВЕДЁТ УЧЁТ кто кого привёл. Никакой системы
  промокодов/автоскидок в заказах НЕ делать.
- Две РАЗНЫЕ картинки: одна для реферера (мотивирует делиться), другая для
  реферала (мотивирует заказать) — у них разные предложения.
- Одна программа на заказ (альбом).
- Глобальность через tenant_id (как у template_sets): NULL = глобальная
  (рекомендованная всем партнёрам), <okeybook> = внутренняя, <партнёр> = своя.

## Структура: две стороны реферала
- РЕФЕРЕР — кто рекомендует, даёт ссылку. Видит свою награду на странице
  «Спасибо». Пример награды: «50% на копию альбома для бабушки».
- РЕФЕРАЛ — кто пришёл по ссылке. Видит свою награду на лендинге. Пример:
  «Скидка 500₽ на первый заказ».

## База данных

### Таблица referral_programs (новая)
```
id                  uuid primary key default gen_random_uuid()
tenant_id           uuid references tenants(id)  -- NULL = глобальная
name                text not null               -- внутреннее имя («Котики»)
is_active           boolean not null default true

-- Награда реферера (кто рекомендует)
referrer_reward_text  text       -- «50% на копию альбома»
referrer_image_url    text       -- картинка для страницы «Спасибо»

-- Награда реферала (кто пришёл по ссылке)
invitee_reward_text   text       -- «Скидка 500₽ на первый заказ»
invitee_image_url     text       -- картинка для лендинга
invitee_headline      text       -- «Вас рекомендует {имя}» (шаблон, {имя}
                                 --  подставляется из реферера)
invitee_description   text       -- условия/подробности

created_at          timestamptz default now()
```
Глобальность: tenant_id IS NULL → программа глобальная (см. логику
template_sets). Переключатель в UI как у дизайнов.

### Поле albums.referral_program_id (новое)
```
ALTER TABLE albums ADD COLUMN referral_program_id uuid
  REFERENCES referral_programs(id) ON DELETE SET NULL;
-- NULL = программа не назначена (показывать дефолтный текст или ничего)
```

### Поле referral_leads.program_id (новое — для аналитики)
```
ALTER TABLE referral_leads ADD COLUMN program_id uuid
  REFERENCES referral_programs(id) ON DELETE SET NULL;
-- фиксируем по какой программе пришла заявка
```
Связь «кто кого привёл» УЖЕ есть через referral_leads.referrer_child_id.

## API

### CRUD программ (через /api/super или /api/admin)
- create_referral_program { name, ... } → новая программа
- update_referral_program { id, поля }
- delete_referral_program { id } — с проверкой: если привязана к альбомам,
  обнулять albums.referral_program_id (ON DELETE SET NULL уже это делает),
  но предупредить «программа используется в N заказах».
- toggle_referral_program_active { id, is_active }
- set_referral_program_global { id, tenant_id: null | '<uuid>' }
- list_referral_programs — для UI (глобальные + tenant'а).
- Загрузка картинок (referrer_image_url, invitee_image_url) в storage
  bucket (например referral-images). Два отдельных аплоада.

### Привязка к заказу
- update_album { id, referral_program_id } — назначить программу альбому.

## UI: конструктор программ (super-admin)

Новая вкладка/страница «Реферальные программы» (рядом с CRM/Партнёры):
- Список программ (название, активна?, глобальная?, превью картинок).
- Кнопка «Создать программу».
- Форма программы:
  - Название (внутреннее)
  - Активна (вкл/выкл)
  - Глобальная (вкл/выкл → tenant_id)
  - БЛОК «Реферер (кто рекомендует)»:
    - Награда (текст)
    - Картинка (загрузка) — показывается на «Спасибо»
  - БЛОК «Реферал (кто приходит)»:
    - Заголовок (шаблон с {имя})
    - Награда (текст)
    - Описание/условия
    - Картинка (загрузка) — показывается на лендинге
  - Превью обеих сторон (как увидят родители).
- Действия: редактировать, дублировать, удалить, вкл/выкл.

## UI: привязка программы к заказу

В настройках альбома (где партнёр настраивает заказ) — селект
«Реферальная программа»:
- Список доступных программ (глобальные + свои активные).
- Выбор одной (или «без программы»).
- Сохраняется в albums.referral_program_id.

## Показ родителям

### Страница «Спасибо» (реферер) — app/[token] финальный экран
Сейчас: зашитый блок «Получите скидку 50%» + реф-ссылка.
Стало: если у альбома назначена программа:
- Показать referrer_image_url (картинка реферера, если задана).
- Показать referrer_reward_text вместо зашитого «50%».
- Реф-ссылка остаётся (/ref/<token>).
Если программа не назначена — дефолтный текст (как сейчас) или скрыть блок.

### Лендинг реферала — app/ref/[token]/page.tsx
Сейчас: «Вас рекомендует <имя>» + общий текст «Что вы получите».
Стало: определить программу (через альбом реферера → referral_program_id):
- invitee_headline с подстановкой имени реферера.
- invitee_image_url (картинка реферала).
- invitee_reward_text + invitee_description.
- При создании заявки — записать program_id в referral_leads (для аналитики).

## Аналитика (Этап 3)

Дашборд по программам:
- На программу: переходов по ссылкам / заявок / конверсий в заказ.
- Разрез по сегменту: садик / 4 класс / 9-11 класс (через albums.text_type
  или тип заведения реферера).
- Показывает «программа X в садиках даёт N% конверсии» — чтобы усиливать
  что работает и делиться с партнёрами.
Метрики переходов: нужно считать клики по реф-ссылке (добавить счётчик/лог
визитов /ref/<token> с program_id). Конверсия = referral_lead со статусом
«заказ».

## Этапы реализации

**Этап 1 — Конструктор + показ (ядро, ~14-18 ч):**
1. БД: referral_programs + albums.referral_program_id +
   referral_leads.program_id. Миграции (применит Сергей).
2. API: CRUD программ, загрузка 2 картинок, глобальность, привязка к альбому.
3. UI super-admin: конструктор программ (2 блока, 2 картинки, превью).
4. UI: селект программы в настройках заказа.
5. Показ на «Спасибо» (реферер) и лендинге (реферал) — динамически из программы.

**Этап 2 — Учёт связей (~2-3 ч):**
6. Запись program_id в referral_leads. Фиксация referrer→invitee (уже есть
   referrer_child_id). Отображение в модалке «Заявки» какая программа.

**Этап 3 — Аналитика (~6-8 ч):**
7. Счётчик переходов по реф-ссылкам.
8. Дашборд: переходы/заявки/конверсии по программам и сегментам.

## Что НЕ делать
- НЕ делать автоматические скидки/промокоды в заказах — награды Сергей
  применяет вручную. Система только показывает и учитывает.
- НЕ делать автогенерацию картинок — Сергей загружает свои (проще, надёжнее
  для маркетинга). В backlog «может потом».
- НЕ трогать существующую логику отбора/submit на родительских страницах —
  только аддитивно менять блок награды на «Спасибо» и контент лендинга.

## Что НЕ трогать
- Родительские страницы деструктивно (только блок реферала аддитивно).
- Существующую referral_leads механику (расширяем полем program_id).
- Логику отбора фото.

## Глобальность (как у template_sets)
- tenant_id IS NULL → глобальная программа (рекомендованная, видят все).
- При выключении глобальности → tenant_id = okeybook (внутренний tenant
  Сергея, slug 'okeybook').
- Партнёр видит: свои программы + глобальные. Может взять готовую проверенную
  связку или создать свою.

## Pre-commit
npx tsc --noEmit && npx next build && npx vitest run — зелёное перед коммитом.
Каждый этап = отдельный коммит. Перед коммитом объяснять простыми словами +
спрашивать подтверждение.
