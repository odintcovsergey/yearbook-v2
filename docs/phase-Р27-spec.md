# Фаза РЭ.27 — `print_type` в альбоме + правила переплёта

**Спецификация v1.0**
**Дата:** 21.05.2026
**Статус:** утверждена 21.05.2026 (browser-чат с Сергеем, после закрытия РЭ.25).
**Зависит от:** РЭ.22 (семантический engine), РЭ.24 (партнёрский каталог шаблонов), РЭ.25 (галка покупки альбома).
**Блокирует:** РЭ.28 (партнёрские дизайны — выиграют от чистой модели пресетов после слияния).
**Эстимация:** ~7-9 коммитов на main + 2 миграции БД (схема + данные). 1-2 дня работы.

---

## 1. Зачем фаза существует

`print_type` (`layflat` / `soft`) — это **тип переплёта** альбома: твёрдые листы или мягкие. Сейчас он живёт в `presets.print_type`. Из-за этого:

1. **Каждый дизайн дублируется в БД ×2.** «Стандарт-layflat» и «Стандарт-soft» — два пресета с идентичной структурой секций и личного раздела, отличаются **только** колонкой `print_type`. После РЭ.28 (партнёрские дизайны) это раздуется ещё больше — каждый партнёр клонирует дубль-пары.

2. **Партнёр в каталоге `/app/templates` мыслит «комплектация Стандарт», а видит два почти одинаковых пресета.** Тип листов — это решение про физический формат, оно принимается **отдельно** от выбора макета.

3. **Тип переплёта влияет на правила сборки альбома**, а правила — это атрибут конкретного альбома, не шаблона.

### 1.1. Что физически означает `print_type`

**layflat (плотные / твёрдые листы)** — переплёт «лежит плоско». Разворот открытого альбома — одна непрерывная поверхность без сгиба, фото может перекинуться с левой страницы на правую и выглядеть как один кадр. Мастер «фото на разворот» допустим.

**soft (мягкие листы)** — обычный книжный переплёт. Корешок съедает пару миллиметров на сгибе, страницы не лежат плоско. Фото не должно пересекать корешок — иначе при печати теряется его центральная часть. Мастер «фото на разворот» **запрещён**. Плюс типографские правила:
- Первая страница после обложки — **правая** (нечётная, титульная). Слева до неё — форзац.
- Последняя страница — **левая** (чётная). Справа после неё — форзац.

### 1.2. Архитектурное решение: редактор всегда разворотный

**Это ключевое отличие от ранней формулировки в `phase-Р24-spec.md §13`.** Там предполагался двойной редактор (постраничный для soft, разворотный для layflat). После обсуждения с Сергеем 21.05.2026 решение поменялось:

> Вёрстка ВСЕГДА разворотная — как в InDesign. Деление на отдельные страницы происходит только при экспорте в PDF.

**Почему так лучше:**
- Фотограф видит картину целиком, проще компоновать.
- Соответствует реальной полиграфической практике (InDesign так работает).
- Единый предсказуемый UI для всех альбомов.
- **Существующие альбомы НЕ меняют поведения** — они и так в разворотном режиме.
- Объём фазы падает с 10-15 коммитов до 7-9.
- Риск регрессий редактора резко снижается — мы не переписываем редактор, а **добавляем правила** поверх существующего.

**Отличия по `print_type` локализованы в трёх местах:**
1. Engine при сборке: для soft → автозаглушка «Форзац» на первой левой и последней правой странице.
2. Каталог мастеров: для soft → мастера с `page_role` «фото на разворот» скрыты/недоступны.
3. Экспорт PDF: для soft — разделение разворота на 2 страницы; для layflat — цельный разворот.

## 2. Модель данных

### 2.0. Что уже было в коде до старта фазы (открыто 21.05.2026)

**При планировании РЭ.27 считалось, что колонки `albums.print_type` ещё нет.** При проверке состояния прод-БД на старте подэтапа 27.1 выяснилось:

1. **Колонка `albums.print_type TEXT NULL` СУЩЕСТВУЕТ с 8 мая 2026.** Добавлена миграцией 1.0 (`migrations/2026-05-08-album-config.sql`). Там же есть `CHECK constraint print_type IN ('layflat', 'soft')`.

2. **API `create_album` УЖЕ копирует `preset.print_type` в `albums.print_type` при создании** (`app/api/tenant/route.ts` строка 3012, переменная `presetPrintType`). Это сделано в неизвестный момент после 8 мая, без явной привязки к РЭ.27.

3. **Распределение значений на момент старта РЭ.27:**
   - `layflat`: 4 альбома (новые, созданные после появления копирования из пресета)
   - `soft`: 2 альбома (новые)
   - `NULL`: 6 альбомов (старые, до появления копирования)

4. **Engine ПРОДОЛЖАЕТ читать `print_type` из пресета**, а не из альбома. Это и есть основная работа подэтапа 27.3.

5. **`update_album` НЕ позволяет менять `print_type` независимо от пресета** — это работа подэтапа 27.2.

### Что меняет это открытие для плана фазы

- **Подэтап 27.1 фактически уже сделан в части схемы** (колонка + CHECK constraint). Реальная работа 27.1 — это **только создание индекса** `idx_albums_print_type` (partial WHERE NOT NULL) и обновление COMMENT с привязкой к РЭ.27.
- **Подэтап 27.2** становится точнее: не «добавить поле в payload» (оно уже принимается косвенно через копирование из пресета), а **«разрешить явно передавать `print_type` в `create_album` и `update_album`»**, чтобы фотограф мог сменить тип листов БЕЗ смены пресета.
- **Подэтап 27.7** актуален: 6 альбомов с NULL нужно заполнить (UPDATE из preset), и слияние дубль-пресетов остаётся в силе.

### 2.2. Миграция 1 — индекс + документирование (27.1)

⚠️ **ПЕРЕОСМЫСЛЕНО после открытия в §2.0.** Изначально планировалось `ADD COLUMN`. После проверки прод-БД оказалось что колонка уже есть. Миграция сохранена (она идемпотентна) и выполняет реальную работу — **создаёт partial index**, которого раньше не было.

```sql
-- Реальные действия миграции:

-- 1. ADD COLUMN IF NOT EXISTS — no-op (колонка существует с 8 мая 2026).
ALTER TABLE albums
  ADD COLUMN IF NOT EXISTS print_type TEXT;

-- 2. COMMENT — обновляет описание колонки с привязкой к РЭ.27.
COMMENT ON COLUMN albums.print_type IS '...';

-- 3. CREATE INDEX IF NOT EXISTS — ✅ ПОЛЕЗНО, индекса не было.
CREATE INDEX IF NOT EXISTS idx_albums_print_type
  ON albums(print_type) WHERE print_type IS NOT NULL;
```

**Безопасность:** аддитивно, NULL допустим, CHECK constraint уже стоит из старой миграции. Старый код игнорирует поле, engine читает с fallback.

### 2.3. Миграция 2 — данные (27.7, после остальной логики)

После того как engine научится работать с `albums.print_type` (27.3) и UI начнёт его проставлять (27.6), переходим к **миграции данных и слиянию дубль-пресетов**.

```sql
-- Шаг 1: Заполнить albums.print_type у существующих альбомов
-- из print_type связанного пресета.
UPDATE albums a
SET print_type = p.print_type
FROM presets p
WHERE a.config_preset_id = p.id
  AND a.print_type IS NULL;

-- Шаг 2: Перепривязать альбомы с soft-пресетов на layflat-пресеты
-- той же структуры. Парами по slug:
--   'standard-soft' ⇄ 'standard-layflat' → оба становятся 'standard'
-- В этом подэтапе вручную составляем UPDATE'ы по реальным slug'ам
-- (см. план подэтапа 27.7).

-- Шаг 3: Удалить осиротевшие soft-пресеты.
DELETE FROM presets
WHERE print_type = 'soft'
  AND id NOT IN (SELECT config_preset_id FROM albums WHERE config_preset_id IS NOT NULL);

-- Шаг 4: Переименовать layflat-пресеты — убрать суффикс из slug/name.
-- Тоже вручную по реальным записям.

-- Шаг 5 (необязательный, опаснее): после успешного слияния можно
-- удалить колонку presets.print_type. Откладываем на отдельный
-- подэтап или вообще на следующую фазу — это уже зачистка, не
-- блокирует функциональность.
```

**Почему миграция данных в отдельном подэтапе:**
- Сначала вся новая логика работает на `albums.print_type` (со fallback на пресет).
- Когда уверены — мигрируем данные и удаляем дубли.
- Если что-то пойдёт не так на этапе слияния — откат через backup, основная функциональность остаётся живой.

## 3. Engine: правила сборки

### 3.1. Где меняется

`lib/album-builder` — единственное место, где появляется новая логика. Smart-fill, UI, экспорт `print_type` не трогают (или трогают минимально — для чтения значения из альбома).

### 3.2. Резолюция `print_type` в `buildAlbum` / `buildFromSectionStructure`

```typescript
function resolvePrintType(album: AlbumInput, preset: Preset): PrintType {
  // РЭ.27: print_type приоритетно из альбома, fallback на пресет
  // для бэк-совместимости со старыми альбомами до миграции данных.
  return album.print_type ?? preset.print_type;
}
```

Этой одной функцией пользуются все три места ниже.

### 3.3. Правила первого/последнего разворота для soft

При сборке альбома, если `resolvedPrintType === 'soft'`:

- **Первый разворот альбома (после обложки):**
  - Левая страница: автозаглушка «Форзац».
  - Правая страница: первый контентный мастер (обычно `intro` или первая страница ученика).

- **Последний разворот альбома:**
  - Левая страница: последний контентный мастер.
  - Правая страница: автозаглушка «Форзац».

Для `layflat` правила НЕ применяются — разворот собирается как обычно.

### 3.4. Автозаглушка «Форзац»

Это не реальный мастер из template_set, а **синтетическая страница**, генерируемая engine. В JSON layout'а появляется новый тип элемента:

```typescript
type EndpaperPlaceholder = {
  type: 'endpaper';
  side: 'first_left' | 'last_right';
  print_type: 'soft';  // эта заглушка появляется только для soft
};
```

В превью SVG и при экспорте PDF отрисовывается как страница с **водяным знаком «Форзац»** (бледным текстом по центру). Это маркер для фотографа и для типографии — «здесь физически нет содержания, тут будет физический форзац альбома».

**Почему именно водяной знак, а не пустота:**
- Партнёр сразу понимает что это специальная страница, а не забытая.
- Типография понимает что страница оставлена пустой намеренно (без логотипа партнёра / без рекламы).
- Можно в будущем заменить на кастомный логотип партнёра — но это уже за рамками РЭ.27.

### 3.5. Фильтр мастера «фото на разворот» для soft

Сейчас в `template_set` могут быть мастера с `page_role` обозначающим «занимает оба листа разворота» (например `student_spread`, `common_spread`). При `print_type=soft` такие мастера запрещены — фото не должно пересекать корешок.

**Защита в двух местах** (см. развилку 4 в обсуждении):

1. **В engine** (`buildFromSectionStructure` / `findMaster`): если `print_type=soft` и кандидат-мастер имеет «spread»-роль — пропускаем, ищем следующий. Warning `spread_master_skipped_for_soft` для наблюдаемости.

2. **В каталоге мастеров UI** (`/app` форма редактирования разворота, super-каталог `/super/templates`): если `album.print_type=soft` — spread-мастера visually disabled с подсказкой «недоступно для мягких листов». Партнёр не видит их как «выберите», вместо этого видит «нельзя».

Реализация в обоих местах потому что:
- Engine-защита — последний рубеж (если UI-баг или старые данные).
- UI-защита — UX (партнёр не натыкается на запрет постфактум).

### 3.6. Тесты

Чистая функция `resolvePrintType` и правила first/last покрываются unit-тестами **без мока Supabase** — по образцу РЭ.25.3:

- `resolvePrintType`: album.print_type → возвращает; null → fallback на пресет.
- `applyEndpaperRules`: для soft вставляет заглушки в нужные места; для layflat не меняет layout.
- `isSpreadMasterAllowedForPrintType`: для soft возвращает false на spread-роли; для layflat true для всех.

Минимум 8-10 новых тестов. Все существующие тесты engine должны продолжать проходить (407/407 на момент старта).

## 4. UI

### 4.1. Форма создания/редактирования альбома (`AlbumFormModal`)

Новый селект **«Тип листов»** рядом с (или внутри) блока выбора шаблона:

```
[ ] Шаблон: Стандарт                          [Изменить шаблон]
       Дизайн: классика 20×30

    Тип листов: ( ● ) Твёрдые (layflat)
                ( ○ ) Мягкие (soft)
       Подсказка: На мягких листах не доступен мастер
       «фото на разворот» (фото пересекает корешок).
```

- При создании альбома обязательное поле.
- При редактировании существующего альбома — менять можно. Если меняем `layflat → soft` и в альбоме есть spread-мастера — показываем подтверждение «эти мастера будут заменены на одностраничные при следующей пересборке».
- Дефолт для нового альбома: `layflat` (как сейчас в большинстве пресетов).

### 4.2. Карточка шаблона в каталоге `/app/templates`

**Убираем бейджи «layflat/soft»** с карточек. Партнёр видит дизайн один раз без дубля. Бейдж типа листов перенесён в форму альбома.

После миграции данных (27.7) в каталоге **остаётся вдвое меньше** карточек шаблонов — оставшиеся теперь представляют «дизайн», а не «дизайн + тип переплёта».

### 4.3. Каталог мастеров (редактор разворота)

Когда фотограф редактирует разворот и выбирает мастер из палитры:
- Если `album.print_type === 'soft'`: spread-мастера показаны серым с тултипом «недоступно для мягких листов».
- Если `'layflat'`: всё доступно.

Точное место зависит от текущей архитектуры редактора — спецификацию подэтапа 27.5 уточним при реализации, не блокер.

### 4.4. Превью разворотов

В превью альбома (после сборки, перед экспортом) для soft:
- Первый разворот: левая страница со значком «Форзац» (бледным курсивом по центру).
- Последний разворот: правая страница со значком «Форзац».
- Остальные развороты — без изменений.

В export PDF — тот же водяной знак, чтобы типография видела намеренность пустых страниц.

## 5. Слияние дубль-пресетов (Вариант A)

Согласовано с Сергеем 21.05.2026: **сливаем дубль-пары** в один пресет, удаляем soft-варианты после миграции данных. Альтернативы (B — deprecated-флаг, C — гибрид) отклонены.

### 5.1. Подготовка к миграции

В подэтапе 27.7 пишем SQL по реальной выгрузке пресетов из прода. **Не предугадываем** какие именно пары есть — снимаем актуальный список в момент работы:

```sql
-- Сначала смотрим что есть:
SELECT id, slug, name, print_type
FROM presets
ORDER BY slug;
```

Партнёр (Сергей) даёт явный список «эти X пар сливаются в Y итоговых пресетов», прежде чем запускаем UPDATE/DELETE.

### 5.2. Безопасность миграции

- **Перед миграцией** — снапшот таблиц `presets` и `albums`.
- **UPDATE'ы атомарные** — одна транзакция на пару пресетов.
- **DELETE только для осиротевших** — пресет без единого альбома можно удалять.
- **Колонку `presets.print_type` не удаляем сразу** — оставляем как deprecated (NULL после слияния). Удалить можно в следующей фазе или после периода стабильности.

### 5.3. Что после миграции

В каталоге `/app/templates`:
- Половина пресетов исчезает (soft-варианты слиты).
- Партнёр видит чистый список «один дизайн = один пресет».
- Тип листов выбирается в форме альбома.

В существующих альбомах:
- `albums.print_type` уже заполнено (шаг 1 миграции).
- Те, что были на soft-пресете → перепривязаны на объединённый.
- Engine читает `print_type` из альбома, выдаёт правильное поведение.

## 7. Архитектурные развилки (зафиксированы)

**A. Постраничный режим редактора?** ✅ **Решение: НЕТ.** Редактор всегда разворотный, отличия по `print_type` только в правилах сборки и экспорта. Это упрощает фазу с 10-15 коммитов до 7-9, риск регрессий минимальный (редактор не переписываем).

**B. Где живёт `print_type` после фазы?** ✅ **Решение: `albums.print_type` (поле в БД) + fallback на `preset.print_type` для бэк-совместимости.** Через `resolvePrintType(album, preset)` в engine.

**C. Что с автозаглушкой «Форзац»?** ✅ **Решение: водяной знак «Форзац» бледным курсивом по центру страницы.** Не пустота (партнёр поймёт что это специальная страница), не логотип партнёра (это уже отдельная фича для будущих фаз).

**D. Миграция print_type существующих альбомов?** ✅ **Решение: автоматически, из `preset.print_type`.** Это безопасно — у каждого существующего альбома есть пресет с заполненным `print_type` (поле уже работает). Заполнение происходит в подэтапе 27.7 одним UPDATE.

**E. Фильтр spread-мастера для soft?** ✅ **Решение: защита в двух местах — engine (последний рубеж) + UI (UX).** В engine — пропуск с warning. В UI — visually disabled с подсказкой.

**F. Что с дубль-пресетами layflat/soft?** ✅ **Решение: Вариант A — слияние.** Перепривязка ссылок + удаление осиротевших soft-пресетов + переименование (убрать суффикс). Колонку `presets.print_type` оставляем как deprecated (NULL) — удалять в следующей фазе или после периода стабильности.

**G. Что если на soft уже выбран spread-мастер при смене типа листов?** ✅ **Решение: при сохранении формы — подтверждение «эти мастера будут заменены на одностраничные при следующей пересборке».** Engine сам разрулит при пересборке (по правилам 3.5 — пропустит spread-кандидата, найдёт следующий подходящий). Реальной потери данных нет.

## 8. Что НЕ входит в РЭ.27 (явно)

- **Удаление колонки `presets.print_type`.** После слияния (27.7) она становится deprecated и NULL. Удалить можно отдельной зачисткой через время. В РЭ.27 — не удаляем, чтобы fallback продолжал работать у альбомов с `albums.print_type IS NULL`.
- **Постраничный режим редактора.** Не делаем (см. развилку A). Если в будущем партнёр попросит — открываем отдельной фазой.
- **Кастомный форзац с логотипом партнёра.** В РЭ.27 — только водяной знак «Форзац». Полноценный форзац-конструктор — будущая фаза.
- **Третий тип переплёта (например `flexbind`).** Архитектура поддерживает (PrintType расширяется в типе), но конкретные правила не реализуются.
- **Принудительное добавление пустых страниц для соблюдения парности у soft.** Сейчас правила first/last работают только если есть достаточно мастеров. Если у партнёра в альбоме непарное число контентных страниц — будет «дырка», но мы не вставляем фейковые мастера. Это сознательное решение: лучше показать партнёру несоответствие, чем тихо подсунуть пустую страницу.

## 9. Принципы

- **Аддитивная миграция схемы.** `albums.print_type NULL`. Старый код, не знающий о поле, продолжает работать через fallback.
- **Engine остаётся семантическим.** Никаких хардкодов «если slug=standard-soft». Только `resolvePrintType` + правила first/last + фильтр по `page_role`.
- **Чистые функции с unit-тестами.** `resolvePrintType`, `applyEndpaperRules`, `isSpreadMasterAllowedForPrintType` — все тестируются без мока Supabase.
- **Защита в двух местах** для spread-мастера: engine (последний рубеж) + UI (UX).
- **Миграция данных в отдельном подэтапе** (27.7). Сначала вся логика на fallback. Когда уверены — мигрируем.
- **Backward-compat.** Все 407 существующих тестов проходят. Новые тесты добавляются точечно.
- **Tenant-aware строго.** Все запросы через `auth.tenantId`. Слияние пресетов не должно затронуть пресеты других партнёров (на момент написания spec'а — пресеты глобальные, но архитектура учитывает будущее).
- **Малые шаги с откатом.** Каждый подэтап — отдельный коммит. Миграция данных — снапшот БД перед запуском.
