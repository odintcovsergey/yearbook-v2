# Инструкция для подэтапа 0.11.4 — Mini-soft

> Это временный файл-инструкция. После выполнения подэтапа удалить и закоммитить вместе с основными изменениями.

## Цель подэтапа

Реализовать особый случай комплектации **Mini с печатью soft** (мягкие листы) — он отличается от обычного Mini-layflat:

1. **Без S-Intro** — первая страница используется под учительский контент
2. **Учительский раздел одностраничный** — используется F-Head-*-R (правый зеркальный мастер) вместо двухстраничной пары F-Head-* + G-*
3. **G-* мастер не используется** — справа от F-*-R сразу начинается ученический контент

Поддерживается только subjects.length 0-8 (F-Head-WithPhoto-R / F-Head-SmallGrid-R / F-Head-LargeGrid-R). При subjects.length ≥ 9 — warning subjects_overflow + обрезка до 8 (используется F-Head-LargeGrid-R).

Это **завершает раздел 0.11**. Остаются только 0.12 (тесты), 0.13 (endpoint+UI) и 0.14 (контекст).

Один коммит:
```
feat(layout/0.11.4): Mini-soft (одностраничная учительская F-*-R, без S-Intro)
```

## Состояние БД (проверено в 0.11.1.5/0.11.3 разведках)

```
F-Head-WithPhoto-R:   default_for_configs=['mini'], slot_capacity={photos_full:1}
F-Head-SmallGrid-R:   default_for_configs=['mini'], slot_capacity={teachers:4, head_teacher:1}
F-Head-LargeGrid-R:   default_for_configs=['mini'], slot_capacity={teachers:8, head_teacher:1}
```

Никаких миграций БД не требуется — все нужные мастера уже на месте с правильными default_for_configs.

## Архитектурные решения

### Решение 1 — отдельная функция `buildMiniSoftTeacherSection`

Не пытаемся использовать существующий `buildTeacherSection` с overrides. Mini-soft логически отличается:
- Берём только левый F-*-R мастер (по subjects.length)
- Не вызываем `pickRightCommonPhotoMaster`
- Не пушим правую страницу

Это проще и явнее чем расширять `TeacherSpreadVariant` новыми полями.

### Решение 2 — `intro_section: IntroSection | null`

Текущий тип `intro_section?: IntroSection` различает только «задано» vs «отсутствует». Для Mini-soft нужно явно сказать «отключено в soft_overrides». Расширяем тип до `IntroSection | null`:
- `undefined` (поле не задано) → используется базовое значение из ScenarioDef
- `null` (явно отключено) → intro не генерируется
- `IntroSection` объект → используется как обычно

В коде buildAlbum проверка меняется с `if (scenario.intro_section)` на `if (scenario.intro_section !== null && scenario.intro_section !== undefined)`. Для существующих сценариев поведение не меняется.

### Решение 3 — `MiniSoftTeacherSection` отдельный тип

Описывает 3 варианта (по subjects.length), без G-* мастеров и без dynamic-выбора правой страницы.

Текущий код вызывает buildTeacherSection всегда. Нужно различать:

// Раздел 2 — учителя
if (scenario.teacher_section) {

### 2.4 Type guard `isMiniSoftTeacherSection`

Поскольку `TeacherSpreadVariant` имеет `subjects_on_left` и `left_filter`, а `MiniSoftTeacherVariant` имеет `subjects_on_page` и `filter` — различение через проверку поля:

```typescript
function isMiniSoftTeacherSection(
  s: TeacherSection | MiniSoftTeacherSection,
): s is MiniSoftTeacherSection {
  return s.variants.length > 0 && 'subjects_on_page' in s.variants[0];
}
```

### 2.5 Функция `buildMiniSoftTeacherSection`

Добавить в build.ts (рядом с `buildTeacherSection`):

```typescript
/**
 * Сборка учительского раздела для Mini-soft.
 *
 * Отличия от buildTeacherSection:
 *  - Только одна страница (F-*-R мастер)
 *  - Без G-* мастера справа
 *  - Без pickRightCommonPhotoMaster (правой страницы вообще нет)
 *
 * При subjects.length > 8 — обрезка до 8 + warning subjects_overflow
 * (используется F-Head-LargeGrid-R).
 */
function buildMiniSoftTeacherSection(
  ctx: BuildContext,
  section: MiniSoftTeacherSection,
): void {
  const head = ctx.input.head_teacher!;
  let subjects = ctx.input.subjects;

  // Degraded для subjects ≥ 9
  if (subjects.length > 8) {
    pushWarning(ctx, {
      code: 'subjects_overflow',
      detail: `Mini-soft: subjects.length=${subjects.length} > 8, обрезано до 8 (F-Head-LargeGrid-R)`,
    });
    subjects = subjects.slice(0, 8);
  }

  const variant = section.variants.find(
    (v) => subjects.length >= v.subjects_min && subjects.length <= v.subjects_max,
  );
  if (!variant) {
    pushWarning(ctx, {
      code: 'master_not_found',
      detail: `Mini-soft: no teacher variant for subjects.length=${subjects.length}`,
    });
    return;
  }

  const filter = { ...variant.filter, applies_to_config: ctx.config.config_type };
  const r = findMaster(ctx.config.template_set.spreads, filter);
  if (!r.ok) {
    pushWarning(ctx, {
      code: 'master_not_found',
      detail: `Mini-soft teacher: ${variant.filter.expected_name_hint ?? '?'}`,
    });
    return;
  }
  if (r.warning) pushWarning(ctx, r.warning);

  const sideTeachers = subjects.slice(0, variant.subjects_on_page);
  const data = buildTeacherLeftData(ctx, r.master, head, sideTeachers);
  // buildTeacherLeftData уже корректно обрабатывает classphotoframe для F-Head-WithPhoto-R

  ctx.spreads.push({
    spread_index: ctx.spreadCounter.value++,
    template_id: r.master.id,
    template_name: r.master.name,
    data,
  });
}
```

`buildTeacherLeftData` уже существует с 0.10b.2 — переиспользуем (она универсально работает с любым F-* мастером, включая F-*-R).
