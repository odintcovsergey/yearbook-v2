# Инструкция для подэтапа 2.1

**Цель:** мигрировать схему `album_layouts` под фазу 2 — добавить флаг
`has_user_edits` для отслеживания правок партнёра в canvas-редакторе.
Дополнительно — расширить endpoint `POST /api/layout?action=build_album`
так, чтобы при пересборке альбома флаг сбрасывался в `false`.

**Зачем:** в фазе 2 партнёр сможет руками редактировать layout (drag-and-drop
фото в placeholder'ах). После каждой правки `save_album_layout` (подэтап
2.5) ставит `has_user_edits=true`. При повторном клике «Собрать
автоматически» UI спрашивает confirm, потому что сборка перезатирает
правки. Сама сборка (build_album) ОБЯЗАНА сбрасывать флаг — иначе после
пересборки UI будет думать, что правки всё ещё есть, и снова спрашивать
confirm на следующем нажатии.

**Контекст:** на момент закрытия фазы 1 в БД 2 layout-записи (Красночетайская
СОШ, Школа 89). Колонка добавляется с `NOT NULL DEFAULT false`, существующие
строки получают `false` автоматически. Никаких данных не теряем.

**Что меняется:**
- Файл миграции `migrations/2026-05-09-album-layouts-has-user-edits.sql`
  (один `ALTER TABLE`)
- `app/api/layout/route.ts` — в `handleBuildAlbum` в объект upsert'а
  добавляется одна строка `has_user_edits: false`
- Файл инструкции `docs/internal/2.1-instructions.md` (этот файл)

**Что НЕ трогаем:**
- `handleBuildAlbumTest` — он не пишет в `album_layouts`, только возвращает
  синтетический JSON
- `handleGetAlbumLayout` — добавление `has_user_edits` в response понадобится
  в подэтапе 2.7 (confirm-диалог), там и сделаем
- Любой другой код фазы 1 — алгоритм builder'а, smart-fill, UI кнопки
  «Собрать автоматически»

## Принципиальные решения

### `has_user_edits BOOLEAN NOT NULL DEFAULT false`

`NOT NULL` с дефолтом — это нужно. Если где-то upsert не передаст поле,
получим `false` (что корректно семантически: «правок нет»). Без default'а
пришлось бы:
- На существующих 2 записях получить NULL → потенциальные баги в UI
  (`if (layout.has_user_edits)` будет фолсить, но `=== true` не будет
  выполняться);
- В каждом upsert явно писать `has_user_edits: false`.

`DEFAULT false` обеспечивает обратную совместимость со старым кодом
(который не знает про флаг) и с потенциальными ad-hoc INSERT'ами из
SQL editor.

### Безопасный re-run миграции

Используем `IF NOT EXISTS`. Если миграция случайно прогонится дважды —
не упадёт. Это типичная защита для SQL-миграций которые применяются
вручную в Supabase SQL editor (как и предыдущие миграции этого репо).

### Почему сбрасываем флаг при каждом build_album, а не только когда `has_user_edits=true`

Можно было бы написать «UPDATE has_user_edits=false WHERE has_user_edits=true»
условно. Но:
1. Endpoint и так делает full upsert — добавление поля бесплатно.
2. Семантически build_album = «обнулить layout до автосборки», и флаг
   правок — часть состояния layout'а. Логичнее сбрасывать всегда.
3. Меньше веток кода = меньше багов. Если вдруг `has_user_edits` каким-то
   образом окажется `true` после неудачной сборки — следующий build_album
   гарантированно его обнулит.

### Передаём `has_user_edits: false` явно (не полагаемся на DEFAULT)

При INSERT default сработает, но при UPDATE — нет. Поскольку у нас upsert
с `onConflict: 'album_id'`, путь UPDATE — самый частый (партнёр пересобирает
альбом). В UPDATE по умолчанию неуказанные поля сохраняются как есть.
Чтобы сбросить флаг, его НУЖНО явно передать.

## Шаги

### Шаг 1. Создать SQL-миграцию

Файл: `migrations/2026-05-09-album-layouts-has-user-edits.sql`

```sql
-- Подэтап 2.1 — флаг has_user_edits в album_layouts.
--
-- Контекст: в фазе 2 партнёр редактирует layout вручную (drag-and-drop).
-- save_album_layout (подэтап 2.5) ставит флаг в true. build_album
-- (подэтап 2.1, этот же подэтап) сбрасывает флаг в false при пересборке.
-- UI 2.7 проверяет флаг и спрашивает confirm перед пересборкой.
--
-- На момент применения в БД 2 layout-записи (Красночетайская СОШ, Школа 89).
-- DEFAULT false → существующие записи получают false автоматически.
-- Никаких данных не теряем.

-- Добавить колонку has_user_edits.
-- IF NOT EXISTS на случай если миграция случайно прогоняется дважды.
ALTER TABLE album_layouts
  ADD COLUMN IF NOT EXISTS has_user_edits boolean NOT NULL DEFAULT false;

-- Проверка структуры
SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'album_layouts'
  AND column_name = 'has_user_edits';
-- Ожидание:
--   has_user_edits | boolean | NO | false

-- Проверка что у существующих записей флаг = false
SELECT id, album_id, has_user_edits FROM album_layouts;
-- Ожидание: 2 записи (или сколько на момент применения), все с
-- has_user_edits = false
```

## Применение миграции

**ВАЖНО — порядок: миграция в Supabase ПЕРВОЙ, потом push.**

Логика:
- Если push идёт первым → Vercel автодеплоит → новый код шлёт upsert
  с `has_user_edits: false` → колонки нет → 500 ошибка на каждом
  build_album пока миграция не применится.
- Если миграция первая → колонка с DEFAULT false → старый Vercel-код
  без поля работает корректно (получает default), новый код после
  деплоя тоже работает. Никакого окна нестабильности.

Порядок:

1. **До push** — Сергей применяет миграцию в Supabase SQL editor:
   - Скопировать содержимое `migrations/2026-05-09-album-layouts-has-user-edits.sql`
   - Прогнать целиком
   - Проверить вывод финальных SELECT'ов:
     - `information_schema.columns` показывает `has_user_edits | boolean | NO | false`
     - `SELECT id, album_id, has_user_edits FROM album_layouts` —
       все существующие записи с `has_user_edits = false`
2. **После применения** — push коммита.
