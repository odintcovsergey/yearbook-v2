# Инструкция для подэтапа 1.3

**Цель:** реализовать endpoint POST `/api/layout?action=build_album`
который принимает `album_id`, прогоняет через `buildAlbumInput` →
`buildAlbum`, классифицирует warning'и по уровням и сохраняет результат
в `album_layouts` (upsert).

**Зачем:** smart-fill helper из 1.2 уже работает локально через CLI.
Endpoint оборачивает его в HTTP с tenant scoping (включая view_as для
OkeyBook staff), сохраняет результат для последующего показа в UI
(подэтап 1.4) и пересборки.

**Что меняется:**
- `lib/album-builder/load-template-set.ts` — добавить `loadPresetById`
  по аналогии с `loadPresetBySlug` (нужно загружать пресет по uuid из
  `albums.config_preset_id`)
- `lib/album-builder/index.ts` — экспортировать `loadPresetById`
- `app/api/layout/route.ts` — добавить новый action `build_album`
  в существующий POST handler (рядом с `build_album_test` и `import_idml`)

**Что НЕ трогаем:**
- `lib/smart-fill/build-album-input.ts` — переиспользуем как есть
- `lib/album-builder/build-from-preset.ts` — buildAlbum не меняется
- UI — это подэтап 1.4

## Принципиальные решения

### Один POST endpoint, три action'а

В `/api/layout` уже есть `build_album_test` (синтетические данные, не
сохраняет) и `import_idml` (multipart, импорт template_set). Добавляем
третий action `build_album` (реальные данные, сохраняет в `album_layouts`).
Дискриминатор — `?action=build_album` в query string, как у соседних.

`build_album_test` остаётся живой — он удобен для проверки изменений
builder'а на синтетических сценариях через UI Build Test
(`/super/templates/[id]`). Не трогаем.

### Tenant scoping + view_as

Endpoint поддерживает три кейса доступа:
1. Партнёр-фотограф: `auth.tenantId` совпадает с `albums.tenant_id`
2. OkeyBook staff с view_as: `auth` в main тенанте, `?view_as=<partner_id>`
3. Superadmin: всё доступно

Реализация — копия паттерна из `/api/tenant/route.ts:131-138`
(определение `tid` через `canViewAs && viewAsTenantId`) +
`assertAlbumAccess(auth, albumId, tid)`.

**Локальная копия `assertAlbumAccess`** в layout route. Альтернатива —
вынести в shared helper `lib/auth.ts` или импортировать из tenant
route. Но импорт между route handler'ами неаккуратен (циркулярные
зависимости риск), а вынос в auth.ts — отдельная рефакторинг-задача
вне scope 1.3. Дублируем 10 строк — работает, легко рефакторится позже.

### loadPresetById vs loadPresetBySlug

`buildAlbumInput` возвращает `template_set_id`, а в БД у альбома
лежит `config_preset_id` (uuid). Нужна функция загрузки пресета по
id. Делаем новую `loadPresetById(supabase, id)` — точная копия
`loadPresetBySlug` с заменой фильтра. Альтернативы (передавать slug
через лишний SELECT) — неоптимально и нелогично.

### Классификация warning'ов

Спека определяет 3 уровня:
- `blocking` — нельзя печатать как есть
- `degraded` — собралось, но компромиссно
- `info` — нормальная ситуация, к сведению

Маппинг (см. `docs/phase-1-spec.md` строки 325-343):

```typescript
const WARNING_LEVELS: Record<string, 'blocking' | 'degraded' | 'info'> = {
  // Builder — blocking
  master_not_found: 'blocking',
  students_empty: 'blocking',

  // Builder — degraded
  students_overflow: 'degraded',
  subjects_overflow: 'degraded',
  students_grid_no_special_master: 'degraded',
  name_mismatch: 'degraded',
  class_photo_missing: 'degraded',
  students_odd_in_standard: 'degraded',

  half_class_missing: 'degraded',
  fallback_used: 'degraded',
  students_too_few: 'degraded',
  adaptive_grid_fallback: 'degraded',

  no_head_teacher: 'info',

  // Smart-fill — info
  students_no_portrait: 'info',
  per_child_override_ignored: 'info',

```

**Что добавилось относительно спеки:** в спеке (строки 326-343)
перечислены 13 кодов, но реальный builder сейчас имеет 14 кодов
(`BuildWarningCode` в types.ts:370-384) + 2 smart-fill = 16. Дополнил
маппинг, чтобы ни один warning не остался без уровня. Если builder
добавит новый код в будущем — fallback на 'degraded' через `??` в
коде endpoint'а (см. ниже).

### Fallback для неизвестного warning кода

```typescript
const level = WARNING_LEVELS[w.code] ?? 'degraded';
```

Если в будущем builder добавит новый warning (например, в фазе 1.5+),
он не уронит endpoint — фолбэк на `degraded`. Это разумный default:
неклассифицированный warning стоит показать, но не как blocking.

### Формат warnings в response и в БД

Каждый warning в response получает `level` (классификацию) и `source`
('builder' или 'smart_fill'). В БД хранится та же расширенная форма,
чтобы UI мог показать категории без повторной классификации.

```typescript
type EnrichedWarning = {
  code: string;
  detail: string;
  level: 'blocking' | 'degraded' | 'info';
  source: 'builder' | 'smart_fill';
};
```

Это аддитивное расширение over `BuildWarning` — старое поле остаётся
доступно. UI 1.4 будет читать `level` напрямую.

### Upsert по album_id

Спека (`docs/phase-1-spec.md:49`) говорит «Upsert по unique album_id».
Перед коммитом проверяем что в БД есть unique-констрейнт на
`album_id`. Если нет — добавляем миграцией в инструкции отдельным
шагом 0 (см. ниже).

Использование Supabase `upsert` с `onConflict: 'album_id'`:

```typescript
const { data, error } = await supabase
  .from('album_layouts')
  .upsert(
    {
      album_id: albumId,
      template_set_id: templateSetId,
      config_preset_id: preset.id,
      spreads: result.spreads,
      warnings: enrichedWarnings,
      updated_at: new Date().toISOString(),
      // status НЕ передаём — у существующих сохраняется, у новых = default 'draft'
    },
    { onConflict: 'album_id' },
  )
  .select('id')
  .single();
```

`status` намеренно не передаётся в upsert — спека (`docs/phase-1-spec.md:49`):
«Всегда 'draft' при первой записи; при upsert — сохраняется».
PostgreSQL `INSERT ON CONFLICT DO UPDATE` (что под капотом у Supabase
upsert) при отсутствии поля в INSERT VALUES (но при использовании
`upsert()` без передачи поля) — поведение зависит от того как
формируется UPDATE clause.

**Важный нюанс:** Supabase `.upsert({...})` транслируется в
`INSERT ... ON CONFLICT DO UPDATE SET column1=excluded.column1, ...`
для каждого переданного поля. Если поле `status` не передано — его
не будет в UPDATE clause, существующее значение сохранится.
**Это и есть нужное поведение.** Передавать `status` нельзя — это
сбросит его до 'draft' на каждом перестроении.

### Что endpoint возвращает

```typescript
{
  spreads: SpreadInstance[],     // result.spreads as-is
  warnings: EnrichedWarning[],   // классифицированные с source
  layout_id: string,             // uuid из album_layouts
  summary: {
    total_spreads: number,
    total_warnings: number,
    warnings_by_level: { blocking: number, degraded: number, info: number },
    preset_slug: string,
    preset_name: string,
  }
}
```

UI 1.4 будет показывать spreads в Canvas-режиме (фаза 2) или JSON
дамп (MVP), warnings — категоризированные секции по level.

### Audit log

Запись в `audit_log` через `logAction(auth, 'album_layout.build', 'album', album_id, {...})`.
Метаданные: `template_set_id`, `preset_slug`, `total_spreads`,
`total_warnings`. Это даёт трейс кто и когда что собирал.

### Шаг 1. Добавить `loadPresetById` в `lib/album-builder/load-template-set.ts`

```typescript
/**
 * Загрузка config_preset по UUID. Парная к loadPresetBySlug — используется
 * в endpoint'ах где у альбома уже есть config_preset_id (FK).
 *
 * @param supabase — клиент с правами SELECT на config_presets
 * @param id — UUID пресета (из albums.config_preset_id)
 */
export async function loadPresetById(
  supabase: SupabaseClient,
  id: string,
): Promise<Preset> {
  const { data, error } = await supabase
    .from('config_presets')
    .select('*')
    .eq('id', id)
    .single();
  if (error || !data) {
    throw new Error(
      `config_preset id=${id} not found: ${error?.message ?? 'no row'}`,
    );
  }
  return data as Preset;
}
```

#### 3.5. Реализация `handleBuildAlbum`

В конец файла (после `handleBuildAlbumTest`) добавить:

```typescript
/**
 * POST /api/layout?action=build_album
 *
 * Smart-fill endpoint: читает реальный альбом из БД, прогоняет через
 * builder, сохраняет результат в album_layouts (upsert), возвращает
 * spreads + классифицированные warnings.
 *
 * Body: { album_id: string }
 *
 * Доступ: owner/manager/viewer тенанта-владельца альбома, OkeyBook staff
 * через ?view_as=<tenant_id>, superadmin без ограничений.
 */
async function handleBuildAlbum(
  req: NextRequest,
  auth: AuthContext,
): Promise<NextResponse> {
  // 1. Validate body
  let body: unknown
  try {
    body = await req.json()
  } catch {
    return NextResponse.json({ error: 'invalid JSON body' }, { status: 400 })
  }

  if (typeof body !== 'object' || body === null) {
    return NextResponse.json({ error: 'body must be object' }, { status: 400 })
  }
  const albumId = (body as Record<string, unknown>).album_id
  if (typeof albumId !== 'string' || !UUID_REGEX.test(albumId)) {
    return NextResponse.json(
      { error: 'album_id is required (uuid)' },
      { status: 400 },
    )
  }

  // 2. Tenant scoping (с view_as для OkeyBook staff)
  const viewAsTenantId = req.nextUrl.searchParams.get('view_as')
  const { data: currentTenantData } = viewAsTenantId
    ? await supabaseAdmin.from('tenants').select('slug').eq('id', auth.tenantId).single()
    : { data: null }
  const canViewAs = auth.role === 'superadmin' || currentTenantData?.slug === 'main'
  const tid = (canViewAs && viewAsTenantId) ? viewAsTenantId : auth.tenantId

  if (!(await assertAlbumAccessLocal(auth, albumId, tid))) {
    return NextResponse.json({ error: 'access denied' }, { status: 403 })
  }

  // 3. Загрузить альбом + проверить пресет
  const { data: album, error: albumErr } = await supabaseAdmin
    .from('albums')
    .select('id, config_preset_id, template_set_id')
    .eq('id', albumId)
    .single()

  if (albumErr || !album) {
    return NextResponse.json({ error: 'album not found' }, { status: 404 })
  }

  if (!album.config_preset_id) {
    return NextResponse.json(
      { error: 'album has no config_preset_id (выберите пресет вёрстки в форме редактирования)' },
      { status: 400 },
    )
  }

  // 4. Собрать AlbumInput из БД (smart-fill helper из 1.2)
  let smartFillResult: { input: AlbumInput; warnings: SmartFillWarning[] }
  try {
    smartFillResult = await buildAlbumInput(supabaseAdmin, albumId)
  } catch (e) {
    return NextResponse.json(
      { error: `smart-fill failed: ${(e as Error).message}` },
      { status: 500 },
    )
  }

  // 5. Загрузить preset + template_set
  let preset: Preset
  try {
    preset = await loadPresetById(supabaseAdmin, album.config_preset_id)
  } catch (e) {
    return NextResponse.json(
      { error: `preset load failed: ${(e as Error).message}` },
      { status: 500 },
    )
  }

  let templateSet: TemplateSet
  try {
    templateSet = await loadTemplateSet(supabaseAdmin)
  } catch (e) {
    return NextResponse.json(
      { error: `template_set load failed: ${(e as Error).message}` },
      { status: 500 },
    )
  }

  // 6. Запустить buildAlbum
  const result = buildAlbum(smartFillResult.input, preset, templateSet)

  // 7. Объединить и классифицировать warnings
  const enrichedWarnings: EnrichedWarning[] = [
    ...result.warnings.map((w) => enrichWarning(w, 'builder')),
    ...smartFillResult.warnings.map((w) => enrichWarning(w, 'smart_fill')),
  ]

  const warningsByLevel = {
    blocking: enrichedWarnings.filter((w) => w.level === 'blocking').length,
    degraded: enrichedWarnings.filter((w) => w.level === 'degraded').length,
    info: enrichedWarnings.filter((w) => w.level === 'info').length,
  }

  // 8. Upsert в album_layouts.
  // status НЕ передаём: при INSERT default='draft', при UPDATE сохраняется
  // существующее значение (см. docs/phase-1-spec.md:49).
  const { data: layoutRow, error: upsertErr } = await supabaseAdmin
    .from('album_layouts')
    .upsert(
      {
        album_id: albumId,
        template_set_id: templateSet.id,
        config_preset_id: preset.id,
        spreads: result.spreads,
        warnings: enrichedWarnings,
        updated_at: new Date().toISOString(),
      },
      { onConflict: 'album_id' },
    )
    .select('id')
    .single()

  if (upsertErr || !layoutRow) {
    return NextResponse.json(
      { error: `album_layouts upsert failed: ${upsertErr?.message ?? 'no row'}` },
      { status: 500 },
    )
  }

  // 9. Audit log
  await logAction(auth, 'album_layout.build', 'album', albumId, {
    template_set_id: templateSet.id,
    preset_slug: preset.slug,
    total_spreads: result.spreads.length,
    total_warnings: enrichedWarnings.length,
    warnings_by_level: warningsByLevel,
  })

  // 10. Response
  return NextResponse.json({
    spreads: result.spreads,
    warnings: enrichedWarnings,
    layout_id: layoutRow.id,
    summary: {
      total_spreads: result.spreads.length,
      total_warnings: enrichedWarnings.length,
      warnings_by_level: warningsByLevel,
      preset_slug: preset.slug,
      preset_name: preset.name,
    },
  })
}
```

## Открытые риски

### UNIQUE на album_id

Если в БД нет UNIQUE, `upsert` с `onConflict: 'album_id'` упадёт.
Шаг 0 обязателен. Если миграция требуется — добавляется в этот же
коммит.

### Длинный response

Один альбом × ~10-15 разворотов × десятки полей в `data` объекте
+ warnings = 50-200 КБ JSON. Это в пределах нормы для Vercel response
(лимит 4.5 МБ). Если реально окажется >2 МБ — компрессия включится
автоматически (Vercel default). Не оптимизируем заранее.

### Параллельные клики «Пересобрать»

Если партнёр кликнет дважды быстро — два upsert'а. Последний
выигрывает по `updated_at`. Никаких race conditions критических нет
(идемпотентная сборка). UI 1.4 будет блокировать кнопку через `busy`
state — это уже там.

### audit_log table missing

Если `logAction` упадёт (например, audit_log в этом тенанте недоступен) —
`logAction` сам ловит ошибки внутри (см. паттерн в существующих route
handlers), endpoint не упадёт. Это документировано в коде
`logAction` — проверять не надо.
