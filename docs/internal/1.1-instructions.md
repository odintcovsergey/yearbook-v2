# Инструкция для подэтапа 1.1

**Цель:** мигрировать схему `album_layouts` под фазу 1 — убрать legacy
`config_type`/`print_type`, добавить `config_preset_id` (FK на
`config_presets`) и `warnings jsonb`.

**Зачем:** в подэтапе 1.3 endpoint `POST /api/layout?action=build_album`
будет делать upsert в `album_layouts` с результатом сборки. В фазе 0.5
`config_type`/`print_type` стали избыточными — пресет содержит обе эти
характеристики. `warnings` — отдельная колонка для list'а warning'ов от
builder + smart-fill, чтобы UI мог их показывать без повторной сборки.

**Контекст:** перед миграцией Сергей подтвердил
`SELECT COUNT(*) FROM album_layouts → 0`. То есть таблица пустая, никаких
данных не теряем. В коде ни одного запроса к таблице нет (проверено
`grep -rn "from.*album_layouts" app/ scripts/` → 0 совпадений). Это
самый безопасный момент для DROP COLUMN.

**Что меняется:**
- Файл миграции `migrations/2026-05-09-album-layouts-preset-fk.sql`
- Больше **ничего**. Кода в репо, который читает/пишет `album_layouts`,
  нет — миграция чисто DDL без правок TypeScript.

**Что НЕ трогаем:**
- `lib/album-builder/types.ts` — там только комментарий упоминающий
  миграцию 0.1; смысл его не меняется
- API endpoint и UI — будут добавлены в подэтапах 1.3/1.4

---

## Принципиальные решения

### `config_preset_id` без NOT NULL

Внешний ключ `config_preset_id uuid REFERENCES config_presets(id)` без
`NOT NULL`. Причина: в endpoint'е 1.3 проверка пресета будет на уровне
`albums.config_preset_id` — если у альбома пресета нет, endpoint вернёт
400 ещё до записи. Но строжайшее `NOT NULL` на `album_layouts` сделало
бы будущие правки болезненными (например, тестовая запись от админа без
пресета — сейчас пусть будет возможна).

В подэтапе 1.3 при upsert всегда пишем `config_preset_id` — это
гарантия на уровне бизнес-логики. БД-constraint не нужен.

### `warnings jsonb NOT NULL DEFAULT '[]'`

`NOT NULL` с дефолтом — это нужно. Если где-то upsert не передаст поле,
получим пустой массив (что корректно семантически: «нет warning'ов»).
Без default'а пришлось бы в каждом upsert явно писать `warnings: []`.

### ON DELETE на FK

По умолчанию `REFERENCES` без `ON DELETE` = `NO ACTION` (запретит
удаление пресета пока есть ссылки). Это безопасно для глобальных
пресетов (их трогать не должны). Для будущих партнёрских пресетов
тоже норм — partner не сможет удалить пресет если он используется в
существующем layout.

Альтернатива `ON DELETE SET NULL` была бы тоже валидна (при удалении
пресета layout остаётся, ссылка обнуляется). Я выбираю `NO ACTION`
потому что нулевой `config_preset_id` в layout'е не имеет смысла —
если хотят пересобрать, должна быть явная ссылка на пресет.

### Не делаем DROP CHECK явно

Спека упоминает «Старый CHECK на config_type/print_type удалится
автоматически с колонками». Это так — PostgreSQL автоматически удаляет
column-level CHECK при `DROP COLUMN`. Если в схеме окажется
table-level CHECK с упоминанием обеих колонок — он тоже удалится при
удалении любой из них (cascade поведение). Дополнительный
`DROP CONSTRAINT` не нужен.

### Безопасный re-run

Миграция использует `IF EXISTS` / `IF NOT EXISTS` где это применимо.
Если случайно прогонится дважды — не упадёт. Это типичная защита для
SQL-миграций которые применяются вручную в Supabase SQL editor.

## Шаги

### Шаг 1. Создать SQL-миграцию

Файл: `migrations/2026-05-09-album-layouts-preset-fk.sql`

```sql
-- Подэтап 1.1 — schema P3 для album_layouts.
-- Контекст: в фазе 0.5 пресет (config_presets) стал единственным источником
-- комплектации и типа печати. Колонки album_layouts.config_type/print_type
-- больше не нужны — их заменяет config_preset_id (FK).
-- Также добавляем warnings jsonb для хранения warning'ов от builder +
-- smart-fill (фаза 1.3).
--
-- Перед применением: SELECT COUNT(*) FROM album_layouts должен быть 0
-- (проверено 09.05.2026). Никаких данных не теряем.
-- В коде ни одного запроса к album_layouts нет — DROP COLUMN не сломает
-- сборку.

-- 1. Удалить legacy колонки.
-- IF EXISTS на случай если миграция случайно прогоняется дважды.
ALTER TABLE album_layouts
  DROP COLUMN IF EXISTS config_type,
  DROP COLUMN IF EXISTS print_type;

-- 2. Добавить FK на config_presets.
-- Без NOT NULL — endpoint 1.3 гарантирует наличие через бизнес-логику.
ALTER TABLE album_layouts
  ADD COLUMN IF NOT EXISTS config_preset_id uuid REFERENCES config_presets(id);

-- 3. Добавить warnings jsonb с default'ом.
-- NOT NULL DEFAULT '[]' — пустой массив если warning'ов нет, чтобы
-- upsert мог не передавать поле явно.
ALTER TABLE album_layouts
  ADD COLUMN IF NOT EXISTS warnings jsonb NOT NULL DEFAULT '[]'::jsonb;

-- Проверка структуры
SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'album_layouts'
ORDER BY ordinal_position;
-- Ожидание (после миграции):
--   id           | uuid                        | NO  | gen_random_uuid()
--   album_id     | uuid                        | NO  |
--   template_set_id | uuid                     | NO  |
--   spreads      | jsonb                       | NO  |
--   status       | text                        | NO  | 'draft'
--   created_at   | timestamp with time zone    | NO  | now()
--   updated_at   | timestamp with time zone    | NO  | now()
--   config_preset_id | uuid                    | YES |
--   warnings     | jsonb                       | NO  | '[]'::jsonb
-- (точный набор может отличаться по количеству и порядку — главное:
-- config_type и print_type отсутствуют, config_preset_id и warnings есть)

-- Проверка количества записей (должно быть 0 — таблица пустая)
SELECT COUNT(*) AS layouts_count FROM album_layouts;
-- Ожидание: 0
```

изменений в коде в этом подэтапе).

## Применение миграции

**ВАЖНО — применить миграцию вручную в Supabase SQL editor.**
Файл коммитится в репо для документации, но автоматически не применяется.

Порядок:

1. **До push** — Сергей применяет миграцию в Supabase SQL editor:
   - Скопировать содержимое `migrations/2026-05-09-album-layouts-preset-fk.sql`
   - Прогнать целиком
   - Проверить вывод финальных SELECT'ов:
     - В `information_schema.columns` нет `config_type` и `print_type`,
       есть `config_preset_id` и `warnings`
     - `layouts_count = 0`
2. **После применения** — push коммита.

Порядок «миграция → push» здесь менее критичен чем в 1.0 (никакой код
не зависит от схемы), но привычка полезная. Если push идёт первым —
ничего не сломается, потому что в коде ни одного запроса к таблице
нет.

## После миграции — что разблокируется

После 1.1 у нас:

- `album_layouts.warnings` — есть, default `[]`

spreads, warnings})`. Без миграции upsert упадёт на отсутствующих
колонках.

читает БД и собирает `AlbumInput`, без endpoint'а).
