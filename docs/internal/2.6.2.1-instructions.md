# Инструкция для подэтапа 2.6.2.1

**Цель:** два точечных UX-фикса в редакторе по результатам визуальной
проверки 2.6.2. Один коммит.

**Зачем:**

1. **Страница «расползается» по высоте.** Палитра с 179 миниатюрами
   занимает ~6000px по высоте. Поскольку корневой `<div>` использует
   `min-h-screen` (а не `h-screen`), контейнер растягивается под
   палитру → canvas «уезжает» вниз → партнёр открывает редактор и
   видит только верхушку палитры, до canvas'а нужно скроллить.

2. **401 на странице редактора при истёкшем токене.** Локальный `api()`
   хелпер не делает refresh — открыли редактор после нескольких часов
   простоя, токен истёк → красный «layout load failed: 401» → нужно
   идти в `/app` чтобы триггернуть refresh, потом возвращаться. Я
   сам ошибочно решил в 2.6.1 что refresh-loop тут не нужен. Это плохой
   UX, особенно для длительной работы в редакторе (drag-and-drop +
   auto-save в 2.6.3-4 будут долгими сессиями).

**Контекст:** оба фикса — точечные, без архитектурных изменений.
В одном коммите потому что оба относятся к UX-надёжности редактора и
оба нашлись в одну проверку.

**Что меняется:**
- `app/app/album/[id]/layout/page.tsx` — три точки:
  1. `min-h-screen` → `h-screen` в корневом `<div>` рендера (фикс 1)
  2. Глобальная `let _refreshing` + функция `refreshAccessToken()`
     (фикс 2)
  3. Расширение `api()` хелпера на 401-ретрай через refresh (фикс 2)
- `docs/internal/2.6.2.1-instructions.md` — этот файл

**Что НЕ трогаем:**
- `min-h-screen` в loading/error блоках (~строки 159, 167) — там
  страница без сложного контента, оставляем как есть. На loading-
  экране палитры нет, страница не «расползается».
- Существующий `api()` хелпер в `app/app/page.tsx` — это другой файл,
  у него своя жизнь. Цель этого подэтапа не унификация, а локальный
  фикс редактора.
- Любой другой код за пределами `page.tsx` редактора.

## Принципиальные решения

### Фикс 1: `h-screen`, не `overflow-hidden` на main

Альтернативное решение: оставить `min-h-screen`, но добавить
`max-h-screen` или `h-[100vh]` где-то ниже. Это работает, но
семантически менее чисто:

- `min-h-screen` = «контейнер минимум на весь viewport, может расти»
- `h-screen` = «контейнер ровно в высоту viewport, не больше и не меньше»

Для fullscreen-редактора нам нужно второе. Дочерний `<div className="flex-1 flex overflow-hidden">` уже корректно настроен
(flex-1 заставит занять всё available, overflow-hidden запретит
утечку). Просто корневой контейнер был неправильно ограничен.

3 символа фикса (`min-` удалить).

### Фикс 2: копия `refreshAccessToken` + расширенный `api()`

В `app/app/page.tsx` уже есть проверенная реализация (строки 90-120).
Копируем целиком в редактор. Микро-дублирование (~30 строк), но это
self-contained — редактор больше ни от чего не зависит.

**Альтернатива** — вынести в `lib/auth-client.ts` и импортировать в
оба места. Это правильное решение «на потом», но:
- Требует трогать `app/app/page.tsx` (7000+ строк) — риск регрессий
- Пара дополнительных коммитов рефакторинга

**Не делаем сейчас**, отметка в backlog: «вынести
refreshAccessToken+api в lib/auth-client.ts (когда будет 3-й consumer
кроме `/app/page.tsx` и редактора)».

### Глобальная переменная `_refreshing`

Это in-flight deduplication: если несколько одновременных запросов
получат 401, они все запустят refresh — но через `_refreshing`
будут ждать одного общего fetch'а. Не убираем, копируем как есть.

Имя `_refreshing` не пересекается с существующим в `page.tsx`
(они в разных модулях). Module-scope переменные в TypeScript/JS
изолированы между файлами.

### Не делаем циклов retry на 401 после refresh

Если refresh не удался → возвращаем 401 response, дальше код handle'ит
как обычную ошибку (показывается «layout load failed: 401» с кнопкой
«В кабинет»). Это лучше чем infinite loop. Партнёр кликает «В кабинет»,
там полноценный refresh цикл, потом обратно в редактор.

## Шаги

### Шаг 1. Фикс 1 — `h-screen` в корневом div рендера

Файл: `app/app/album/[id]/layout/page.tsx`

Найди (~строка 205):

**БЫЛО:**

```tsx
  return (
    <div className="min-h-screen flex flex-col bg-gray-50">
      {/* ═══ Header ═══ */}
```

**СТАЛО:**

```tsx
  return (
    <div className="h-screen flex flex-col bg-gray-50">
      {/* ═══ Header ═══ */}
```

Изменение: `min-h-screen` → `h-screen` (удалили `min-`).

**Loading и error блоки не трогаем** — они на ~строках 159 и 167,
там просто центрированный текст без палитры, страница не
«расползается». `min-h-screen` там корректен.

### Шаг 2. Фикс 2а — глобальная `_refreshing` + `refreshAccessToken`

В том же файле `app/app/album/[id]/layout/page.tsx`. Найди существующий
блок:

```typescript
// ─── Хелпер api() — копия из app/app/page.tsx, но без refresh-loop'а ─────
// (refresh-логика тут не нужна: при истёкшем токене страница редактора
// просто покажет ошибку и партнёр откроет её заново из /app)
async function api(path: string, opts?: RequestInit): Promise<Response> {
  return fetch(path, {
    ...opts,
    credentials: 'include',
    headers: { 'Content-Type': 'application/json', ...(opts?.headers ?? {}) },
  })
}
```

Полностью замени его на:

```typescript
// ─── Refresh-aware api() — копия из app/app/page.tsx ─────────────────────
//
// При 401 пытается отрефрешить access token через /api/auth и
// повторить исходный запрос. Дедупликация одновременных refresh'ей
// через module-scope `_refreshing`.
//
// Изначально в 2.6.1 я не копировал эту логику (надеясь, что редактор
// не нужна), но визуальная проверка 2.6.2 показала: партнёр открывает
// редактор после нескольких часов простоя → 401 → пустой экран. Не
// годится для долгих editing-сессий. См. 2.6.2.1.
let _refreshing: Promise<boolean> | null = null

async function refreshAccessToken(): Promise<boolean> {
  if (_refreshing) return _refreshing
  _refreshing = fetch('/api/auth', {
    method: 'POST',
    credentials: 'include',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ action: 'refresh' }),
  }).then(r => r.ok).catch(() => false).finally(() => { _refreshing = null })
  return _refreshing
}

async function api(path: string, opts?: RequestInit): Promise<Response> {
  const res = await fetch(path, {
    ...opts,
    credentials: 'include',
    headers: { 'Content-Type': 'application/json', ...(opts?.headers ?? {}) },
  })
  if (res.status === 401) {
    const ok = await refreshAccessToken()
    if (ok) {
      return fetch(path, {
        ...opts,
        credentials: 'include',
        headers: { 'Content-Type': 'application/json', ...(opts?.headers ?? {}) },
      })
    }
  }
  return res
}
```

**Замечания:**

1. **`opts?.headers ?? {}`** — копия паттерна из 2.6.1 (этот вариант
   защищает от undefined, в `app/app/page.tsx` написано `...opts?.headers`
   что эквивалентно). Сохраняем оба варианта работоспособными.

2. **Module-scope `_refreshing`** — выше функций (паттерн как в
   `page.tsx:90`).

3. **Position в файле**: оригинальный `api()` сейчас находится сразу
   после type-определений и до `LayoutEditorPage`. Замена в этом
   же месте — лишних перестановок не нужно.

## Применение и верификация в проде

### Порядок

Миграций нет. Vercel задеплоит за ~30-60 секунд.

### Проверка визуально

После деплоя открой `https://yearbook-v2.vercel.app/app/album/3fccf1dc-3e3d-402b-8c0c-7036e3496418/layout`:

**Чек-лист:**

1. **Canvas виден сразу после загрузки** — без скролла. Должна быть
   стандартная картина: header сверху, canvas в центре левой колонки,
   палитра справа.
2. **Палитра скроллится только внутри своей колонки** — main-область
   с canvas остаётся на месте при скролле палитры
3. **Страница НЕ скроллится по вертикали целиком** (попробуй покрутить
   колесом мыши на main-области) — только палитра скроллится
4. **401 после длительного простоя** — если оставить вкладку открытой
   на час+ и потом обновить страницу, она должна загрузиться без
   красной ошибки (refresh сработает автоматически). Это проверить
   сложно одномоментно — оставим в backlog как «проверить через час».

### Что **может** не сработать (followup'ы если что)

- **Палитра имеет фиксированную высоту равную viewport'у, миниатюры
  не помещаются**: должна быть прокрутка внутри `<aside>`. Если
  её нет (overflow scrollbar не появляется) → нужен `h-full` на
  `<aside>` в `PhotoPalette.tsx`. Проверка визуально.

- **Sticky-заголовки секций «Портреты»/«Учителя» внутри палитры
  ломаются** при изменении высоты — должны работать как раньше.

- **На очень коротких альбомах (1-2 разворота)** низкая палитра
  в высоту виден большой пустой низ — не критично.

## Что НЕ делаем в 2.6.2.1

- **Унификация `api()`/`refreshAccessToken` в общий module** —
  backlog (когда будет 3-й consumer)
- **Refresh-loop с retry-logic'ом** — однократный refresh достаточен,
  если refresh упал → пользователь идёт в `/app`
- **Контекст-файл v43 → v44** — обновляется в 2.8
