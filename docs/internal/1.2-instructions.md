# Инструкция для подэтапа 1.2

**Цель:** реализовать `lib/smart-fill/build-album-input.ts` — helper,
который читает реальный альбом из БД и собирает `AlbumInput` для
`buildAlbum`. Это первый содержательный код фазы 1: тонкая обёртка
между Supabase и builder'ом.

**Зачем:** в подэтапе 1.3 endpoint `POST /api/layout?action=build_album`
будет вызывать этот helper, потом передавать результат в `buildAlbum`,
сохранять spreads + warnings в `album_layouts`. В 1.2 endpoint ещё нет —
helper тестируется через одноразовый CLI-скрипт. Это даёт раннее
обнаружение багов в маппинге БД → AlbumInput до того как UI завязан
на endpoint.

**Что меняется:**
- Новый файл `lib/smart-fill/build-album-input.ts` — основная логика
- Новый файл `lib/smart-fill/index.ts` — публичный API модуля
- Никаких правок в существующем коде

**Что НЕ трогаем:**
- `lib/album-builder/*` — типы builder'а используем как импорт
- `app/api/tenant/route.ts` — туда будет интеграция через endpoint в 1.3
- `app/app/page.tsx` — UI кнопки добавим в 1.4

---

## Принципиальные решения

### Архитектура: тонкая обёртка, без бизнес-логики

Smart-fill читает БД → переводит в `AlbumInput` → возвращает. Все
ограничения, fallback'и, edge-cases остаются в builder'е (как
зафиксировано в `docs/phase-1-spec.md` раздел «Главное решение»).

Помимо `AlbumInput` helper возвращает массив **smart-fill warnings** —
это отдельные коды от `BuildWarning` builder'а. В фазе 1 их два:
- `students_no_portrait` — N из M учеников без selection portrait_page
- `per_child_override_ignored` — N учеников имеют свой `config_preset_id`,
  игнорируется на MVP

### Сигнатура

```typescript
export async function buildAlbumInput(
  supabase: SupabaseClient,
  albumId: string,
): Promise<{
  input: AlbumInput;
  warnings: SmartFillWarning[];
}>
```

`supabase` инжектится параметром — тот же паттерн что у `loadTemplateSet`.
Это даёт:
- Лёгкое тестирование в CLI-скрипте (создаётся свой клиент)
- В endpoint'е передаётся `supabaseAdmin`
- Никаких прямых импортов из `@/lib/supabase` в этой папке (изолировано)

### Auto-resolve template_set_id

Если `albums.template_set_id IS NULL` — резолвим через
`getDefaultTemplateSetId()` (локальная функция в helper'е, копия
из `app/api/tenant/route.ts:110`). Если и `getDefaultTemplateSetId()`
вернул null — кидаем явную ошибку.

После бэкфила 08.05.2026 `template_set_id` не должен быть NULL у альбомов
в проде (см. `docs/phase-1-spec.md` строка 56). Auto-resolve остаётся
как защита.

### Параллельные SELECT'ы

Все запросы к БД идут через `Promise.all`. Это критично: 6 таблиц,
последовательно работало бы 0.5-1 секунду, параллельно — 100-200 мс.

Группа запросов (после `albums` + `template_set_id` resolve):
- `teachers` (фильтр + сортировка) — для head_teacher и subjects
- `children` (сортировка) — для students
- `photo_teachers + photos` (join) — фото учителей
- `selections + photos` (join) — портреты и friend фото учеников
- `student_texts` — quotes
- (опционально) `children.config_preset_id` — для warning'а
  per_child_override

### Маппинг fields

#### head_teacher

```sql
SELECT id, full_name, position, description, is_head_teacher
FROM teachers
WHERE album_id = X
ORDER BY created_at
```

После 1.0 у каждого альбома ровно один head (по partial unique index),
или ноль если нет учителей.

```typescript
const head = teachers.find(t => t.is_head_teacher);
const headPhoto = head ? photoByTeacher[head.id] : null;
const head_teacher: HeadTeacher | null = head ? {
  name: head.full_name ?? '',
  role: head.position ?? '',
  text: head.description ?? '',
  photo: headPhoto ? getPhotoUrl(headPhoto.storage_path) : null,
} : null;
```

Если `head_teacher === null` — builder сам пишет `no_head_teacher`
warning. Smart-fill свой не добавляет — дублирование.

#### subjects

Все остальные учителя (`!is_head_teacher`), сортировка по `created_at`:

```typescript
const subjects: Subject[] = teachers
  .filter(t => !t.is_head_teacher)
  .map(t => ({
    name: t.full_name ?? '',
    role: t.position ?? '',
    photo: photoByTeacher[t.id]
      ? getPhotoUrl(photoByTeacher[t.id].storage_path)
      : null,
  }));
```

#### students

```sql
SELECT id, full_name, class, config_preset_id
FROM children
WHERE album_id = X
ORDER BY class ASC, full_name ASC
```

**Все ученики**, без фильтра по `submitted_at` (стабильность layout'а).

```typescript
const students: Student[] = children.map(c => ({
  full_name: c.full_name ?? '',
  quote: textByChild[c.id] ?? '',
  portrait: portraitByChild[c.id]
    ? getPhotoUrl(portraitByChild[c.id].storage_path)
    : null,
  friend_photos: (friendsByChild[c.id] ?? [])
    .map(p => getPhotoUrl(p.storage_path)),
}));
```

`portraitByChild` строится из `selections WHERE selection_type =
'portrait_page'`, по одному на ребёнка (берём первый если внезапно
несколько).

`friendsByChild` строится из `selections WHERE selection_type = 'group'`,
сортировка по `created_at ASC`, **cap 10** (защита от мусора). Builder
дальше режет до `preset.config.student_section.first_spread_content.
friend_photos.max`.

#### common_photos

В MVP всегда пустой объект:

```typescript
const common_photos: CommonPhotos = {
  full_class: [],
  half: [],
  quarter: [],
  sixth: [],
  collage: [],
};
```

Причина: builder фазы 0.5 общий раздел не генерирует автоматически
(см. `docs/phase-1-spec.md` строки 213-216).

### Warnings агрегированные

```typescript
const studentsNoPortrait = students.filter(s => !s.portrait).length;
if (studentsNoPortrait > 0) {
  warnings.push({
    code: 'students_no_portrait',
    detail: `${studentsNoPortrait} из ${students.length} учеников без портрета`,
  });
}

const overrideCount = children.filter(c => c.config_preset_id).length;
if (overrideCount > 0) {
  warnings.push({
    code: 'per_child_override_ignored',
    detail: `${overrideCount} учеников имеют свой пресет — игнорируется на MVP`,
  });
}
```

### Паттерн получения URL фото

`getPhotoUrl` из `@/lib/supabase`. Это уже работающая функция, ходит
в YC. Альтернатива (городить URL руками) — нарушение правила
«URL фото только через getPhotoUrl/getThumbUrl» из контекста.

```typescript
import { getPhotoUrl } from '@/lib/supabase';
```

### Не используем `getThumbUrl`

`AlbumInput.Photo = string` — builder работает с реальными URL
оригиналов, не миниатюрами. В будущем (фаза 2 Canvas-рендер) будет
смысл подгружать миниатюры для preview, но это не сейчас.

### Возврат при пустом альбоме

Если в альбоме 0 учеников — возвращаем `students: []`. Builder в
`buildAlbum` вернёт warning `students_empty` (это уже его задача).
Smart-fill свою валидацию не делает.

## Шаги

### Шаг 1. Создать `lib/smart-fill/build-album-input.ts`

```typescript
/**
 * Smart-fill: чтение реального альбома из БД и сборка AlbumInput для buildAlbum.
 *
 * Архитектурный принцип (см. docs/phase-1-spec.md): тонкая обёртка между БД
 * и builder'ом. Никакой бизнес-логики, никаких ограничений — только маппинг.
 *
 * Возвращает:
 *  - input: AlbumInput — готовый объект для передачи в buildAlbum
 *  - warnings: SmartFillWarning[] — агрегированные предупреждения чтения БД
 *    (отдельно от BuildWarning builder'а)
 *
 * Используется в:
 *  - /tmp/test-build-album-input.ts — одноразовая проверка (не коммитим)
 *  - app/api/layout/route.ts (action=build_album) — endpoint 1.3
 */

import type { SupabaseClient } from '@supabase/supabase-js';
import { getPhotoUrl } from '@/lib/supabase';
import type {
  AlbumInput,
  HeadTeacher,
  Subject,
  Student,
  CommonPhotos,
} from '@/lib/album-builder';

export type SmartFillWarningCode =
  | 'students_no_portrait'
  | 'per_child_override_ignored';

export type SmartFillWarning = {
  code: SmartFillWarningCode;
  detail: string;
};

export type BuildAlbumInputResult = {
  input: AlbumInput;
  warnings: SmartFillWarning[];
};

/**
 * Резолв template_set_id для альбома: если NULL у альбома, ищем глобальный
 * okeybook-default. Копия паттерна из app/api/tenant/route.ts:110.
 */
async function getDefaultTemplateSetId(
  supabase: SupabaseClient,
): Promise<string | null> {
  const { data, error } = await supabase
    .from('template_sets')
    .select('id')
    .eq('slug', 'okeybook-default')
    .is('tenant_id', null)
    .single();
  if (error || !data) return null;
  return data.id;
}

export async function buildAlbumInput(
  supabase: SupabaseClient,
  albumId: string,
): Promise<BuildAlbumInputResult> {
  // 1. Загрузить альбом + проверить наличие
  const { data: album, error: albumErr } = await supabase
    .from('albums')
    .select('id, template_set_id')
    .eq('id', albumId)
    .single();

  if (albumErr || !album) {
    throw new Error(
      `album ${albumId} not found: ${albumErr?.message ?? 'no row'}`,
    );
  }

  // 2. Auto-resolve template_set_id если NULL
  let templateSetId: string | null = album.template_set_id;
  if (!templateSetId) {
    templateSetId = await getDefaultTemplateSetId(supabase);
    if (!templateSetId) {
      throw new Error(
        `album ${albumId} has no template_set_id and no default okeybook-default in DB`,
      );
    }
  }

  // 3. Параллельный SELECT всех связанных таблиц
  const [
    teachersRes,
    childrenRes,
  ] = await Promise.all([
    supabase
      .from('teachers')
      .select('id, full_name, position, description, is_head_teacher')
      .eq('album_id', albumId)
      .order('created_at'),
    supabase
      .from('children')
      .select('id, full_name, class, config_preset_id')
      .eq('album_id', albumId)
      .order('class')
      .order('full_name'),
  ]);

  if (teachersRes.error) {
    throw new Error(`teachers load failed: ${teachersRes.error.message}`);
  }
  if (childrenRes.error) {
    throw new Error(`children load failed: ${childrenRes.error.message}`);
  }

  const teachers = teachersRes.data ?? [];
  const children = childrenRes.data ?? [];
  const teacherIds = teachers.map((t: any) => t.id);
  const childIds = children.map((c: any) => c.id);

  // 4. Параллельный SELECT photo links и текстов
  const [
    teacherPhotoLinksRes,
    selectionsRes,
    textsRes,
  ] = await Promise.all([
    teacherIds.length > 0
      ? supabase
          .from('photo_teachers')
          .select('teacher_id, photos(storage_path)')
          .in('teacher_id', teacherIds)
      : Promise.resolve({ data: [], error: null }),
    childIds.length > 0
      ? supabase
          .from('selections')
          .select(
            'child_id, selection_type, created_at, photos(storage_path)',
          )
          .in('child_id', childIds)
          .order('created_at')
      : Promise.resolve({ data: [], error: null }),
    childIds.length > 0
      ? supabase
          .from('student_texts')
          .select('child_id, text')
          .in('child_id', childIds)
      : Promise.resolve({ data: [], error: null }),
  ]);

  // Ошибки — кидаем сразу, чтобы получить ясное сообщение
  if ((teacherPhotoLinksRes as any).error) {
    throw new Error(
      `photo_teachers load failed: ${(teacherPhotoLinksRes as any).error.message}`,
    );
  }
  if ((selectionsRes as any).error) {
    throw new Error(
      `selections load failed: ${(selectionsRes as any).error.message}`,
    );
  }
  if ((textsRes as any).error) {
    throw new Error(
      `student_texts load failed: ${(textsRes as any).error.message}`,
    );
  }

  // 5. Построить maps для быстрого lookup
  const photoByTeacher: Record<string, { storage_path: string }> = {};
  for (const link of (teacherPhotoLinksRes as any).data ?? []) {
    const ph = link.photos;
    if (ph?.storage_path) {
      photoByTeacher[link.teacher_id] = { storage_path: ph.storage_path };
    }
  }

  const portraitByChild: Record<string, { storage_path: string }> = {};
  const friendsByChild: Record<string, { storage_path: string }[]> = {};
  for (const sel of (selectionsRes as any).data ?? []) {
    const ph = sel.photos;
    if (!ph?.storage_path) continue;
    if (sel.selection_type === 'portrait_page' && !portraitByChild[sel.child_id]) {
      portraitByChild[sel.child_id] = { storage_path: ph.storage_path };
    } else if (sel.selection_type === 'group') {
      if (!friendsByChild[sel.child_id]) friendsByChild[sel.child_id] = [];
      friendsByChild[sel.child_id].push({ storage_path: ph.storage_path });
    }
  }

  // Cap 10 защита от мусора (builder режет дальше по preset.max)
  for (const k of Object.keys(friendsByChild)) {
    if (friendsByChild[k].length > 10) {
      friendsByChild[k] = friendsByChild[k].slice(0, 10);
    }
  }

  const textByChild: Record<string, string> = {};
  for (const t of (textsRes as any).data ?? []) {
    textByChild[t.child_id] = t.text ?? '';
  }

  // 6. Собрать AlbumInput
  const headTeacherRow = teachers.find((t: any) => t.is_head_teacher);
  const head_teacher: HeadTeacher | null = headTeacherRow
    ? {
        name: headTeacherRow.full_name ?? '',
        role: headTeacherRow.position ?? '',
        text: headTeacherRow.description ?? '',
        photo: photoByTeacher[headTeacherRow.id]
          ? getPhotoUrl(photoByTeacher[headTeacherRow.id].storage_path)
          : null,
      }
    : null;

  const subjects: Subject[] = teachers
    .filter((t: any) => !t.is_head_teacher)
    .map((t: any) => ({
      name: t.full_name ?? '',
      role: t.position ?? '',
      photo: photoByTeacher[t.id]
        ? getPhotoUrl(photoByTeacher[t.id].storage_path)
        : null,
    }));

  const students: Student[] = children.map((c: any) => ({
    full_name: c.full_name ?? '',
    quote: textByChild[c.id] ?? '',
    portrait: portraitByChild[c.id]
      ? getPhotoUrl(portraitByChild[c.id].storage_path)
      : null,
    friend_photos: (friendsByChild[c.id] ?? []).map((p) =>
      getPhotoUrl(p.storage_path),
    ),
  }));

  const common_photos: CommonPhotos = {
    full_class: [],
    half: [],
    quarter: [],
    sixth: [],
    collage: [],
  };

  const input: AlbumInput = {
    template_set_id: templateSetId,
    head_teacher,
    subjects,
    students,
    common_photos,
  };

  // 7. Smart-fill warnings
  const warnings: SmartFillWarning[] = [];

  const studentsNoPortrait = students.filter((s) => !s.portrait).length;
  if (studentsNoPortrait > 0) {
    warnings.push({
      code: 'students_no_portrait',
      detail: `${studentsNoPortrait} из ${students.length} учеников без портрета`,
    });
  }

  const overrideCount = children.filter((c: any) => c.config_preset_id).length;
  if (overrideCount > 0) {
    warnings.push({
      code: 'per_child_override_ignored',
      detail: `${overrideCount} учеников имеют свой пресет — игнорируется на MVP`,
    });
  }

  return { input, warnings };
}
```

```typescript
/**
```

```typescript
import { buildAlbumInput } from '../yearbook-v2/lib/smart-fill';

const albumId = process.argv[2];
if (!albumId) {

const supabase = createClient(

  process.env.SUPABASE_SERVICE_ROLE_KEY!,

  for (const s of input.students.slice(0, 5)) {
    console.log(

  console.log('━━━ Warnings ━━━');
  if (warnings.length === 0) {
    console.log('(none)');
  } else {
    for (const w of warnings) {
      console.log(`[${w.code}] ${w.detail}`);

main().catch((e) => {

  process.exit(1);
```

### Шаг 5. Тест на реальной БД

После успешного build — Сергей выберет реальный album_id из dev-БД
для проверки helper'а.

Прежде чем тестировать на dev-БД, нужно знать какой альбом взять.
В Supabase SQL editor:

```sql
SELECT id, title, city, year,
  (SELECT COUNT(*) FROM children WHERE album_id = albums.id) AS students,
  (SELECT COUNT(*) FROM teachers WHERE album_id = albums.id) AS teachers
FROM albums
ORDER BY created_at DESC
LIMIT 10;
```

Сергей выбирает альбом с >0 учениками и >0 учителями (например, тот же
тестовый «Красночетайская СОШ»). Копирует `id` и запускает:

```bash

set -a && . ./.env.local && set +a && \
  npx tsx /tmp/test-build-album-input.ts <album_id>
```

## Открытые риски

### Производительность

Helper делает 2 + 3 = 5 SELECT'ов параллельно. На альбоме с 30 учениками
+ 10 учителями ожидаемое время: 100-300 мс. Если медленнее — смотреть
в логах Supabase, возможно нужны индексы (но они уже есть на FK).

### Большие альбомы (>500 учеников)

Не должно быть в проде, но если попадётся — лимиты Supabase Postgres
(payload до ~1 МБ на response). На 1000 учеников + 4 фото каждый =
4000 selections × 200 байт ≈ 800 КБ — приемлемо. На 5000 — упадёт.
Это будущая проблема, не для MVP.

### Storage path с pre-yc legacy

Все пути в `photos.storage_path` имеют префикс `yc:` после миграции
02.05.2026 (см. `lib/storage.ts:65`). `getPhotoUrl` обрабатывает обе
формы. Если попадётся pre-yc путь — сработает, но пора чистить.
