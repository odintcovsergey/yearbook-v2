# Инструкция для подэтапа 2.3

**Цель:** показать партнёру горизонтальную полосу визуальных миниатюр всех
разворотов автособранного layout'а в Обзоре альбома (`AlbumDetailModal`).
Каждая миниатюра рендерится через `AlbumSpreadCanvas` (созданный в 2.2)
с реальным контентом — фото, имена, тексты. Это первый момент, когда
партнёр **видит** результат автосборки, а не только текст «Layout
собран · 11 разворотов».

**Зачем:** до 2.3 партнёр в Обзоре видит только текстовый result-блок
(статус, количество разворотов, warning'и, кнопку «Скопировать JSON»).
После 2.3 — миниатюры разворотов с реально расставленными фото.
Это закрывает базовый use-case: «убедиться, что автосборка дала
вменяемый результат, не открывая редактор».

**Контекст:** `AlbumSpreadCanvas` (2.2) принимает `instance: SpreadInstance`
+ `template: SpreadTemplate`. У layout'а есть `spreads` (массив instance'ов
с `template_id` каждого), но нет самих `template`'ов с placeholder'ами —
их нужно загрузить отдельно через `GET /api/layout?action=template_set_detail`.
Соответственно, чтобы `LayoutPreviewStrip` мог запросить detail, у layout'а
должен быть известен `template_set_id`. Сейчас (после 1.4) ни
`GET ?action=album_layout`, ни `POST ?action=build_album` не возвращают
это поле — нужно расширить оба endpoint'а.

**Что меняется:**

1. `app/api/layout/route.ts` — расширение response двух endpoint'ов
   (`handleGetAlbumLayout` и `handleBuildAlbum`): добавляется поле
   `template_set_id` на top-level layout.
2. `app/app/page.tsx` — три точки:
   - Тип `SmartFillLayout` расширяется на `template_set_id: string`
   - `runSmartFill`: при успехе кладёт `template_set_id: data.template_set_id`
     в `setLayout({...})`
   - В JSX Обзора после result-блока (после строки ~1187) добавляется
     `{layout && <LayoutPreviewStrip layout={layout} onOpenEditor={...} />}`
   - В верхнем dynamic-блоке файла появляется
     `const LayoutPreviewStrip = dynamic(() => import('./_components/LayoutPreviewStrip'), { ssr: false, loading: () => null })`
3. `app/app/_components/LayoutPreviewStrip.tsx` — новый компонент:
   - Загружает `template_set_detail` лениво через useEffect
   - Рендерит горизонтальную полосу миниатюр через `AlbumSpreadCanvas`
   - Кнопка «Открыть редактор» (disabled до 2.6, с tooltip)
4. `docs/internal/2.3-instructions.md` — этот файл

**Что НЕ трогаем:**

- `lib/album-builder/` — чисто frontend задача
- Миграции БД — нет. Поле `template_set_id` уже есть в `album_layouts`
  (миграция 0.1, NOT NULL). Просто добавляем его в SELECT/JSON.
- `app/app/_components/AlbumSpreadCanvas.tsx` — компонент 2.2 без правок
- Страница `/app/album/[id]/layout` — это 2.6
- Авторизацию `template_set_detail` — оно уже доступно
  owner/manager/viewer (см. `app/api/layout/route.ts:121`)

## Принципиальные решения

### `template_set_id` на top-level layout (а не внутри `summary`)

Текущий `summary` содержит метрики: `total_spreads`, `total_warnings`,
`warnings_by_level`, `preset_slug`, `preset_name`. Это «свёрнутая статистика»
для UI блока. `template_set_id` — не статистика, а ссылочный идентификатор
для рендера. Логически он принадлежит layout'у, не summary. Поэтому кладём
на верхний уровень рядом с `layout_id`:

```typescript
type SmartFillLayout = {
  layout_id: string
  template_set_id: string  // ← новое в 2.3
  spreads: unknown[]
  warnings: EnrichedWarning[]
  summary: SmartFillSummary
}
```

### Ленивая загрузка `template_set_detail` внутри `LayoutPreviewStrip`

Спека §4.4 предполагает что `template` приходит в props компонента.
Я отступаю от этого: грузим detail внутри компонента через useEffect.

**Аргументы за ленивую загрузку:**
- Call site в `/app/page.tsx` остаётся минимальным:
  `<LayoutPreviewStrip layout={layout} onOpenEditor={...} />` — без
  лишнего state в `AlbumDetailModal`.
- Логика загрузки в одном месте (компонент сам отвечает за свои данные).
- При появлении layout'а (после `runSmartFill`) detail подтянется
  автоматически — не надо дублировать вызов в callback'е.

**Аргументы против (и почему игнорируем):**
- Extra-запрос на каждый mount LayoutPreviewStrip. Размер JSON — ~30KB,
  кешируется браузером. Незначимо.
- В 2.6 редактор тоже будет грузить detail (свой). Дублирование?
  Нет — у компонентов разные жизненные циклы (preview-strip mount при
  открытии модала, редактор — отдельная страница). Кеш браузера
  справится.

### Кнопка «Открыть редактор» — disabled в 2.3

Спека §3.2 предписывает существование страницы `/app/album/[id]/layout`,
но создаётся она в подэтапе 2.6. Если в 2.3 кнопка ведёт по этому URL —
партнёр в проде получит 404. Это плохой UX и плохой сигнал нашей
надёжности.

Решение: кнопка **видимая, но disabled** с tooltip «Редактор скоро
будет доступен». В подэтапе 2.6 убираем `disabled` атрибут и связываем
с роутером.

```tsx
<button
  type="button"
  disabled
  className="btn-secondary text-xs disabled:opacity-50 disabled:cursor-not-allowed"
  title="Редактор скоро будет доступен (подэтап 2.6)"
>
  Открыть редактор
</button>
```

### Размер миниатюр — фиксированная высота, не ширина

В шаблоне `okeybook-default` есть мастера двух типов:
- `is_spread=false` — одностраничный (226×226 мм, квадрат)
- `is_spread=true` — двухстраничный разворот (452×226 мм, ландшафт 2:1)

Если фиксировать ширину (как в спеке §4.4 — 250px), одностраничные
миниатюры будут квадратами 250×250, а двустраничные — лежачими 250×125.
Это визуальный мусор: в одной полосе элементы прыгают по высоте.

Решение: фиксируем **высоту** = 175px. Ширина считается из соотношения
сторон шаблона:

```typescript
const targetHeight = 175
const containerWidth = (tmpl.width_mm / tmpl.height_mm) * targetHeight
```

Тогда:
- Одностраничный 226×226 → 175×175 (квадрат)
- Двустраничный 452×226 → 350×175 (ландшафт)

Все миниатюры выровнены по высоте, ширина варьируется естественно.

`AlbumSpreadCanvas` (2.2) принимает `containerWidth` — этого достаточно
(scale Stage'а вычисляется из неё, высота = `template.height_mm * scale`
автоматически). Не трогаем компонент 2.2.

### Динамический импорт `AlbumSpreadCanvas` через `next/dynamic`

Konva использует `window.Image` (см. `useImage` хелпер в 2.2). На SSR
у Next.js нет `window` → бэкап рухнет на этапе серверного рендера.
Стандартный паттерн из проекта — `dynamic(() => import(...), { ssr: false })`
(см. `app/super/templates/[id]/page.tsx:48`).

Применяем дважды:
1. Внутри `LayoutPreviewStrip.tsx` — для импорта `AlbumSpreadCanvas`.
2. В `app/app/page.tsx` — для импорта `LayoutPreviewStrip` (он внутри
   уже использует Konva-зависимый компонент).

Двойная защита — пусть будет. Один level dynamic'а — гарантия для
конкретного call site'а.

### Loading state — без скелетонов, минимальный текст

Пока detail грузится (~0.3-1с при холодном кеше), показываем
«Загружаем шаблон…». Без скелетонов миниатюр — лишняя сложность для
30KB запроса. Если пойдут жалобы на «мигает» — добавим.

### Error state — текстовое сообщение, не падение всего модала

Если `template_set_detail` не загрузился (network error, 404),
LayoutPreviewStrip показывает: «Не удалось загрузить шаблон. Попробуйте
открыть модал заново.». Альбомные страницы и result-блок продолжают
работать — компонент изолирован.

### Ключ навигации миниатюры — `spread_index`, не `template_id`

В `instance.spread_index` — позиция разворота в layout'е (1, 2, 3 ...
для облавы). У некоторых ученических разворотов один и тот же
`template_id` (например все Стандарт-ученики используют
`E-Student-Standard`). Если использовать `template_id` как key React,
будет коллизия. Используем `spread_index`.

Дублёр для подписи под миниатюрой — тоже `spread_index` (а не
`template.name`, который слишком длинный — «E-Student-Standard»).

### Шаг 2. Создать компонент `LayoutPreviewStrip`

Файл: `app/app/_components/LayoutPreviewStrip.tsx`

```tsx
'use client'

import { useEffect, useState, useMemo } from 'react'
import dynamic from 'next/dynamic'
import type { SpreadInstance, SpreadTemplate } from '@/lib/album-builder/types'

// Dynamic import: AlbumSpreadCanvas использует window.Image (Konva), SSR-incompatible.
const AlbumSpreadCanvas = dynamic(
  () => import('./AlbumSpreadCanvas'),
  { ssr: false, loading: () => null },
)

// ─── Минимальный publicный contract layout'а (не SmartFillLayout) ────────
// Принимаем layout с конкретными полями которые нужны компоненту, без
// необходимости тащить SmartFillLayout (он определён в page.tsx).
type LayoutShape = {
  template_set_id: string
  spreads: unknown[]  // narrowing к SpreadInstance[] делаем внутри
}

type Props = {
  layout: LayoutShape
  onOpenEditor: () => void
}

type TemplateDetailResponse = {
  template_set: { id: string; spread_width_mm: number; spread_height_mm: number }
  spread_templates: SpreadTemplate[]
}

const TARGET_HEIGHT_PX = 175  // см. инструкцию 2.3 «Размер миниатюр»

// ─── Хелпер: API-fetch (тот же стиль что в page.tsx — credentials: include) ──
async function fetchTemplateDetail(templateSetId: string): Promise<TemplateDetailResponse> {
  const r = await fetch(
    `/api/layout?action=template_set_detail&id=${templateSetId}`,
    { credentials: 'include', headers: { 'Content-Type': 'application/json' } },
  )
  if (!r.ok) {
    throw new Error(`template_set_detail failed: ${r.status}`)
  }
  return r.json()
}

export default function LayoutPreviewStrip({ layout, onOpenEditor }: Props) {
  const [detail, setDetail] = useState<TemplateDetailResponse | null>(null)
  const [error, setError] = useState<string | null>(null)
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    let cancelled = false
    setLoading(true)
    setError(null)
    fetchTemplateDetail(layout.template_set_id)
      .then((d) => {
        if (!cancelled) setDetail(d)
      })
      .catch(() => {
        if (!cancelled) setError('Не удалось загрузить шаблон')
      })
      .finally(() => {
        if (!cancelled) setLoading(false)
      })
    return () => {
      cancelled = true
    }
  }, [layout.template_set_id])

  // Map id → SpreadTemplate для O(1) lookup'а внутри map'а миниатюр.
  const templateById = useMemo(() => {
    const m = new Map<string, SpreadTemplate>()
    if (!detail) return m
    for (const t of detail.spread_templates) m.set(t.id, t)
    return m
  }, [detail])

  const spreads = layout.spreads as SpreadInstance[]

  return (
    <div className="bg-gray-50 rounded-lg p-3 mb-4">
      <div className="flex items-center justify-between mb-2 flex-wrap gap-2">
        <div className="text-xs text-gray-500 uppercase">
          Превью разворотов ({spreads.length})
        </div>
        <button
          type="button"
          onClick={onOpenEditor}
          disabled
          className="text-xs px-3 py-1.5 rounded border border-gray-300 bg-white text-gray-500 disabled:opacity-50 disabled:cursor-not-allowed"
          title="Редактор скоро будет доступен (подэтап 2.6)"
        >
          Открыть редактор
        </button>
      </div>

      {loading && (
        <div className="text-xs text-gray-400 py-4">Загружаем шаблон…</div>
      )}

      {error && (
        <div className="text-xs text-red-600 py-4">{error}</div>
      )}

      {!loading && !error && detail && (
        <div className="flex gap-2 overflow-x-auto pb-2">
          {spreads.map((s) => {
            const tmpl = templateById.get(s.template_id)
            if (!tmpl) {
              return (
                <div
                  key={s.spread_index}
                  className="flex-shrink-0 w-[175px] h-[175px] border border-red-200 bg-red-50 flex items-center justify-center text-xs text-red-500"
                >
                  Шаблон не найден
                </div>
              )
            }
            const containerWidth = (tmpl.width_mm / tmpl.height_mm) * TARGET_HEIGHT_PX
            return (
              <div key={s.spread_index} className="flex-shrink-0">
                <div className="bg-white rounded shadow-sm border border-gray-200">
                  <AlbumSpreadCanvas
                    instance={s}
                    template={tmpl}
                    containerWidth={containerWidth}
                    mode="preview"
                  />
                </div>
                <div className="text-[10px] text-center text-gray-500 mt-1">
                  {s.spread_index}
                </div>
              </div>
            )
          })}
        </div>
      )}
    </div>
  )
}
```

**Замечания по реализации:**

1. **`LayoutShape` вместо `SmartFillLayout`** — `SmartFillLayout` определён
   локально в `page.tsx`. Чтобы не выносить его в общий module и не плодить
   циклические зависимости, в компоненте принимаем минимальный shape с
   нужными полями. TypeScript сатисфирует структурно — `SmartFillLayout`
   будет совместим с `LayoutShape`.

2. **`spreads as SpreadInstance[]`** — на стороне `page.tsx` тип `spreads:
   unknown[]` (см. строку 704). Cast — необходимое зло до тех пор пока
   не унифицируем тип SpreadInstance между API и UI (это можно
   сделать позже отдельным рефакторингом).

3. **`onOpenEditor` принимается, но не вызывается** — кнопка disabled.
   Параметр оставлен в API компонента чтобы в 2.6 включить кнопку
   одной правкой (disabled→handler).

4. **`flex-shrink-0` на каждой миниатюре** — без него flex-родитель
   ужмёт миниатюры под доступную ширину, и они исказятся. Skipping
   `overflow-x-auto` тогда не сработает.

5. **«Шаблон не найден»** — оборона на случай если `template_id` в
   spread не нашёлся в template_set_detail. Не должно случаться, но
   защита от рассинхрона данных в БД.

## Применение и верификация в проде

### Порядок

Миграций нет. После push Vercel задеплоит за ~30-60 секунд. Никаких
ручных действий в Supabase.

### Проверка визуально

После деплоя:

1. Открыть `/app` → один из smoke-альбомов (Красночетайская СОШ или
   Школа 89) → AlbumDetailModal → вкладка **Обзор**.
2. Прокрутить вниз до result-блока «Layout собран · 11 разворотов».
3. **Сразу под** result-блоком должна появиться полоса миниатюр:
   - Заголовок: «Превью разворотов (11)»
   - Кнопка справа: «Открыть редактор» (серая, disabled, при hover —
     tooltip «Редактор скоро будет доступен (подэтап 2.6)»)
   - Горизонтальный скролл с 11 миниатюрами разной ширины
4. На миниатюрах должны быть видны:
   - Реальные фото портретов учеников (из YC)
   - Имена учеников (текст, шрифт Arial)
   - Учительский разворот (фото + имена в круглых рамках, если is_circle)
   - Тонкие dashed-рамки в местах где фото нет (например пустые слоты
     `studentPhoto3..4` если у ученика только 2 фото с друзьями)

### Что должно работать

- Нажатие «Пересобрать» → миниатюры обновляются (после 1-2 сек)
- Скролл миниатюр горизонтальный, мышью или тачпадом
- При открытии модала с уже сохранённым layout (из БД) миниатюры
  подгружаются после короткого «Загружаем шаблон…»

### Что **может** не работать (если всплывут — фиксим в 2.3-followup)

- **CORS на storage.yandexcloud.net**: `<KonvaImage>` с
  `crossOrigin='anonymous'` (см. 2.2 useImage). Бакет `yearbook-photos`
  публичный, CORS должен быть открыт. Если миниатюры рисуют только
  текст и пустые рамки (но фото отсутствуют) — проверить настройки
  CORS бакета в YC console.
- **Производительность**: 11 Stage'ей с реальными фото. На современных
  машинах — нормально. Если тормозит — добавить `pixelRatio={1}` в
  `AlbumSpreadCanvas` (но это правка 2.2, отдельным коммитом).
- **Шрифты**: текст рисуется Arial, не Inter/Open Sans оригиналов.
  Это ожидаемо (фаза 3).
- **Aspect ratio при разном `is_spread`**: одностраничные миниатюры
  175×175, двустраничные 350×175. В одной полосе высоты выровнены —
  должно смотреться.

## Замечания (если что-то всплывёт)

### `dynamic(...)` для `LayoutPreviewStrip` в `page.tsx`

В `app/app/page.tsx` сейчас, по grep'у, dynamic'ов нет вообще. Это
значит:
- `import dynamic from 'next/dynamic'` нужно добавить в импорты
- Декларация `const LayoutPreviewStrip = dynamic(...)` — в module-scope
  (вне `AlbumDetailModal`)

Если `next/dynamic` вдруг там УЖЕ импортирован (например, появился
в каком-то промежуточном коммите — проверь grep'ом), второй раз
не импортируй.

### Подзаголовок миниатюры

В спеке §4.4 — `{i + 1}`, я пишу `{s.spread_index}`. Это одно и то же
если builder заполняет `spread_index` начиная с 1 последовательно.
Проверить можно в `lib/album-builder/build-from-preset.ts` —
`spreadCounter.value` начинается с 1 и инкрементится. То есть
`spread_index = i + 1`. Используем `s.spread_index` для консистентности
с серверной нумерацией (видимой в JSON, audit log и т.д.).

### Тестирование с альбомом без layout

Пока партнёр не нажал «Собрать автоматически» — `layout` в state равен
null. `{layout && <LayoutPreviewStrip ...>}` рендерится только при
наличии layout — это уже учтено в JSX. Дополнительная проверка не
нужна.
