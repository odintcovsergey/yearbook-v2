# Инструкция для подэтапа 2.6.4

**Цель:** превратить редактор из «локального песочника» в полнофункциональный
инструмент с persistent изменениями. Четыре связанных фичи в одном
коммите:
1. **Auto-save** через POST `/api/layout?action=save_album_layout`
   (endpoint 2.5) с debounce 2 секунды
2. **Swap** между photo-placeholder'ами (drag из placeholder'а в
   placeholder')
3. **SaveIndicator** в header'е — показывает статус «Сохраняется…» /
   «Сохранено» / «Ошибка»
4. **beforeunload** предупреждение если есть несохранённые правки

**Зачем:** до 2.6.4 партнёр терял все правки при F5. Это блокирует
любой реальный workflow — невозможно редактировать ~30-минутный layout
если он стирается на каждом случайном reload'е. После 2.6.4 редактор
становится production-ready: drag → 2 сек → автосохранение, индикатор
видимый, защита от случайной потери при закрытии вкладки.

**Контекст:** все участники готовы:
- Endpoint `save_album_layout` (2.5) — принимает `{ album_id, spreads }`,
  возвращает `{ success, layout_id }`, ставит `has_user_edits=true`
- Drag из палитры (2.6.3) — мутирует layout state, готов триггерить save
- `_refreshing` + `api()` хелпер (2.6.2.1) — refresh-aware, не упадёт
  на 401 при долгой сессии
- Swap-логика — props `onSwap?` зарезервирован в `AlbumSpreadCanvas` с 2.2

**Что меняется:**
- `app/app/_components/AlbumSpreadCanvas.tsx` — `DropZone` теперь не
  только droppable, но и **draggable** для swap'а: photo-placeholder
  с непустым `data[label]` можно тащить в другой placeholder
- `app/app/album/[id]/layout/page.tsx` — четыре блока:
  1. State `saveStatus`, `lastSavedSpreads` (для diff'а)
  2. useEffect с debounce 2s который триггерит save при изменении
     `layout.spreads`
  3. `handleDragEnd` расширяется на ветку `placeholder → placeholder`
     (swap)
  4. Header с `SaveIndicator`, beforeunload listener
- Новый компонент `app/app/_components/SaveIndicator.tsx` (~30 строк)
- `docs/internal/2.6.4-instructions.md` — этот файл

**Что НЕ трогаем:**
- Endpoint `save_album_layout` — без изменений (он уже умеет всё что
  нужно)
- `PhotoPalette` — без изменений
- Миграции БД — нет
- Кнопка «Открыть редактор» в Обзоре — это 2.6.5

## Принципиальные решения

### Debounce 2 секунды

Спека phase-2-spec §4.5 предписывает 2 секунды. Это компромисс:
- **Меньше** (например 0.5s) — каждый drag триггерит save → если партнёр
  быстро делает 5 drag'ов подряд, получим 5 запросов вместо 1. Лишняя
  нагрузка на БД и flicker SaveIndicator'а.
- **Больше** (например 5s) — партнёр может закрыть вкладку до того
  как auto-save сработает, потерять правку.
- 2 секунды — баланс: после последнего drag'а партнёр видит save
  через 2с, BeforeUnload подстрахует если закрыли раньше.

### Реализация debounce — через useEffect cleanup

Не используем lodash debounce или сторонние хуки. Стандартный паттерн:

```typescript
useEffect(() => {
  if (!layout || !hasChanges) return
  const timer = setTimeout(() => {
    saveLayout(layout)
  }, 2000)
  return () => clearTimeout(timer)
}, [layout, hasChanges])
```

При каждом изменении `layout` старый таймер отменяется (cleanup),
новый ставится на 2 секунды. Если за 2 секунды происходит ещё одно
изменение — снова cleanup и перезапуск. После 2 секунд тишины — save.

### Determining `hasChanges` — diff с `lastSavedSpreads`

Простое сравнение: если `JSON.stringify(layout.spreads) !==
JSON.stringify(lastSavedSpreads)` → есть несохранённые изменения.
JSON-сериализация для diff'а — медленно для больших layouts (~5-10ms
для 11 spreads × 30 placeholders × 100 chars), но в нашем масштабе
не критично, и проще чем deep-equal.

Алтернатива: `JSON.stringify` каждого spread'а отдельно с шорт-сирку.
Делаем простой подход.

`lastSavedSpreads` — отдельный useState, обновляется после успешного
save. При первой загрузке layout'а — равен исходным spread'ам (т.е.
сразу `hasChanges = false`).

### saveStatus state machine

Четыре состояния:
- `'saved'` — изменений нет, всё сохранено (default)
- `'pending'` — есть изменения, ждём debounce
- `'saving'` — POST в полёте
- `'error'` — последний POST упал

Переходы:
```
saved → pending (изменение)
pending → saved (cancellation: например ничего не изменилось после ещё drag'ов)
pending → saving (debounce истёк, отправляем POST)
saving → saved (200 OK)
saving → error (500 / network)
error → pending (новое изменение после ошибки)
```

### Save при `pending → saving` — без race conditions

Если партнёр продолжает драгать пока POST в полёте — следующий drag
ставит `pending`, debounce запускается заново. После завершения
текущего save'а сразу запускается следующий. Это корректное поведение,
last-write-wins.

Защита от out-of-order responses (если save N завершится раньше save
N+1): отслеживаем счётчик `saveCounter`, при ответе сравниваем с
текущим — если устарел, игнорируем.

```typescript
const saveCounterRef = useRef(0)

async function saveLayout(spreads: SpreadInstance[]) {
  const myCounter = ++saveCounterRef.current
  setSaveStatus('saving')
  try {
    const res = await api('/api/layout?action=save_album_layout', {
      method: 'POST',
      body: JSON.stringify({ album_id: albumId, spreads }),
    })
    if (myCounter !== saveCounterRef.current) return  // устарел, игнорируем
    if (res.ok) {
      setLastSavedSpreads(spreads)
      setSaveStatus('saved')
    } else {
      setSaveStatus('error')
    }
  } catch {
    if (myCounter !== saveCounterRef.current) return
    setSaveStatus('error')
  }
}
```

### Swap — drag из placeholder'а в placeholder

Логика в `AlbumSpreadCanvas`:
- Каждая `DropZone` теперь и **draggable** (через `useDraggable`),
  и **droppable** (как раньше)
- Только если `data[label]` непустой (есть фото) — иначе тащить нечего

Драгабельность вкл/выкл через `disabled` пропс useDraggable. Если
photo-слот пуст — драг disabled. Если есть фото — можно тащить.

В `handleDragEnd` editor'а — новая ветка:

```typescript
if (sourceData?.type === 'placeholder') {
  const sourceLabel = sourceData.label as string
  const targetLabel = String(over.id)
  if (sourceLabel === targetLabel) return  // drop в себя — ничего не делаем

  setLayout((prev) => {
    if (!prev) return prev
    const newSpreads = prev.spreads.map((s, idx) => {
      if (idx !== currentIdx) return s
      const valueA = s.data[sourceLabel]
      const valueB = s.data[targetLabel]
      return { ...s, data: { ...s.data, [sourceLabel]: valueB, [targetLabel]: valueA } }
    })
    return { ...prev, spreads: newSpreads }
  })
}
```

Swap **двусторонний** — value меняются местами. Если target пустой
(URL = null), source становится пустым, target получает source URL.
Это equivalent «переместить из X в Y, очистить X».

### Показывать DragOverlay для swap'а

Ровно та же логика что и для палитры: при `onDragStart` определяем
тип source. Если `placeholder` — рендерим в DragOverlay миниатюру
из палитры по URL. Найти photo по URL — обходом массива photos:

```typescript
function findPhotoByUrl(url: string, photos: AlbumPhoto[]): AlbumPhoto | null {
  return photos.find(p => p.url === url) ?? null
}
```

В onDragStart:
```typescript
if (sourceData?.type === 'placeholder') {
  const url = sourceData.url
  const photo = url ? findPhotoByUrl(url, photos) : null
  if (photo) setActiveDrag(photo)
}
```

Если photo не найдено в палитре (битый URL) — overlay не показывается,
но drag всё равно работает.

### SaveIndicator визуально

Минималистичный pill в правой части header'а:

```tsx
{saveStatus === 'saving' && <span className="text-xs text-gray-500">Сохраняется…</span>}
{saveStatus === 'saved' && <span className="text-xs text-green-600">✓ Сохранено</span>}
{saveStatus === 'pending' && <span className="text-xs text-amber-600">● Не сохранено</span>}
{saveStatus === 'error' && <span className="text-xs text-red-600">⚠ Ошибка сохранения</span>}
```

Без spinner-icon'а — текст достаточен и не отвлекает. Если в будущем
жалобы на «не понятно что происходит» — добавим иконки.

### beforeunload — только при `pending`/`saving`/`error`

```typescript
useEffect(() => {
  const handler = (e: BeforeUnloadEvent) => {
    if (saveStatus !== 'saved') {
      e.preventDefault()
      e.returnValue = ''
    }
  }
  window.addEventListener('beforeunload', handler)
  return () => window.removeEventListener('beforeunload', handler)
}, [saveStatus])
```

Браузер сам показывает свой диалог «Изменения могут не сохраниться».
Кастомизировать текст невозможно (browsers ограничили API из-за
phishing-атак). Это норма.

При `saved` listener не срабатывает — навигация без подтверждения.

### Disabled drag для пустых photo-слотов

Если `data[label]` пуст (нет URL'а) — нет смысла позволять drag.
Реализуем через `disabled: !hasValue` в useDraggable:

```typescript
const hasValue = !!instance.data[placeholder.label]
const { attributes, listeners, setNodeRef: setDragRef, isDragging } = useDraggable({
  id: `placeholder-${placeholder.label}`,
  data: { type: 'placeholder', label: placeholder.label, url: instance.data[placeholder.label] },
  disabled: !hasValue,
})
```

**Важный момент:** id должен быть уникальным между палитрой и
placeholder'ами. Использую префикс `placeholder-${label}` для
draggable, а droppable id остаётся просто `label`. Так dnd-kit
не путает.

При drop'е `over.id` — это **droppable id** = `label` без префикса.
Мы это обрабатываем правильно.

### Курсор для draggable photo-placeholder

Если placeholder имеет фото — курсор `cursor-move` (не `cursor-grab`,
чтобы отличаться от палитры — там палитра «бери», тут placeholder
«перемести»). Если пусто — `cursor-default`.

### TypeScript: расширить тип source data

В `handleDragEnd` теперь два типа source: `'palette'` и `'placeholder'`.
Уточняем discriminated union:

```typescript
type DragSourceData =
  | { type: 'palette'; photo: AlbumPhoto }
  | { type: 'placeholder'; label: string; url: string | null }
```

Cast в `event.active.data.current as DragSourceData | undefined`.

### `instance.data[label]` — может быть `null`?

Смотрю на тип `SpreadInstance`. По factу да, но фактически builder
не пишет null'ы — кладёт URL или просто не вставляет ключ. Защита
через `!!instance.data[placeholder.label]` обрабатывает both cases
(null, undefined, '').

#### 2а. Расширить импорт

#### 2б. Заменить компонент `DropZone` целиком

Найти существующий `DropZone` функциональный компонент. Заменить
**целиком**:

```tsx
// ─── Drop-target над Konva-placeholder'ом + draggable если есть фото ─────
//
// 2.6.3 — только droppable. 2.6.4 — добавлен draggable для swap'а
// между placeholder'ами. Если photo-слот пуст (нет url), drag
// disabled.
function DropZone({
  placeholder,
  scale,
  url,
}: {
  placeholder: PhotoPlaceholder
  scale: number
  url: string | null
}) {
  const hasValue = !!url
  const { setNodeRef: setDropRef, isOver } = useDroppable({
    id: placeholder.label,
  })
  const {
    attributes,
    listeners,
    setNodeRef: setDragRef,
    isDragging,
  } = useDraggable({
    id: `placeholder-${placeholder.label}`,
    data: { type: 'placeholder', label: placeholder.label, url },
    disabled: !hasValue,
  })

  // Объединяем оба ref'а: один div одновременно droppable И draggable
  const setRef = (node: HTMLElement | null) => {
    setDropRef(node)
    setDragRef(node)
  }

  return (
    <div
      ref={setRef}
      {...(hasValue ? { ...attributes, ...listeners } : {})}
      className={`absolute pointer-events-auto transition-all ${
        isOver
          ? 'ring-2 ring-blue-500 bg-blue-100/40'
          : 'ring-1 ring-transparent hover:ring-blue-300/50'
      } ${
        hasValue ? 'cursor-move' : 'cursor-default'
      } ${
        isDragging ? 'opacity-40' : ''
      }`}
      style={{
        left: `${placeholder.x_mm * scale}px`,
        top: `${placeholder.y_mm * scale}px`,
        width: `${placeholder.width_mm * scale}px`,
        height: `${placeholder.height_mm * scale}px`,
      }}
    />
  )
}
```

Изменения:
- Добавлен пропс `url`
- `useDraggable` с `disabled: !hasValue`, id=`placeholder-${label}`
- Объединённый ref через `setRef` функцию
- `attributes`+`listeners` распыляются только если `hasValue`
- `cursor-move` для непустых, `cursor-default` для пустых
- `isDragging` — opacity-40 как у палитры

#### 3г. useEffect — debounce save

После функции `saveLayout`:

```typescript
  // Debounce auto-save: при изменении layout.spreads ждём 2с тишины,
  // потом отправляем POST. При новом изменении старый таймер отменяется.
  useEffect(() => {

    const isUnchanged =

    if (isUnchanged) {

      saveLayout(layout.spreads)
```

```typescript
    }
```
```typescript
  }
```

## Применение и верификация в проде

### Порядок

Миграций нет. Vercel задеплоит за ~30-60 секунд.

### Проверка визуально — на тестовом альбоме «тест»

**Важно:** проверки делаем на новом тестовом альбоме (не на Красночетайке
и не на Школе 89 — это реальные заказы). Открой UUID этого альбома
из URL когда зайдёшь в кабинет → Обзор → ссылка «открыть редактор»
пока заблокирована, поэтому открой `/app/album/<его UUID>/layout`
вручную в адресной строке.

**Чек-лист функциональности:**

1. **Header показывает «✓ Сохранено»** при загрузке (нет несохранённых
   изменений)
2. **Drag фото из палитры → drop в placeholder** — фото вставляется,
   header переходит в **«● Не сохранено»** (амбер)
3. **Подождать 2 секунды** — header показывает **«Сохраняется…»**,
   потом **«✓ Сохранено»**
4. **F5** (refresh страницы) — после загрузки фото **остаётся на месте**
   (главный признак работы save'а)
5. **В Supabase проверить:**
   ```sql
   SELECT album_id, has_user_edits, updated_at FROM album_layouts
   WHERE album_id = '<UUID тест-альбома>';
   ```
   `has_user_edits = true`, `updated_at` свежий

6. **Swap placeholder ↔ placeholder**:
   - Перейти на разворот с двумя ученическими фото
   - Курсор `move` при hover на placeholder с фото
   - Тащить фото из одного placeholder'а в другой → они меняются местами
   - Через 2с — save → ✓ Сохранено

7. **beforeunload**:
   - Сделать drag (header → ● Не сохранено)
   - **Не дожидаясь** 2 секунд (или сразу) пробуй закрыть вкладку
     или нажать F5
   - Браузер должен показать диалог «Изменения могут не сохраниться»
   - Cancel — остаёшься на странице. OK — уходишь, изменения теряются
     (это правильно)

8. **Когда всё сохранено** (✓), F5 не должен показывать диалог

9. **Кликнуть «Пересобрать»** в Обзоре альбома → POST `build_album` →
   `has_user_edits` сбрасывается в `false`. Это интеграция с 2.1
   (мы это уже проверяли).

### Что **может** не работать (followup'ы)

- **Save Indicator застревает в pending** — значит useEffect debounce
  не отработал. Проверить console на ошибки React (не зарегистрировал
  изменение `layout.spreads`).
- **Out-of-order race**: серия быстрых drag'ов → старый ответ перезаписал
  свежий. `saveCounterRef` должен это предотвращать.
- **beforeunload не показывается** — некоторые современные браузеры
  игнорируют пустой `returnValue`. Стандарт выполняется, диалог
  должен быть.
- **Swap из пустого placeholder'а** — disabled через `!hasValue`,
  но проверить что курсор default и нет ghost-движения.
