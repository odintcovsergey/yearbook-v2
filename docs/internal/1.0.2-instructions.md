# Инструкция для подэтапа 1.0.2

**Цель:** показать в кабинете партнёра (`/app` → вкладка «Учителя»)
выбранные родителем фотографии учителей. Сейчас фото есть в БД
(`photo_teachers` ⇄ `photos`), их видно в CSV экспорте, но в UI кабинета
этих превью нет — партнёр не понимает, чьё фото отобрал родитель.

**Зачем:** партнёр готовит альбом к печати и должен видеть финальный
набор фото учителей, не выгружая CSV. Это чистая read-only фича —
переназначать фото партнёр не может (это делает ответственный родитель
через `/teacher/[token]`).

**Что меняется:**
- `app/api/tenant/route.ts` → `action=teachers` GET: добавить join с
  `photo_teachers → photos` и отдать `photo_storage_path` + `photo_filename`
- `app/app/page.tsx`:
  - тип `Teacher` + поля `photo_storage_path: string | null` и
    `photo_filename: string | null`
  - в `TeachersTab` помощник `photoUrl` (как в SpreadTab)
  - в readonly-блоке: миниатюра 64×64 слева от текстового блока
    (с placeholder если фото нет), кликом → `<a>` ссылка на оригинал в
    новом табе

**Что НЕ меняем:**
- `update_teacher` API не трогаем — назначение фото идёт через
  `/api/teacher` (родитель)
- В edit-режиме партнёра фото не показываем (партнёр не назначает фото)
- `/teacher/[token]` не трогаем
- Тип чтения `photos` через join по primary key — `photos.id` уже PK,
  storage_path nullable не бывает (см. как работает CSV экспорт)

---

## Принципиальные решения

### Read-only превью

Партнёр видит, но не трогает. Если ему нужно изменить фото — он
отправляет ссылку ответственного родителя из соседней вкладки, как
указано в подсказке наверху TeachersTab.

### Миниатюра 64×64 слева

Сходство с readonly-режимом `/teacher/[token]` (там тоже 64×64). Это
держит вкладку компактной — при 8+ учителях список не разъезжается
вертикально.

### Placeholder когда фото нет

Серый квадрат с «?» — как на teacher-странице. Это даёт партнёру
наглядный сигнал: «у этого учителя ещё нет фото».

### Клик на превью → открыть оригинал в новом табе

Простой `<a target="_blank">` с прямым URL Yandex Storage. Никаких
lightbox'ов в кабинете не делаем (теряет фокус, лишняя сложность).
Партнёр может посмотреть детальнее в новом табе, если надо.

### Storage path → URL через локальный helper

В `TeachersTab` добавляем такую же утилиту как в `SpreadTab`:

```typescript
const ycBase = 'https://storage.yandexcloud.net/yearbook-photos/'
const photoUrl = (path: string) => ycBase + path.replace('yc:', '')
```

Это совместимо с существующими паттернами (страница 5818-5819) и не
требует нового импорта.

## Шаги

### Шаг 1. API: расширить `action=teachers`

Файл: `app/api/tenant/route.ts`, блок `action === 'teachers' && albumId`
(строка ~439).

**Текущий код** (после правок 1.0):

```typescript
if (action === 'teachers' && albumId) {
  if (!(await assertAlbumAccess(auth, albumId, tid))) {
    return NextResponse.json({ error: 'Альбом не найден' }, { status: 404 })
  }

  const { data } = await supabaseAdmin
    .from('teachers')
    .select('id, full_name, position, description, access_token, submitted_at, created_at, is_head_teacher')
    .eq('album_id', albumId)
    .order('created_at')

  return NextResponse.json(data ?? [])
}
```

**Новый код** (заменить блок целиком):

```typescript
if (action === 'teachers' && albumId) {
  if (!(await assertAlbumAccess(auth, albumId, tid))) {
    return NextResponse.json({ error: 'Альбом не найден' }, { status: 404 })
  }

  const { data: teachers } = await supabaseAdmin
    .from('teachers')
    .select('id, full_name, position, description, access_token, submitted_at, created_at, is_head_teacher')
    .eq('album_id', albumId)
    .order('created_at')

  // Подгружаем фото отдельным запросом (паттерн как в CSV экспорте,
  // строка ~951): photo_teachers (teacher_id, photo_id) → photos.
  const teacherIds = (teachers ?? []).map((t: any) => t.id)
  const { data: photoLinks } = teacherIds.length > 0
    ? await supabaseAdmin
        .from('photo_teachers')
        .select('teacher_id, photos(filename, storage_path)')
        .in('teacher_id', teacherIds)
    : { data: [] }

  const photoByTeacher: Record<string, { filename: string | null; storage_path: string | null }> = {}
  for (const link of photoLinks ?? []) {
    const ph = (link as any).photos
    if (ph) {
      photoByTeacher[(link as any).teacher_id] = {
        filename: ph.filename ?? null,
        storage_path: ph.storage_path ?? null,
      }
    }
  }

  const enriched = (teachers ?? []).map((t: any) => ({
    ...t,
    photo_storage_path: photoByTeacher[t.id]?.storage_path ?? null,
    photo_filename: photoByTeacher[t.id]?.filename ?? null,
  }))

  return NextResponse.json(enriched)
}
```

Ключевые моменты:
- Точно тот же паттерн что в CSV экспорте (строки 944-961) — два запроса
  + объединение в map. Не используем nested select в одном запросе чтобы
  не зависеть от настроек FK у Supabase (в CSV экспорте этот же паттерн
  работает стабильно).
- Поля `photo_storage_path` и `photo_filename` `null` если фото не
  выбрано (или выбрано но фото удалили — теоретически невозможно но
  страхуемся через `if (ph)`).

## Что НЕ делаем в 1.0.2

- UI назначения фото в кабинете партнёра — фаза неопределённая. Сейчас
  родитель отвечает за фото через свою ссылку
- Lightbox с навигацией между фото — простой `<a target="_blank">`
  достаточен
- CSV-экспорт не трогаем — он уже отдаёт `Портрет_страница: filename`
  для учителей (строка 972 в tenant API)
- Контекст-файл v42 → v43 — обновляется по плану в 1.5
