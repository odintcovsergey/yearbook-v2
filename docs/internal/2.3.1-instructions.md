# Инструкция для подэтапа 2.3.1

**Цель:** косметический патч к фазе 2.3 + один фикс устойчивости в
smart-fill. Один коммит, три точечных правки.

**Зачем:** после ручной проверки 2.3 в проде обнаружились три проблемы:
1. Пустые photo-слоты на миниатюрах **невидимы** — рамка `strokeWidth=0.3 mm`
   на скейле миниатюры превращается в ~0.2px, для глаза неотличимо от
   фона. Партнёр не понимает «здесь должно быть фото, но его нет».
2. Подпись миниатюр идёт с `0` («0, 1, 2…»), потому что в коде builder'а
   `spreadCounter.value` стартует с 0. Партнёру привычнее с 1.
3. В smart-fill для портрета ученика выбирается **самое старое**
   selection — `order('created_at')` без аргумента = ASC + first-wins
   через `!portraitByChild[...]`. Если ребёнок поменял выбор фото
   (старая запись остаётся в `selections`), в layout пойдёт устаревшее.
   Это не объясняет битые URL в smoke-альбомах (там файлов в YC нет
   вовсе), но лечит класс будущих багов для реальных партнёров.

**Контекст:** диагностика 2.3 показала что код Konva-рендера и компонент
`LayoutPreviewStrip` работают корректно. Текст и кадрирование — ок.
Битые URL миниатюр — следствие рассинхрона `photos.storage_path` ↔ YC
для smoke-альбомов (часть `storage_path` указывает на отсутствующие
файлы). Это **проблема данных**, чистится отдельным SQL-cleanup'ом —
не часть кода, не часть 2.3.1, отложено в backlog.

**Что меняется:**
- `app/app/_components/AlbumSpreadCanvas.tsx` — пустой photo-слот теперь
  с заливкой и видимой обводкой (3 строки: убрать `dash`, добавить
  `fill`, поднять `strokeWidth`)
- `app/app/_components/LayoutPreviewStrip.tsx` — `{s.spread_index + 1}`
  вместо `{s.spread_index}` (одна строка)
- `lib/smart-fill/build-album-input.ts` — `order('created_at',
  { ascending: false })` (одна строка)
- `docs/internal/2.3.1-instructions.md` — этот файл

**Что НЕ трогаем:**
- Логика выбора группового фото (`group` selection) — там работает
  массовая агрегация через `friendsByChild[].push(...)`, на порядок
  не завязана. Latest-wins только для `portrait_page`.
- Фильтр `photos.type === 'portrait'` для portrait_page selections —
  потенциально полезен, но риск сломать legacy данные где типы
  фото были записаны иначе. Откладываем в backlog.
- SQL-cleanup для smoke-альбомов — отдельная задача, не код.
- Endpoint API — не меняется.

## Принципиальные решения

### Сплошная заливка вместо dashed-рамки

Текущий пустой слот:

```tsx
<Rect ... stroke="#cbd5e1" strokeWidth={0.3} dash={[1, 1]} />
```

Проблемы:
- `strokeWidth=0.3 mm` — на скейле миниатюры `175 / 226 ≈ 0.77` это
  ~0.23px. Subpixel рендер размывает до невидимого.
- `dash=[1, 1]` — пунктир 1mm × 1mm на том же скейле = ~0.77px штрихи,
  которые в антиалиасинге сливаются.

Новый пустой слот:

```tsx
<Rect ... fill="#f3f4f6" stroke="#cbd5e1" strokeWidth={0.5} />
```

`fill="#f3f4f6"` — Tailwind `gray-100`, очень светлый серый. На общем
`bg-#fafafa` фоне разворота даёт видимый контраст «здесь зона фото».
`strokeWidth=0.5` — на скейле ~0.4px, тоже не идеально, но `fill`
делает всю работу.

**Альтернатива которую не делаем**: иконка камеры в центре пустого
слота. Это требовало бы Konva `<Image>` на SVG-icon с asset'ом, или
`<Path>` с инлайн SVG-data. Слишком много кода для косметики; в
фазе 2.6 при добавлении hover-state можно будет добавить.

### `spread_index + 1` в UI, не в builder

Альтернатива: поменять `spreadCounter.value` стартовое значение на 1
в `lib/album-builder/build-from-preset.ts`. Не делаю — это изменит
semantics в JSON-выводе layout'а (`spread_index` начнёт с 1 везде:
audit_log, build_album response, /super инструменты). Может быть
несовместимо с какими-то ad-hoc скриптами / тестами. Минимальное
безопасное изменение — только в UI: `+ 1` при отображении.

### Latest selection wins — `order('created_at', { ascending: false })`

Текущий код:

```typescript
.from('selections')
.select('child_id, selection_type, created_at, photos(storage_path)')
.in('child_id', childIds)
.order('created_at')  // = ASC по умолчанию

// Дальше в цикле:
if (sel.selection_type === 'portrait_page' && !portraitByChild[sel.child_id]) {
  portraitByChild[sel.child_id] = { storage_path: ph.storage_path };
}
```

В этой комбинации (ASC + first-wins через `!map[...]`) выигрывает
**самая ранняя** запись. Если у ученика два `portrait_page` selection
(старый + новый), сейчас в layout пойдёт старый.

Фикс — поменять только `order(...)` на DESC. Логика first-wins тогда
выбирает самое свежее. Минимально-инвазивно.

**Почему НЕ через `.distinct()` или `MAX(created_at)`**: эти конструкции
требуют либо raw SQL, либо более сложных Supabase-запросов. Для двух
параллельных условий (`portrait_page` first-wins + `group` массовый)
накладно. Текущий цикл универсален, нужен только правильный порядок.

### `friendsByChild` — порядок не меняется

Сейчас group-фото добавляются массово (`push`). После смены order'а
на DESC порядок group-фото в массиве станет обратным (от свежего к
старому). Для UI миниатюр это не критично — фото с друзьями просто
поменяются местами. Если builder использует первые `friend_photos`
(см. `lib/album-builder/build-from-preset.ts` где есть friend_photos),
свежие пойдут в первые слоты — это **лучше** старого поведения, не
хуже. Принимаем как побочный эффект.

### Защитный slice до 10

Текущий код:
```typescript
for (const k of Object.keys(friendsByChild)) {
  if (friendsByChild[k].length > 10) {
    friendsByChild[k] = friendsByChild[k].slice(0, 10);
  }
}
```

Не трогаем — лимит 10 это защита от extreme-cases (родитель
выбрал 50 group-фото). На реальных альбомах max 4 friend_photos
по spec, 10 — комфортный запас.


**СТАЛО:**

```tsx

  // Партнёру важно видеть, где должно быть фото — на скейле миниатюры
  // тонкая dashed-рамка не различима, поэтому используем fill.
  if (!url || !img) {
    return (
      <Rect
        x={placeholder.x_mm}
        y={placeholder.y_mm}
        width={placeholder.width_mm}
        height={placeholder.height_mm}
        fill="#f3f4f6"
        stroke="#cbd5e1"
        strokeWidth={0.5}
      />
    )
  }
```

Изменения: убрана `dash`, добавлен `fill="#f3f4f6"`, `strokeWidth` с
0.3 на 0.5, обновлён комментарий.

## Применение и верификация в проде

### Порядок

Миграций нет. После push Vercel задеплоит за ~30-60 секунд.

### Проверка визуально

После деплоя открыть Красночетайскую СОШ → Обзор:

1. Под подписями миниатюр должно быть **«1, 2, 3…»** вместо «0, 1, 2…».
2. Пустые места под фото на миниатюрах должны быть **видимы** —
   светло-серые прямоугольники с обводкой. Раньше они были
   неотличимы от фона.

### Что **не** изменится визуально

- Фото в миниатюрах **по-прежнему не появятся** — это рассинхрон
  данных в БД (см. диагноз 2.3 + SQL-проверки). Лечится отдельно.
- Учительский разворот первой миниатюры будет выглядеть так же
  (текст рендерится корректно).

### Smart-fill latest-wins — побочный эффект

После пересборки альбома (нажатие «Пересобрать») URL'ы portrait в
layout могут измениться, если у каких-то учеников было больше одного
`portrait_page` selection в БД. На smoke-альбомах с битыми данными
это проблему не решит (там нет рабочих файлов в принципе), но и не
ухудшит ситуацию.

## Что НЕ делаем в 2.3.1

- **SQL-cleanup мусорных `photos` записей** для smoke-альбомов — это
  инфраструктурная задача, отдельный SQL-скрипт «найти photos где
  файла в YC нет → DELETE». В backlog.
- **Фильтр `photos.type === 'portrait'` для portrait_page selections** —
  риск сломать legacy данные, отложено в backlog.
- **Иконка камеры в пустом photo-слоте** — косметика следующего
  уровня, при добавлении hover-state в фазе 2.6.
- **Изменение `spreadCounter.value` стартового значения в builder'е** —
  затронет JSON формат, audit_log, ad-hoc tests; не делаем.
- **Контекст-файл v43 → v44** — обновляется в 2.8.
