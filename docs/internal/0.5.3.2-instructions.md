# Инструкция для подэтапа 0.5.3.2

**Цель:** Реализовать ученические секции для 4 пресетов в новом
`build-from-preset.ts`:
- `buildSinglePagePerStudent` для **standard** + **universal**
- `buildSpreadPerStudent` для **maximum** + **individual** (без thumbnails)

После 0.5.3.2 в новом builder'е работают: intro, teacher, и 4 из 7
комплектаций. Останутся 3 grid-комплектации (Медиум/Лайт/Мини) и
thumbnails — это 0.5.3.3.

**Архитектурный путь:** Г1, продолжение. Старый `build.ts` и
`scenarios.ts` остаются параллельно. Smoke 58/58 на старом — обязателен.

## Принципиальные решения

### Решение по Универсалу — путь (a)

Сергей разрешил выбор: либо (a) одностраничный E-Student-Left/Right
alternate как сейчас в фазе 0, либо (b) попытка двухстраничного
E-Student-Default первым с fallback на одностраничные.

**В 0.5.3.2 делаем только путь (a).** Решение (b) записываем в
`docs/templates/master-cleanup-tz.md` как улучшение для будущего
подэтапа после фазы 0.5. Причины: (b) требует знания placeholder-имён
E-Student-Default из БД, расширения findMaster, и потенциально миграции
default_for_configs — всё это удлиняет 0.5.3.2 без блокирующей
потребности.

### Семантический выбор правого мастера в `buildSpreadPerStudent`

Builder обобщает Maximum и Individual через **capacity-pool**:

```typescript
const rightCandidates = templateSet.spreads.filter(s =>
  s.page_role === 'student_right' &&
  s.default_for_configs.includes(cfgType) &&
  s.slot_capacity?.students === 1
);
// сортировка asc по photos_friend
```

- Maximum: pool = `[E-Max-Right]` (capacity=4) → всегда один мастер
- Individual: pool = `[E-Ind-Right-3, E-Max-Right]` (capacity=3,4) → per-student

Per-student логика — общая для обоих:
1. friendCount = `student.friend_photos.length`, обрезается до
   `preset.config.student_section.first_spread_content.friend_photos.max`
2. Выбираем первый кандидат с `slot_capacity.photos_friend >= friendCount`
   (минимально-достаточный)
3. Если ни один не подходит — берём с максимальным capacity + warning

Это обеспечивает:
- Идентичное поведение текущему build.ts:buildMaximumStudents
- Идентичное поведение текущему build.ts:buildIndividualStudents
- Гибкость: при добавлении в БД новых вариантов (E-Max-Right-2 и т.п.)
  builder автоматически перейдёт на per-student режим

#### 2.1. Найти левый мастер E-Max-Left

```typescript
const leftFilter: MasterFilter = {
  page_role: 'student_left',
```

#### 2.2. Собрать pool правых кандидатов

Прямой filter (не findMaster, потому что нужен список а не один):

```typescript
const rightCandidates = ctx.templateSet.spreads
  .filter(s => {
    if (s.page_role !== 'student_right') return false;
    if (s.default_for_configs.indexOf(ctx.cfgType) < 0) return false;
    if (s.is_fallback) return false;
    if (s.slot_capacity === null) return false;
    if (typeof s.slot_capacity.students !== 'number' || s.slot_capacity.students < 1) return false;
    return true;
  })
  .sort((a, b) =>
    (a.slot_capacity?.photos_friend ?? 0) - (b.slot_capacity?.photos_friend ?? 0)
  );

if (rightCandidates.length === 0) {
  pushWarning(ctx, {
    code: 'master_not_found',
    detail: `spread_per_student: no right candidates for cfgType=${ctx.cfgType}`,
  });
  return;
}
```

#### 2.3. Per-student loop

```typescript
const friendMaxFromPreset = ss.first_spread_content.friend_photos?.max ?? 0;

for (const student of ctx.input.students) {
  // Левая страница: портрет + имя
  ctx.spreads.push({
    spread_index: ctx.spreadCounter.value++,
    template_id:  left.master.id,
    template_name: left.master.name,
    data: {
      studentportrait: student.portrait,
      studentname:     student.full_name,
    },
  });

  // Правая страница: выбор по friend_photos.length
  let friendCount = student.friend_photos.length;
  if (friendCount > friendMaxFromPreset) {
    pushWarning(ctx, {
      code: 'students_overflow',
      detail: `spread_per_student: student ${student.full_name} has ${friendCount} friend_photos, обрезано до ${friendMaxFromPreset} (preset.max)`,
    });
    friendCount = friendMaxFromPreset;
  }

  // Минимально-достаточный мастер
  let rightMaster = rightCandidates.find(c =>
    (c.slot_capacity?.photos_friend ?? 0) >= friendCount
  );
  let usedSlots: number;

  if (rightMaster) {
    usedSlots = friendCount;
  } else {
    // У всех capacity ниже friendCount — берём максимальный
    rightMaster = rightCandidates[rightCandidates.length - 1];
    usedSlots = rightMaster.slot_capacity?.photos_friend ?? 0;
    pushWarning(ctx, {
      code: 'students_overflow',
      detail: `spread_per_student: student ${student.full_name} needs ${friendCount} slots, max available ${usedSlots} (${rightMaster.name})`,
    });
  }

  ctx.spreads.push({
    spread_index: ctx.spreadCounter.value++,
    template_id:  rightMaster.id,
    template_name: rightMaster.name,
    data: studentSinglePageData(student, usedSlots),
  });
}
```

### Шаг 3. Helper `studentSinglePageData`

Новая локальная функция в build-from-preset.ts (та же что в текущем
build.ts:391-401):

```typescript
function studentSinglePageData(
  s: Student,
  friendSlots: number,
): Record<string, string | null> {
  const data: Record<string, string | null> = {
    studentportrait: s.portrait,
    studentname:     s.full_name,
    studentquote:    s.quote,
  };
  for (let i = 0; i < friendSlots; i++) {
    data['studentphoto' + (i + 1)] = s.friend_photos[i] ?? null;
  }
  return data;
}
```

```markdown
### §B2 — Двухстраничный E-Student-Default для Универсала (решение b из фазы 0.5)

**Контекст:** В фазе 0 builder использует для Универсала одностраничные

вывод для чётного числа учеников (нет «висящего» одностраничного last).

**Что делать:**
1. SQL-проверка: `SELECT name, default_for_configs, slot_capacity, is_fallback
   FROM spread_templates WHERE name LIKE 'E-Student-Default%';`
2. Если default_for_configs не включает `universal` — UPDATE-миграция
3. В `lib/album-builder/build-from-preset.ts:buildSinglePagePerStudent`
   ветка `friend_photos !== null` сначала пытается findMaster
   двухстраничного с `slot_capacity_min.photos_friend = friend_photos.max`
4. При успехе — flow аналогичный buildStandardLikeFlow (chunk 2,
   студенты parами на двухстраничный мастер)
5. При неудаче — fallback на текущий одностраничный alternate-flow

**Сложность:** ~1.5-2 ч. Минимальный риск регрессии (только Универсал
затрагивается). Можно делать в любой момент после 0.5.3.2.

**Приоритет:** P2 (после фазы 0.5).
```

```typescript
const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,

async function loadPreset(slug: string): Promise<Preset> {
  const { data, error } = await supabase.from('config_presets')
    .select('*').eq('slug', slug).single();
  if (error || !data) throw new Error(`preset ${slug} not found: ${error?.message}`);

function makeStudent(name: string, friendCount: number): Student {
```
