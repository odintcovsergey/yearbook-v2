# Инструкция для подэтапа 2.6.4.1

**Цель:** косметический фикс. Палитра в `PhotoPalette` сейчас
показывает фото в порядке загрузки в систему (`created_at` из endpoint'а
2.4) — это даёт хаотичный порядок, потому что партнёры обычно
загружают фото пачками вне алфавитного порядка. Партнёру удобнее
видеть фото отсортированными **по filename** (DSC_xxxxx.jpg) — это
стандартный порядок камеры.

**Зачем:** на тестовом альбоме партнёр обнаружил что фото в палитре
идут вразнобой (один и тот же ученик в разных частях списка из-за
порядка загрузки). Сортировка по filename даёт предсказуемый порядок
и упрощает поиск визуально.

**Контекст:** эта правка **только в client-side компоненте**.
Endpoint `/api/tenant?action=album_photos` (2.4) НЕ меняем — он отдаёт
данные в порядке `created_at`, что является разумным default'ом для
других потенциальных consumer'ов. Сортировка для UI палитры —
обязанность UI компонента.

**Что меняется:**
- `app/app/_components/PhotoPalette.tsx` — добавление `.sort` в нужных
  местах после фильтрации по типу (~5 строк)
- `docs/internal/2.6.4.1-instructions.md` — этот файл

**Что НЕ трогаем:**
- Endpoint `album_photos` (2.4)
- API структура response
- Тип AlbumPhoto

---

## Принципиальные решения

### Сортировка после фильтрации, не до

В коде есть три массива: `portraits`, `groups`, `teachers`, `originals`,
полученные через `filtered.filter(...)`. Сортируем каждый отдельно
**после** фильтрации:

```typescript
const portraits = filtered
  .filter((p) => p.source === 'selections' && p.type === 'portrait')
  .sort((a, b) => a.filename.localeCompare(b.filename))
```

Сортировать **до** filter'а тоже работает, но дороже — сортируем 200
фото вместо 4-х групп по 50.

### `localeCompare` без options

JavaScript `String.prototype.localeCompare` работает natural — `DSC_2`
идёт **до** `DSC_10`? Нет — `localeCompare` без опций даёт
**lexicographic** порядок: `DSC_10 < DSC_2`. Для имён файлов с
числовыми суффиксами это может быть неудобно.

**Но** в реальности filename'ы имеют формат `DSC_5570.jpg`,
`OVK_2233.jpg` — четырёхзначные номера. Лексический порядок
совпадает с числовым (одинаковая длина). Не делаем `numeric: true`
options — оверкилл для нашего случая.

Если позже всплывут проблемы (camera пишет `IMG_1.jpg`, `IMG_10.jpg`,
`IMG_100.jpg` — разная длина) — добавим `localeCompare(b.filename,
undefined, { numeric: true })`.

### Не сортируем originals

`originals` — массив из второго источника (`original_photos` table).
В нём filename'ы тоже, и сортировка применима. Сортируем для
единообразия.

### `useMemo` обертка не нужна

Сортировка применяется к уже-`useMemo`'д `filtered`. Сами `.filter`
+ `.sort` уже под `useMemo` через ссылочную стабильность filtered'а.
Не делаем дополнительный useMemo.

Альтернативно: завернуть всё в один useMemo возвращающий объект
`{ portraits, groups, teachers, originals }`. Это microoptimisation
не нужная пока — проблем с производительностью нет.

## Шаги

### Шаг 1. Изменить группировку в PhotoPalette

Файл: `app/app/_components/PhotoPalette.tsx`

Найди блок группировки (~строки 130-145):

**БЫЛО:**

```typescript
  // Группировка по source/type.
  const portraits = filtered.filter(
    (p) => p.source === 'selections' && p.type === 'portrait',
  )
  const groups = filtered.filter(
    (p) => p.source === 'selections' && p.type === 'group',
  )
  const teachers = filtered.filter(
    (p) => p.source === 'selections' && p.type === 'teacher',
  )
  const originals = filtered.filter((p) => p.source === 'originals')
```

**СТАЛО:**

```typescript
  // Группировка по source/type + сортировка по filename внутри каждой
  // группы (DSC_xxxx.jpg → лексический порядок, см. 2.6.4.1).
  const byFilename = (a: AlbumPhoto, b: AlbumPhoto) =>
    a.filename.localeCompare(b.filename)

  const portraits = filtered
    .filter((p) => p.source === 'selections' && p.type === 'portrait')
    .sort(byFilename)
  const groups = filtered
    .filter((p) => p.source === 'selections' && p.type === 'group')
    .sort(byFilename)
  const teachers = filtered
    .filter((p) => p.source === 'selections' && p.type === 'teacher')
    .sort(byFilename)
  const originals = filtered
    .filter((p) => p.source === 'originals')
    .sort(byFilename)
```

Изменения:
- Хелпер `byFilename` для DRY
- `.sort(byFilename)` после каждого `.filter`

## Применение и верификация в проде

### Порядок

Миграций нет. Vercel задеплоит за ~30-60 секунд.

### Проверка визуально

После деплоя открой редактор тестового альбома. В палитре справа:

1. **Все портреты** идут по filename A-Z (DSC_08369, DSC_08373, DSC_08376...)
2. **Группы** аналогично
3. **Учителя** аналогично
4. Поиск работает как раньше — фильтрует и сортированный список

Партнёр должен увидеть **более логичный порядок** — фото идут в той
последовательности что отдала камера, а не вразнобой.

## Что НЕ делаем в 2.6.4.1

- **Сортировка по имени ученика** — backlog, требует подтянуть
  child names в endpoint
- **Toggle сортировки в UI** (по filename / дате / алфавиту имён) —
  backlog, лишняя сложность для MVP
- **Numeric-aware сортировка** (`numeric: true`) — пока не нужна,
  filename'ы одинаковой длины
