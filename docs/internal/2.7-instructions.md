# Инструкция для подэтапа 2.7

**Цель:** добавить защиту от случайной потери ручных правок партнёра.
Кнопка «Пересобрать» в Обзоре альбома вызывает builder-автосборку
(`POST /api/layout?action=build_album`), которая полностью **переписывает**
layout. Если партнёр уже сделал ручные правки в редакторе (есть
сохранённые drag'и) — пересборка их сотрёт.

В 2.7 перед POST'ом проверяем флаг `has_user_edits` в БД и при
`true` показываем нативный `confirm()`: «Вы вручную правили layout.
Пересборка их сотрёт. Продолжить?». При cancel — ничего не происходит,
правки сохранены.

**Зачем:** это последняя защита перед закрытием функционала фазы 2.
Без неё партнёр может полчаса собирать layout, потом случайно
кликнуть «Пересобрать» (думая что это safe-операция) и всё потерять.
После 2.7 — кликнул, увидел warning, отменил.

**Контекст:** все механизмы готовы:
- В таблице `album_layouts` есть колонка `has_user_edits` (миграция 2.1)
- POST `build_album` (2.1) сбрасывает `has_user_edits=false` после
  пересборки
- POST `save_album_layout` (2.5) ставит `has_user_edits=true` при
  ручной правке
- В редакторе (2.6.4) auto-save выставляет флаг

То есть **флаг уже** правильно живёт в БД. Нужно только:
1. Расширить `GET ?action=album_layout` чтобы возвращал `has_user_edits`
2. На клиенте (`runSmartFill`) перед POST `build_album` читать этот
   флаг и показывать confirm если он true

**Что меняется:**
- `app/api/layout/route.ts` — `handleGetAlbumLayout` дополнить выбор
  колонки `has_user_edits` и вернуть её в response
- `app/app/page.tsx` — две точки:
  1. Тип `SmartFillLayout` дополнить `has_user_edits`
  2. `runSmartFill` перед POST проверять флаг и показывать `confirm`
- `docs/internal/2.7-instructions.md` — этот файл

**Что НЕ трогаем:**
- POST `build_album` — без изменений (он уже сбрасывает флаг)
- POST `save_album_layout` — без изменений (он уже ставит флаг)
- Миграции БД — нет (колонка уже существует)
- Редактор `/app/album/[id]/layout/page.tsx` — без изменений (он
  работает с тем же `LayoutData` типом который не нуждается в
  `has_user_edits`)

## Принципиальные решения

### Нативный `confirm()` против кастомного модала

Альтернатива: красивый модальный диалог с кнопками «Пересобрать» /
«Отмена», стилизованный под проект. Это дополнительные ~50 строк
JSX + UX-вопросы (focus management, ESC, backdrop).

**Не делаем** — нативный `confirm()` достаточен для destructive-операции:
- Гарантированно блокирует UI до ответа
- Партнёр привычен к таким диалогам
- Браузер сам разруливает focus, ESC, accessibility
- Никаких dependencies

Если в будущем нужна кастомизация (например брендинг OkeyBook) —
backlog.

### Текст confirm

Стандартный паттерн «destructive action with explanation»:

```typescript
const ok = window.confirm(
  'У вас есть несохранённые правки в редакторе. ' +
  'Пересборка их сотрёт. Продолжить?'
)
if (!ok) return
```

Текст явно говорит:
1. Что произошло (правки есть)
2. Что случится (сотрутся)
3. Спрашивает подтверждение (продолжить?)

**Не делаем:**
- "Are you sure?" — слабая формулировка, не объясняет последствий
- Длинные технические объяснения (`has_user_edits=true` и т.д.) —
  partner это не понимает

### Где проверять — на клиенте или на сервере

Альтернатива: добавить проверку на сервере в `handleBuildAlbum` —
если `has_user_edits=true` без специального флага в body
(`?force=true`) — отказать с ошибкой. Затем клиент при этой ошибке
показывает confirm и ретраит с флагом.

Не делаем — это double-work (два запроса для одного действия) и
сложнее код. Клиент-side проверка нормально для warn-flow:

- Если партнёр обходит проверку (например через DevTools отправляет
  POST напрямую) — он сознательно знает что делает, защищать его не
  нужно
- Если партнёр изменил `has_user_edits` через консоль и обманул —
  это не урон, просто потеря его собственных правок

### Где брать актуальный `has_user_edits`

Два варианта:

**A.** Использовать значение из state (загруженного при mount'е модала)
**B.** Делать дополнительный запрос `GET ?action=album_layout` перед
конфирмом для свежих данных

**Делаем A** — простота. Если партнёр полчаса назад открыл альбом, а
параллельно менеджер OkeyBook через view_as сделал правки и сохранил
их — state партнёра устарел, флаг будет `false` в его state. Партнёр
не увидит confirm. Это **редкий edge case** для текущей фазы.

В фазе 3+ когда добавим real-time sync (или хотя бы 30-сек polling)
— можно будет переключиться на B без боли.

### Тип в response endpoint'а

`has_user_edits` в БД — `boolean NOT NULL DEFAULT false`. В response
возвращаем как `boolean`. В `SmartFillLayout` тип:

```typescript
type SmartFillLayout = {
  layout_id: string
  template_set_id: string
  spreads: unknown[]
  warnings: EnrichedWarning[]
  summary: SmartFillSummary
  has_user_edits: boolean  // ← новое поле
}
```

`SmartFillSummary` не трогаем — `has_user_edits` логически отдельная
концепция, не part of summary.

### Опциональная защитная проверка

Если по какой-то причине `has_user_edits` отсутствует в response (старый
кеш, ошибка endpoint'а, race condition) — `layout.has_user_edits ===
true` будет false, confirm не показывается. Это safe-default —
партнёр продолжит как раньше без раздражения. **Не валидируем** строго.

### Шаг 3. Добавить confirm в runSmartFill

Тот же файл `app/app/page.tsx`. Найди функцию `runSmartFill` (~строка 919):

**БЫЛО:**

```typescript
  const runSmartFill = async () => {
    if (!album.config_preset_id || smartFillBusy) return
    setSmartFillBusy(true)
    try {
      const r = await apiVA('/api/layout?action=build_album', {
        method: 'POST',
        body: JSON.stringify({ album_id: album.id }),
      })
      if (r.ok) {
        const data = await r.json()
        setLayout({
          layout_id: data.layout_id,
          template_set_id: data.template_set_id,
          spreads: data.spreads,
          warnings: data.warnings,
          summary: data.summary,
        })
        onNotify(`Layout собран: ${data.summary.total_spreads} разворотов, ${data.summary.total_warnings} предупреждений`)
      } else {
        const d = await r.json().catch(() => ({}))
        onError(d.error ?? 'Сборка не удалась')
      }
    } catch {
      onError('Ошибка сети при сборке')
    } finally {
      setSmartFillBusy(false)
    }
  }
```

**СТАЛО:**

```typescript
  const runSmartFill = async () => {
    if (!album.config_preset_id || smartFillBusy) return

    // Защита: если в layout есть несохранённые ручные правки партнёра —
    // подтверждаем destructive-операцию. has_user_edits=true появляется
    // после save_album_layout (2.5) и сбрасывается в false после
    // build_album (2.1).
    if (layout?.has_user_edits) {
      const ok = window.confirm(
        'У вас есть ручные правки в редакторе. Пересборка их сотрёт. Продолжить?'
      )
      if (!ok) return
    }

    setSmartFillBusy(true)
    try {
      const r = await apiVA('/api/layout?action=build_album', {
        method: 'POST',
        body: JSON.stringify({ album_id: album.id }),
      })
      if (r.ok) {
        const data = await r.json()
        setLayout({
          layout_id: data.layout_id,
          template_set_id: data.template_set_id,
          spreads: data.spreads,
          warnings: data.warnings,
          summary: data.summary,
          has_user_edits: false,  // build_album сбрасывает флаг (см. 2.1)
        })
        onNotify(`Layout собран: ${data.summary.total_spreads} разворотов, ${data.summary.total_warnings} предупреждений`)
      } else {
        const d = await r.json().catch(() => ({}))
        onError(d.error ?? 'Сборка не удалась')
      }
    } catch {
      onError('Ошибка сети при сборке')
    } finally {
      setSmartFillBusy(false)
    }
  }
```

Изменения:
- Перед `setSmartFillBusy(true)` добавлен блок проверки + confirm
- В вызове `setLayout({...})` после успешного build добавлено поле
  `has_user_edits: false` (build всегда сбрасывает)

**Замечание:** в build_album response сейчас может НЕ возвращаться
`has_user_edits` (мы его не ставили в response 2.1). Поэтому ставим
вручную `false` — это семантически правильно (build только что
сбросил флаг в БД).

## Применение и верификация в проде

### Порядок

Миграций нет. Vercel задеплоит за ~30-60 секунд.

### Проверка визуально (полный сценарий)

**Сценарий A — без правок** (нет confirm'а):

1. Открой тестовый альбом → Обзор
2. Нажми «Пересобрать»
3. Должна сразу пойти пересборка без диалога (потому что
   `has_user_edits=false`)

**Сценарий B — с правками** (есть confirm):

1. Открой редактор тестового альбома
2. Сделай хотя бы один drag (любой)
3. Дождись «✓ Сохранено» (auto-save через 2с)
4. Вернись в Обзор через «← К альбому»
5. Нажми «Пересобрать» — должен появиться нативный диалог браузера:
   > «У вас есть ручные правки в редакторе. Пересборка их сотрёт.
   > Продолжить?»
   > [Cancel] [OK]
6. Cancel → ничего не происходит, миниатюры остаются с твоими правками
7. SQL `has_user_edits` всё ещё `true`

**Сценарий C — продолжение confirm**:

1. Снова нажми «Пересобрать»
2. В диалоге OK
3. Layout пересобирается, миниатюры показывают builder-результат
   (твои правки потеряны)
4. SQL `has_user_edits` теперь `false`

**Сценарий D — повторно после reset**:

1. Сразу же ещё раз «Пересобрать»
2. Confirm НЕ показывается (потому что флаг уже false после
   предыдущего build'а)
3. Сборка идёт сразу

### Что **может** не работать

- **Confirm не появляется хотя должен** — возможно state `layout.has_user_edits`
  не обновился после save'а в редакторе. Логично — главная страница
  не перезагружает layout пока модал открыт. Когда партнёр
  возвращается из редактора через query-deeplink, модал re-mount'ится
  и `useEffect` re-fetch'нет layout с актуальным флагом. Если этот
  случай не работает → починим (но скорее всего работает).

## Что НЕ делаем в 2.7

- **Кастомный стилизованный модал** вместо нативного confirm — backlog
- **Real-time sync флага** при правках другим пользователем — backlog
  (фаза 3+, из spec §7.4)
- **Серверная защита от build_album при has_user_edits** — backlog
  (для admin-tools или CLI-сценариев)
- **Контекст-файл v43 → v44** — это 2.8
