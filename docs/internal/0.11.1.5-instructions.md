# Инструкция для подэтапа 0.11.1.5 — архитектурный фикс default_for_configs

> Это временный файл-инструкция. После выполнения подэтапа удалить и закоммитить вместе с основными изменениями.

## Цель подэтапа

Разделить две семантики тегов мастеров:
- **`applies_to_configs`** (существующий) — «технически совместим с этой комплектацией, можно использовать вручную в UI редактора фаз 2-4». Остаётся как есть.
- **`default_for_configs`** (новый) — «для этой комплектации мастер выбирается автоматически в buildAlbum». Используется в коде фильтров.

Это решает проблему обнаруженную в первой попытке 0.11.2: для Лайт-27 учеников `pickAdaptiveGridMaxCapacity` брал N-12 вместо L-6, потому что `applies_to_configs` у обоих включал 'light'. После фикса — для Лайт автовёрстка видит только L-* мастера, для Мини только N-*.

**Этот подэтап НЕ добавляет нового функционала.** Все 36 smoke-сцен из 0.11.1 должны продолжать проходить (поведение не меняется для уже работающих комплектаций). Овёрфлоу Лайт/Мини сделаем в **0.11.2 после этого подэтапа**.

Один коммит:
```
refactor(layout/0.11.1.5): default_for_configs — разделение совместимости и автовыбора
```

ADD COLUMN default_for_configs text[] NOT NULL DEFAULT '{}';

## 3. `lib/album-builder/find-master.ts`

Здесь сейчас фильтрация по `applies_to_configs` в `matchesBaseFilters` или подобной функции. Найти все места где код проверяет:
```typescript
if (s.applies_to_configs.indexOf(filter.applies_to_config) < 0) return false;
```
или
```typescript
if (!s.applies_to_configs.includes(filter.applies_to_config)) return false;
```

**Заменить на проверку по `default_for_configs`:**
```typescript
if (s.default_for_configs.indexOf(filter.applies_to_config) < 0) return false;
```

**ВАЖНО:** имя поля в `MasterFilter` — `applies_to_config` (singular) — НЕ переименовываем. Это просто строковый «искомый config_type». А вот свойство в `SpreadTemplate` теперь читаем из `default_for_configs`.

Если в коде есть несколько мест с фильтрацией — все нужно заменить. Должны быть как минимум:
- `matchesBaseFilters` (или эквивалент) в find-master.ts
- `pickAdaptiveGrid` в build.ts
- `pickAdaptiveGridMaxCapacity` (был добавлен в 0.11.2 v2 — но эту версию мы откатили; она будет создана заново в 0.11.2)

В **этом** подэтапе правим только find-master.ts и build.ts (если там есть прямой доступ к `applies_to_configs`).

const candidates = ctx.config.template_set.spreads.filter((s) => {

Грепом проверь что в build.ts больше нет обращений к `applies_to_configs`. Если есть — все заменить на `default_for_configs`.

## 5. Загрузка template_set из БД

Где-то в коде template_set читается из Supabase. Скорее всего это в одном из:
- `lib/album-builder/load-template-set.ts` (если такой существует)
- `scripts/smoke-album-builder.ts` (точно есть)
- `app/api/layout/route.ts` (action='template_set_detail')

В select-выражении нужно добавить `default_for_configs` к списку колонок.

**Грепом найти все места где читается `applies_to_configs`** — и добавить рядом `default_for_configs`:

```bash
grep -rn "applies_to_configs" --include="*.ts" --include="*.tsx" .
```

В каждом select/mapping добавить `default_for_configs`. Если поле не пробрасывается — TypeScript падёт при сборке (тип `SpreadTemplate` теперь требует это поле).

В `scripts/smoke-album-builder.ts` загрузка скорее всего что-то вроде:
```typescript
const { data } = await supabase
  .from('spread_templates')
  .select('id, name, applies_to_configs, slot_capacity, ...')
  ...
```
Добавить `default_for_configs` в список.

## 6. (Дополнительно) `lib/idml-converter/parse.ts` и `scripts/convert-idml.ts`

При импорте IDML создаются записи spread_templates с пустым `default_for_configs` (default БД '{}'). Это нормально — после импорта администратор вручную обновит default_for_configs через SQL/UI. Менять парсер пока НЕ нужно.

Но в типах TS, которые формирует парсер (если они представляют SpreadTemplate без `default_for_configs`), нужно либо добавить поле как опциональное, либо при insert'е пробрасывать пустой массив. Зависит от текущей структуры. Если упадёт TS — поправим.
