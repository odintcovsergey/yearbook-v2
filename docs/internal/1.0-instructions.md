# Инструкция для подэтапа 1.0

**Цель:** ввести явный флаг `teachers.is_head_teacher` (классный руководитель)
вместо неявной логики «первый учитель = классный руководитель» в кабинете
партнёра. Открывает фазу 1 (Smart-fill).

**Зачем:** в фазе 1 builder будет читать классного руководителя из БД для
генерации учительского разворота (E-Head-* мастера). Builder уже принимает
`AlbumInput.head_teacher: HeadTeacher | null` и пишет warning `no_head_teacher`
если null — нужно из БД честно сказать кто head. Сейчас в кабинете партнёра
description (текст от кл. рук.) показывается только у учителя с `idx === 0`
по `created_at` — это работает, но не даёт партнёру выбрать другого учителя
в роли классного руководителя.

**Что меняется:**
- БД: новая колонка `teachers.is_head_teacher BOOLEAN NOT NULL DEFAULT false`
  + partial unique index «один head на альбом» + бэкфил (первый по `created_at` → head)
- `app/api/tenant/route.ts`:
  - `action=teachers` GET → SELECT добавить `is_head_teacher`
  - `action=update_teacher` POST → принимать `is_head_teacher`, реализовать
    radio-pattern (при `is_head_teacher=true` сначала сбросить флаг у других
    учителей альбома, потом установить у текущего)
- `app/app/page.tsx` → `TeachersTab`:
  - тип `Teacher` + поле `is_head_teacher: boolean`
  - в edit-форме: чекбокс «Классный руководитель»; поле description показываем
    когда чекбокс отмечен (вне зависимости от исходного значения — реактивно)
  - в readonly виде: бейдж «Классный руководитель» и блок description
    показываем по `t.is_head_teacher` (вместо `i === 0`)

**Что НЕ трогаем:**
- `app/teacher/[token]/page.tsx` — страница ответственного родителя
  оставляем как есть (логика `idx === 0`). Известное расхождение: после
  миграции бэкфил отметит первого по `created_at` как head — поведение
  совпадёт. Если партнёр позже сменит head в `/app`, на странице
  ответственного родителя по-прежнему будет показан первый учитель —
  это допустимый компромисс MVP, поправим в фазе 1.X если будет фидбек.
- `app/api/teacher/route.ts` — родительский API учителей не трогаем.

## Принципиальные решения

### Radio-pattern в update_teacher

Когда партнёр отмечает чекбокс «Классный руководитель» у учителя X в альбоме A,
нужно гарантировать что в альбоме A только один head. Реализация в API
(не на клиенте) — сервер делает 2 шага:

1. `UPDATE teachers SET is_head_teacher=false WHERE album_id=A AND id!=X` — сбросить у остальных
2. `UPDATE teachers SET is_head_teacher=true WHERE id=X` — установить у текущего

Порядок важен: если поменять (сначала установить → потом сбросить других),
partial unique index `teachers_one_head_per_album` выстрелит ошибкой.
Не нужно транзакции (Supabase RPC не используем) — на одиночную ошибку
маловероятна, fallback простой: партнёр повторит клик.

Когда `is_head_teacher=false` приходит в payload — просто `UPDATE teachers
SET is_head_teacher=false WHERE id=X`. После этого может оказаться 0 head
в альбоме — это допустимо. Builder в фазе 1 напишет `no_head_teacher` warning.

### `is_head_teacher` опционален в payload

Если поле не пришло (например, при сохранении только `position` через
существующий вызов) — флаг не меняется. Это значит существующие места
которые делают `update_teacher` без флага продолжают работать.

### Чекбокс реактивно меняет видимость description

В edit-форме поле description показываем по `editForm.is_head_teacher`,
а не по сохранённому `t.is_head_teacher`. Это значит:
- Если партнёр отмечает чекбокс при создании head — поле description
  появляется сразу (до сохранения)
- Если партнёр снимает чекбокс — поле description скрывается, но текст
  не теряется (остаётся в `editForm.description` пока редактор открыт);
  после save если is_head_teacher=false → description в БД сохраняется
  как есть (сервер не очищает description при снятии флага).

### Бэкфил миграции — первый по `created_at`

Соответствует текущей UI-логике (`idx === 0` после `order('created_at')`).
Это **безопасный** бэкфил: после миграции каждый альбом который видел
description в `/app` продолжает видеть тот же description у того же
учителя — никаких видимых изменений для пользователя.

### Бейдж «Классный руководитель» — синий, как раньше

Существующий `<span className="badge-blue">Классный руководитель</span>`
переиспользуем; меняется только условие рендера (`t.is_head_teacher`
вместо `i === 0`).

## Шаги

### Шаг 1. Создать SQL-миграцию

Файл: `migrations/2026-05-09-teachers-head-flag.sql`

```sql
-- Подэтап 1.0 — флаг классного руководителя для учителей.
-- До этой миграции UI определял классного руководителя по позиции в списке
-- (idx === 0 после ORDER BY created_at). Нужно явное поле — Smart-fill будет
-- читать head_teacher из БД для генерации учительского разворота.

-- 1. Колонка флага
ALTER TABLE teachers
  ADD COLUMN is_head_teacher BOOLEAN NOT NULL DEFAULT false;

-- 2. Один head на альбом (partial unique index)
CREATE UNIQUE INDEX teachers_one_head_per_album
  ON teachers (album_id)
  WHERE is_head_teacher = true;

-- 3. Бэкфил: для каждого альбома где есть учителя — отметить первого
-- по created_at как head. Это совпадает с текущей UI-логикой
-- (TeachersTab показывает description у teachers[0]).
WITH first_per_album AS (
  SELECT DISTINCT ON (album_id) id
  FROM teachers
  ORDER BY album_id, created_at ASC
)
UPDATE teachers
SET is_head_teacher = true
WHERE id IN (SELECT id FROM first_per_album);

-- Проверка
SELECT
  album_id,
  COUNT(*) AS teachers_total,
  COUNT(*) FILTER (WHERE is_head_teacher) AS heads
FROM teachers
GROUP BY album_id
ORDER BY album_id
LIMIT 20;
-- Ожидание: heads = 1 для каждого альбома (или 0 если в альбоме 0 учителей).
```

**ВАЖНО — применить миграцию вручную в Supabase SQL editor**
(production БД). Файл коммитится в репо для документации, но автоматически
не применяется. Сергей применит миграцию до запуска builder'а на живых
данных. До применения миграции код всё равно собирается — TypeScript про
поле узнаёт через явные SELECT'ы и payloads.

### Шаг 3. API: расширить `action=update_teacher` (radio-pattern)

Файл: `app/api/tenant/route.ts`

Найти блок `body.action === 'update_teacher'` (около строки 1842). Текущий
код принимает `full_name`, `position`, `description` и делает один UPDATE.
Нужно добавить поддержку `is_head_teacher` с radio-pattern.

**Полная новая версия handler'а** (заменить блок целиком, начиная от
комментария `// update_teacher — редактирование данных учителя` до
`return NextResponse.json({ ok: true })`):

```typescript
  // ----------------------------------------------------------
  // update_teacher — редактирование данных учителя
  // ----------------------------------------------------------
  if (body.action === 'update_teacher') {
    const { teacher_id, full_name, position, description, is_head_teacher } = body
    if (!teacher_id) {
      return NextResponse.json({ error: 'teacher_id обязателен' }, { status: 400 })
    }

    if (!(await assertTeacherAccess(auth, teacher_id))) {
      return NextResponse.json({ error: 'Учитель не найден' }, { status: 404 })
    }

    const updates: Record<string, unknown> = {}
    if (full_name !== undefined) updates.full_name = full_name?.trim() || null
    if (position !== undefined) updates.position = position?.trim() || null
    if (description !== undefined) updates.description = description?.trim() || ''

    const headFlagProvided = is_head_teacher !== undefined
    const wantHead = is_head_teacher === true

    if (Object.keys(updates).length === 0 && !headFlagProvided) {
      return NextResponse.json({ error: 'Нет полей для обновления' }, { status: 400 })
    }

    // Radio-pattern: если выставляем is_head_teacher=true — сначала
    // сбросить флаг у остальных учителей того же альбома (partial unique
    // index не разрешит двух head на альбом).
    if (headFlagProvided && wantHead) {
      // Получить album_id текущего учителя
      const { data: teacherRow, error: fetchErr } = await supabaseAdmin
        .from('teachers')
        .select('album_id')
        .eq('id', teacher_id)
        .single()

      if (fetchErr || !teacherRow) {
        return NextResponse.json({ error: 'Учитель не найден' }, { status: 404 })
      }

      // Сбросить head у остальных в этом альбоме
      const { error: resetErr } = await supabaseAdmin
        .from('teachers')
        .update({ is_head_teacher: false })
        .eq('album_id', teacherRow.album_id)
        .neq('id', teacher_id)

      if (resetErr) {
        return NextResponse.json({ error: resetErr.message }, { status: 500 })
      }
    }

    if (headFlagProvided) {
      updates.is_head_teacher = wantHead
    }

    const { error } = await supabaseAdmin
      .from('teachers')
      .update(updates)
      .eq('id', teacher_id)

    if (error) {
      return NextResponse.json({ error: error.message }, { status: 500 })
    }

    await logAction(auth, 'teacher.update', 'teacher', teacher_id, {
      fields: Object.keys(updates),
    })

    return NextResponse.json({ ok: true })
  }
```

Ключевые отличия от старого кода:
- Добавлен парсинг `is_head_teacher` и derived `headFlagProvided`/`wantHead`
- Условие «нет полей для обновления» учитывает `headFlagProvided`
- Перед основным UPDATE — если ставим head=true, делаем pre-update reset
  у остальных учителей альбома (через дополнительный SELECT album_id)
- В `updates` объект флаг добавляется только если `headFlagProvided`
  (если не пришёл — поле не меняется, как у других полей)

## Открытые риски и проверки вручную

### После применения миграции в Supabase

Сергей проверяет:
```sql
-- Каждый альбом имеет ровно одного head (или ноль если 0 учителей)
SELECT album_id, COUNT(*) FILTER (WHERE is_head_teacher) AS heads
FROM teachers
GROUP BY album_id
HAVING COUNT(*) FILTER (WHERE is_head_teacher) > 1;
-- Ожидание: 0 строк (никаких альбомов с >1 head)
```

### Smoke для UI (после применения миграции и деплоя)

1. Открыть существующий альбом в `/app` → вкладка «Учителя» → у первого
   видно бейдж «Классный руководитель» и блок description (если был
   заполнен). Это поведение должно совпадать с тем что было до миграции.
2. Нажать «Редактировать» у второго учителя → отметить чекбокс
   «Классный руководитель» → Сохранить. После reload список:
   у первого учителя бейджа больше нет, у второго появился. Description
   у первого скрыт (но в БД остался).
3. У выбранного второго отметить «Редактировать» → снять чекбокс →
   Сохранить. После reload: ни у кого нет бейджа «Классный руководитель»
   (нулевая ситуация — допустима).
4. У третьего учителя в edit-форме отметить чекбокс — текст description
   появляется без сохранения. Снять чекбокс — текст скрывается, но
   editForm.description не теряется. Снова отметить — текст возвращается.

### `/teacher/[token]` (страница ответственного родителя)

После миграции открыть существующую teacher-ссылку (если есть в dev) —
описание классного руководителя должно показываться у того же учителя
что и до миграции (idx === 0 ⇄ первый по created_at ⇄ помеченный head).

Если партнёр позже сменит head в `/app`, на `/teacher` это не отразится —
там старая логика `idx === 0`. Известный компромисс. Если Сергей решит
выровнять — фаза 1.X (отдельная задача).
