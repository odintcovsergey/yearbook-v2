# Инструкция для подэтапа 2.6.3.1

**Цель:** два связанных фикса по результатам визуальной проверки 2.6.3.
Один коммит.

**Зачем:**

1. **Фото после drop'а не появляется в Konva canvas** (но бейдж «✓»
   на миниатюре в палитре появляется). Это значит state мутируется
   корректно (`spread.data[label] = photo.url`), но Konva-слой
   `<KonvaImage>` не загружает изображение.

   Корень: в `useImage` хук `app/app/_components/AlbumSpreadCanvas.tsx:56`
   стоит `i.crossOrigin = 'anonymous'`. Этот атрибут требует от сервера
   возвращать `Access-Control-Allow-Origin` header. YC бакет
   `yearbook-photos` для GET-запросов **не возвращает CORS headers**
   (он публичный для чтения, но без CORS configuration), поэтому
   браузер блокирует загрузку → onerror → `setImg(null)` → пустая
   рамка в Konva.

   **Это объясняет** почему миниатюры в `LayoutPreviewStrip` (2.3) и
   canvas в редакторе показывали только серые рамки даже на «правильных»
   URL'ах. Изначально мы списали на «битые URL'ы smoke-альбома», а
   корень был в crossOrigin.

   Фикс: убрать `i.crossOrigin = 'anonymous'`. Браузер загрузит
   изображение через obычный (no-cors) запрос. **Минус:** Konva canvas
   станет «tainted» — нельзя будет экспортировать его в PNG через
   `stage.toDataURL()`. Но эта функциональность нам не нужна на фазе 2
   (PDF-экспорт — фаза 3, и там используется отдельный pipeline на
   сервере, не canvas-rasterization).

2. **Drag'ое фото уезжает «под» canvas-зону.** Когда партнёр тащит
   миниатюру за границу палитры в сторону canvas'а, она визуально
   скрывается под `<div className="relative">` в `AlbumSpreadCanvas`.

   Корень: `position: relative` создаёт новый stacking context. Внутри
   него Stage'овый `<canvas>` HTML-элемент. Параллельно в палитре
   `<PhotoTile>` имеет `style={{ transform: ..., zIndex: 50 }}`. Но
   `zIndex: 50` действует только в собственном stacking context'е
   палитры, который НЕ выше canvas'ового context'а. Локальный
   `zIndex: 9999` тоже не поможет — это две разные stacking
   иерархии.

   **Правильное решение** — `<DragOverlay>` от dnd-kit. Он рендерится
   через React Portal в `<body>` (на самом верху DOM-tree), и
   draggable во время drag'а отображается из этого portal'а — он
   гарантированно поверх всего. В 2.6.3 я отказался от `<DragOverlay>`
   из-за UX-соображений («дёрганый»), но баг с z-index'ом важнее.

**Контекст:** оба фикса точечные, в разных компонентах. В одном коммите
потому что обнаружены в одну проверку и оба критичны для работы 2.6.3.

**Что меняется:**
- `app/app/_components/AlbumSpreadCanvas.tsx` — убрать
  `i.crossOrigin = 'anonymous'` (1 строка)
- `app/app/_components/PhotoPalette.tsx` — убрать `transform`/`zIndex`
  style из `PhotoTile` (теперь рендером во время drag'а займётся
  DragOverlay)
- `app/app/album/[id]/layout/page.tsx` — добавить `<DragOverlay>` в
  `<DndContext>`, рендерить там «тень» draggable'а с тем же `<img>`
- `docs/internal/2.6.3.1-instructions.md` — этот файл

**Что НЕ трогаем:**
- DropZone в `AlbumSpreadCanvas` — без изменений
- `handleDragEnd` — без изменений
- `useDraggable` логика в `PhotoTile` — без изменений (только style)
- Вся остальная функциональность редактора

## Принципиальные решения

### Фикс 1: убрать crossOrigin полностью, не делать condition

Альтернатива: `try-with-cors-then-fallback`. Сложно, требует второй
useState, обработка race conditions. Не делаем.

Альтернатива: настроить CORS на YC бакете. Технически правильно,
**но** требует доступа в YC console + знания CORS XML-формата +
тестирования. Можно сделать в backlog, не блокирует фазу 2.

**Принимаем:** просто убрать crossOrigin. Это backwards-compatible
с существующим UI (palette `<img>` уже работает без crossOrigin,
LayoutPreviewStrip заработает корректно после деплоя). Tainted canvas
для фазы 2 — non-issue.

### Фикс 2: DragOverlay рендерит копию миниатюры

DragOverlay принимает children и рендерит их в Portal'е. Контент
определяет родитель (LayoutEditorPage). Чтобы знать **что именно
рендерить**, нам нужно отслеживать активный draggable через
`onDragStart`:

```typescript
const [activeDrag, setActiveDrag] = useState<AlbumPhoto | null>(null)

function handleDragStart(event: DragStartEvent) {
  const sourceData = event.active.data.current as { type?: string; photo?: AlbumPhoto } | undefined
  if (sourceData?.type === 'palette' && sourceData.photo) {
    setActiveDrag(sourceData.photo)
  }
}

function handleDragEnd(event: DragEndEvent) {
  setActiveDrag(null)  // сбрасываем в любом случае
  // ... existing logic ...
}
```

И в JSX:

```tsx
<DndContext sensors={...} onDragStart={handleDragStart} onDragEnd={handleDragEnd}>
  {/* ... main content ... */}
  <DragOverlay>
    {activeDrag && (
      <div className="aspect-[3/4] w-[120px] bg-gray-100 rounded overflow-hidden border-2 border-blue-500 shadow-xl">
        <img
          src={activeDrag.thumb_url}
          alt={activeDrag.filename}
          draggable={false}
          className="w-full h-full object-cover pointer-events-none"
        />
      </div>
    )}
  </DragOverlay>
</DndContext>
```

Размер 120px подобран по миниатюрам палитры (`w-[120px]` ≈ существующая
ширина одной tile при `grid-cols-3` и palette ~400px). Он будет
точно не больше оригинала, partner'у удобно.

`shadow-xl` + `border-2 border-blue-500` — визуальная подчёркнутость
draggable'а как «парящего объекта».

### `PhotoTile` теряет `transform` и становится `opacity-40`-only

Когда DragOverlay активен, dnd-kit **сам** делает оригинальный
draggable полупрозрачным через свой механизм. Но мы оставим явный
`isDragging ? 'opacity-40'` для чёткости — он не мешает.

Убираем только **style с transform/zIndex**. Без него миниатюра
остаётся на своём месте в сетке палитры, не сдвигается. Drag-визуал
происходит в DragOverlay.

```tsx
// БЫЛО:
const style = transform
  ? { transform: `translate(${transform.x}px, ${transform.y}px)`, zIndex: 50 }
  : undefined

return (
  <div
    ref={setNodeRef}
    style={style}
    {...attributes}
    {...listeners}
    ...
```

```tsx
// СТАЛО:
return (
  <div
    ref={setNodeRef}
    {...attributes}
    {...listeners}
    ...
```

`transform` из `useDraggable` хука всё равно возвращается, но мы его
игнорируем. dnd-kit это нормально воспринимает — DragOverlay берёт
управление визуалом на себя.

### Не используем dropAnimation

`<DragOverlay>` имеет проп `dropAnimation` для анимации возврата при
drop вне drop-zone. По умолчанию включена. Для нашего UX (drop в
canvas или ничего) — это нормально, не убираем.

### `useState<AlbumPhoto | null>` против ref

`useRef` не триггерит re-render при изменении. Но `<DragOverlay>`
читает children из state JSX — нужен re-render когда меняется содержимое.
Используем useState.

### Никаких изменений в DropZone

Drop-target'ы продолжают работать как раньше. Когда DragOverlay в
портале над canvas'ом, hit-testing dnd-kit использует **collision
detection** между DragOverlay-rectом и DropZone'ами — это работает
out-of-the-box.

### Шаг 1. Фикс 1 — убрать crossOrigin

Файл: `app/app/_components/AlbumSpreadCanvas.tsx`

Найди useImage (~строка 48-62):

**БЫЛО:**

```typescript
function useImage(url: string | null): HTMLImageElement | null {
  const [img, setImg] = useState<HTMLImageElement | null>(null)
  useEffect(() => {
    if (!url) {
      setImg(null)
      return
    }
    const i = new window.Image()
    i.crossOrigin = 'anonymous'
    i.src = url
    i.onload = () => setImg(i)
    i.onerror = () => setImg(null)
  }, [url])
  return img
}
```

**СТАЛО:**

```typescript
function useImage(url: string | null): HTMLImageElement | null {
  const [img, setImg] = useState<HTMLImageElement | null>(null)
  useEffect(() => {
    if (!url) {
      setImg(null)
      return
    }
    // crossOrigin не используем: YC bucket не возвращает CORS headers,
    // и crossOrigin='anonymous' блокирует загрузку. Tainted canvas
    // допустимо в фазе 2 (PDF-экспорт через серверный pipeline в фазе 3).
    const i = new window.Image()
    i.src = url
    i.onload = () => setImg(i)
    i.onerror = () => setImg(null)
  }, [url])
  return img
}
```

Изменения: удалена строка `i.crossOrigin = 'anonymous'`, добавлен
комментарий-обоснование.

#### 3б. State activeDrag и handleDragStart

Найди в `LayoutEditorPage` блок с `useState`'ами (~строки 67-74).
**После** последнего useState (`viewport`) добавь:

```typescript
  const [activeDrag, setActiveDrag] = useState<AlbumPhoto | null>(null)
```

Затем найди функцию `handleDragEnd`. **Перед** ней добавь:

```typescript
  function handleDragStart(event: DragStartEvent) {
    const sourceData = event.active.data.current as
      | { type?: string; photo?: AlbumPhoto }
      | undefined
    if (sourceData?.type === 'palette' && sourceData.photo) {
      setActiveDrag(sourceData.photo)
    }
  }
```

Внутри существующего `handleDragEnd`, в самом начале (первой строкой
тела функции) добавь:

```typescript
    setActiveDrag(null)
```

Это сбросит state в любом случае — успешный drop, drop вне зоны, или
cancel. ВАЖНО: добавить **первой строкой**, до `if (!over) return`.
Иначе при drop вне зоны overlay останется висеть после release.

Полный обновлённый `handleDragEnd`:

```typescript
  function handleDragEnd(event: DragEndEvent) {
    setActiveDrag(null)  // ← новая строка, всегда сбрасываем
    const { active, over } = event
    if (!over) return

    const sourceData = active.data.current as
      | { type?: string; photo?: AlbumPhoto }
      | undefined
    if (sourceData?.type !== 'palette') return

    const photo = sourceData.photo
    if (!photo) return

    const targetLabel = String(over.id)

    setLayout((prev) => {
      if (!prev) return prev
      const newSpreads = prev.spreads.map((s, idx) =>
        idx === currentIdx
          ? { ...s, data: { ...s.data, [targetLabel]: photo.url } }
          : s,
      )
      return { ...prev, spreads: newSpreads }
    })
  }
```

## Применение и верификация в проде

### Порядок

Миграций нет. Vercel задеплоит за ~30-60 секунд.

### Проверка визуально

После деплоя открой `https://yearbook-v2.vercel.app/app/album/3fccf1dc-3e3d-402b-8c0c-7036e3496418/layout`:

**Чек-лист:**

1. **Бонус-фикс**: на canvas в учительском развороте 1 теперь должны
   появиться **реальные фото** учителей (если для них в `instance.data`
   есть рабочие URL'ы в YC). Раньше видны были только серые рамки.

2. **Бонус-фикс на миниатюрах в Обзоре**: вернись в `/app` →
   Красночетайка → Обзор. Миниатюры разворотов в `LayoutPreviewStrip`
   тоже должны теперь показывать **реальные фото** для тех ученков
   у которых рабочие URL'ы.

3. **Drag**: вернись в редактор. Тащишь миниатюру из палитры — теперь
   во время drag'а видишь «парящую» копию миниатюры с синей рамкой
   и тенью. Она рисуется в DragOverlay (Portal), поэтому **больше не
   уезжает под canvas-зону** — всегда видна сверху.

4. **Drop в placeholder** — фото **появляется в Konva-canvas'е**
   (главный фикс 2.6.3.1).

5. **Бейдж «✓ N»** на миниатюре в палитре — как и раньше.

6. **Drop вне placeholder'а** — overlay-копия исчезает, draggable
   возвращается в палитру (анимация dropAnimation от dnd-kit).

### Что **может** не работать (followup'ы)

- **На smoke-альбоме часть фото всё равно остаётся пустой** — это
  ожидаемо, потому что в БД для этого альбома есть `selections`
  указывающие на `photos` записи с битыми (несуществующими в YC)
  файлами. Проблема данных, не кода. См. диагноз 2.3.

- **DragOverlay появляется со смещением от курсора** — dnd-kit
  использует point-of-grab, должно быть нормально. Если жалобы —
  можно настроить `transform: { x: offsetX, y: offsetY }`.

- **Производительность во время drag'а** — overlay рендерит ОДИН
  `<img>` из thumb_url, должно быть мгновенно даже на больших thumbs.
