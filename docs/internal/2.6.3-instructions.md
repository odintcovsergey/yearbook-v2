# Инструкция для подэтапа 2.6.3

**Цель:** реализовать drag фотографии из палитры (`PhotoPalette`) на
photo-placeholder в canvas'е спреда. После drop'а фото визуально
вставляется в placeholder, layout мутируется в client state. Сохранения
на сервер пока **нет** — это подэтап 2.6.4.

**Зачем:** до 2.6.3 редактор был read-only — партнёр мог только смотреть
автособранный layout. С 2.6.3 он впервые получает возможность **менять
layout вручную**: «эта Архалькина не в фокусе на портрете → перетащу из
палитры её другое, более удачное».

**Контекст:** все участники готовы:
- `PhotoPalette` (2.6.2) рендерит фото-миниатюры с уникальными `photo.id`
- `AlbumSpreadCanvas` (2.2) знает координаты placeholder'ов, имеет
  prop `onDrop?: (label, photoId) => void` зарезервированный с 2.2
- `LayoutEditorPage` (2.6.1) держит `layout` в useState — мутировать
  через setLayout
- `@dnd-kit/core` уже стоит и используется в `app/app/CRMModal.tsx`
  (паттерн `useDraggable` + `useDroppable` + `DndContext` +
  `handleDragEnd`)

**Что меняется:**
- `app/app/_components/PhotoPalette.tsx` — `PhotoTile` оборачивается в
  `useDraggable` с `id=photo.id`, `data={ type: 'palette', photo }`
- `app/app/_components/AlbumSpreadCanvas.tsx` — рядом со Stage'ем
  параллельно рендерится DOM-overlay с прозрачными drop-target'ами
  для каждого `PhotoPlaceholder`. Применяется только в `mode='edit'`.
- `app/app/album/[id]/layout/page.tsx` — оборачивает всё в `<DndContext>`,
  реализует `handleDragEnd` который мутирует `layout.spreads[currentIdx].data`
- `docs/internal/2.6.3-instructions.md` — этот файл

**Что НЕ трогаем:**
- Endpoint'ы — нет
- Миграции БД — нет
- `LayoutPreviewStrip` (2.3) — drag там не нужен, остаётся read-only
  (use mode='preview', overlay не рисуется)

## Принципиальные решения

### Drop-target'ы — DOM-overlay поверх Konva

**Главный архитектурный вопрос:** как сделать Konva-placeholder
drop-target'ом для `useDroppable`?

Konva рендерит всё внутрь HTML5 `<canvas>` тегом, всё содержимое — pixels.
Konva-элементы НЕ являются DOM-узлами, и `useDroppable` их не видит.

**Решение:** параллельно со `<Stage>` рендерим **прозрачный DOM-overlay**
с `position: absolute`, в нём для каждого `PhotoPlaceholder` создаём
`<div>` с координатами и размерами совпадающими с Konva-bbox'ом. Каждый
div — отдельный `useDroppable` с `id={placeholder.label}`.

```tsx
<div className="relative">
  <Stage ...>
    <Layer>
      {/* Konva-рендер как раньше */}
    </Layer>
  </Stage>

  {mode === 'edit' && (
    <div className="absolute inset-0 pointer-events-none">
      {placeholders.filter(p => p.type === 'photo').map(p => (
        <DropZone
          key={p.label}
          placeholder={p}
          scale={scale}
        />
      ))}
    </div>
  )}
</div>
```

`pointer-events-none` на родителе overlay'а — чтобы Konva-события
(если в edit-режиме listening=true) проходили насквозь. На самих
DropZone-div'ах включается `pointer-events-auto` чтобы они принимали
drag-события.

**Координаты div'а** считаются из placeholder в мм + scale:
```tsx
style={{
  left: `${placeholder.x_mm * scale}px`,
  top: `${placeholder.y_mm * scale}px`,
  width: `${placeholder.width_mm * scale}px`,
  height: `${placeholder.height_mm * scale}px`,
}}
```

`scale` = `containerWidth / template.width_mm` — тот же что в Stage.

### Подсветка hover при isOver

`useDroppable` возвращает `isOver: boolean` когда draggable находится
над зоной. Подсвечиваем синей рамкой:

```tsx
className={`absolute pointer-events-auto transition-all ${
  isOver
    ? 'ring-2 ring-blue-500 bg-blue-50/30'
    : 'ring-1 ring-transparent'
}`}
```

`ring` (Tailwind) рисует обводку **без** изменения box-model — не
сдвинет элемент при подсветке. `bg-blue-50/30` — лёгкая полупрозрачная
заливка для понятности.

### Только photo-placeholder'ы как drop-target'ы

Text-placeholder'ы пропускаем — текст drag'ом не редактируется. Фильтр
`p.type === 'photo'` в `.map()`.

### `handleDragEnd` — single-source mutation в editor

Логика drop'а централизована в `LayoutEditorPage`. Палитра только
сообщает «начался drag фото X», canvas/overlay — «drop в placeholder Y»,
а слепить и применить мутацию — обязанность editor'а.

```typescript
function handleDragEnd(event: DragEndEvent) {
  const { active, over } = event
  if (!over) return  // drop вне drop-target'а

  const sourceData = active.data.current as { type: string; photo?: AlbumPhoto } | undefined
  if (sourceData?.type !== 'palette') return  // в 2.6.3 только палитра → canvas

  const photo = sourceData.photo
  if (!photo) return

  const targetLabel = String(over.id)

  // Mutate layout: spreads[currentIdx].data[targetLabel] = photo.url
  setLayout((prev) => {
    if (!prev) return prev
    const newSpreads = prev.spreads.map((s, idx) =>
      idx === currentIdx
        ? { ...s, data: { ...s.data, [targetLabel]: photo.url } }
        : s,
    )
    return { ...prev, spreads: newSpreads }
  })
}
```

**Immutable update** — обязательно копировать spread + data, иначе
React не зарегистрирует изменение и canvas не перерисуется.

### Что происходит с photoId фактически

В `instance.data[label]` лежит **URL фото**, не `photo.id`. Это важно
для совместимости с builder'ом: builder тоже кладёт URL'ы. Сохраняем
тот же contract.

При drop'е партнёра тоже кладём URL (`photo.url`). Бейдж «✓ N» в
палитре после этого пересчитается автоматически (useMemo пересоберёт
usageMap при изменении spreads).

### Перезаписываем без подтверждения

Если в placeholder'е уже было фото — drag нового просто заменяет
старое. Confirmation dialog не нужен — действие легко отменяется
повторным drag'ом нужного фото обратно. Альтернативно в 2.6.5+ можно
добавить undo/redo (Ctrl+Z), пока нет.

### `pointer-events-auto` для overlay-div'ов

Без этого они не получают `mousedown`. `pointer-events-none` на
родителе overlay'а пропускает события на Stage и канвас. На каждом
конкретном DropZone-div'е через `auto` локально включаем приём событий.

### Курсор при drag

Палитра: при hover на miniature `cursor-grab`, при активном drag
`cursor-grabbing`. dnd-kit сам управляет cursor'ом во время drag'а
через свои listeners — мы только ставим `cursor-grab` для idle state.

### Sensors: PointerSensor с distance constraint

Без `distance: 5px` любой клик на PhotoTile становится drag'ом —
неудобно. dnd-kit рекомендует ставить minimum 5-10px.

```typescript
const sensors = useSensors(
  useSensor(PointerSensor, { activationConstraint: { distance: 5 } })
)
```

### NO `<DragOverlay>` в 2.6.3

`<DragOverlay>` это компонент dnd-kit для рендера «парящего» элемента
во время drag'а. Без него сам draggable-элемент сдвигается через
transform. Для палитры это создаёт дёрганый UX (миниатюра «убегает»
из своего места в сетке).

**В 2.6.3 пока без DragOverlay** — миниатюра просто становится
полупрозрачной во время drag'а через `isDragging ? opacity-40 : ''`.
Это identical паттерну CRM (`app/app/CRMModal.tsx:322`).

В 2.6.4 если будут жалобы на UX — добавим DragOverlay с улучшенным
визуалом.

### Drag не должен срабатывать на клик заголовка/поиск

`useDraggable` крепится только на конкретные `PhotoTile` div'ы. Поиск
и заголовки палитры — отдельные элементы, их `useDraggable` не
касается.

### Не нужен collision detection

dnd-kit по умолчанию использует `rectIntersection`. Этого хватает —
у нас один draggable во время drag'а и непересекающиеся drop-зоны
(placeholder'ы не накладываются друг на друга).

**Не используем** `closestCenter` (как в CRM) потому что там для
вертикальных стейджей с близко стоящими карточками. У нас фото-зоны
прямоугольные с гарантированными зазорами.

```tsx
function PhotoTile({
  photo,
  usage,
}: {
  photo: AlbumPhoto
  usage: number[]
}) {
  // useDraggable: id=photo.id, data содержит сам photo чтобы handleDragEnd
  // мог сразу взять url без поиска по id.
  const { attributes, listeners, setNodeRef, transform, isDragging } =
    useDraggable({
      id: photo.id,
      data: { type: 'palette', photo },
    })

  const style = transform
    ? { transform: `translate(${transform.x}px, ${transform.y}px)`, zIndex: 50 }
    : undefined

  return (
    <div
      ref={setNodeRef}
      style={style}
      {...attributes}
      {...listeners}
      className={`relative aspect-[3/4] bg-gray-100 rounded overflow-hidden border border-gray-200 hover:ring-2 hover:ring-blue-300 transition cursor-grab active:cursor-grabbing ${
        isDragging ? 'opacity-40' : ''
      }`}
      title={photo.filename}
    >
      <img
        src={photo.thumb_url}
        alt={photo.filename}
        loading="lazy"
        draggable={false}
        className="w-full h-full object-cover pointer-events-none"
        onError={(e) => {
          ;(e.target as HTMLImageElement).style.display = 'none'
        }}
      />
      <UsageBadge usage={usage} />
      <div className="absolute bottom-0 left-0 right-0 px-1.5 py-1 text-[10px] bg-black/60 text-white opacity-0 hover:opacity-100 transition truncate pointer-events-none">
        {photo.filename}
      </div>
    </div>
  )
}
```

**Изменения:**
- `useDraggable` с `id=photo.id`, `data={ type: 'palette', photo }`
- `ref={setNodeRef}`, `style`, `{...attributes} {...listeners}` —
  стандартный паттерн dnd-kit
- `cursor-grab active:cursor-grabbing` (вместо `cursor-default`)
- `isDragging ? 'opacity-40'` для визуального feedback'а
- `<img draggable={false}>` — выключаем нативный HTML drag, конфликтующий
  с dnd-kit
- `pointer-events-none` на `<img>` и filename overlay — чтобы события
  drag'а ловились родительским div'ом, а не уходили в child'ы

#### 2б. Добавить компонент `DropZone` перед `AlbumSpreadCanvas`

Найди начало компонента `export default function AlbumSpreadCanvas`
(~строка 200). **Перед ним** добавь:

```tsx
// ─── Прозрачный drop-target над Konva-placeholder'ом (mode='edit') ───────
//
// Konva рендерит на canvas (pixels), не в DOM. @dnd-kit видит только
// DOM-узлы. Решение: рисуем поверх Stage'а абсолютно-позиционированный
// <div> для каждого photo-placeholder с теми же координатами что у
// Konva-элемента. Этот div принимает drop через useDroppable.
//
// Размеры: placeholder.x_mm * scale → пиксели в Stage'овой системе
// координат (которая равна containerWidth × containerHeight).
function DropZone({
  placeholder,
  scale,
}: {
  placeholder: PhotoPlaceholder
  scale: number
}) {
  const { setNodeRef, isOver } = useDroppable({ id: placeholder.label })
  return (
    <div
      ref={setNodeRef}
      className={`absolute pointer-events-auto transition-all ${
        isOver
          ? 'ring-2 ring-blue-500 bg-blue-100/40'
          : 'ring-1 ring-transparent hover:ring-blue-300/50'
      }`}
      style={{
        left: `${placeholder.x_mm * scale}px`,
        top: `${placeholder.y_mm * scale}px`,
        width: `${placeholder.width_mm * scale}px`,
        height: `${placeholder.height_mm * scale}px`,
      }}
    />
  )
}
```

#### 2в. Обернуть Stage в относительно-позиционированный `<div>` и добавить overlay

Найди return JSX компонента `AlbumSpreadCanvas` (~строка 215, начинается
с `return (` + `<Stage`). Сейчас просто `<Stage>...</Stage>`. Заменить
**целиком** на:

**БЫЛО:**

**СТАЛО:**

```tsx

      <Stage
        width={stageWidth}
        height={stageHeight}
        scaleX={scale}
        scaleY={scale}
```

  и сужает тип для TS

### Проверка визуально

После деплоя открой `https://yearbook-v2.vercel.app/app/album/3fccf1dc-3e3d-402b-8c0c-7036e3496418/layout`:

**Чек-лист:**

2. **Кликнуть и удерживать** на любую миниатюру в палитре, потащить —

4. **Отпустить над placeholder'ом** — фото из палитры должно появиться
   в этом placeholder'е canvas'а (заменив пустой серый слот или
   предыдущее фото)
5. **Бейдж в палитре** — на перетащенной миниатюре должен появиться
   зелёный pill «✓ N» с номером текущего разворота
6. **Перейти на другой разворот** через ◀▶ — изменения сохраняются
   в state, при возврате назад фото на месте
7. **Обновить страницу (F5)** — изменения **исчезают** (это ожидаемо,
   save в 2.6.4)
8. **Drop вне placeholder'а** — ничего не происходит, draggable
   возвращается на место в палитру

### Что **может** не работать (followup'ы)

- **Hover state DropZone не виден** — overlay рендерится но прозрачный.
  Возможная причина: `<div className="absolute inset-0">` не получает
  размеры от родителя. Проверка: в DevTools посмотреть размеры
  overlay-div'а (должны совпадать со Stage'ом).

- **Drop не срабатывает** — `useDroppable` не подцепился к div'у.
  Проверка: console.log в `handleDragEnd` — приходит ли event с
  `over !== null`?

- **Перетаскивание ломает скролл палитры** — `activationConstraint:
  { distance: 5 }` должен это решать, но если жалобы — увеличить
  до 10.

- **Производительность при наличии 30 placeholder'ов на разворте** —
  30 useDroppable не должно влиять на FPS, но если будут жалобы —
  оптимизация в 2.6.5.

### Что **не** проверяем в 2.6.3

- Save на сервер — это 2.6.4. Изменения после F5 теряются. Это
  ожидаемо.
- Swap между placeholder'ами — это 2.6.4
- has_user_edits в БД — это часть save'а, тоже 2.6.4
- Beforeunload подтверждение «несохранённые изменения» — 2.6.4
