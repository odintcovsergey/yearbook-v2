# Инструкция для подэтапа 2.4

**Цель:** добавить endpoint `GET /api/tenant?action=album_photos` —
универсальный список всех фото альбома для палитры будущего редактора.
Объединяет два источника: `photos` (фото с выбором родителей через
`selections`) и `original_photos` (оригиналы фотографа из workflow).
Для каждого фото — флаг источника, привязка к детям/учителям, список
типов селекшенов.

**Зачем:** в подэтапе 2.6 будет fullscreen-редактор с палитрой
фотографий. Партнёр сможет drag-and-drop'ом заменить фото в layout'е.
Палитре нужен ОДИН источник всех фото альбома с готовыми URL'ами и
метаданными — endpoint и есть этот источник.

В фазе 2.4 делаем **только endpoint** (бэкенд + JSON). UI палитры
(`PhotoPalette` компонент) добавится в подэтапе 2.6 вместе с самим
редактором. В 2.4 проверяем endpoint через DevTools fetch — без UI.

**Контекст:** в `app/api/tenant/route.ts` уже есть action `photos`
(строки 497-545) — он отдаёт только записи из `photos` table,
сгруппированные по детям через `photo_children`. Этого недостаточно
для палитры:
- Не отдаёт originals (оригиналы фотографа из `original_photos`)
- Не отдаёт `selection_types` (где это фото фигурирует в выборах
  родителей: portrait_page / portrait_cover / group)
- Не отдаёт `teacher_ids` (привязка к учителям через `photo_teachers`)

Существующий action `photos` НЕ трогаем — он используется во вкладке
«Фото» в /app, ломать обратную совместимость нельзя. Делаем
**параллельный** action `album_photos`.

**Что меняется:**
- `app/api/tenant/route.ts` — новый блок `if (action === 'album_photos')`
  внутри GET-ветки роута, рядом с существующим `action === 'photos'`
- `docs/internal/2.4-instructions.md` — этот файл

**Что НЕ трогаем:**
- Существующий `action === 'photos'` — без изменений
- Любой другой код кабинета или builder'а
- Миграции БД — все нужные таблицы (`photos`, `original_photos`,
  `photo_children`, `photo_teachers`, `selections`) уже существуют
- UI — компонент палитры будет в 2.6

## Принципиальные решения

### Отступление от phase-2-spec §4.1: `child_ids[]` вместо `child_id`

Спека phase-2-spec предполагает singular `child_id: string | null` для
каждого фото. После изучения схемы понятно, что это упрощение —
`photo_children` это связь many-to-many: одно групповое фото может
быть привязано к нескольким ученикам, и наоборот один ученик может
иметь несколько фото-привязок.

Возвращаем `child_ids: string[]` (пустой массив = нет привязки). UI
палитры в 2.6 сможет:
- Для portrait: показать фото в группе любого из linked-учеников
  (обычно `child_ids` это один ученик)
- Для group: показать в общей палитре или сгруппировать по любому из
  child_ids (партнёр выбирает)

Аналогично `teacher_ids: string[]` для учительских фото через
`photo_teachers`.

Спека phase-2-spec будет уточнена в 2.8 при обновлении контекста v44.

### Не дублирую фото в response для каждой связи

Альтернативный путь: если фото связано с тремя учениками — вернуть
три записи фото в массиве (по одной на каждого linked-ребёнка) и
плоский `child_id: string`. Не делаем — это усложняет dedupe в UI
(одно и то же фото id будет повторяться) и раздувает payload.
Один объект на одно фото, привязки — массив.

### Source = 'selections' | 'originals'

Два источника:
1. `photos` table — фото загруженные через UI родителем или партнёром
   (с типизацией portrait/group/teacher, селекшенами, привязками)
2. `original_photos` table — оригиналы фотографа загруженные через
   workflow (без типа, без селекшенов, без привязок — просто файлы)

В response единый массив, различимый по полю `source`. Фронт сам
решает как показывать (например, отдельный таб «Оригиналы» или
переключатель).

Семантика:
- `source: 'selections'` — фото из `photos`, у него есть `type`,
  `child_ids`, `teacher_ids`, `selection_types`
- `source: 'originals'` — фото из `original_photos`, у него
  `type: null`, `child_ids: []`, `teacher_ids: []`,
  `selection_types: []`

### `type: null` для originals (не выдумываем 'original')

Существующая колонка `photos.type` имеет CHECK-constraint на значения
`'portrait' | 'group' | 'teacher'`. Я **не** добавляю `'original'` —
потому что originals не из `photos`, они вообще из другой таблицы.
В response для originals `type: null`. UI в 2.6 будет различать
через `source`, не через `type`.

### URL и thumb_url — формируем на сервере

Возвращаем готовые URL'ы (`url`, `thumb_url` для UI). Это переиспользует
паттерн существующего `action=photos`:

```typescript
url: getPhotoUrl(p.storage_path),
thumb_url: getThumbUrl(p.storage_path, p.thumb_path),
```

Для originals `thumb_path` нет — там `getThumbUrl` фолбекнется на
`getPhotoUrl(storage_path)`. Это нормально для оригиналов (партнёр
обычно загружает не очень тяжёлые JPG, миниатюра не нужна).

### Selection_types — distinct по типу

В `selections` table одна запись = один selection_type для одного
photo одним child. Если три ребёнка выбрали одно и то же фото как
`portrait_page`, в response будет `selection_types: ['portrait_page']`
(один раз, через `Set` dedupe).

Это справедливо для всех типов: фото фигурирует «в каком-то
селекшене типа X», без подсчёта количества. Для подсчёта (если
понадобится в 2.6) UI может посчитать самостоятельно через
дополнительный запрос — но это вряд ли нужно.

### Авторизация — тот же паттерн что у `action=photos`

Используем существующий `assertAlbumAccess(auth, albumId, tid)` где
`tid` уже учитывает view_as. Доступ — owner/manager/viewer тенанта-
владельца альбома + staff с view_as + superadmin.

### Параллельные SELECT'ы через Promise.all

Endpoint делает 5 запросов: photos, original_photos, photo_children,
photo_teachers, selections. Все можно параллелить — они read-only
из независимых таблиц. Используем `Promise.all` (паттерн уже есть
в `lib/smart-fill/build-album-input.ts`).

### Audit log — не пишем

Read-операции на уровне API без audit_log — это договорённость
проекта (см. `app/api/layout/route.ts:117` в комментарии к GET
endpoint'у). album_photos — read-only, audit не нужен.

## Шаги

### Шаг 1. Добавить action `album_photos` в `app/api/tenant/route.ts`

Найди существующий блок `if (action === 'photos' && albumId)` (~строки
497-545). После его закрывающей фигурной скобки **добавь** новый
блок. **Не модифицируй** существующий photos.

```typescript
// ----------------------------------------------------------
// album_photos — единый список всех фото альбома для палитры
// будущего редактора (фаза 2.4 / phase-2-spec §4.1).
//
// Объединяет два источника:
//   - photos (с типом portrait/group/teacher, привязками, селекшенами)
//   - original_photos (оригиналы фотографа из workflow)
//
// Для photos подтягивает:
//   - child_ids[] из photo_children
//   - teacher_ids[] из photo_teachers
//   - selection_types[] из selections (distinct)
//
// Для original_photos все привязки/селекшены пусты, type=null.
// ----------------------------------------------------------
if (action === 'album_photos' && albumId) {
  if (!(await assertAlbumAccess(auth, albumId, tid))) {
    return NextResponse.json({ error: 'Альбом не найден' }, { status: 404 })
  }

  // 1. Параллельная загрузка из всех источников.
  const [photosRes, originalsRes] = await Promise.all([
    supabaseAdmin
      .from('photos')
      .select('id, filename, storage_path, thumb_path, type, created_at')
      .eq('album_id', albumId)
      .order('created_at'),
    supabaseAdmin
      .from('original_photos')
      .select('id, filename, storage_path, file_size, created_at')
      .eq('album_id', albumId)
      .order('created_at'),
  ])

  if (photosRes.error) {
    return NextResponse.json({ error: photosRes.error.message }, { status: 500 })
  }
  if (originalsRes.error) {
    return NextResponse.json({ error: originalsRes.error.message }, { status: 500 })
  }

  const photos = photosRes.data ?? []
  const originals = originalsRes.data ?? []
  const photoIds = photos.map((p: any) => p.id)

  // 2. Подгрузить связи и селекшены параллельно (только если есть photos).
  const [childLinksRes, teacherLinksRes, selectionsRes] = photoIds.length > 0
    ? await Promise.all([
        supabaseAdmin
          .from('photo_children')
          .select('photo_id, child_id')
          .in('photo_id', photoIds),
        supabaseAdmin
          .from('photo_teachers')
          .select('photo_id, teacher_id')
          .in('photo_id', photoIds),
        supabaseAdmin
          .from('selections')
          .select('photo_id, selection_type')
          .in('photo_id', photoIds),
      ])
    : [
        { data: [], error: null },
        { data: [], error: null },
        { data: [], error: null },
      ]

  // Ошибки джоинов не валим запрос — просто отдаём фото с пустыми связями.
  // Если что-то пошло не так — это видно в логах Vercel; UI сможет
  // показать фото без привязок (читабельно деградировать).

  // 3. Группировка связей по photo_id.
  const childIdsByPhoto: Record<string, string[]> = {}
  for (const link of childLinksRes.data ?? []) {
    const pid = (link as any).photo_id as string
    const cid = (link as any).child_id as string
    if (!childIdsByPhoto[pid]) childIdsByPhoto[pid] = []
    childIdsByPhoto[pid].push(cid)
  }

  const teacherIdsByPhoto: Record<string, string[]> = {}
  for (const link of teacherLinksRes.data ?? []) {
    const pid = (link as any).photo_id as string
    const tidRow = (link as any).teacher_id as string
    if (!teacherIdsByPhoto[pid]) teacherIdsByPhoto[pid] = []
    teacherIdsByPhoto[pid].push(tidRow)
  }

  // selection_types: distinct через Set, потом обратно в массив.
  const selectionTypesByPhoto: Record<string, Set<string>> = {}
  for (const sel of selectionsRes.data ?? []) {
    const pid = (sel as any).photo_id as string
    const stype = (sel as any).selection_type as string
    if (!selectionTypesByPhoto[pid]) selectionTypesByPhoto[pid] = new Set()
    selectionTypesByPhoto[pid].add(stype)
  }

  // 4. Сборка результата: photos + originals в одном массиве.
  const fromPhotos = photos.map((p: any) => ({
    id: p.id,
    filename: p.filename,
    storage_path: p.storage_path,
    thumb_path: p.thumb_path,
    type: p.type as 'portrait' | 'group' | 'teacher',
    source: 'selections' as const,
    child_ids: childIdsByPhoto[p.id] ?? [],
    teacher_ids: teacherIdsByPhoto[p.id] ?? [],
    selection_types: Array.from(selectionTypesByPhoto[p.id] ?? []),
    url: getPhotoUrl(p.storage_path),
    thumb_url: getThumbUrl(p.storage_path, p.thumb_path),
    created_at: p.created_at,
  }))

  const fromOriginals = originals.map((o: any) => ({
    id: o.id,
    filename: o.filename,
    storage_path: o.storage_path,
    thumb_path: null,
    type: null,
    source: 'originals' as const,
    child_ids: [] as string[],
    teacher_ids: [] as string[],
    selection_types: [] as string[],
    url: getPhotoUrl(o.storage_path),
    thumb_url: getPhotoUrl(o.storage_path),  // у originals нет thumb_path
    created_at: o.created_at,
  }))

  return NextResponse.json({
    photos: [...fromPhotos, ...fromOriginals],
  })
}
```

**Замечания по реализации:**

1. **Деструктуризация массива результатов `Promise.all`** — возвращаем
   литерал из объектов с `data: []` если `photoIds.length === 0`.
   Это упрощает дальнейший код (всегда работаем с массивами).

2. **`as any` для строк из Supabase** — Supabase TypeScript-types для
   динамических join'ов слабые. Существующий код в этом файле
   повсюду использует `(row as any).field` — следуем стилю.

3. **Distinct selection_types через `Set`** — гарантирует что
   `['portrait_page', 'portrait_cover']` не превратится в
   `['portrait_page', 'portrait_page', 'portrait_cover']` если
   несколько детей выбрали как portrait_page.

4. **`type` для originals = `null`** — TypeScript-narrowing в UI
   палитры будет через discriminated union по `source`, не по `type`.

5. **`thumb_url` для originals = `getPhotoUrl(storage_path)`** —
   fallback на полное изображение. Для оригиналов это нормально
   (партнёр загружает разумного размера JPG).

## Применение и верификация в проде

### Порядок

Миграций нет. После push Vercel задеплоит за ~30-60 секунд.

### Проверка через DevTools

UI пока не использует endpoint, поэтому проверяем напрямую через
консоль браузера.

После деплоя открыть `/app` (любая страница, главное чтобы был
авторизованный cookie), затем DevTools → Console:

```js
fetch('/api/tenant?action=album_photos&album_id=3fccf1dc-3e3d-402b-8c0c-7036e3496418', {
  credentials: 'include'
})
  .then(r => r.json())
  .then(d => {
    console.log('TOTAL:', d.photos.length)
    console.log('BY SOURCE:', d.photos.reduce((acc, p) => {
      acc[p.source] = (acc[p.source] || 0) + 1
      return acc
    }, {}))
    console.log('BY TYPE:', d.photos.filter(p => p.source === 'selections').reduce((acc, p) => {
      acc[p.type] = (acc[p.type] || 0) + 1
      return acc
    }, {}))
    console.log('SAMPLE:', d.photos[0])
    console.log('SAMPLE WITH SELECTIONS:', d.photos.find(p => p.selection_types.length > 0))
    console.log('SAMPLE TEACHER:', d.photos.find(p => p.type === 'teacher'))
  })
```

Ожидание (для альбома Красночетайская СОШ):
- **TOTAL** ≈ 163 photos + N originals (originals может быть 0 если
  партнёр их не загружал)
- **BY SOURCE**: `{ selections: 163, originals: <N> }`
- **BY TYPE** для selections: `{ portrait: 163, ... }` или с разбивкой
  если есть group/teacher
- **SAMPLE** — объект с `source`, `type`, `url`, `thumb_url`, и т.д.
- **SAMPLE WITH SELECTIONS** — фото у которого `selection_types`
  непустой (= кто-то из родителей его выбрал)
- **SAMPLE TEACHER** — учительское фото с `teacher_ids` непустым

### Проверка для тенанта без доступа

// Должен вернуть 404 «Альбом не найден»

## Что НЕ делаем в 2.4

- **`PhotoPalette` компонент** — это подэтап 2.6 (вместе с редактором)
- **Фильтрация по `type` через query param** — в 2.6 палитра сама
  фильтрует на клиенте; на сервере не нужно
- **Пагинация** — для альбомов с <500 фото не нужна, payload в JSON
  будет ~150KB, приемлемо
- **`POST /api/layout?action=save_album_layout`** — это подэтап 2.5
- **Endpoint для удаления/добавления привязок** — палитра в 2.6
  работает read-only, drag'n'drop меняет только layout, не привязки
  фото к детям
- **Контекст-файл v43 → v44** — обновляется в 2.8

## После 2.4 — что разблокируется

После 2.4 у нас:
- Endpoint `GET /api/tenant?action=album_photos` готов и проверен
- UI палитры может быть собран с реальными данными в 2.6

Это разблокирует подэтап 2.5 — endpoint `POST /api/layout?action=save_album_layout`
для сохранения правок партнёра. Они независимы (можно делать в любом
порядке), но 2.5 короче (~30 минут).

После 2.5 идём в 2.6 — fullscreen-редактор, где впервые соединяются
все компоненты: миниатюры, палитра, drag'n'drop, auto-save.
