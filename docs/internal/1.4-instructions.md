# Инструкция для подэтапа 1.4

**Цель:** добавить в кабинет партнёра (`/app` → AlbumDetailModal → Обзор)
кнопку «Собрать автоматически», запускающую endpoint 1.3, и result-блок
показывающий итог (количество разворотов, категоризированные warning'и,
JSON для копирования). Persisted state — при открытии Обзора подгружать
existing layout из БД, если уже есть.

**Зачем:** без UI partner-фотограф не может запустить smart-fill из
кабинета. После 1.4 функция доступна в продуктиве, остаётся только
end-to-end smoke в 1.5.

**Что меняется:**
- `app/api/layout/route.ts` — новый GET action `album_layout` для
  загрузки existing layout при открытии модала
- `app/app/page.tsx`:
  - `AlbumDetailModal` — state для layout/busy/error
  - `load()` — параллельный запрос existing layout
  - блок «Пресет вёрстки» в Обзоре расширен кнопкой «Собрать
    автоматически» (если есть пресет) + result-блок (если есть layout)
  - функция `runSmartFill` (POST /api/layout?action=build_album)
  - функция `copyLayoutJson`
  - вспомогательный компонент `CollapseSection` для warning'ов
- (по желанию) расширение типа `Layout` для frontend

**Что НЕ трогаем:**
- buildAlbum, smart-fill, endpoint POST `build_album` — стабильны
- Другие табы (Ученики, Учителя, Фото, etc.) — не касаемся
- Другие страницы (`/super`, `/[token]`, etc.)

## Принципиальные решения

### Загрузка existing layout — отдельный GET endpoint

Добавляем `GET /api/layout?action=album_layout&album_id=X`. Возвращает
тот же формат что POST `build_album` (spreads, warnings, layout_id,
summary), либо `{ layout: null }` если записи нет.

Альтернатива (передавать в `dashboard` GET tenant API) — нет, layout
не имеет отношения к tenant, лежит в layout API.

### Состояние result-блока

Один объект `layout` или `null`:

```typescript
type LayoutData = {
  layout_id: string;
  spreads: SpreadInstance[];
  warnings: EnrichedWarning[];
  summary: {
    total_spreads: number;
    total_warnings: number;
    warnings_by_level: { blocking: number; degraded: number; info: number };
    preset_slug: string;
    preset_name: string;
  };
};

const [layout, setLayout] = useState<LayoutData | null>(null);
const [busy, setBusy] = useState(false);
```

`null` — нет существующего layout, показываем только кнопку «Собрать».
`LayoutData` — показываем кнопку «Пересобрать» + result-блок.

### Кнопка disabled когда нет пресета

Спека: «Только при `config_preset_id IS NULL`». В коде `album.config_preset_id`
nullable. Пока пресета нет — кнопка disabled с tooltip. Это естественный
guard — endpoint 1.3 в любом случае вернёт 400, но проверка на клиенте
улучшает UX.

### Загрузка existing layout — НЕ блокирует остальной модал

В `load()` Promise.all с остальными запросами. Если layout endpoint
упадёт — loading state остальных вкладок не сломается. Layout запрос
оборачиваем в `.catch(() => null)` — fallback на null.

### busy на кнопке + блокировка повторных запусков

`busy=true` пока POST в процессе. Кнопка disabled, текст «Сборка...».
По завершении — `busy=false`, обновляем `layout`. Race-condition
защита.

### Категоризированные warning'и через native `<details>`

Без JS, без анимаций. Native HTML — клик по `<summary>` разворачивает
секцию. По умолчанию свёрнуты. Цветовая дифференциация по уровню:
- blocking — красный фон/текст
- degraded — янтарный
- info — серый

Если у уровня 0 warning'ов — секция вообще не рендерится.

### «Скопировать JSON»

`navigator.clipboard.writeText(JSON.stringify(layout, null, 2))`. Доступен
на HTTPS (Vercel) и localhost. Подсветить успех тостом
`onNotify('Layout JSON скопирован')`.

### «Пересобрать» = тот же runSmartFill

Не отдельная функция. Просто повторный POST. Endpoint 1.3 делает upsert,
существующая запись обновляется. UX: «Пересобрать» с busy → новый
layout приходит → state перезаписывается.

### viewAsTenantId в новом endpoint

Существующий POST `build_album` поддерживает `?view_as=...` (1.3).
Новый GET `album_layout` тоже должен поддерживать — для OkeyBook
staff просматривающих partner альбомы. Тот же паттерн что в POST.

### Не показываем кнопку viewer'у партнёра

`canEdit` уже правильно фильтрует роли (owner/manager). У viewer'а
кнопка не появится. OkeyBook staff в режиме view_as имеет
canEdit=true (см. AlbumDetailModal pattern в существующем коде).

### Тип EnrichedWarning на клиенте

Дублируем структуру:
```typescript
type WarningLevel = 'blocking' | 'degraded' | 'info';
type EnrichedWarning = {
  code: string;
  detail: string;
  level: WarningLevel;
  source: 'builder' | 'smart_fill';
};
```

Альтернатива (импортировать из layout/route.ts) — нет, route handlers
не должны экспортироваться в клиентский код. Это ~5 строк, дублируем.

### Размер layout JSON — не оптимизируем

11-15 разворотов × десятки полей `data` ≈ 50-200 КБ. Передаём целиком
через REST. Если станет проблемой (фаза 2 Canvas — там придётся думать
про lazy load) — оптимизируем тогда. Сейчас держать всё в state модала —
ок.

#### 1.2. Реализовать `handleGetAlbumLayout`

В конец файла (после `handleBuildAlbum`) добавить:

```typescript
/**
 * GET /api/layout?action=album_layout&album_id=<uuid>
 *
 * Загружает existing layout из album_layouts. Используется в UI 1.4 для
 * persisted state — при открытии Обзора в AlbumDetailModal.
 *
 * Возвращает:
 *   { layout: null } — записи нет (alboum ещё не собирали)
 *   { layout: { layout_id, spreads, warnings, summary } } — есть запись
 *
 * Доступ: тот же что у build_album (owner/manager/viewer + view_as).
 */
async function handleGetAlbumLayout(
  req: NextRequest,
  auth: AuthContext,
): Promise<NextResponse> {
  const albumId = req.nextUrl.searchParams.get('album_id')
  if (!albumId || !UUID_REGEX.test(albumId)) {
    return NextResponse.json(
      { error: 'album_id is required (uuid)' },
      { status: 400 },
    )
  }

  // Tenant scoping (тот же паттерн что в handleBuildAlbum)
  const viewAsTenantId = req.nextUrl.searchParams.get('view_as')
  const { data: currentTenantData } = viewAsTenantId
    ? await supabaseAdmin.from('tenants').select('slug').eq('id', auth.tenantId).single()
    : { data: null }
  const canViewAs = auth.role === 'superadmin' || currentTenantData?.slug === 'main'
  const tid = (canViewAs && viewAsTenantId) ? viewAsTenantId : auth.tenantId

  if (!(await assertAlbumAccessLocal(auth, albumId, tid))) {
    return NextResponse.json({ error: 'access denied' }, { status: 403 })
  }

  // Загрузить layout. Joinим config_presets для получения preset_slug/name
  // (нужно для UI summary).
  const { data: layoutRow, error } = await supabaseAdmin
    .from('album_layouts')
    .select(`
      id, spreads, warnings,
      config_presets ( slug, name )
    `)
    .eq('album_id', albumId)
    .maybeSingle()

  if (error) {
    return NextResponse.json(
      { error: `album_layouts load failed: ${error.message}` },
      { status: 500 },
    )
  }

  if (!layoutRow) {
    return NextResponse.json({ layout: null })
  }

  const spreads = (layoutRow.spreads ?? []) as Array<Record<string, unknown>>
  const warnings = (layoutRow.warnings ?? []) as Array<{
    code: string
    detail: string
    level: 'blocking' | 'degraded' | 'info'
    source: 'builder' | 'smart_fill'
  }>

  const warningsByLevel = {
    blocking: warnings.filter((w) => w.level === 'blocking').length,
    degraded: warnings.filter((w) => w.level === 'degraded').length,
    info: warnings.filter((w) => w.level === 'info').length,
  }

  const presetData = (layoutRow as any).config_presets as { slug?: string; name?: string } | null

  return NextResponse.json({
    layout: {
      layout_id: layoutRow.id,
      spreads,
      warnings,
      summary: {
        total_spreads: spreads.length,
        total_warnings: warnings.length,
        warnings_by_level: warningsByLevel,
        preset_slug: presetData?.slug ?? null,
        preset_name: presetData?.name ?? null,
      },
    },
  })
}
```

### Шаг 3. UI — функции runSmartFill и copyLayoutJson

const runSmartFill = async () => {
  if (!album.config_preset_id || smartFillBusy) return
  setSmartFillBusy(true)
  try {
    const r = await apiVA('/api/layout?action=build_album', {
      method: 'POST',
      body: JSON.stringify({ album_id: album.id }),
    })
    if (r.ok) {
      const data = await r.json()
      setLayout({
        layout_id: data.layout_id,
        spreads: data.spreads,
        warnings: data.warnings,
        summary: data.summary,
      })
      onNotify(`Layout собран: ${data.summary.total_spreads} разворотов, ${data.summary.total_warnings} предупреждений`)
    } else {
      const d = await r.json().catch(() => ({}))
      onError(d.error ?? 'Сборка не удалась')
    }
  } catch (e) {
    onError('Ошибка сети при сборке')
  } finally {
    setSmartFillBusy(false)
  }
}

const copyLayoutJson = async () => {
  if (!layout) return
  try {
    await navigator.clipboard.writeText(JSON.stringify(layout, null, 2))
    onNotify('Layout JSON скопирован')
  } catch {
    onError('Не удалось скопировать (нет доступа к буферу обмена)')
  }
}

### Шаг 5. UI — компонент `CollapseSection`

В `app/app/page.tsx` где определены другие вспомогательные компоненты
(например, `MiniStat`, посмотреть рядом). Если такого места нет — добавить

```tsx
// ─── Категоризированный warning блок (фаза 1.4) ────────────────────────────
function CollapseSection({
  level,
  warnings,
}: {
  level: WarningLevel
  warnings: EnrichedWarning[]
}) {
  if (warnings.length === 0) return null

  const config = {
    blocking: {
      label: 'Критично',
      classes: 'bg-red-50 text-red-700 border-red-200',
      summaryClasses: 'text-red-700',
    },
    degraded: {
      label: 'Требует внимания',
      classes: 'bg-amber-50 text-amber-700 border-amber-200',
      summaryClasses: 'text-amber-700',
    },
    info: {
      label: 'К сведению',
      classes: 'bg-gray-50 text-gray-700 border-gray-200',
      summaryClasses: 'text-gray-700',
    },
  }[level]

  return (
    <details className={`rounded border ${config.classes}`}>
      <summary className={`px-3 py-1.5 text-sm cursor-pointer select-none ${config.summaryClasses}`}>
        {config.label} ({warnings.length})
      </summary>
      <ul className="px-3 py-2 text-xs space-y-1">
        {warnings.map((w, i) => (
          <li key={`${w.code}-${i}`} className="leading-relaxed">
            <span className="font-mono opacity-70">[{w.code}]</span>{' '}
            <span>{w.detail}</span>
          </li>
        ))}
      </ul>
    </details>
  )
}
```

## Открытые риски

### Большой layout JSON в state

11-15 разворотов × ~5 КБ JSON каждый ≈ 60-150 КБ в state модала.
Не критично для современных браузеров, но если попадёт альбом
с 100+ разворотами (фаза 4 многоальбомная сборка) — нужно будет
lazy load. Сейчас не оптимизируем.

### `navigator.clipboard.writeText` недоступен на HTTP

Vercel прод — HTTPS, всё ок. Локальная dev (`localhost:3000`) — тоже
работает (browser whitelist). На IP-адресах HTTP — упадёт. Catch
обрабатывает.

### `<details>` без анимации

UX — неидеален (snap), но в MVP норм. Фаза 4 переделать на radix-ui
или подобное.

### preset_slug/name загружается через nested join

В endpoint'е `album_layout` идёт SELECT с join `config_presets(slug, name)`.
Если у layout'а `config_preset_id IS NULL` (теоретически возможно если
вручную писали в БД) — join вернёт null, в response `preset_slug=null,
preset_name=null`. UI просто покажет meta без названия пресета. Не
критично, но в проде такого быть не должно.
