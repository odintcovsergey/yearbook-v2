# Инструкция для подэтапа 2.6.1.1

**Цель:** косметический патч после визуальной проверки 2.6.1. Три
точечных фикса в одном коммите.

**Зачем:** на проверке 2.6.1 партнёр обнаружил три бага:

1. **Заголовок «Альбом»** вместо реального названия. В инструкции я
   ошибочно написал action `albums` (plural), а правильный action в
   `/api/tenant` — `album` (singular). Endpoint вернул то ли 400 то ли
   что-то неожиданное → `title` остался пустым строкой → fallback на
   слово «Альбом».

2. **Двустраничный разворот меньше одностраничного.** Захардкоженная
   `containerWidth = 800` фиксирует **ширину** Stage'а независимо от
   аспекта. Одностраничный 226×226 → 800×800 (квадрат). Двустраничный
   452×226 → 800×400 (ландшафт сжат в 2 раза по высоте). Визуально
   двустраничный кажется **в 2 раза меньше** одностраничного — это
   обратное от ожидаемого.

3. **Текст имён «Осипов Роман» / «Савельева Анжелика» нечитаем** на
   двустраничном развороте. Они рендерятся поверх пустого photo-слота
   `fill="#f3f4f6"` (заливка из 2.3.1), сам текст светло-серый
   (исходные `placeholder.color` из IDML, типа `#aaaaaa`). Серое на
   сером — низкий контраст. На миниатюрах LayoutPreviewStrip это
   незаметно (size мелкий), на большом редакторском canvas'е —
   неприемлемо.

**Контекст:** все три фикса — точечные, без архитектурных изменений.
Изменения только в редакторе и в `AlbumSpreadCanvas` (общий компонент).
Проверки `LayoutPreviewStrip` (миниатюры в Обзоре) тоже улучшатся
от фикса 3.

**Что меняется:**
- `app/app/album/[id]/layout/page.tsx`:
  1. Action `albums` → `album` в URL запроса (фикс 1)
  2. `containerWidth = 800` → динамический расчёт по высоте viewport'а
     с учётом аспекта шаблона (фикс 2)
- `app/app/_components/AlbumSpreadCanvas.tsx`:
  3. `TextSlot` — fallback color `#000000` если `placeholder.color`
     пустой/прозрачный/светлее threshold'а (фикс 3)
- `docs/internal/2.6.1.1-instructions.md` — этот файл

**Что НЕ трогаем:**
- Список endpoint'ов и их response структуру — все остальные действуют корректно
- `LayoutPreviewStrip` — не нужно править отдельно, фикс 3 в `AlbumSpreadCanvas`
  улучшит и его
- 2.6.2+ функциональность не трогаем

## Принципиальные решения

### Фикс 1: правильное имя action — `album`

В `app/api/tenant/route.ts:257`:
```typescript
if (action === 'album' && albumId) { ... return NextResponse.json(album) }
```

Возвращает плоский объект `{ id, title, ... }`. Уже сейчас в коде
редактора есть `Array.isArray` ветка — она остаётся как defensive
защита (не повредит). Меняем только URL.

### Фикс 2: containerWidth по высоте viewport'а

Идея — как в `LayoutPreviewStrip` (2.3): фиксируем **высоту** доступной
области, ширина = `(width_mm / height_mm) * height`. Тогда оба типа
spread'ов имеют одинаковую высоту, ландшафтные — шире, квадратные —
уже.

```typescript
// Доступная высота: viewport минус header (~60px), навигация (~60px),
// padding'и (~80px). Запас 200px. Берём 70vh как комфортный размер.
const targetHeight = Math.max(400, window.innerHeight * 0.7)
const aspectRatio = currentTemplate.width_mm / currentTemplate.height_mm
const containerWidth = targetHeight * aspectRatio
```

Защита `Math.max(400, ...)` — чтобы на крошечных окнах (мобильник в
горизонтальной альбомной ориентации) не получить совсем мизерные
размеры.

**Граничный случай — двустраничный на узком экране:**
если `containerWidth > availableWidth` — spread будет торчать за
пределы левой колонки. На стандартных ноутбучных экранах (1440px)
с палитрой 30% (≈430px) на canvas остаётся ~1000px. Двустраничный
452×226 при targetHeight=600 даст containerWidth = 600 × (452/226)
= 1200px — **больше** доступного 1000px.

Решение: ограничить `containerWidth` по доступной ширине, и если
ширина оказывается limiting factor — пересчитать высоту:

```typescript
// Доступная область: 70% ширины окна (палитра 30%) минус padding.
const availableWidth = Math.max(400, window.innerWidth * 0.7 - 80)
const availableHeight = Math.max(400, window.innerHeight * 0.7)

const aspectRatio = currentTemplate.width_mm / currentTemplate.height_mm
// Кандидат по высоте: фиксируем height, считаем width.
const widthByHeight = availableHeight * aspectRatio
// Если кандидат не помещается по ширине — фиксируем width вместо высоты.
const containerWidth = Math.min(widthByHeight, availableWidth)
```

Stage в `AlbumSpreadCanvas` сам считает scale = `containerWidth /
template.width_mm` и применит к высоте — то есть рендер всегда
пропорционален. Достаточно отдать только `containerWidth`.

**SSR safety:** `window.innerWidth` доступно только на клиенте.
Страница уже `'use client'`, и расчёт идёт **внутри render function**.
Но при первом render (до useEffect'а) на стороне Next.js компиляции
`window` не существует — нужно отлавливать через `typeof window !==
'undefined'` либо через useState + useEffect.

Проще через useState с defaults для SSR:

```typescript
const [viewport, setViewport] = useState({ width: 1440, height: 900 })

useEffect(() => {
  const update = () => setViewport({
    width: window.innerWidth,
    height: window.innerHeight,
  })
  update()  // первоначальный замер
  window.addEventListener('resize', update)
  return () => window.removeEventListener('resize', update)
}, [])
```

Дополнительный bonus: при resize окна спред будет адаптироваться. Это
именно то что нужно.

### Фикс 3: дефолтный цвет текста — чёрный, если placeholder.color светлый

Проблема: некоторые `placeholder.color` из IDML — светло-серые
(`#cccccc`, `#aaaaaa`), они задумывались для специальных эффектов на
тёмных фонах оригинального дизайна. На нашем сером пустом слоте
`#f3f4f6` они становятся невидимыми.

**Простое решение:** в `TextSlot`, если `placeholder.color` пустой
или **слишком светлый** — использовать `#000000`. Threshold'ом для
«светлый» считать «luminance > 0.7».

```typescript
function isTooLight(hex: string | null | undefined): boolean {
  if (!hex) return true
  // hex может быть #RGB или #RRGGBB
  const m = hex.replace('#', '')
  if (m.length !== 3 && m.length !== 6) return true
  const expand = m.length === 3
    ? m.split('').map((c) => c + c).join('')
    : m
  const r = parseInt(expand.slice(0, 2), 16) / 255
  const g = parseInt(expand.slice(2, 4), 16) / 255
  const b = parseInt(expand.slice(4, 6), 16) / 255
  // Простой luminance (без gamma correction — для нашей задачи достаточно)
  const luminance = 0.299 * r + 0.587 * g + 0.114 * b
  return luminance > 0.7
}

// Использование:
const fillColor = isTooLight(placeholder.color) ? '#000000' : placeholder.color
```

**Не делаем:** автоопределение цвета фона за placeholder'ом и
выбор контрастного. Слишком сложно; на большинстве шаблонов фон
белый или светлый, чёрный текст всегда читаем.

**Альтернатива which I rejected:** хардкодить чёрный для всех
placeholder'ов и игнорировать `placeholder.color` целиком. Это
сломает дизайнерские шаблоны где text цветной по задумке (например,
красные подзаголовки на белом фоне). Не делаем.

### Шаг 2. Фикс 2 — динамический containerWidth

#### 2а. Добавить state для viewport

Найди блок `const [error, setError] = useState<string | null>(null)`
и **после** него добавь:

```typescript
  const [viewport, setViewport] = useState({ width: 1440, height: 900 })
```

#### 2б. Заменить захардкоженный containerWidth

Найди строку:

```typescript
  const canvasContainerWidth = 800
```

Замени на:

```typescript
  // Динамический расчёт canvas: вписываем spread в доступное пространство
  // с сохранением аспекта. Если по ширине шире чем доступно (двустраничный
  // на узком окне) — limiting factor становится ширина.
  const availableWidth = Math.max(400, viewport.width * 0.7 - 80)
  const availableHeight = Math.max(400, viewport.height * 0.7)
  const aspectRatio = currentTemplate
    ? currentTemplate.width_mm / currentTemplate.height_mm
    : 1
  const widthByHeight = availableHeight * aspectRatio
  const canvasContainerWidth = Math.min(widthByHeight, availableWidth)
```

**Важно:** `currentTemplate` объявляется в коде ниже (после fallback
`if (!layout) return null`). Я перенесу расчёт **после** объявления
`currentTemplate`. То есть переменную `canvasContainerWidth` объявить
там же где сейчас (~строка 156 в твоём коде, ~между объявлением
currentTemplate и return блоком).

Проверь: `currentTemplate` имеет тип `SpreadTemplate | undefined`,
поэтому в расчёте использован тернарный `currentTemplate ? ... : 1` —
fallback на квадрат.

### Шаг 3. Фикс 3 — fallback color в TextSlot

Файл: `app/app/_components/AlbumSpreadCanvas.tsx`

Найди функцию `TextSlot`. Она принимает `placeholder` и в `<Text>`
передаёт `fill={placeholder.color || '#000000'}`. Это неработающий
fallback — он срабатывает только при пустом color, но не при светлом.

#### 3а. Добавить хелпер `isTooLight` рядом с `getCoverCrop`

Найди определение `getCoverCrop` (~строка 60). **После** него
добавь:

```typescript
// ─── Хелпер: проверка яркости цвета ──────────────────────────────────────
//
// Возвращает true если цвет слишком светлый (luminance > 0.7) или
// невалидный. Используется в TextSlot для fallback на чёрный когда
// placeholder.color из IDML светло-серый, который нечитаем на пустом
// светло-сером photo-слоте (см. инструкцию 2.6.1.1).
function isTooLight(hex: string | null | undefined): boolean {
  if (!hex) return true
  const m = hex.replace('#', '')
  if (m.length !== 3 && m.length !== 6) return true
  const expand = m.length === 3
    ? m.split('').map((c) => c + c).join('')
    : m
  const r = parseInt(expand.slice(0, 2), 16) / 255
  const g = parseInt(expand.slice(2, 4), 16) / 255
  const b = parseInt(expand.slice(4, 6), 16) / 255
  if (Number.isNaN(r) || Number.isNaN(g) || Number.isNaN(b)) return true
  const luminance = 0.299 * r + 0.587 * g + 0.114 * b
  return luminance > 0.7
}
```

#### 3б. Использовать хелпер в TextSlot

Найди в `TextSlot`:

**БЫЛО:**

```tsx
      fill={placeholder.color || '#000000'}
```

**СТАЛО:**

```tsx
      fill={isTooLight(placeholder.color) ? '#000000' : placeholder.color || '#000000'}
```

Логика: если color невалидный/пустой/слишком светлый → чёрный.
Иначе → используем как есть. Двойной `|| '#000000'` в else-ветке —
TypeScript narrowing safety (хотя `isTooLight` уже отвечает за пустые).

## Применение и верификация в проде

### Порядок

Миграций нет. После push Vercel задеплоит за ~30-60 секунд.

### Проверка визуально

После деплоя открыть `https://yearbook-v2.vercel.app/app/album/3fccf1dc-3e3d-402b-8c0c-7036e3496418/layout`:

1. **Заголовок** должен показать **«Красночетайская СОШ»** вместо «Альбом»
2. **Разворот 1** (учительский, одностраничный) — должен быть квадратной
   формы, занимать ~70% высоты viewport'а
3. **Разворот 7** (двустраничный) — должен быть **в 2 раза шире** разворота
   1, при той же высоте. Партнёр должен видеть «развёрнутую книгу»,
   а не сплющенный прямоугольник
4. **Текст имён учеников** на пустых слотах разворота 7 — должен быть
   **читаемым** (тёмный или чёрный, не сливается с серым фоном)
5. **При изменении размера окна браузера** (drag за угол) — canvas
   должен динамически перерисовываться
6. **Миниатюры в Обзоре** (`/app` → Красночетайка → Обзор) — текст
   тоже должен стать читаемым (фикс 3 распространяется)

### Что **не** должно сломаться

- Навигация ◀▶ работает как раньше
- Палитра справа выглядит как раньше («179 фото загружено...»)
- Кнопка «← К альбому» работает
