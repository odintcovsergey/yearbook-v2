# Инструкция для подэтапа 2.5

**Цель:** добавить endpoint `POST /api/layout?action=save_album_layout`
для сохранения ручных правок партнёра в layout альбома. Это complement
к `build_album` (2.1): пара формирует пометку «есть несохранённые
правки», которую UI 2.7 будет проверять перед confirm-диалогом
пересборки.

**Зачем:** в подэтапе 2.6 партнёр будет редактировать layout drag-and-
drop'ом (заменять фото в placeholder'ах). После каждого drag'а нужно
сохранить новое состояние layout'а с пометкой «правки есть». Эта
пометка — `has_user_edits=true` в `album_layouts`. При следующем
нажатии «Пересобрать» UI 2.7 покажет confirm: «правки будут перезатёрты
автосборкой».

Логическая пара:
- **2.1** `build_album` сбрасывает `has_user_edits=false` при пересборке
- **2.5** `save_album_layout` ставит `has_user_edits=true` при ручной правке

В 2.5 делаем **только endpoint** (бэкенд + JSON). UI вызова появится в
2.6 при auto-save'е editor'а с debounce ~1 секунда.

**Контекст:** в `app/api/layout/route.ts` уже есть POST router (строки
224-252) с тремя existing actions: `build_album`, `build_album_test`,
`import_idml`. Добавляем четвёртый — `save_album_layout`. Используем
тот же паттерн авторизации и view_as что у `handleBuildAlbum` —
максимальная консистентность кода.

**Что меняется:**
- `app/api/layout/route.ts` — две точки:
  1. Регистрация в POST router (1 if-блок ~3 строки)
  2. Новая функция `handleSaveAlbumLayout` (~80 строк)
- `docs/internal/2.5-instructions.md` — этот файл

**Что НЕ трогаем:**
- Существующие actions `build_album`, `build_album_test`, `import_idml`
- `handleBuildAlbum`, `handleGetAlbumLayout` — без изменений
- Миграции БД — колонка `has_user_edits` уже есть (миграция 2.1
  применена 09.05.2026)
- UI — drag-and-drop логика и auto-save в 2.6

## Принципиальные решения

### Body: `{ album_id, spreads }`, без `has_user_edits`

Принимаем только `album_id` (UUID) и `spreads` (массив). Не принимаем
`has_user_edits` от клиента — endpoint **всегда** ставит `true`. Иначе
клиент мог бы случайно (или специально) прислать `false` и обнулить
семантику флага.

```typescript
const { album_id, spreads } = body
// ... валидация ...
.update({ spreads, has_user_edits: true, updated_at: new Date().toISOString() })
```

### Не валидируем структуру `spreads`

Клиент шлёт `spreads` как есть — массив `SpreadInstance[]` который
получил из предыдущего `build_album` или предыдущего `save_album_layout`.
Глубокая валидация структуры (наличие template_id у каждого spread'а,
data объект, label'ы) — это бессмысленная защита: layout'ы строятся
билдером, не вручную, и партнёр не может «сломать» структуру через UI.

Что валидируем:
- `album_id` — UUID (через существующий `UUID_REGEX`)
- `spreads` — массив (`Array.isArray`)

Этого достаточно. Postgres `jsonb` поле примет любой валидный JSON;
если структура битая — следующий рендер `LayoutPreviewStrip`
обработает graceful (template_id не найдётся → "Шаблон не найден"
плашка).

### UPDATE, не UPSERT

В 2.1 `handleBuildAlbum` использует upsert потому что layout может
**не существовать** для альбома (первый build) — нужно или INSERT
или UPDATE. В 2.5 ситуация противоположная: `save_album_layout`
вызывается из редактора, а редактор открывается только когда
**layout уже есть** (после первого build_album). Если layout'а нет —
сохранять нечего.

Используем UPDATE с `.eq('album_id', albumId)`. Если строка не
найдена (rowsAffected=0) — возвращаем 404 «layout not found, run
build_album first». Это понятная ошибка, не молчаливое создание
layout'а в обход pipeline'а.

### Не трогаем `template_set_id`, `config_preset_id`, `warnings`

В 2.5 сохраняем только данные правок — `spreads` и пометку флагом.
Шаблон и пресет неизменны (они зафиксированы при build_album и
не могут быть «переопределены» сохранением правок). Warnings
существующие тоже не перетираем — они описывают исходную автосборку,
а не ручные правки.

```typescript
.update({
  spreads,
  has_user_edits: true,
  updated_at: new Date().toISOString(),
})
// template_set_id, config_preset_id, warnings — НЕ трогаем
```

### Авторизация: тот же паттерн что у build_album

```typescript
const auth = await requireAuth(req, ['superadmin', 'owner', 'manager', 'viewer'])
```

Уже сделано на уровне POST router (строка 228 в текущей версии). Внутри
handler'а — view_as через slug='main' + `assertAlbumAccessLocal`.
Идентично `handleBuildAlbum`.

**Нюанс по viewer:** viewer-роль читает данные но не должна писать.
Здесь принципиально НЕ повышаем требование до `['owner', 'manager']`
по двум причинам:
1. Существующая POST-роутер открыт для viewer'а (см. строки 228 +
   logical interpretation: build_album тоже доступен viewer'у —
   спорное архитектурное решение, но оно уже есть, и менять его в
   2.5 значит трогать существующее поведение).
2. View_as делает «чужой» tenant эффективным `tid`, и
   `assertAlbumAccessLocal` проверит права доступа к альбому.

Если в будущем потребуется ужесточить — это отдельная задача
«ужесточение ролей в /api/layout», в backlog.

### `updated_at` явно

Postgres мог бы выставить `updated_at` через триггер, но в текущей
схеме `album_layouts` триггера нет (см. миграцию 0.1). Передаём
явно — `new Date().toISOString()`. Та же практика что в `handleBuildAlbum`
(строка ~624 текущего кода).

### Возвращаем `{ success: true, layout_id }`

Минимальный response. UI в 2.6 после auto-save'а просто проверит
`response.ok` и обновит локальный флаг `has_user_edits = true`. Полный
layout возвращать не нужно — клиент только что отправил его сам.

`layout_id` оставляем для consistency с `handleBuildAlbum` response
(там `layout_id` в response есть) и для удобства логирования / debug'а.

### Шаг 2. Реализация `handleSaveAlbumLayout`

В том же файле `app/api/layout/route.ts`. **После** `handleBuildAlbum`
(заканчивается ~строка 678) и **перед** `handleGetAlbumLayout` (~679)
добавь новую функцию:

```typescript
// ─────────────────────────────────────────────────────────────────────────
// handleSaveAlbumLayout — POST ?action=save_album_layout
//
// Сохраняет ручные правки партнёра в layout альбома (drag-and-drop фото
// в placeholder'ах из палитры редактора). Всегда ставит
// has_user_edits=true; флаг сбрасывается обратно в false при следующем
// build_album (см. handleBuildAlbum / 2.1).
//
// Body: { album_id: uuid, spreads: SpreadInstance[] }
// Авторизация: owner/manager/viewer тенанта-владельца + view_as как
// у handleBuildAlbum (тот же паттерн).
//
// 404 если layout для альбома ещё не существует (нужно сначала
// build_album). 400 если body невалидный.
// ─────────────────────────────────────────────────────────────────────────
async function handleSaveAlbumLayout(
  req: NextRequest,
  auth: AuthContext,
): Promise<NextResponse> {
  let body: unknown
  try {
    body = await req.json()
  } catch {
    return NextResponse.json({ error: 'invalid JSON body' }, { status: 400 })
  }

  if (typeof body !== 'object' || body === null) {
    return NextResponse.json({ error: 'body must be object' }, { status: 400 })
  }

  const albumId = (body as Record<string, unknown>).album_id
  if (typeof albumId !== 'string' || !UUID_REGEX.test(albumId)) {
    return NextResponse.json(
      { error: 'album_id is required (uuid)' },
      { status: 400 },
    )
  }

  const spreads = (body as Record<string, unknown>).spreads
  if (!Array.isArray(spreads)) {
    return NextResponse.json(
      { error: 'spreads is required (array)' },
      { status: 400 },
    )
  }

  // view_as: тот же паттерн что у handleBuildAlbum.
  const viewAsTenantId = req.nextUrl.searchParams.get('view_as')
  const { data: currentTenantData } = viewAsTenantId
    ? await supabaseAdmin.from('tenants').select('slug').eq('id', auth.tenantId).single()
    : { data: null }
  const canViewAs = auth.role === 'superadmin' || currentTenantData?.slug === 'main'
  const tid = (canViewAs && viewAsTenantId) ? viewAsTenantId : auth.tenantId

  if (!(await assertAlbumAccessLocal(auth, albumId, tid))) {
    return NextResponse.json({ error: 'access denied' }, { status: 403 })
  }

  // UPDATE с has_user_edits=true. Если строки нет (rowsAffected=0) →
  // layout ещё не строился, нужно сначала build_album.
  const { data: layoutRow, error } = await supabaseAdmin
    .from('album_layouts')
    .update({
      spreads,
      has_user_edits: true,
      updated_at: new Date().toISOString(),
    })
    .eq('album_id', albumId)
    .select('id')
    .maybeSingle()

  if (error) {
    return NextResponse.json(
      { error: `save failed: ${error.message}` },
      { status: 500 },
    )
  }

  if (!layoutRow) {
    return NextResponse.json(
      { error: 'layout not found for this album, run build_album first' },
      { status: 404 },
    )
  }

  return NextResponse.json({
    success: true,
    layout_id: layoutRow.id,
  })
}
```

**Замечания по реализации:**

1. **`as Record<string, unknown>`** — паттерн валидации body, идентичный
   `handleBuildAlbum`. Не используем `zod` или другие схема-либы —
   проект их не использует, и текущая валидация достаточна.

2. **`.maybeSingle()` вместо `.single()`** — `single()` упадёт с
   ошибкой если строки нет. `maybeSingle()` вернёт `null` —
   мы перехватываем и отдаём 404. Это семантически верно (отсутствие
   layout'а — не серверная ошибка, а ожидаемая ситуация).

3. **Имя `assertAlbumAccessLocal`** — это локальная функция в файле
   `app/api/layout/route.ts`, отличная от `assertAlbumAccess` в
   `tenant/route.ts`. Используется уже в `handleBuildAlbum` —
   импорт не нужен, она в том же файле.

4. **`auth: AuthContext`** — тип уже импортирован в файле (используется
   в существующих handler'ах). Импорт не меняем.

### Проверка через DevTools Console

Открыть `/app` → DevTools (Cmd+Option+I) → вкладка Console.

**Сценарий 1: Нормальное сохранение**

Скопируй и вставь в Console одной строкой:

```js
fetch('/api/tenant?action=album_layout&album_id=3fccf1dc-3e3d-402b-8c0c-7036e3496418', {credentials:'include'}).then(r=>r.json()).then(layoutData=>{const spreads=layoutData.layout.spreads;return fetch('/api/layout?action=save_album_layout',{method:'POST',credentials:'include',headers:{'Content-Type':'application/json'},body:JSON.stringify({album_id:'3fccf1dc-3e3d-402b-8c0c-7036e3496418',spreads})}).then(r=>r.json())}).then(d=>console.log('SAVE RESULT:',d))
```

(Этот код сначала читает текущий layout через action=album_layout
из tenant API — wait, проверю, откуда читать layout. Layout
читается через **/api/layout?action=album_layout**, не через tenant.)

**Корректный код:**

```js
fetch('/api/layout?action=album_layout&album_id=3fccf1dc-3e3d-402b-8c0c-7036e3496418', {credentials:'include'}).then(r=>r.json()).then(layoutData=>{const spreads=layoutData.layout.spreads;return fetch('/api/layout?action=save_album_layout',{method:'POST',credentials:'include',headers:{'Content-Type':'application/json'},body:JSON.stringify({album_id:'3fccf1dc-3e3d-402b-8c0c-7036e3496418',spreads})}).then(r=>r.json())}).then(d=>console.log('SAVE RESULT:',d))
```

Что делает:
1. Читает текущий layout альбома (action=album_layout)
2. Берёт его `spreads` массив
3. Шлёт его обратно через `save_album_layout` (имитация «партнёр
   сохранил ту же структуру что была»)

Ожидание: `SAVE RESULT: { success: true, layout_id: '<uuid>' }`

## Что НЕ делаем в 2.5

- **UI вызов endpoint'а из редактора** — это подэтап 2.6 (auto-save
  с debounce 1s)
- **Confirm-диалог при «Пересобрать» если `has_user_edits=true`** —
  это подэтап 2.7
- **Расширение `GET ?action=album_layout` чтобы возвращать
  `has_user_edits`** — это тоже 2.7 (там же где и confirm-диалог)
- **Optimistic locking (`If-Match` через `updated_at`)** — для одного
  партнёра-редактора race-condition'ы маловероятны; добавить можно
  будет в фазе 3+ если несколько менеджеров OkeyBook одновременно
  правят один альбом
- **Audit log записи save** — read/write на стороне layout'а пока
  без аудита (см. договорённость в комментариях `/api/layout`)
- **Rate-limiting** — auto-save с debounce 1s = ~60 запросов/мин
  максимум, защита не нужна на этом этапе
- **Контекст-файл v43 → v44** — обновляется в 2.8

После 2.5 у нас:
- Endpoint `POST /api/layout?action=save_album_layout` готов и проверен
- Парность `build_album` (reset флага) ↔ `save_album_layout`

с debounce ~1s.

`has_user_edits` flag (2.5 ставит, 2.1 сбрасывает).
