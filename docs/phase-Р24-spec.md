# Фаза РЭ.24 — Готовые шаблоны для быстрого старта партнёра

**Спецификация v1.4**
**Дата:** 20.05.2026
**Статус:** v1.0 утверждена 20.05.2026; v1.1 расширена секцией §12
(архитектура multi-design); v1.2 расширена секцией §13 (архитектурный
долг: print_type должен жить в альбоме — будущая фаза РЭ.27);
v1.3 расширена секцией §14 (архитектурный долг: партнёрские дизайны —
будущие фазы РЭ.28 и РЭ.29); v1.4 расширена секцией §15 (терминология).
**Зависит от:** РЭ.22 (двух-осевая модель + семантический engine), РЭ.23 (renderPreviewSvg + display_label).
**Эстимация:** 6-8 коммитов основной фазы + 3 коммита РЭ.24.5b. Цельный день работы.

---

## 1. Зачем фаза существует

После РЭ.22/23 engine работает, превью генерируется, каталог `/super/master-catalog` доступен. Но **новый партнёр** при заходе на платформу всё ещё не имеет доступной точки старта: пресеты есть только в `/super` (доступ к нему — только у Сергея), при создании альбома партнёр выбирает что-то из `presets` напрямую без визуальной анонсации.

РЭ.24 — это **витрина** для партнёра. Партнёр видит:
- 7 готовых шаблонов OkeyBook (мини / лайт / медиум / стандарт / универсал / максимум / индивидуальный)
- Свои клонированные/созданные шаблоны
- Карточку с превью ключевых разворотов
- Кнопку «Сохранить в мои» для клонирования
- Возможность редактировать свои шаблоны (наследует РЭ.22-логику через существующий `PresetEditorModal`)

При создании альбома — обязательный шаг «выбрать шаблон». Без шаблона альбом не создаётся.

## 2. Архитектура: 4 уровня сущностей

```
┌──────────────────────────────────────────────────────────────────┐
│ 1. ГЛОБАЛЬНЫЕ ШАБЛОНЫ OkeyBook                                   │
│    presets WHERE tenant_id IS NULL                               │
│    Создаёт Сергей в /super/presets. Партнёры видят только        │
│    для клонирования. Редактировать не могут.                     │
└──────────────────────────────────────────────────────────────────┘
                            ↓ клонирование
┌──────────────────────────────────────────────────────────────────┐
│ 2. КАТАЛОГ ШАБЛОНОВ /app/templates (НОВАЯ СТРАНИЦА)             │
│    Две секции:                                                   │
│      • «Готовые от OkeyBook» — карточки глобальных шаблонов      │
│        с кнопкой «Сохранить в мои» (клонирует с tenant_id        │
│        партнёра)                                                 │
│      • «Мои шаблоны» — собственная библиотека партнёра           │
└──────────────────────────────────────────────────────────────────┘
                            ↓ выбор при создании
┌──────────────────────────────────────────────────────────────────┐
│ 3. ЛИЧНАЯ БИБЛИОТЕКА ПАРТНЁРА                                    │
│    presets WHERE tenant_id = X                                   │
│    Клонированные + созданные с нуля. Партнёр редактирует через   │
│    PresetEditorModal (используется тот же из /super/presets).    │
│    Создание с нуля — кнопка «Создать свой шаблон».               │
└──────────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────────┐
│ 4. ВЫБОР ПРИ СОЗДАНИИ АЛЬБОМА (модалка)                          │
│    Сначала «Мои шаблоны» (если есть), под ними отдельным         │
│    блоком «Готовые от OkeyBook» (быстрый клон + создание).       │
│    Невалидные шаблоны → disabled с пометкой «Доработай».         │
└──────────────────────────────────────────────────────────────────┘
```

**Ключевое:**
- Глобальные шаблоны редактирует **только Сергей** в `/super/presets`. Партнёр их видит read-only.
- При клонировании создаётся **независимая копия** — улучшения наших шаблонов **не догоняют** клонов автоматически.
- Удаление шаблона партнёра — с блокировкой если есть активные альбомы (см. §6).

## 3. Валидация шаблона

Шаблон валиден (показывается партнёру, можно выбрать при создании альбома), если выполнены **все**:

1. `display_name` непустое
2. `print_type` ∈ `{'layflat', 'soft'}`
3. `template_set_id` не NULL и ссылается на существующий template_set
4. `section_structure` — массив длиной ≥ 1
5. Для секций `students` — `student_layout_mode` не NULL (page / spread / grid)
6. Если `student_layout_mode='grid'` — `student_grid_size` не NULL и в диапазоне 2..12

**Поведение:**
- **Глобальные**: невалидный глобальный шаблон **не отдаётся API** партнёру (фильтрация на сервере). В логе сервера warning «preset 'мини' (id=...) невалиден: student_layout_mode=NULL».
- **Партнёрские**: невалидные **отдаются API** с флагом `valid: false` и подсвечиваются в UI с пометкой «Доработай». Партнёр может открыть редактор и доделать. В выпадашке выбора при создании альбома — `disabled`.

Утилита `validatePreset(preset): { valid: boolean, errors: string[] }` — общая для сервера и UI.

## 4. Карточка шаблона (превью)

Фиксированный набор миниатюр в каждой карточке:

```
┌─────────────────────────────────────┐
│  ╔═══════════════════════════════╗  │  ← Крупное превью:
│  ║                               ║  │     личный раздел (главное)
│  ║      [SVG личного]            ║  │
│  ║                               ║  │
│  ╚═══════════════════════════════╝  │
│                                     │
│  ┌──┐ ┌──┐ ┌──┐ ┌──┐                │  ← Малые превью:
│  │..│ │..│ │..│ │..│                │     обложка, учительский,
│  └──┘ └──┘ └──┘ └──┘                │     soft-intro/final
│  обл  уч.  intro                    │
│                                     │
│  Стандарт                           │  ← display_name пресета
│  4 ученика, цитата, мягкий          │  ← короткое описание
│  переплёт                           │
│                                     │
│  [ Сохранить в мои ]                │  ← кнопка (для глобальных)
│  или                                │
│  [ Редактировать ] [ ... ]          │  ← для «моих»
└─────────────────────────────────────┘
```

**Выбор ключевых разворотов** для превью:
- **Крупное**: первый разворот секции `students` (mode='page' → личная страница ученика; 'spread' → разворот; 'grid' → сетка)
- **Малое 1**: cover (если есть в section_structure)
- **Малое 2**: teachers (первая правая страница учительского разворота)
- **Малое 3**: soft_intro или soft_final (если есть)

Если у шаблона нет какой-то секции — на её месте просто пусто. Жёстких требований нет.

**Эти превью генерируются на сервере**: для каждого пресета engine делает dry-run сборку с фиктивными данными (10 учеников, 5 предметников, 1 фото класса) → собирает 4 ключевых разворота → каждый рендерится через `renderPreviewSvg` → результат отдаётся клиенту как 4 SVG-строки в одной payload.

Это «тяжело» при первом фетче (1-2 секунды), но клиент кэширует.

## 5. Что добавляется в БД

**Поле `presets.is_recommended BOOLEAN DEFAULT false`** — фильтр для каталога. Только глобальные шаблоны (`tenant_id=NULL`) могут быть рекомендованными. Сергей в `/super/presets` помечает.

В РЭ.24 партнёр видит в каталоге **только** глобальные шаблоны с `is_recommended=true`. Это позволяет тебе держать в БД и черновые пресеты — они не покажутся партнёрам пока не пометишь.

**Поле `presets.parent_preset_id UUID NULL`** — для отслеживания «откуда клонирован». NULL для глобальных и созданных с нуля. У клонированных = id оригинального глобального. Используется для:
- Аналитики («какой шаблон чаще всего клонируют»)
- Возможной будущей фичи «обновить из шаблона» (не в РЭ.24)
- UX подсказки в карточке («клон из Стандарт»)

Миграция: `2026-05-20-presets-recommended-and-parent.sql` — две колонки, обе nullable.

## 7. Архитектурные развилки (зафиксированы)

**A. Хранение «рекомендованности»** ✅ **Решение**: новое поле `presets.is_recommended BOOLEAN`. Простой фильтр. Порядок отображения — по `display_name` (или добавим `display_order` позже если понадобится).

**B. Что в карточке** ✅ **Решение**: 1 крупное превью (личный раздел) + 3 малых (cover, teachers, soft_intro/final). Фиксированный набор.

**C. Что значит «использовать пресет»** ✅ **Решение**: клонирование. Копия пресета с `tenant_id=партнёр`. Партнёр в библиотеке независим. Поле `parent_preset_id` фиксирует происхождение.

**D. Где партнёр видит каталог** ✅ **Решение**: отдельная страница `/app/templates` (раздел «Шаблоны»). Доступна из главного меню партнёра. Каталог независим от создания альбома.

**E. Валидация** ✅ **Решение**: строгая. Глобальные невалидные не отдаются партнёру. Партнёрские невалидные отдаются с флагом `valid: false`, подсвечиваются «Доработай», disabled при выборе альбома.

**F. Кнопки на карточке глобального** ✅ **Решение**: одна — «Сохранить в мои». Не клонируем при создании альбома напрямую — партнёр сначала собирает библиотеку.

**G. Что при удалении шаблона у партнёра** ✅ **Решение**: проверка активных альбомов. Если есть — блокируем удаление с понятным сообщением «Этот шаблон используется в N альбомах: …». Если нет — удаляем без вопросов.

**H. Где видит выбор при создании альбома** ✅ **Решение**: вариант C из обсуждения. Модалка показывает «Мои шаблоны» сверху, под ними отдельным блоком «Готовые от OkeyBook». Невалидные disabled.

**I. Когда генерируются превью** ✅ **Решение**: серверный dry-run при запросе API `templates_list_global`/`_my`. Клиент кэширует. Альтернатива (сохранять в БД) — отложили.

## 8. Что НЕ входит в РЭ.24 (явно)

- **Конструктор шаблонов с галереей вариантов** — это **РЭ.25**. В РЭ.24 партнёр редактирует свои шаблоны через тот же `PresetEditorModal` что и в `/super/presets` (двух-осевая модель из РЭ.22). Без галерейного выбора альтернативных layout'ов.
- **Обновление шаблонов от OkeyBook задним числом** (когда Сергей улучшит «Стандарт», у партнёров остаются старые копии). Это будущая фича, не в РЭ.24.
- **Шеринг шаблонов между партнёрами** — нет, не планируется.
- **Импорт/экспорт шаблонов** — нет.
- **История изменений шаблона** — нет.
- **Превью в виде реальной сборки с реальными фото** — нет, только SVG-схемы из РЭ.23.

## 9. Связь с РЭ.25

После РЭ.24 партнёр может:
- Клонировать готовый шаблон → редактировать в `PresetEditorModal` (как Сергей в `/super`)
- Создать свой с нуля → редактировать там же
- Использовать в альбомах

РЭ.25 поверх этого добавит: в редакторе пресета для каждой роли (cover, student_left, teacher_left и т.д.) — галерею альтернативных layout'ов с превью. Партнёр выбирает не «4 ученика с цитатой» а «вот этот конкретный layout с 4 учениками». Это другой UI поверх той же модели данных.

## 10. Принципы

- **Не ломать `/super/presets`.** Существующая страница Сергея продолжает работать. РЭ.24 только добавляет: фильтр `is_recommended`, чекбокс в редакторе, метку на карточке.
- **Не ломать создание альбома.** Старый flow (выбор пресета через старую модалку) продолжает работать пока не выкатим новый. Новая модалка заменяет старую атомарно в одном подэтапе.
- **Tenant-aware строго.** API templates_list_my — только свои + глобальные (для отображения). API template_clone — клонирует в `tenant_id=auth.tenantId`. API template_delete — только свои.
- **Глобальные read-only для партнёра.** Любой POST/UPDATE/DELETE на пресете с `tenant_id=NULL` от не-superadmin → 403.
- **Бэк-совместимость.** Существующие альбомы с старым `preset_id` продолжают собираться. Поля `is_recommended` и `parent_preset_id` опциональны.

## 12. Архитектура multi-design (добавлено в v1.1)

**Контекст**: после реализации 24.5 (страница /app/templates) выявился
архитектурный вопрос: как встраивать в систему разные дизайны (белый,
сказочный для садиков, школьный строгий и т.д.), которых в планах ~20-30.

**Решение (зафиксировано 20.05.2026)**:

### 12.1. Дизайн = template_set

В БД таблица `template_sets` хранит дизайны. Каждый template_set содержит:
- Свою линейку мастеров (`spread_templates.template_set_id`)
- Свои размеры страницы, тип бумаги, переплёт
- Свой визуальный язык (фоны, рамки, декор — на уровне placeholders'ов
  мастеров в InDesign)

Сейчас в БД: один template_set `okeybook-default` («белый, без фона»).
В планах: 20-30 разных дизайнов с уникальными мастерами под каждый.

### 12.2. Пресет жёстко привязан к дизайну

Поле `presets.template_set_id` обязательно указывает на конкретный
template_set. Это означает:
- Один пресет работает только с одним дизайном
- Мастера сказочного дизайна не подходят к шаблонам белого (разные
  размеры, разные семантические теги, разный визуал)
- Клонирование пресета (`template_clone`) **сохраняет** template_set_id
  оригинала — нельзя «переодеть» клон в другой дизайн через эту операцию

Если партнёр хочет тот же набор настроек в другом дизайне — создаёт
новый шаблон с нуля под нужный дизайн.

### 12.3. Каталог двухуровневый (РЭ.24.5b)

`/app/templates` теперь страница **выбора дизайна**. Партнёр видит
большие карточки template_set'ов (свои + глобальные). Каждая карточка:
- Название дизайна
- 3-4 мини-превью характерных мастеров из этого template_set
- Счётчики: «N готовых от OkeyBook», «M моих шаблонов»

`/app/templates/[designId]` — каталог шаблонов внутри выбранного дизайна
(то что было в 24.5: две секции «Готовые от OkeyBook» / «Мои»).

При создании альбома (24.6) — двухшаговая модалка: дизайн → шаблон.

### 12.4. Глобальные пресеты могут быть разные для разных дизайнов

Стартовая линейка Сергея (мини / лайт / медиум / стандарт / универсал /
максимум / индивидуальный) **не дублируется автоматически** под каждый
новый дизайн. Для каждого дизайна Сергей сам решает какие комплектации
имеют смысл:
- Для садиков, например, 3-4 пресета (Лайт/Медиум, без Максимума с
  цитатами — потому что детям до 7 лет цитаты не пишут)
- Для школ — другой набор (Стандарт/Универсал/Максимум, без некоторых
  садиковых)

Это естественное следствие модели: фильтрация в API
`templates_list_global` уже работает по `template_set_id`, новых полей
не нужно.

### 12.5. Загрузка дизайнов

В `/super` уже есть кнопка «Загрузить InDesign документ». Pipeline
сейчас создаёт `template_set` + мастера в нём. После РЭ.24.5b новый
template_set автоматически появится в списке `designs_list` для
партнёров — никакой дополнительной работы при добавлении дизайна.

### 12.6. Tenant-аспект

template_set'ы имеют `tenant_id`:
- `tenant_id IS NULL` — глобальный дизайн от OkeyBook
- `tenant_id = X` — кастомный дизайн партнёра X (на будущее — пока
  ни у кого нет, но архитектура готова)

В каталоге партнёр видит все доступные ему дизайны: свои + глобальные.

### 12.7. Что меняется в реализации (РЭ.24.5b)

**API:**
- Новый action `designs_list` — список доступных template_sets с
  превью и счётчиками
- `templates_list_global` и `_my` — добавить параметр `?design_id=...`

**UI:**
- Текущая страница `/app/templates/page.tsx` → перенесена в
  `/app/templates/[designId]/page.tsx`
- Новая страница `/app/templates/page.tsx` — выбор дизайна

**Что НЕ меняется:**
- БД (template_sets и presets.template_set_id уже есть)
- Engine
- /super/master-catalog
- IDML-загрузка

### 12.8. Эстимация РЭ.24.5b

~3 коммита: spec → API → UI. Полдня работы.

## 13. Архитектурный долг: print_type должен жить в альбоме (РЭ.27)

**Добавлено в v1.2** (20.05.2026). Фаза не выполняется в РЭ.24 —
это запись будущей фазы РЭ.27, чтобы знание о долге не потерялось.

### 13.1. Проблема

Сейчас `print_type` (layflat / soft) хранится в `presets.print_type`.
В каталоге `/app/templates/[designId]` партнёр видит шаблоны типа
«Стандарт-layflat» и «Стандарт-soft» как два разных пресета — хотя
содержательно это одно и то же, отличается только тип переплёта.

Это **избыточно** в БД (двойное число пресетов) и **неудобно** для
партнёра — он мыслит «комплектация Стандарт», тип листов выбирает
отдельно от шаблона по цене.

### 13.2. Почему print_type не атрибут пресета

Тип листов — это **не только** про типографию и цену клиента. Это про
**визуальную модель редактора**:

- **layflat (плотные листы)** — переплёт «лежит плоско». Разворот — одна
  непрерывная поверхность. Фотограф **должен** видеть и компоновать
  разворот целиком, иначе крупный объект может попасть на сгиб и быть
  обрезан. Редактор должен показывать **две страницы рядом** как одну
  единицу работы.

- **soft (мягкие листы)** — обычный книжный переплёт. Между левой и
  правой страницей корешок съедает миллиметры, разворот не плоский.
  Каждая страница самостоятельна. Плюс типографские правила: первая
  страница после обложки **обязательно правая** (титульная), последняя
  — левая. Редактор работает **постранично**.

Это означает: тип листов влияет на **режим работы редактора**, а не
только на цену. Этот режим — атрибут конкретного **альбома**, не
шаблона. Шаблон описывает **что** в альбоме (структура секций), тип
листов — **как с этим работать**.

### 13.3. Целевая модель

```
ПРЕСЕТ (preset)
  • Дизайн (template_set_id)
  • Структура секций (section_structure)
  • Личный раздел (student_layout_mode)
  • НЕ содержит print_type

АЛЬБОМ (album)
  • Шаблон (опционально, section_structure_preset_id)
  • Тип листов (print_type) — выбирается отдельно при создании
  • Загруженные фото, ученики, прочее
```

### 13.4. Что меняется

- **БД**: миграция `albums ADD COLUMN print_type` + миграция данных
  (заполнить из preset.print_type у существующих альбомов).
- **Engine**: `loadBundle` берёт print_type из альбома, fallback на
  пресет для совместимости со старыми альбомами.
- **Редактор**: переключение режима по `album.print_type`:
  - layflat → разворотный режим (две страницы рядом, общая зона работы)
  - soft → постраничный режим (как сейчас) + правила первой/последней
    страницы.
- **Каталог `/app/templates`**: убрать бейдж «layflat/soft» с карточек
  шаблонов (это атрибут альбома, не шаблона). Уменьшается число пресетов
  в БД примерно вдвое.
- **Форма создания альбома**: добавить селект «Тип листов» рядом с
  выбором шаблона.
- **Экспорт в типографию**: проверить что использует правильный
  источник print_type.

### 13.5. Эстимация

Большая фаза. ~10-15 коммитов. Самый рискованный подэтап — редактор
(переключение между разворотным и постраничным режимами с сохранением
существующего поведения для всех текущих альбомов).

### 13.6. Когда

После закрытия фазы РЭ.24. Если Сергей готов сразу после РЭ.24
заняться этим — открываем РЭ.27 следующей сессией. Если есть другие
приоритеты — фиксируем долг и возвращаемся когда станет реальной
болью (партнёр первый раз спросит «а можно тот же шаблон на других
листах?»).

### 13.7. Что НЕ делаем в РЭ.24

В РЭ.24 (текущая фаза) продолжаем работать с `print_type` в пресете
как сейчас. Каталог шаблонов показывает бейджи «твёрдая обложка» /
«мягкая обложка». Это **не идеально**, но это **то что работает в
проде**, и мы не ломаем его сегодня.

## 14. Архитектурный долг: партнёрские дизайны (РЭ.28+)

**Добавлено в v1.3** (20.05.2026). Фиксирует план работы с разными
типографиями — кандидат на отдельные фазы после РЭ.24/25/27.

### 14.1. Проблема

Партнёры работают с разными типографиями, у которых разные технические
требования: размер разворота, зоны обрезки, плотность бумаги, типы
переплёта. Сейчас все наши `template_set`'ы создаются Сергеем через
загрузку IDML в /super — партнёр не может **под себя** подстроить
размеры под свою типографию.

Это блокирует масштабирование платформы на партнёров с собственными
типографиями.

### 14.2. Текущее состояние архитектуры

**Что уже есть в БД:**
- `template_sets.tenant_id` поддерживает партнёрские дизайны
  (NULL = глобальный, иначе id партнёра)
- API `designs_list` уже фильтрует «свои + глобальные» — если в БД
  появится партнёрский template_set, он сразу будет виден в каталоге

**Чего НЕТ:**
- UI для партнёра «создать свой template_set»
- Логики копирования глобального template_set'а в личный с заменой
  размеров
- Ресайза мастеров под другие размеры страницы (engine **частично**
  это уже умеет — placeholder'ы пересчитываются пропорционально)
- Защиты от плохой адаптации (несовместимые пропорции)

### 14.3. Два сценария партнёрской работы

**Сценарий A: «Похожая типография с другими размерами»** —
самый частый случай. Партнёр работает с одной типографией для обычных
альбомов и другой для «премиум» формата. Обе печатают похожие альбомы,
но размеры немного разные (например 20×30 vs 19×29).

В этом сценарии:
- Партнёр копирует глобальный template_set с переопределением размеров
- Мастера пересчитываются пропорционально (engine уже это умеет)
- Партнёр видит свой дизайн в каталоге рядом с глобальными
- Объём: отдельная фаза ~5-7 коммитов

**Сценарий B: «Своя типография, всё с нуля»** — редкий, но возможный.
Партнёр работает с локальной типографией со специфическими требованиями
(другие пропорции, другие техн.зоны), и хочет создать **с нуля** свой
template_set с своими мастерами.

В этом сценарии:
- Партнёр загружает свой IDML (как сейчас Сергей в /super)
- Создаются мастера с произвольными placeholder'ами
- Это **большая отдельная фаза** (10+ коммитов, риск регрессий)
- Открываем когда придёт первый реальный кейс

### 14.4. Ключевое ограничение: пропорции

Engine умеет ресайзить placeholder'ы пропорционально размеру страницы,
но это работает **только когда пропорции похожи**.

**Хорошо работает (визуально незаметная разница):**
- 20×30 → 19×29 (разница пропорций ~1.7%)
- 200×280 → 210×297 (разница ~3%)

**Плохо работает (видимые искажения):**
- 20×30 → 21×30 (соотношение меняется на 5%) — на грани
- 20×30 → 18×24 (разные пропорции совсем) — нужны другие мастера

**Граница**: разница пропорций ≤ 5% — адаптация автоматическая.
> 5% — система предупреждает партнёра «эти размеры значительно
отличаются, могут быть искажения; рекомендуется выбрать дизайн
с подходящими пропорциями».

**Технические нюансы (плохое поведение при больших расхождениях):**
- Круги (учительские аватарки) превращаются в овалы — engine не
  отслеживает «это должен оставаться круг»
- Фиксированные отступы превращаются в дробные миллиметры
- Тонкие рамки могут «съехать» по краям при сильном растяжении

### 14.5. План — сценарий A (РЭ.28)

**Подэтапы**:
- Миграция БД: добавить поле `template_sets.parent_template_set_id`
  (аналог parent_preset_id из РЭ.24) для трейсинга клонов
- Утилита `copyTemplateSetWithResize(originalId, newSizes, tenantId)`:
  создаёт копию template_set + копии всех мастеров с пересчётом
  placeholder'ов
- Утилита проверки совместимости пропорций (warning > 5% разницы)
- API endpoints:
  - `template_set_clone` — копирование с новыми размерами
  - `template_set_my_list` — список своих template_set'ов
  - `template_set_delete` — удаление своего template_set'а (только
    если нет привязанных пресетов)
- UI в /app:
  - Раздел «Мои дизайны» в каталоге `/app/templates`
  - Кнопка «Создать свой дизайн на основе…» на карточке глобального
  - Модалка с полями размеров + предупреждение о пропорциях

Эстимация: ~5-7 коммитов, 1-2 дня работы.

### 14.6. План — сценарий B (РЭ.29+)

**В разработке как фаза**, когда будет первый реальный партнёр с
запросом «хочу свою типографию полностью». Не делаем пока не появится
кейс.

Возможные подэтапы (предварительно):
- Перенос UI загрузки IDML из /super в /app/templates
- Защита от плохо размеченных IDML (валидация placeholder'ов,
  page_role, slot_capacity)
- Песочница для тестирования собственного template_set'а перед
  использованием в реальных альбомах
- Возможно — graphic-editor placeholder'ов прямо в UI

### 14.7. Что НЕ делаем сейчас

В РЭ.24 (текущая фаза) **не реализуем** партнёрские дизайны. Все
дизайны — глобальные от OkeyBook. Партнёры пользуются тем что есть.

После РЭ.24 → РЭ.25 (галка покупки) → РЭ.27 (print_type в альбом) →
**потом** уже РЭ.28 (партнёрские дизайны A).

РЭ.29 (сценарий B) — без даты, по запросу первого партнёра.

## 15. Терминология (зафиксировано в v1.4)

Добавлено в v1.4 (20.05.2026). Закрепляет термины которые используются
в коде, документации и в разговоре с пользователем. Цель — устранить
путаницу между legacy-смыслом «комплектация» и новой моделью.

**Шаблон (Template)** — пресет в БД с привязкой к одному дизайну.
Содержит:
- структуру альбома (`section_structure`)
- режим личного раздела (`student_layout_mode` + `student_grid_size`)
- тип листов (`print_type`) *— долг РЭ.27, см. §13*
- ссылку на дизайн (`template_set_id`)

В UI партнёра — «шаблон». В БД — таблица `presets`. Выбирается из
каталога `/app/templates/[designId]` или модалкой `TemplatePickerModal`
при создании альбома.

**Дизайн (Design)** — `template_set` в БД. Это **визуальный стиль**:
размеры страницы, набор мастеров с placeholder'ами, фоновая графика.
Один дизайн — один InDesign-проект.

В UI партнёра — «дизайн». Выбирается на верхнем уровне каталога
`/app/templates` или фильтром в `TemplatePickerModal`.

**Комплектация (legacy)** — устаревший термин из старой системы.
В коде это `config_type` (`standard | universal | maximum | medium |
light | mini | individual`) + `print_type` → `preset_slug`. В форме
альбома эти селекты остаются как fallback для альбомов без шаблона.
После полного перехода на новые шаблоны — выводится из активного
использования. **Не использовать в новом UI и документации** — вместо
этого говорим «шаблон».

**Тип листов (print_type)** — `layflat` (твёрдые) или `soft` (мягкие).
Влияет на:
- цену для клиента
- технологию переплёта типографии
- **визуальную модель редактора** (разворотная vs постраничная) —
  это долг РЭ.27, см. §13

Сейчас атрибут пресета, должен стать атрибутом альбома (РЭ.27).

**Секции структуры (section_structure)** — массив типов разделов
альбома в порядке появления. Существующие типы: `intro`, `students`,
`teachers`, `common`, `final`. **Не привязаны к типу листов.** Если
в `section_structure` есть `intro` — engine добавит вступительную
страницу независимо от print_type. То что в layflat обычно нет
intro — историческое следствие, не архитектурное правило.

**Личный раздел / страница ученика (student_layout_mode)** — режим
вёрстки страницы ученика:
- `page` — один ученик на одну страницу
- `spread` — один ученик на разворот
- `grid` — несколько учеников в сетке (`student_grid_size` штук)

**Общий раздел (common section)** — секция с фотографиями класса
целиком (полный класс, половина класса, четверть класса, шестая
часть). В типах фото — `full_class`, `half_class`, `spread`,
`quarter`, `sixth`.

**Учительский раздел (teachers)** — секция с фотографиями
предметников и главного учителя (классный руководитель / директор).

**Intro / Final** — семантические секции структуры альбома, не
привязаны к типу листов. Может быть в любом шаблоне если описано
в `section_structure`. Реализуются как мастера с `page_role='intro'`
или `'final'`.
