# Жизненный цикл архива — отложенное автоудаление исходников

Архивный заказ через **90 дней** автоматически освобождает место: удаляются
ТОЛЬКО исходники фотографа (тяжёлые JPG, `photos.original_path`), превью
(`storage_path`) и записи `photos` остаются — архив можно смотреть. Удаление
необратимо (живое хранилище Timeweb) → фича с буфером, отменой и предохранителями.

**Происхождение.** Закрывает баг архивации: `archive_album`/`delete_album` били
удалением в Supabase Storage API вместо Timeweb (`yc:`-ключи) → место не
освобождалось. Здесь — правильный путь: «архив помечает → чистильщик удаляет
отложенно через `ycDelete`».

## КРИТИЧНОЕ ядро безопасности — анти-шаринг
~13% записей `photos` делят файлы (клоны ссылаются на storage заказа-источника;
клонирование копирует `storage_path`). **Нельзя удалять по префиксу
`<album_id>/originals/`** — промахнётся по клонам и сломает живой клон.
**Единственный безопасный способ:** оригинал удаляем, ТОЛЬКО если на этот
`original_path` не ссылается НИ ОДИН не-истёкший заказ (живой / в пределах 90д /
`keep_originals_forever` / `archived_at=null` / уже-почищенный).

## Технические факты (диагностика)
- Оригиналы — поле `photos.original_path` (1981 запись). `original_photos` —
  LEGACY и пустая, не использовать.
- Ключи: превью `yc:<prefix>/<type>/...webp`; оригинал `yc:<prefix>/originals/...jpg`.
- Удаление — `ycDelete` (→ `twcDelete(stripYcPrefix)`), НЕ Supabase API.
- Фоновая инфра — systemd (образец `yearbook-backup.timer`, ежедневно 03:00 UTC).

## Параметры (владелец)
90 дней; удаляем только исходники; хранилище Timeweb; UI с отсчётом и кнопками
«Продлить на 90 дней»/«Оставить навсегда»; разархивация снимает таймер
(`archived_at=null`); каждое автоудаление → `audit_log` (чёрный ящик).

## Фазы
- **Фаза 1 — миграция (✅, `3ad5142`).** `albums`: `archived_at`,
  `keep_originals_forever`, `originals_deleted_at` (миграция
  `2026-06-25-archive-lifecycle.sql`, применена + reload PostgREST). `archive_album`
  ставит `archived_at=now()`, `unarchive_album` снимает. 11 уже-архивных НЕ
  бэкфилл (archived_at=null — safe default). Проверено: `archived_with_date=0`.
- **Фаза 2 — ядро (✅).** `lib/archive-cleanup/`:
  - `core.ts` (чистое): `isExpiredAlbum`, `computeDeletableOriginals` (анти-шаринг
    по `original_path`). `DEFAULT_TTL_DAYS=90`.
  - `run.ts` (загрузчик): `runCleanup({dryRun})` — `true` только считает (заказы/
    ключи/объём), `false` удаляет (`ycDelete` → null `original_path` →
    `originals_deleted_at` → `audit_log`). Не вызывается автоматически.
  - 14 тестов, вкл. анти-шаринг (падает без фильтра), защиту бэкфилла.
  - Dry-run на проде: сейчас 0 (бэкфилл-защита); симуляция «Тест 30 истёк, клон
    Тест 13 живой» → все 242 оригинала исключены анти-шарингом (клон защищён).
- **Фаза 3 — расписание (СЛЕДУЮЩАЯ).** `yearbook-cleanup.service`+`.timer`
  (ежедневно ~04:00 UTC), Node через tsx → `runCleanup({dryRun:false})`. **Таймер
  создаётся, но НЕ включается** (включение — фаза 5).
- **Фаза 4 — UI.** Отсчёт «исходники удалятся через N дней», кнопки «Продлить на
  90»/«Оставить навсегда», статус «исходники удалены».
- **Фаза 5 — боевое включение (необратимо, по явному «ок»).** dry-run на проде →
  проба на одном заказе → полный прогон вручную → `systemctl enable --now timer`.

## Формула отсчёта (Фаза 4)
`N = 90 − floor((now − archived_at)/день)`. `archived_at=null` → «отсчёт не начат».
`originals_deleted_at` задан → «исходники удалены, превью доступны».
