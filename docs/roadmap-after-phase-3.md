## Зачем этот документ

Закрыли большую фазу 3 (PDF-экспорт, 17 коммитов). Когда после деплоя начали смотреть реальное использование, выяснилось что система **не готова к запуску партнёрской программы** — отсутствуют критичные функции (общий раздел, оригиналы для печати, кастомные профили типографий, обложки и т.д.).

Этот документ фиксирует:
1. Что мы поняли про архитектуру системы 10.05
2. Какие задачи нужно сделать для боеготовности
3. Какие ответы ждём от дизайнера прежде чем кодить определённые задачи
4. Приоритеты и порядок выполнения

**Цель документа** — чтобы при возврате к работе через несколько дней (или в новом чате/у Claude Code) была полная картина без необходимости поднимать всю историю обсуждения.

## Главные открытия про систему

### 1. Что РЕАЛЬНО работает сейчас

- Отбор **личных страниц** родителями (портрет + цитата + личный разворот)
- Отбор **обложки** (для альбомов где она есть)
- **Группы для личной страницы** Универсала/Максимума/Индивидуального — родители выбирают «фото с друзьями» которые попадают в их раздел
- **Учителя** — фотограф загружает, builder ставит в учительский раздел
- **PDF-экспорт** через ExportPanel в Обзоре альбома (фаза 3)
- Workflow производства от фотографа в OkeyBook
- CRM с воронкой и сделками

### 2. Чего НЕТ совсем — критические блокеры

#### Блокер 1: Общий раздел в конце альбома

**Проблема:** Альбом не собирается целиком — общий раздел (фото класса разной величины) **не загружается в систему вообще**.

**Контекст:** Изначально система задумывалась как инструмент **отбора** портретов и групповых фото на личную страницу/обложку/учителей. Сейчас же без общего раздела альбом не собирается.

**Что должно быть:** Фотограф в разделе Фото загружает фото общего раздела по категориям. Родители НЕ голосуют за эти фото. Builder сам раскладывает их в общий раздел в конце альбома. В будущем может появиться возможность отбора этих фото родителями, но пока не нужно.

**Категории общего раздела:**
- `common_spread` — фото на разворот
- `common_full` — весь класс на странице (полностраничные)
- `common_half` — полкласса (половинки)
- `common_quarter` — четверть класса (четвертинки)
- `common_sixth` — одна шестая

**Сколько фото в среднем:** ~30 фото на класс

**Какие комплектации имеют общий раздел:** Все. Только количество разворотов и фото отличается. Точное количество для каждой комплектации — есть в IDML-скрипте автовёрстки (старая система).

**Размер хранилища:** При 5 МБ оригинал × 30 фото × 100 альбомов = ~15 ГБ только на общий раздел.

#### Блокер 2: Оригиналы для печати

**Проблема:** Сейчас фото загружаются и сжимаются в WebP. Оригиналов в системе нет вообще. PDF-экспорт берёт WebP версии — для печати недостаточно качественно.

**Что должно быть:** При загрузке фотограф даёт оригинал (5-10 МБ JPEG обычно). Сервер сжимает до 5-6 МБ JPEG 95% quality (без видимых потерь для печати на 300 dpi) и кладёт в `album_id/originals/<filename>`. Параллельно создаёт WebP для UI/отбора. PDF-export использует оригиналы.

**Подтверждённые числа:** 5-10 МБ оригинал → 5-6 МБ JPEG 95% → нормально для печати 300 dpi разворота.

### 3. Архитектурные находки про обложку

- Обложка состоит из **передней + корешка + задней стороны**
- Корешок есть **физически всегда** даже у тонких альбомов (заполнен фоном, без текста)
- Сейчас **информацию на корешке не размещаем** — только лицо (правая) и оборот (левая)
- На задней стороне нужна **рамка для карточки фотографа** (лого + контакты) — маркетинговая фича
- 4 варианта дизайна обложки: Свой портрет / Другой портрет / Общая фотография / В стиле дизайна без фото
- Обложки только **твёрдые**, независимо от типа листов (мягкие/твёрдые) внутри

### 4. Архитектурные находки про индивидуальные комплекты

- Это полноценная фотокнига структуры:
  1. Учителя (одинаковая часть у всех)
  2. Личный раздел (1-3 разворота, у каждого свой)
  3. Виньетки/миниатюры всего класса
  4. Общий раздел (одинаковая часть у всех)
- **Виньетки = автоматический разворот** из `portrait_page` selections + ФИО детей. Builder сам генерирует. Фотограф ничего отдельно не загружает.
- Размер индивидуальных страниц = такой же как у общей части (для конкретного альбома)

### 5. Что не нужно делать

**Backgrounds (фоны разворотов):**
- Это **часть дизайна template_set**, а не загрузка фотографом
- В IDML мастера уже содержат `background_url`
- Фотограф НЕ должен иметь возможность загружать свои фоны — это нарушит дизайн
- **Никакой отдельной категории `common_background`** в системе быть не должно

**Виньетки как отдельная загрузка:**
- Это **автоматическая генерация из portrait selections**
- Builder сам собирает миниатюры всех детей в виньеточный разворот
- Фотограф ничего не загружает отдельно

### 6. Архитектура хранилища (финальная)

```
yearbook-photos/
  album_id/
    portrait/<filename>          ← портреты (WebP)
    group/<filename>             ← группы для личных страниц (WebP)
    teacher/<filename>           ← учителя (WebP)
    common_spread/<filename>     ← фото на разворот (новое)
    common_full/<filename>       ← полностраничные (новое)
    common_half/<filename>       ← половинки (новое)
    common_quarter/<filename>    ← четвертинки (новое)
    common_sixth/<filename>      ← шестые (новое)
    personal/<child_id>/<filename>   ← личный разворот ребёнка
    originals/<filename>         ← оригиналы для печати (новое)
    exports/<filename>.pdf       ← PDF/JPG экспорты (90 дней)
    delivery/<filename>          ← доставка от OkeyBook (6 мес)
  tenants/<tenant_id>/
    logo.webp                    ← логотип в брендинге
    photographer_card_logo.png   ← карточка для задней обложки (новое, фаза Е)
```

## Архитектурные решения

### Кастомные комплектации фотографов

**Из ответа Сергея на вопрос 16:** «У фотографов могут быть свои комплектации, отличающиеся от моих внутренних».

**Решение:** В будущем (фаза Ж) — `config_presets` становится **tenant-aware** с inheritance:
- Базовый набор пресетов от OkeyBook (Стандарт, Универсал, Максимум, Медиум, Лайт, Мини, Индивидуальный, Трюмо)
- Фотограф в своём кабинете может **скопировать** пресет OkeyBook → изменить → сохранить как свой
- Может **создать с нуля** свою комплектацию

Сейчас (до фазы Ж) — все пользуются глобальными пресетами OkeyBook.

### Размеры альбомов и пропорции

**Из ответа Сергея на вопрос 17:** «Сейчас делаем 2 разных IDML под 226×288 и 200×280, но ты обещал автоподгонку».

**Реальность:**
- Автоподгонка работает **если пропорции страницы одинаковые**
- 226×288 (~0.785) и 200×255 (~0.784) — практически идентичны → можно одним IDML, scaling × 0.885
- 226×288 (~0.785) и 250×250 (1.0) — разные пропорции → нужны разные IDML

**Решение (фаза Д):**
- Концепция **template_sets family** — группа template_sets одного дизайна для разных пропорций
- Builder выбирает подходящий по пропорции, потом scaling под точный размер
- В настройках альбома выбор размера

### Типографии — разные требования

**Из ответа Сергея на вопрос 6:** «В моей типографии — один PDF на одинаковые альбомы. В другой — JPG постранично с определённым naming чтобы их система собрала».

**Решение (фаза Г):**
- Профили экспорта становятся **tenant-aware** — каждый фотограф добавляет свою типографию как профиль
- Глобальные профили OkeyBook остаются как пресеты
- Конкретные требования каждой типографии (формат, naming, размеры, цвет) собираем у дизайнера

## План фаз — приоритет от срочного к менее срочному

### 🔴 Фаза А — Общий раздел (3-5 дней)

**Критический блокер. Без этого альбом не собирается целиком.**

#### А.1 — Расширение photos.type + UI загрузки (~1.5 дня)

**Цель:** Фотограф может загружать фото общего раздела через раздел Фото в админке.

**Задачи:**
- Миграция БД: расширить enum `photos.type` или добавить колонку `photos.subtype`. Новые значения:
  - `common_spread` — фото на разворот
  - `common_full` — полностраничные (весь класс)
  - `common_half` — половинки
  - `common_quarter` — четвертинки
  - `common_sixth` — шестые
- Backend `/api/upload`: поддержать новые категории, складывать в `album_id/<category>/`
- Frontend в разделе Фото: 5 новых блоков загрузки с подходящими названиями
- WebP-сжатие как у других категорий (для UI/preview)

**Зависимости:** Никаких. Можно делать прямо сейчас.

#### А.2 — Builder использует общий раздел (~1.5 дня)

**Цель:** Smart-fill при `Собрать автоматически` заполняет общий раздел из загруженных фото.

**Задачи:**
- В `lib/album-builder/`: после личных разворотов и виньеток заполняем common-развороты
- Алгоритм: builder читает фото из новых категорий, мапит на мастера с подходящим `slot_capacity` (`photos_full`, `photos_half`, `photos_quarter`, `photos_sixth`)
- Использует уже имеющиеся семантические теги мастеров (`page_role='common'`)
- Лимит: настройка альбома «сколько разворотов в общем разделе» (по умолчанию из пресета комплектации)

**Зависимости:** А.1 должна быть готова (нужны загруженные фото).

#### А.3 — Виньетки как автоматический разворот (~1 день)

**Цель:** В Индивидуальных комплектах builder автоматически генерирует разворот с миниатюрами всех детей.

**Задачи:**
- В smart-fill: после личных разворотов вставляем виньеточный разворот
- Берёт всех детей альбома + их `portrait_page` selections + ФИО
- Использует существующие мастера с `page_role='student_grid'` (N-12-Left/Right, L-6-Left/Right + overflow-варианты) — они уже в БД и уже работают в Мини
- Логика выбора по числу учеников: N-12 при 19-36, L-6 при 13-24 (точное распределение из A1/A2 в master-cleanup-tz)

**Поправка от 11.05.2026:** ранее в этом пункте была пометка «зависит от дизайнера — возможно требует новых мастеров». Это была ошибка — виньеточные мастера (`page_role='student_grid'`) уже существуют в template_set okeybook-default, builder умеет с ними работать в Мини. Для Индивидуальной комплектации задача сводится к добавлению виньеточного разворота в её сценарий — никаких новых мастеров от дизайнера не требуется.

**Зависимости:** нет.

#### А.4 — UI настройки общего раздела в альбоме (~0.5 дня)

**Цель:** Фотограф/менеджер может настроить «сколько разворотов в общем разделе» для конкретного альбома.

**Задачи:**
- В настройках альбома: поле «Разворотов в общем разделе» с дефолтом из пресета
- Builder учитывает этот лимит при заполнении

### 🟠 Фаза В — Lifecycle YC + хранилище (полдня)

**Цель:** Снизить стоимость хранилища через Cold storage и автоматическое удаление старого.

**Текущая ситуация (10.05.2026):**
- Yandex Cloud Object Storage, 1 бакет `yearbook-photos`
- Занято: 598 МБ (2010 объектов)
- Все Standard класса (~₽1.95/ГБ/мес)
- Бакет публичный

**Прогноз нагрузки на сезон:**
- Средний альбом класса: 30 учеников × 100 фото × 5 МБ = 15 ГБ оригиналов + 3 ГБ WebP + 2 ГБ экспортов = **20 ГБ на альбом**
- 100 альбомов в сезон ≈ **2 ТБ**
- При 2 ТБ Standard = ₽3900/мес
- При lifecycle (Cold для завершённых после 60 дней) = ~₽1300/мес → экономия 60-70%

**Задачи:**
- Lifecycle policy в YC:
  - Активные альбомы (`workflow_status` active/ready/submitted/in_production) → Standard
  - Завершённые (delivered) после 60 дней → Cold (₽0.65/ГБ/мес)
  - `delivery/` после 6 месяцев → удаление (уже в БД через `expires_at`)
  - `exports/` после 90 дней → удаление (уже в БД через `expires_at`)
- Виджет в `/super`: «Yandex Cloud: занято X ГБ, прогноз ₽Y/мес»

**Стоимость YC (для справки):**
- Standard storage: ~₽1.95/ГБ/мес
- Cold storage: ~₽0.65/ГБ/мес
- Egress (скачивание родителями/builder'ом): ~₽1/ГБ
- GET-запросы: ~₽0.005 за 1000
- PUT-запросы: ~₽0.05 за 1000 (копейки)
- Биллинг: postpaid, списания посуточно. Можно зарядить условно ₽3000 на 6 месяцев.

### 🟡 Фаза Ж — Редактор пресетов комплектаций (7-10 дней)

**Эволюция из старого плана «кастомные комплектации» в полноценный визуальный редактор. Поднята в приоритете после обсуждения с Сергеем 11.05.2026.**

**Контекст:** А.4 (UI лимита разворотов общего раздела) сделана как ВРЕМЕННОЕ решение — даёт одно число «не более N разворотов» с хардкоженным приоритетом крупных фото. Но в реальности фотограф/дизайнер должен решать сам какой именно состав идёт в общий раздел. Например лимит «2 разворота» не должен означать «оба заполнятся только common_full», правильное поведение — «full × 1 + sixth × 1» как осознанный выбор.

**А.4 эволюционирует в Ж** — числовой инпут заменяется выбором пресета или прямой настройкой состава.

**Архитектура:**

`config_presets.config.common_section` обогащается полем `spread_slots`:
```jsonc
common_section: {
  enabled: true,
  spread_slots: [
    { type: "full_class", count: 1 },    // 1 разворот с фото класса
    { type: "half",       count: 2 },    // 2 разворота половин
    { type: "sixth",      count: 1 }     // 1 разворот коллажей
  ]
}
```

Builder тогда не «приоретизирует», а **точно знает** что и в каком порядке вставить. Если фото в категории не хватает — warning + пропуск. Если фото больше — лишние остаются в БД, не в layout.

**Задачи фазы Ж:**

1. **Миграция БД:** расширить структуру `config_presets.config.common_section` со `spread_slots`. Существующие глобальные пресеты получают дефолтные слоты.

2. **`config_presets` становится tenant-aware** с inheritance от глобальных. Партнёр может скопировать глобальный пресет → изменить → сохранить как свой.

3. **UI «Мои пресеты»** в кабинете партнёра:
   - Список своих пресетов + просмотр глобальных
   - «Скопировать пресет → изменить → сохранить как свой»
   - Создание пресета с нуля
   - Редактор формы:
     * Какие разделы включены (intro / учителя / личный / общий / виньетки)
     * Сколько разворотов в каждом разделе
     * Какие комплектации обложки доступны
     * Базовый layout_mode (single_page / spread / grid)
     * print_type compatibility
     * Состав общего раздела (spread_slots)

4. **Builder подстраивается:** `buildCommonSection` читает `spread_slots` вместо текущей приоритезации. А.4 fallback остаётся для пресетов без `spread_slots` (backward-compat).

5. **UI override на уровне альбома:** заменить текущее одиночное поле «Разворотов в общем разделе» на полноценный «настроить состав» — модалка где видны слоты пресета и можно поштучно изменить count или категорию.

**Зависимости:** нет (можно начинать сейчас, не нужны ответы дизайнера).

**Связь с А.4:** А.4 (поле common_section_max_spreads) остаётся как fallback и для простых случаев. После Ж появится главный механизм spread_slots, а common_section_max_spreads становится «общим суммарным лимитом сверху».

### ✅ Фаза К — Workflow цветокора и ретуши (ЗАКРЫТА 12.05.2026, 5 коммитов)

**ЗАКРЫТА** ~6 часов работы (вместо 3-4 дней по оценке) — поднялась в приоритете 11.05.2026 как критический блокер запуска партнёрки, выполнена 12.05.2026.

**Контекст:** в Б.1 уже работает автоматическая загрузка оригиналов фото для печати. Но процесс ретуши/цветокора нигде не описан и не интегрирован: фотограф/ретушёр должны выйти из системы, отретушировать локально в Lightroom/Photoshop, и как-то вернуть результат обратно — этого механизма нет.

**Сценарий партнёра:**
1. Все ученики собрали данные → альбом готов к производству
2. Фотограф (или его ретушёр) скачивает ZIP оригиналов
3. Локально в Lightroom/Photoshop ретуширует портреты, делает цветокор
4. Загружает обработанные файлы обратно в систему → они заменяют оригиналы в `photos.original_path`
5. PDF-экспорт автоматически использует новые версии

**Сделано:**

1. **Backend endpoint скачивания ZIP оригиналов (К.1, `0a8ebc6`):**
   - `GET /api/workflow/originals-zip?album_id=X` — серверный ZIP всех оригиналов
   - Опциональный `?categories=portrait,group,...` (CSV-фильтр по 8 категориям)
   - `?view_as=<tid>` для сотрудников OkeyBook от имени партнёра
   - Лимит 200 фото/запрос (~1 ГБ) → 413 + `by_category` для UI частичной выгрузки
   - Подпапки внутри ZIP по типу + `manifest.json` (id, filename, type, привязки к детям/учителям, zip_path, failures) + `README.txt` с инструкцией
   - Коллизии filename → префикс `<photo_id[:8]>_<filename>`
   - X-Originals-Total/Downloaded/Failed заголовки
   - audit_log: `workflow.download_originals_zip`

2. **UI «📥 Скачать оригиналы» во вкладке Производство (К.2, `409d150`):**
   - Новый блок «Цветокор и ретушь» (bg-blue-50) над legacy «Оригинальные фото», виден когда canEdit=true
   - При 413 раскрывается панель выбора категорий — чекбоксы с русскими лейблами и счётчиками, сумма vs лимит, кнопка «Скачать выбранные»
   - viewAsTenantId пробрасывается через props из AlbumDetailModal

3. **Backend register/rebind/discard_retouched (К.3, `9c92729`):**
   - `register_retouched` — body `{album_id, files:[{filename,storage_path}]}`. Безопасность: все storage_path должны начинаться с `yc:<album_id>/originals/`. Матчит по filename, UPDATE photos.original_path, удаляет старый файл из YC параллельно. Response `{matched, unmatched_count, unmatched, replaced}`
   - `rebind_retouched` — для К.5, ручная привязка по photo_id
   - `discard_retouched` — для К.5, удаление лишнего файла
   - audit_log на каждое действие

4. **UI загрузки обработанных + summary (К.4, `ff05e60`):**
   - Кнопка «📤 Загрузить обработанные» (input multiple)
   - Для каждого файла: presigned URL через `/api/upload-url` upload_type='originals' → PUT прямо в YC (обход 4.5 МБ лимита) → потом единый register_retouched
   - Progress-bar «Загружаем N/M»
   - 🟢 «✅ Обновлено N оригиналов» / 🟡 «⚠️ Не найдено K файлов: list»

5. **UI ручной привязки unmatched (К.5, `d89f633`):**
   - Lazy-load фото альбома через `/api/tenant?action=album_photos` (с view_as)
   - Inline autocomplete через нативный `<datalist>` — без отдельного модала
   - Кнопка «Привязать» (disabled пока photo_id не зарезолвлен) + 🗑 удаление
   - После rebind: убираем из unmatched, инкрементим matched в summary

**Отложено в Фазу Л.3 (редактор макета):**

- **Точечная замена одного оригинала** — в редакторе клик на фото → «Заменить оригинал» → загрузить файл. Используется для сценария «клиент попросил отретушировать одну фотографию после согласования». Технически делается через тот же `/api/upload-url` + `rebind_retouched`, но UI должен быть в редакторе макета (Фаза Л), не в ProductionTab.

**Известное ограничение (backlog):**

- `/api/upload-url` и actions `register_retouched/rebind_retouched/discard_retouched` в `/api/workflow` пока без поддержки `view_as` для не-superadmin. Это значит: сотрудник OkeyBook через partner cabinet (если он не superadmin) не сможет загружать retouched от имени партнёра — только сам партнёр или superadmin. Фикс: расширить эти endpoint'ы той же view_as логикой что в `/api/tenant`. Не блокер для запуска партнёрки (основной сценарий — партнёр сам грузит свои файлы).

**Зависимости:** Б.1 (сделано), не зависело от ответов дизайнера.

**Связь с PDF:** layout не пересобирается, потому что `original_path` обновляется in-place. При следующем экспорте PDF — новые версии используются автоматически.
