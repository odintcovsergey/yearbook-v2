## 1. Что такое переходный раздел

Это секция между students и common_required (общим обязательным разделом).
Её задача — **аккуратно закрыть раздел учеников** так, чтобы он выходил на
чётный разворот, и чтобы хвостовая часть выглядела сбалансированно.

Конкретные случаи:
- Личный раздел закончился на **левой** странице (1-4 ученика остались)
  → нужно достроить и низ левой, и правую целиком
- Личный раздел закончился на **правой** странице (1-3 ученика)
  → левая остаётся как есть (это последняя полная страница students),
  правую нужно достроить
- Личный раздел кончился ровно (нет хвоста)
  → переходного нет, сразу common_required

## 2. Полная таблица (xlsx)

См. `transition-section-spec.xlsx`, лист «Общий раздел».

Структура колонок:
| Колонка | Содержимое |
|---|---|
| 1 | Комплектация: Мини / Лайт / Медиум / Стандарт+Универсал / Максимум |
| 2 | Тип листов: плотные / мягкие |
| 3 | Диапазон учеников |

Колонки 6-8 — уже работают через `common_required` секцию, не трогаем.

## 3. Ключевые наблюдения

### 3.1 Комплектации — НОВОЕ понятие

Сейчас в системе:
- **Плотность** (1, 4, 6, 12 учеников/страницу) — параметр шаблона
- **Тип мастера students** — выбирается партнёром (E-Student-Left, N-Grid-12 и т.п.)

В таблице **комплектация** — широкое понятие, объединяющее и плотность,
и характер мастера students:
| Комплектация | Плотность | Что на странице |
|---|---|---|
| Мини | 12 | 12 портретов в сетке |
| Лайт | 6 | 6 портретов в сетке |
| Медиум | 4 | 4 портрета в сетке |
| Стандарт | 1 | 1 портрет + ФИО + текст (E-Student) |
| Универсал | 1 | 1 портрет + ФИО + текст + 2 фото с друзьями |
| Максимум | 1 на разворот | слева 1 портрет + ФИО, справа 4 фото с друзьями |

**КЛЮЧЕВОЙ ВОПРОС:** комплектация — это новое поле шаблона, или это
производное от выбранных мастеров students? Скорее всего — **новое поле**,
определяющее набор доступных мастеров для students-секции.

### 3.2 Combo-мастер «N учеников + classphoto» — один универсальный

Сергей подтвердил: **один универсальный мастер с максимальным числом
позиций** (например `J-Combined-4` с 4 позициями для портретов + 1 classphoto),
где лишние позиции **скрываются через `__hidden__<label>`**.

**Прецедент в коде существует:**
- `lib/balance-overrides/index.ts` — парсер ключей `__hidden__<label>` /
  `__pos__<label>`, применяемый одинаково в Canvas (превью) и PDF
- `lib/rule-engine/sections/teachers.ts:516-548` — рабочий пример:
  если в мастере teachers слотов больше чем subjects[], engine пишет
  `__hidden__subjectphoto_N` для каждой лишней позиции
- `BalanceClause` (types.ts:240) — флаги `placeholder_centering` и
  `hide_unfilled` уже зарезервированы, но автоматическая балансировка
  через rule-engine на данном этапе не используется (всё per-section)

Для combo-мастера нужен **этот же механизм**:
- Мастер `J-Combined-Tail-4` в InDesign содержит 4 портрет-слота +
  classphoto-слот в одной композиции
- Engine при сборке знает что хвост = N учеников (1, 2, 3 или 4)
- Заполняет первые N слотов учениками, лишние помечает `__hidden__`
- В мастере InDesign дополнительно потребуется **центрирование** —
  если 2 ученика, они должны быть посередине, а не слева

**Альтернатива:** семейство отдельных мастеров `J-Combined-1`,
`J-Combined-2`, `J-Combined-3`, `J-Combined-4` — каждый со своей геометрией.
Не выбрано — Сергей предпочитает универсальный.

### 3.3 Приоритетные альтернативы — НЕ жёсткое правило в коде

Для правой стороны переходного таблица даёт цепочки вида:
> «в приоритете 2 фото по 1/2 класса, либо 6 фото 1/6, либо 1 общая»

**Принятое решение:** в код таблицу жёстко НЕ зашиваем. Логика выбора
будет настраиваема через UI шаблона. См. секцию 4 ниже — это самый
интересный UX-вопрос фазы РЭ.37.

### 3.4 Хвост известен только в момент сборки, не в момент шаблона

Партнёр настраивает шаблон **заранее**, не зная сколько учеников
будет в конкретном альбоме. Поэтому в шаблоне партнёр настраивает
**сценарии** для всех возможных хвостов, а engine при сборке выбирает
нужный.

Это переворачивает текущую модель: было «один мастер на переходный»,
становится «таблица сценариев в шаблоне».

### 3.5 Симметризация — отдельная опциональная фича

См. РЭ.37.5 ниже. Применяется только к **Мини** и **Лайт**
(сетки 12 и 6). Для Медиум, Стандарт, Универсал, Максимум — бессмысленна.

## 4. Открытые UX-вопросы

### 4.1 Как выразить сценарии переходного в UI шаблона

**Контекст:** партнёр в редакторе шаблона должен настроить переходный
раздел. Сейчас один dropdown «Мастер переходной страницы». Этого мало —
нужно различать как минимум:
- Хвост попал на левую (нужно достроить лево + право)
- Хвост попал на правую (нужно достроить только право)
- Нет хвоста (нет переходного)

**Предложенный UX (требует подтверждения Сергея):**

Три режима, выбираемых радио-кнопкой:

**Режим A — «Стандартная логика OkeyBook» (рекомендуемая для новичков)**
Engine применяет правила из таблицы (см. xlsx). Партнёр ничего не настраивает.

**Режим B — «Один мастер на переходный» (текущее поведение)**
Партнёр выбирает один мастер из dropdown'а. Engine кладёт его на
правую сторону хвоста students (как сейчас). Для тех кому не нужны
сложные таблицы.

**Режим C — «Свой сценарий: настроить лево/право вручную» (для продвинутых)**
Партнёр в UI видит **визуальный конструктор**: два мокапа разворотов.

```
┌─ Сценарий A: хвост на ЛЕВОЙ странице ─────────┐
│ ┌──────────────────┐  ┌──────────────────┐    │
│ │  ЛЕВАЯ           │  │  ПРАВАЯ          │    │
│ │  ┌─ выбор ─┐     │  │  ┌─ выбор ─┐     │    │
│ │  │J-Comb-4 │     │  │  │J-Half   │     │    │
│ │  └─────────┘     │  │  └─────────┘     │    │
│ │  до 4 уч+classph │  │  2 фото 1/2 кл   │    │
│ └──────────────────┘  └──────────────────┘    │
└────────────────────────────────────────────────┘

┌─ Сценарий B: хвост на ПРАВОЙ странице ────────┐
│ ┌──────────────────┐  ┌──────────────────┐    │
│ │  ЛЕВАЯ           │  │  ПРАВАЯ          │    │
│ │  (последняя      │  │  ┌─ выбор ─┐     │    │
│ │   полная стр.    │  │  │J-Comb-3 │     │    │
│ │   students —     │  │  └─────────┘     │    │
│ │   авто)          │  │  до 3 уч+classph │    │
│ └──────────────────┘  └──────────────────┘    │
└────────────────────────────────────────────────┘
```

Для каждого слота — dropdown с подходящими мастерами (фильтруется
по комплектации и пулу доступных combo-N мастеров).

**Альтернатива:** более простой UI — таблица «хвост = N → мастер X»
без визуальных мокапов. Менее наглядно, но проще реализовать.

### 4.2 Как ввести понятие «комплектация» в систему

Варианты:
1. **Новое поле template_set'а** — выбирается один раз при создании дизайна,
   определяет набор доступных мастеров students. Партнёр не может смешивать
   Мини с Лайт в одном альбоме.
2. **Производное от выбранных мастеров** — engine анализирует какие
   мастера students использованы в шаблоне и определяет комплектацию.
   Гибче, но сложнее логически.
3. **Поле пресета (config_preset)** — комплектация привязана к конкретному
   пресету шаблона. Один template_set может иметь несколько пресетов под
   разные комплектации.

Голос за вариант 3 (по аналогии с тем как пресет уже определяет
section_structure).

### 4.3 Что делать при mismatch (партнёр настроил Режим C но фото нет)

Например: партнёр указал «правая = 2 фото 1/2 класса», но в альбоме
не загружены half_class фото. Что делает engine?

Варианты:
1. Применяет настройку, оставляет пустые слоты → плохо визуально
2. Падает к следующему по приоритету варианту (нужна цепочка) → как
   в Режиме A
3. Кидает warning «настройка переходного не сработала, нужны фото X»
   и пропускает страницу

Голос за **3** — честно говорим что не получилось. В UI редактора
warning будет видим через РЭ.36.UI плашку.

## 5. Структурные изменения в коде (черновик)

### 5.1 Тип SectionStructureEntry для transition

Старый формат:
```ts
{ type: 'transition', master_name?: string }  // один мастер или 'auto'
```

Новый формат (черновик):
```ts
{
  type: 'transition',
  mode: 'okeybook_default' | 'single_master' | 'custom',
  // Режим B: текущее поведение
  single_master?: string,
  // Режим C: пользовательский сценарий
  custom?: {
    // когда хвост попал на левую сторону — нужны оба мастера
    tail_left: {
      left: { master_name: string },  // combo-N
      right: { master_name: string }, // J-Half / J-Full / ...
    },
    // когда хвост попал на правую — только правый
    tail_right: {
      right: { master_name: string }, // combo-N
    },
  },
  // Симметризация — отдельный флаг
  symmetrize_tail?: boolean,  // только для Мини/Лайт
}
```

### 5.2 fillTransitionSection — анализ хвоста

```ts
function fillTransitionSection(ctx, params) {
  // 1. Определить где закончились ученики (left/right последнего разворота)
  // 2. Применить симметризацию если включена И комплектация Мини/Лайт
  //    (перебалансировать предпоследнюю страницу)
  // 3. Применить выбранный режим:
  //    - okeybook_default → engine.applyTransitionTable()
  //    - single_master → engine.fillRightWithMaster()
  //    - custom → engine.fillFromCustomScenario()
  // 4. Для combo-N: bindings заполняют первые N слотов, остальные __hidden__
  // 5. Возможно — placeholder_centering для combo-N (если 2 ученика — посередине)
}
```

### 5.3 Combo-мастера в каталоге (требуют дорисовки в InDesign)

| Имя | Назначение | Placeholders |
|---|---|---|
| `J-Combined-Tail-2` | До 2 учеников + classphoto | student_photo_1..2, student_name_1..2, classphoto |
| `J-Combined-Tail-3` | До 3 учеников + classphoto | student_photo_1..3, student_name_1..3, classphoto |
| `J-Combined-Tail-4` | До 4 учеников + classphoto | student_photo_1..4, student_name_1..4, classphoto |

Возможно — единый `J-Combined-Tail-4` с placeholder'ами под все 4
ученика, где лишние скрываются (тот самый универсальный подход).

Тогда мастеров combo нужно 1, а не 3. Это правильно если 4-местная
композиция при скрытии 2 слотов и центрировании оставшихся даёт
визуально приемлемый результат. Если нет — нужно семейство.

**Это вопрос к Сергею после набросков в InDesign.**
