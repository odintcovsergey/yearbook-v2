# РЭ.37 — Конструктор переходного раздела (спецификация, финал)

**Статус:** требования зафиксированы. Готово к реализации в новом чате под РЭ.37.
**Источник истины:** `docs/transition-section-spec.xlsx` (загружен Сергеем 24.05.2026).
**Создан:** v154. Финализирован после диалога Сергея 24.05.2026 вечер.

---

## 1. Что такое переходный раздел

Секция между students и common_required (обязательным общим разделом).
Её задача — аккуратно закрыть раздел учеников, чтобы он выходил на чётный
разворот, и чтобы хвостовая часть выглядела сбалансированно.

Случаи хвоста:
- Хвост попал на **левую** страницу (1-4 ученика) → достроить низ левой
  (combo-мастером N портретов + classphoto) + всю правую (вариант общего
  раздела)
- Хвост попал на **правую** страницу (1-3 ученика) → левая остаётся как
  есть (последняя полная страница students), правую достраиваем combo-мастером
- Нет хвоста → переходного нет, сразу common_required

## 3. Концептуальные решения

### 3.1 Комплектация — характеристика шаблона, выводится автоматически

**Комплектация = существующее понятие шаблона**, не отдельное поле. Engine
определяет её по мастерам students-секции:

| Комплектация | Признак (мастер students) |
|---|---|
| Мини | N-Grid-12 (12 портретов в сетке на странице) |
| Лайт | N-Grid-6 (6 портретов) |
| Медиум | N-Grid-4 (4 портрета) |
| Стандарт | E-Student-Left/Right без слотов друзей |
| Универсал | E-Student-Left/Right с 2 слотами друзей |
| Максимум | M-Student-Spread (разворот на ученика) |

**Mismatch case:** шаблон может содержать несколько students-секций с
разными мастерами. Например: основной режим «1 ученик/страницу» (E-Student)
+ дополнительная секция-сетка для детских фото (N-Grid-12 — для детей из
детского сада тех же выпускников). В таком случае engine ориентируется на
**наличие сетки**: если есть N-Grid-12 → применяются правила Мини, если
есть N-Grid-6 → Лайт, и т.д. Это позволяет переходному применить правильные
combo-мастера.

### 3.2 Combo-мастер «N учеников + classphoto» — 3 универсальных мастера

**Решение Сергея:** один универсальный мастер на каждую комплектацию-сетку,
с максимальным числом слотов. Лишние слоты скрываются через `__hidden__<label>`.

| Мастер | Комплектация | Слоты | Сценарий использования |
|---|---|---|---|
| `J-Combined-Tail-4` | Мини | 4 портрета + 1 classphoto | Хвост 1-4 ученика после N-Grid-12 |
| `J-Combined-Tail-3` | Лайт | 3 портрета + 1 classphoto | Хвост 1-3 ученика после N-Grid-6 |
| `J-Combined-Tail-2` | Медиум | 2 портрета + 1 classphoto | Хвост 1-2 ученика после N-Grid-4 |

**Геометрия и пропорции у разных мастеров разные** (Мини — мелкие портреты,
Медиум — крупные). Универсальность работает **внутри** одной комплектации:
J-Combined-Tail-4 при хвосте 2 ученика → 2 заполненных слота + 2 скрытых
+ центрирование оставшихся (флаг `placeholder_centering` в BalanceClause).

**Прецедент скрытия слотов в коде:**
- `lib/balance-overrides/index.ts` — парсер ключей `__hidden__<label>` /
  `__pos__<label>`, применяется одинаково в Canvas (превью) и PDF
- `lib/rule-engine/sections/teachers.ts:516-548` — рабочий пример: если в
  мастере teachers слотов больше чем subjects[], engine пишет
  `__hidden__subjectphoto_N` для каждой лишней позиции

Для combo-N нужен **тот же механизм** + автоматическое центрирование
оставшихся слотов (опционально).

### 3.3 Приоритетные альтернативы для правой стороны переходного

В таблице цепочки вида «в приоритете 2 фото по 1/2 класса, либо 6 фото 1/6,
либо 1 общая». **Жёстко в код не зашиваем** — будет частью «Стандартной
логики OkeyBook» как настройка по умолчанию. Engine знает приоритетный
порядок из конфига, при сборке падает по цепочке: пробует первый вариант,
если фото нет (пустой пул half_class) — пробует следующий.

### 3.4 Симметризация — опциональная галочка, только сетки 12 и 6

Применяется к комплектациям **Мини и Лайт**. Бессмысленна для Медиум,
Стандарт, Универсал, Максимум.

**Что делает:**
- Если хвост учеников = 1 (одинокий портрет в большой сетке) → engine
  забирает 1 ученика с предыдущей полной страницы → теперь на хвостовой
  странице 2 ученика, на предыдущей — на 1 меньше (неполный ряд)
- Оба «дефицита» центрируются: хвостовая пара по центру combo-мастера,
  неполный последний ряд предыдущей страницы — по центру сетки

**Без симметризации** combo-мастера применяются как есть, но визуально
менее красиво (одинокий портрет с краю).

## 4. UI шаблона — что меняется

### 4.1 Секция «Личный раздел» — добавляется галочка симметризации

Сейчас (см. скриншот PresetEditorModal):
```
ЛИЧНЫЙ РАЗДЕЛ
[Режим: Сетка из N учеников на страницу]
[Учеников на страницу: 6]
[Цитата: нет]
```

После РЭ.37:
```
ЛИЧНЫЙ РАЗДЕЛ
[Режим: Сетка из N учеников на страницу]
[Учеников на страницу: 6]
[Цитата: нет]
[☑ Симметризировать хвост]  ← НОВОЕ, активна только для N ∈ {6, 12}
  ↳ tooltip: «Если в последнем ряду остался один ученик, движок
     возьмёт ещё одного с предыдущей страницы, чтобы хвост был
     парным. Применяется только к сеткам 6 и 12.»
```

Галочка disabled для N ∈ {1, 4}, скрыта для не-grid режимов.

### 4.2 Секция «Структура альбома» → переходный имеет 2 режима

Сейчас элемент transition в section_structure — один dropdown «Мастер
переходной страницы».

После РЭ.37 — кликабельный элемент transition открывает мини-редактор
с radio:

**Радио 1: Стандартная логика OkeyBook (рекомендуется)** ← дефолт
> Движок сам решит как достроить хвост учеников, по правилам OkeyBook
> (см. таблицу). Применяет combo-мастера + цепочку приоритетов для
> правой стороны.

**Радио 2: Свой сценарий**
> Открывается визуальный конструктор с двумя мокапами разворотов:

```
СЦЕНАРИЙ A — хвост попал на ЛЕВУЮ страницу:
┌─────────────────┐  ┌─────────────────┐
│  ЛЕВАЯ          │  │  ПРАВАЯ         │
│  [Combo-4    ▼] │  │  [J-Half      ▼]│
│  до 4 уч + cls  │  │  2 фото 1/2     │
└─────────────────┘  └─────────────────┘

СЦЕНАРИЙ B — хвост попал на ПРАВУЮ страницу:
┌─────────────────┐  ┌─────────────────┐
│  ЛЕВАЯ          │  │  ПРАВАЯ         │
│  (auto:         │  │  [Combo-3    ▼] │
│  последняя      │  │  до 3 уч + cls  │
│  полная стр.    │  │                 │
│  students)      │  │                 │
└─────────────────┘  └─────────────────┘
```

Dropdown'ы фильтруются по комплектации (для Лайт показываются только
combo-3 и подходящие правые мастера).

### 4.3 Правки на уровне альбома (не шаблона) — БЕЗ ИЗМЕНЕНИЙ

Существующая кнопка «Заменить шаблон правой страницы» в редакторе альбома
продолжает работать как есть. Партнёр после сборки может вручную поменять
мастер любой страницы — это отдельный уровень правок, не пересекающийся
с настройками шаблона.

`has_user_edits=true` уважается при ре-сборке (engine не перезаписывает
правки партнёра).

## 5. Структурные изменения в коде

### 5.1 Тип SectionStructureEntry для transition

Старый формат:
```ts
{ type: 'transition', master_name?: string }
```

Новый формат:
```ts
{
  type: 'transition',
  mode: 'okeybook_default' | 'custom',
  // Только для mode='custom':
  custom?: {
    tail_left: {  // хвост попал на левую страницу
      left: { master_name: string },   // combo-N
      right: { master_name: string },  // J-Half / J-Full / J-Collage-6 / ...
    },
    tail_right: {  // хвост попал на правую страницу
      right: { master_name: string },  // combo-N
    },
  },
}
```

### 5.2 Симметризация — флаг в preset, не в section_structure

```ts
// config_presets table — новое поле
symmetrize_students_tail: boolean  // default false
```

Применяется только если комплектация = Мини или Лайт (engine проверяет
наличие N-Grid-12 или N-Grid-6).

### 5.3 fillTransitionSection — новая логика

```ts
function fillTransitionSection(ctx, params) {
  // 1. Определить комплектацию: посмотреть мастера students-секций,
  //    если есть N-Grid-12 → Мини, N-Grid-6 → Лайт, и т.д.
  // 2. Если symmetrize_students_tail И комплектация Мини/Лайт И хвост=1:
  //    забрать 1 ученика с предыдущей полной страницы students,
  //    перебалансировать предыдущую (placeholder_centering).
  // 3. Определить где сел хвост (left/right).
  // 4. Применить выбранный режим:
  //    - okeybook_default → engine.applyTransitionTable(complectation, students_count)
  //    - custom → engine.fillFromCustomScenario(custom.tail_left | custom.tail_right)
  // 5. Для combo-N: заполнить первые K слотов учениками, остальные __hidden__.
  //    Опционально — placeholder_centering для оставшихся.
  // 6. Если ни один вариант приоритетной цепочки не сработал → warning.
}
```

### 5.4 Стандартная логика OkeyBook — данные

Таблица из xlsx переводится в TS-структуру (можно в конфиге пресета или
в коде engine — это деталь реализации, обсудим в РЭ.37.2):

```ts
type TransitionRule = {
  complectation: 'mini' | 'light' | 'medium' | 'standard' | 'universal' | 'maximum',
  print_type: 'soft' | 'hard',
  students_range: [min: number, max: number],
  tail_left: {
    left_master: string | null,   // null = нет хвоста на левой
    right_priority: string[],     // цепочка приоритетов
  },
  tail_right: {
    right_master: string | null,
    right_priority: string[],
  },
}
```

## 7. Открытые риски

1. **Центрирование внутри combo-N** — нужен ли алгоритм автоматического
   центрирования placeholder'ов при скрытии, или Сергей в InDesign рисует
   несколько композиций под каждое количество слотов? Сейчас в коде есть
   флаг `placeholder_centering` (BalanceClause), но автоматическая
   реализация на уровне engine не написана. **Решить в РЭ.37.4.**

2. **Mismatch комплектаций — детский сад + школа** — нужно ли engine
   запускать переходный **отдельно** для секции детского сада (N-Grid-12)
   и для школы (E-Student)? Или достаточно одного переходного перед
   common_required? **Решить в РЭ.37.1 с Сергеем на примере.**

3. **Соответствие dropdown'а custom-режима реальному набору мастеров
   template_set** — если в дизайне нет J-Combined-Tail-3 (Сергей ещё не
   нарисовал), что показывать в dropdown'е custom? Скорее всего нужно
   фильтровать по фактическому slot_capacity и комплектации. **Решить в
   РЭ.37.6.**
