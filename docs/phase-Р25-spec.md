# Фаза РЭ.25 — Галка покупки альбома ребёнком

**Спецификация v1.0**
**Дата:** 21.05.2026
**Статус:** утверждена 21.05.2026 (browser-чат с Сергеем, после закрытия РЭ.24).
**Зависит от:** РЭ.22 (семантический engine), РЭ.24 (партнёрский каталог шаблонов — для понимания, что секция `students` строится через `section_structure`).
**Эстимация:** ~6-8 коммитов на main + 1 миграция БД. Один день работы.

---

## 1. Зачем фаза существует

Не все ученики класса заказывают выпускной альбом. Сейчас система этого не учитывает: engine собирает в секцию `students` (личные страницы) **всех** учеников из таблицы `children`. Если в классе 25 учеников и 3 не заказали альбом — фотограф получает 25 личных страниц вместо 22, что:
- увеличивает себестоимость печати на необоснованные страницы,
- ломает соответствие «оплачено = выдано» (фотограф фактически дарит страницы тем, кто не платил),
- противоречит реальной практике школьной фотографии.

При этом важно: **факт неучастия в заказе не означает удаления ребёнка из альбома**. На общих фото класса все 25 человек физически стоят рядом, виньетка — это **состав класса**, а не «список заказчиков». Фильтр должен быть **точечным**: только секция `students`, всё остальное — полный класс.

Дополнительно фаза вводит **переопределение на уровне альбома**: бывают ситуации (мало не-заказчиков, фотограф хочет дать всем по странице), когда нужно **выключить фильтр для конкретного альбома** одной галкой, не трогая `is_purchased` каждому ребёнку.

## 2. Модель данных (две галки)

### 2.1. `children.is_purchased BOOLEAN`

Поле на каждом ученике. Означает «этот ребёнок заказывает альбом».

- `true` — заказывает (по умолчанию)
- `false` — не заказывает

**Кто меняет:**
- **фотограф / админ** через UI `/app` (форма редактирования ученика);
- **родитель** через UI `/[token]` (страница отбора фото ребёнка).

Оба канала пишут в **одно поле**. Последнее изменение побеждает (last-write-wins) — никакой блокировки, никаких флагов «фотограф запретил менять». Логика реальная: родитель сначала отказался, после отбора фото передумал, фотограф увидел в списке и проставил обратно.

**Дефолт:** `true`. Это критично для миграции существующих альбомов: все существующие дети считаются заказчиками, engine ведёт себя как до фазы.

### 2.2. `albums.include_non_purchasers BOOLEAN`

Поле на каждом альбоме. Переопределяет фильтр секции `students` для всего класса.

- `false` (по умолчанию) — **строгое поведение**: не-заказчики (`is_purchased=false`) НЕ попадают в личные страницы.
- `true` — **мягкое поведение**: все ученики попадают в личные страницы независимо от `is_purchased`.

**Когда `true` нужен:**
- не-заказчиков мало (2-3 на класс), фотограф решает «не плодить дырки, дам всем по странице»;
- партнёр не хочет вести учёт заказов вообще (старая модель работы — все одинаково);
- старые альбомы после миграции (дефолт = `false`, но эффекта нет, потому что `is_purchased=true` у всех детей).

**Кто меняет:** фотограф / админ через UI `/app` (форма альбома, рядом с выбором шаблона). Родитель этот флаг не видит.

**Дефолт:** `false`. Безопасен для существующих альбомов (см. п. 2.1).

### 2.3. Per-child переопределение НЕ вводится

Третий флаг «всё равно дать страницу этому ребёнку» **не добавляется**. Если фотограф хочет вставить конкретного не-заказчика в личный раздел — он поднимает `is_purchased=true` именно этому ребёнку. Одна сущность, одна галка, один механизм. Не усложняем.

## 3. Что меняется в БД

Миграция `migrations/2026-05-21-children-is-purchased-and-album-include-non-purchasers.sql` (аддитивная, обратно совместимая):

```sql
-- Галка покупки альбома ребёнком (РЭ.25).
ALTER TABLE children
  ADD COLUMN is_purchased BOOLEAN NOT NULL DEFAULT true;

-- Переопределение на уровне альбома: включить ли не-заказчиков
-- в личный раздел. По умолчанию выключено = строгое поведение.
ALTER TABLE albums
  ADD COLUMN include_non_purchasers BOOLEAN NOT NULL DEFAULT false;

-- Индекс для будущих агрегаций «N из M заказали» в карточке альбома.
CREATE INDEX IF NOT EXISTS idx_children_is_purchased_album
  ON children (album_id, is_purchased);
```

**Безопасность миграции:** все колонки `NOT NULL DEFAULT`, существующие записи получают дефолты автоматически. Старый код, не знающий о новых колонках, продолжает работать — он просто их не читает и не пишет. Это zero-downtime изменение (см. правила миграций в `yearbook-context-v117.md`).

## 4. Engine: фильтр в секции `students`

### 4.1. Где применяется фильтр

**Единственное место** — в `lib/smart-fill/build-album-input.ts` при наполнении `AlbumInput.students` из таблицы `children`.

Логика:
```
if (album.include_non_purchasers === true) {
  // мягкий режим: всех в личный раздел
  students = все children
} else {
  // строгий режим: фильтруем не-заказчиков
  students = children.filter(c => c.is_purchased === true)
}
```

Это значит:
- `AlbumInput.students` уже **отфильтрован** к моменту входа в `buildAlbum`. Engine остаётся **чистым**: ничего не знает про `is_purchased`, работает только со списком учеников который ему дали.
- Фильтр **не размазан** по builder'у. Один источник истины, легко отлаживать.

### 4.2. Где фильтр НЕ применяется

Никогда, ни при каких условиях:
- **teachers** (учительская секция) — не зависит от учеников.
- **common** (общие фото класса: `full_class`, `half_class`, `spread`, `quarter`, `sixth`) — физически на фото все ученики класса. Не-заказчики остаются.
- **виньетки** (если в дизайне есть мастер с виньеткой класса) — виньетка отражает **состав класса**, а не список заказчиков. Все ученики на ней присутствуют.
- **intro / final** — не зависят от списка учеников.

Это значит: ребёнок с `is_purchased=false` (при `include_non_purchasers=false`):
- **не получает** персональной страницы,
- но **остаётся видимым** в общих фото и виньетке как часть класса.

### 4.3. Что НЕ делаем в engine

- **Не добавляем** проверку `is_purchased` внутрь `fillStudentsSection` / `buildFromSectionStructure`. Engine продолжает работать с `students[]` как с непрозрачным списком.
- **Не добавляем** поля в `Student` тип (`AlbumInput`). Внутрь движка попадают только те, кто должен попасть.
- **Не меняем** баланс / find-master / load-bundle / preview-bundle. Они не знают про `is_purchased`.

## 5. UI контракт

### 5.1. UI фотографа: форма ученика в `/app`

Список учеников альбома (`/app/album/[id]` или модалка редактирования) получает чекбокс:

```
[✓] Заказывает альбом
```

- По умолчанию **проставлен** (`is_purchased=true`).
- Снятие → `false`. Поднятие обратно → `true`.
- Сохраняется через `update_child` action (новый, см. §6).

**Визуальный маркер в списке:** не-заказчики (`is_purchased=false`) отображаются с приглушённым стилем (например, иконка-метка «не заказывает» или серая полоска). Чтобы фотограф мгновенно видел кто без альбома, не открывая каждого.

### 5.2. UI фотографа: форма альбома в `/app`

В `AlbumFormFields` (форма создания / редактирования альбома) ниже блока выбора шаблона:

```
[ ] Включить в личный раздел всех учеников, даже не заказавших альбом

  Подсказка: если выключено (по умолчанию) — личные страницы получают
  только те ученики, у которых проставлена галка «Заказывает альбом».
  Включи, если хочешь дать страницу всему классу независимо от заказа.
```

- По умолчанию **снят** (`include_non_purchasers=false`).
- Сохраняется через `update_album` / `create_album` (новое поле в payload, см. §6).

### 5.3. UI родителя: страница ребёнка в `/[token]`

На странице отбора фото (`app/[token]/page.tsx`) добавляется блок (вверху или в боковой панели):

```
[✓] Мы заказываем альбом

  Если вы не покупаете альбом, снимите галочку. Личной страницы для
  ребёнка в альбоме не будет, но он останется на общих классных фото
  и в виньетке класса.
```

- По умолчанию **проставлен** (родитель видит текущее значение из БД).
- Снятие → `false`, проставление обратно → `true`. Изменение **доступно всегда**, даже после `submitted_at` — потому что фактически родитель может передумать в любой момент.
- Сохраняется через **новый action в `/api/select` или `/api/draft`** (см. §6) — токен-аутентификация уже работает в этих endpoint'ах.

### 5.4. UI карточки альбома: счётчик заказов

В `/app` (список альбомов) на карточке альбома показывается:

```
Заказали: 22 из 25
```

- Цифры берутся из `children` через `COUNT WHERE is_purchased=true` и `COUNT total`.
- Если `include_non_purchasers=true` — счётчик **не показывается** (не релевантен, в альбоме все 25 в любом случае).
- Если у альбома 0 учеников — счётчик не показывается.

Это **наблюдаемость**, не обязательная для базовой работы фазы фича. Можно вынести в отдельный подэтап 25.6.

## 6. API контракт

### 6.1. Новый action: `update_child` (POST `/api/tenant`)

Универсальный action редактирования ребёнка (раньше был только `update_child_preset`). Принимает частичный patch:

```typescript
// Request
{
  action: 'update_child',
  child_id: string,
  full_name?: string,        // если передано → обновить
  class?: string,            // если передано → обновить
  is_purchased?: boolean,    // если передано → обновить (РЭ.25)
}

// Response
{ ok: true } | { error: string }
```

- Проверка доступа через `assertChildAccess(auth, child_id)` (уже есть в коде).
- Аудит-лог `child.update` с указанием изменённых полей.
- Если все поля undefined → ответ `{ error: 'нечего обновлять' }` 400.

**Почему универсальный, а не `update_child_purchased`:** разрастание action'ов «одно поле = один action» приведёт к лесу `update_child_full_name` / `update_child_class` / `update_child_purchased`. Уже сейчас `update_child_preset` отдельный — это историческое легаси. Лучше один атомарный patch-action.

`update_child_preset` оставляем как есть для обратной совместимости — не трогаем, чтобы не ломать существующий UI.

### 6.2. Расширение action'ов `create_album` и `update_album`

Добавить опциональное поле `include_non_purchasers: boolean`:

```typescript
// create_album / update_album request — добавляется:
{
  // ...существующие поля
  include_non_purchasers?: boolean,
}
```

Если не передано — в `create_album` пишется дефолт `false`, в `update_album` поле не трогается (текущее значение остаётся).

### 6.3. Новый action для родителя: `child_purchase` (PUT `/api/select`)

В `/api/select` (которое уже работает по токену ребёнка) добавляется новый action:

```typescript
// PUT /api/select
{
  token: string,
  action: 'set_purchased',
  value: boolean,
}
```

- Аутентификация — по `access_token` ребёнка (уже работает в `/api/select`).
- Доступно **всегда**, даже при `submitted_at !== null` (родитель может передумать).
- При смене с `true` → `false` после `submitted_at` — в будущем (вне РЭ.25) можно добавить уведомление фотографу через `parent_messages`. В этой фазе **не делаем** — фиксируем как возможное расширение.

**Почему `/api/select`, а не `/api/tenant`:** `/api/tenant` требует cookie-аутентификацию партнёра. Родитель работает только с токеном. `/api/select` уже принимает токен, добавление одного action туда — естественное расширение.

### 6.4. Расширение CSV-экспорта (`action=export_csv`)

В CSV-экспорт добавляется колонка `Заказ` (значения `да` / `нет`). Это нужно второму проекту (автовёрстка InDesign) чтобы он тоже мог фильтровать.

Позиция колонки — сразу после `Класс` / `Ученик` / `Портрет_страница`, перед остальными. Конкретное место уточнить при реализации (25.5), не критично.

## 8. Архитектурные развилки (зафиксированы)

**A. Кто проставляет галку — фотограф или родитель?** ✅ **Решение: оба, last-write-wins.** Родитель может передумать после отбора, фотограф увидит и подтвердит / поправит. Одно поле, две точки входа.

**B. Не-покупатели в общих фото класса (секция `common`)?** ✅ **Решение: попадают.** Фото класса — это фото класса, не «фото заказчиков». Дырки в кадре невозможны. Виньетка тоже отражает **состав класса**.

**C. Не-покупатели в личных страницах (секция `students`)?** ✅ **Решение: НЕ попадают по умолчанию (`include_non_purchasers=false`), но партнёр может включить через флаг альбома.** Per-child переопределение НЕ добавляется (избыточно — поднимай `is_purchased` обратно).

**D. Дефолт миграции для существующих альбомов?** ✅ **Решение: `children.is_purchased DEFAULT true`, `albums.include_non_purchasers DEFAULT false`.** Существующие альбомы работают как до фазы (все ученики в личном разделе, потому что у всех `is_purchased=true`).

**E. Где живёт фильтр — в engine или в smart-fill?** ✅ **Решение: в smart-fill.** Engine получает уже отфильтрованный `AlbumInput.students[]` и не знает про `is_purchased`. Один источник истины, чистый engine.

**F. CSV-экспорт колонка `Заказ`?** ✅ **Решение: добавляем.** Второй проект (автовёрстка InDesign) тоже должен видеть статус — это потенциально влияет на пагинацию в его собственной логике.

**G. Уведомление фотографу при смене родителем `true→false` после `submitted_at`?** ✅ **Решение: в РЭ.25 НЕ делаем.** Возможное расширение через `parent_messages`, открываем когда станет реальной болью.

## 9. Что НЕ входит в РЭ.25 (явно)

- **Уведомления** фотографу о смене статуса родителем — отложено (см. §8 G).
- **История изменений** `is_purchased` (audit log на уровне поля, кто и когда менял) — пока не требуется. Стандартный аудит-лог `child.update` зафиксирует факт изменения, но не значение до/после на уровне колонок.
- **Per-child переопределение** «всё равно дать страницу» — не делаем (см. §2.3).
- **Фильтр для виньетки класса** — не делаем, виньетка всегда полная (см. §4.2).
- **Per-album тип счётчика** в карточке (например, «3 не заказали» вместо «22 из 25») — оставляем «22 из 25» как универсальную формулу.
- **Связь со скидками / биллингом** — биллинг живёт у Сергея отдельно через менеджера. `is_purchased` это только про вёрстку, не про деньги.
- **Engine ничего не знает про `is_purchased`** — фильтр в smart-fill (§4.3).

## 10. Принципы

- **Аддитивная миграция.** Все колонки `NOT NULL DEFAULT`, существующие записи получают дефолты автоматически. Старый код, не знающий о новых колонках, продолжает работать.
- **Не трогать защищённые страницы.** `/[token]` редактируется **аддитивно** — только добавляется один чекбокс + один запрос. Логика отбора фото, draft, submit не меняется.
- **Tenant-aware строго.** `update_child` идёт через `assertChildAccess(auth, child_id)`. `set_purchased` идёт через token-аутентификацию ребёнка (уже работает).
- **Чистый engine.** Builder остаётся непрозрачным к `is_purchased`. Фильтр применён до входа в `buildAlbum`. Тесты engine не меняются.
- **Backward-compat.** Все 395 существующих тестов проходят. Новые тесты добавляются точечно (smart-fill фильтр).
- **Малые шаги с откатом.** Каждый подэтап — отдельный коммит. Если что-то ломается — откатывается одной командой.

## 11. Связь с предыдущими и будущими фазами

**РЭ.24 (закрыта):**
- Партнёр уже выбирает шаблон через каталог. РЭ.25 добавляет к этому **галку «включить всех учеников»** рядом с выбором шаблона. Никакого конфликта.
- Engine с `section_structure` и `student_layout_mode` остаётся в работе. Секция `students` строится через тот же путь, просто на отфильтрованных учениках.

**РЭ.27 (`print_type` в альбоме, после РЭ.25):**
- Параллельная фаза. Никакого пересечения с `is_purchased` / `include_non_purchasers`.
- Когда РЭ.27 закроется и редактор получит двойную модель (разворотная / постраничная) — фильтр учеников остаётся в smart-fill, ничего не меняется.

**РЭ.28 (партнёрские дизайны, после РЭ.27):**
- Не пересекается. `is_purchased` — атрибут конкретного ребёнка в конкретном альбоме, не зависит от дизайна.
