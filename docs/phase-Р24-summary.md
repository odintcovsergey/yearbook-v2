# Фаза РЭ.24 — Готовые шаблоны для быстрого старта партнёра

**Статус**: ✅ Закрыта
**Сроки**: 20.05.2026 (одна большая сессия от РЭ.24.0 до РЭ.24.8)
**Главная цель**: дать партнёру **витрину** готовых шаблонов от OkeyBook и личную библиотеку для их клонирования и редактирования. Партнёр выбирает шаблон при создании альбома вместо манипуляций с legacy-селектами «комплектация + тип печати».

## Что было до фазы

После РЭ.22 (двух-осевая модель + семантический engine) и РЭ.23 (визуальный каталог мастеров + автогенерация SVG-превью) у нас были:
- Engine с поддержкой произвольных пресетов через `section_structure` и `student_layout_mode`
- Каталог мастеров `/super/master-catalog` с SVG-превью
- Чистая функция `renderPreviewSvg` для генерации схематических превью

Но **партнёр** этого ничего не видел. У него:
- В `/super/presets` доступа нет (это административная зона)
- При создании альбома — два legacy-селекта `config_type` + `print_type`, собирающих `preset_slug` для resolvePresetBySlug
- Никакой возможности увидеть **что получится** до создания альбома
- Никакой личной библиотеки настроек

## Что сделано

### 1. Подготовка БД (РЭ.24.1)

Миграция `2026-05-20-presets-recommended-and-parent.sql`:
- `presets.is_recommended BOOLEAN NOT NULL DEFAULT false` — фильтр для каталога. Только глобальные (`tenant_id IS NULL`) с `is_recommended=true` показываются партнёрам.
- `presets.parent_preset_id UUID NULL REFERENCES presets(id) ON DELETE SET NULL` — для трейсинга клонов. У клонированных = id оригинального глобального.
- Индекс `idx_presets_parent_preset_id` (partial WHERE NOT NULL).

Аддитивная миграция, существующие записи не задеты.

### 2. Чистые утилиты для каталога (РЭ.24.2 + 24.3)

**`lib/presets/validate.ts`** — `validatePreset(preset) → { valid, errors[] }`. 6 правил валидности шаблона:
1. `display_name` непустое
2. `print_type` ∈ `{layflat, soft}`
3. `template_set_id` не NULL
4. `section_structure` массив длиной ≥ 1
5. Если в `section_structure` есть `students` → `student_layout_mode` задан
6. Если `mode='grid'` → `student_grid_size` в 2..12

`errors[]` — массив человеко-читаемых строк на русском, готовых к показу в UI («Доработай в Шаблонах»). 18 unit-тестов.

**`lib/presets/preview-bundle.ts`** — `buildPresetPreviewBundle(bundle) → { students, cover, teachers, soft }`. Чистая функция возвращает 4 SVG (или null). Двухпроходный алгоритм:
1. Cover ищется напрямую по `page_role='cover'` (не через engine — обложка не в section_structure)
2. Dry-run `buildFromSectionStructure` с фиктивными данными (10 учеников, 5 предметников, 1 фото класса, head_teacher) → распределение страниц по `family_id` из `decision_trace` → первый из корзины
3. Fallback: пустые поля → поиск по `page_role` в template_set

Никогда не бросает исключения — каждый этап в try/catch. Если engine упал — cover всё ещё работает через прямой поиск. 8 unit-тестов.

### 3. API endpoints (РЭ.24.4 + 24.5b)

В `app/api/tenant/route.ts`:

**GET actions:**
- `designs_list` (РЭ.24.5b) — список доступных `template_set`'ов (свои + глобальные) с превью 3 характерных мастеров и счётчиками (`recommended_count`, `my_count`). Для не-superadmin отсекает технические TEST-наборы фильтром `name NOT ILIKE 'TEST%'`.
- `templates_list_global?design_id=...` — каталог глобальных шаблонов (`is_recommended=true`) с фильтром по дизайну. Для каждого — validatePreset → loadBundle → buildPresetPreviewBundle → 4 SVG.
- `templates_list_my?design_id=...` — личная библиотека партнёра. Возвращает все шаблоны с флагом `valid` + `errors[]`. Невалидные отдаются с пустыми превью.

**POST actions:**
- `template_clone` — клонирование глобального шаблона в свой `tenant_id`. Копия с `parent_preset_id` ссылающимся на оригинал, `is_recommended=false`.
- `template_create_blank` — пустой шаблон с нуля. Опциональный `template_set_id` сразу привязывает к выбранному дизайну.
- `template_delete` — удаление с проверкой активных альбомов (409 со списком если > 0).

Tenant-aware строго: глобальные read-only для не-superadmin, чужие тенант'овские → 404.

### 4. UI каталога — двухуровневая навигация (РЭ.24.5 + 24.5b)

**Кнопка `📐 Шаблоны`** в шапке `/app`.

**Уровень 1: `/app/templates`** — выбор дизайна. Карточки доступных `template_set`'ов:
- 3 мини-превью характерных мастеров
- Название + бейджи (от OkeyBook / мой / тип печати)
- Счётчики «N готовых», «M моих»

**Уровень 2: `/app/templates/[designId]`** — каталог шаблонов внутри дизайна:
- Секция «Готовые от OkeyBook» — глобальные с `is_recommended=true` + кнопка «Сохранить в мои» (клонирование)
- Секция «Мои шаблоны» — личная библиотека + «Создать свой шаблон»
- Карточка: крупное SVG-превью students + 3 малых (cover/teachers/soft) + бейджи + название
- Для невалидных партнёрских — красная рамка + пометка «Доработай» + список ошибок
- Редактирование через переиспользование `PresetEditorModal` из `/super/presets` (без копипасты)

### 5. Галка «Показывать в каталоге» (РЭ.24.7)

В `PresetEditorModal` (используется и в `/super/presets`, и в `/app/templates/[designId]`) добавлен чекбокс `is_recommended` с пояснением. Появляется **только для глобальных** пресетов (партнёрские не могут быть recommended — каталог показывает только глобальные). Сергей через UI помечает стартовую линейку шаблонов; SQL-обновление больше не нужно.

API `rule_preset_update` принимает boolean + проверяет only-global ограничение. API `rule_presets_list` отдаёт `is_recommended` в SELECT (иначе чекбокс при первом открытии всегда показывал бы false).

### 6. Модалка выбора шаблона при создании альбома (РЭ.24.6)

В форме создания/редактирования альбома (`AlbumFormFields`) добавлено опциональное поле «Шаблон»:
- Если не выбран → серая кнопка «Выбрать шаблон (опционально)»
- Если выбран → синий бейдж «Название · Дизайн · сменить · ×»
- Старые селекты `config_type` + `print_type` затемняются (opacity 50%) при выборе нового шаблона

**`TemplatePickerModal`** — модалка выбора с фильтром по дизайну, секциями «Мои шаблоны» и «Готовые от OkeyBook» (с группировкой по дизайнам), disabled-карточками для невалидных шаблонов, кнопками «Создать без шаблона» и «Отмена».

**Confirm-диалог при смене/снятии** существующего шаблона предупреждает о возможной потере drafts из редактора и проблемах с пропорциями.

API `create_album` и `update_album` обрабатывают `section_structure_preset_id`:
- Если передан → валидируется доступ (свой/глобальный), `template_set_id` и `print_type` авто-резолвятся из preset'а, legacy `config_preset_id` очищается
- Если не передан → старая логика (preset_slug → config_preset_id, default template_set_id)

Новые альбомы без шаблона создаются без проблем — `section_structure_preset_id` опциональный.

## Архитектура multi-design (§12)

Зафиксирована в spec'е после обсуждения с Сергеем:
- **Дизайн = template_set** (уже есть в БД)
- **Пресет жёстко привязан к дизайну** через `template_set_id` — клонирование сохраняет дизайн
- **Каталог двухуровневый**: дизайны → шаблоны внутри
- **Глобальные пресеты могут быть разные для разных дизайнов** — садики и школы получают разные комплектации, не дублирование линейки

В планах ~20-30 дизайнов. Когда новый дизайн загружается через IDML в `/super` — он автоматически появляется в `designs_list` для партнёров. Никакой дополнительной работы не требуется.

## Что НЕ сделано в фазе (зафиксировано как долг)

### §13 — `print_type` должен жить в альбоме (фаза РЭ.27)

Сейчас `print_type` — атрибут пресета. Это значит «Стандарт-layflat» и «Стандарт-soft» — два разных пресета с одинаковой структурой. Избыточно в БД, неудобно для партнёра.

Правильная модель: `print_type` атрибут **альбома**. Один пресет «Стандарт» нейтрален, тип листов выбирается при создании альбома отдельно.

Главное обоснование: **визуальная модель редактора зависит от типа листов**:
- `layflat` (плотные) — переплёт лежит плоско, разворот = одна непрерывная поверхность. Редактор должен показывать **две страницы рядом** как одну зону работы.
- `soft` (мягкие) — обычный книжный переплёт, между страницами съедающий миллиметры корешок. Редактор работает **постранично** + правила «первая страница правая, последняя левая».

Эстимация РЭ.27: ~10-15 коммитов, большая отдельная фаза.

### §14 — Партнёрские дизайны (фазы РЭ.28 и РЭ.29+)

Партнёры работают с разными типографиями. Сейчас `template_set`'ы создаются только Сергеем через `/super`.

**Сценарий A (РЭ.28)** — копирование глобального дизайна с переопределением размеров под свою типографию. Engine частично уже умеет ресайз placeholder'ов. **Ключевое ограничение**: разница пропорций ≤ 5% → автомат, > 5% → предупреждение партнёру.

**Сценарий B (РЭ.29+)** — загрузка собственного IDML с нуля. Большая фаза, открываем по запросу первого реального партнёра.

### §15 — Терминология (документация)

Зафиксированы термины: **шаблон** (preset), **дизайн** (template_set), **комплектация** (legacy, не использовать), **тип листов** (`print_type`), секции структуры, intro/final.

## Скрытые баги (закрыты по ходу)

1. **РЭ.24.3** — пустой `section_structure=[]` давал engine `status='ok'` с 0 spreads → ранний выход из `buildPresetPreviewBundle` до fallback. Закрыт флагом `engineProducedPages`.
2. **РЭ.24.7** — проверка `Object.keys(patch).length === 0` стояла **до** обработки `is_recommended` → API отвечал «нечего обновлять» при изменении только галки. Перенесена выше проверки.
