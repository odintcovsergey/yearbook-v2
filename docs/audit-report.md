## 3. Категория B — проверить перед удалением (НЕ удалять)

«Мёртвые» action-эндпоинты: статически у них нет вызывающей стороны, НО роуты диспетчеризуются по строке `action`, поэтому теоретически может звать динамически построенная строка. Динамических вызовов (`action=${...}`) к этим хендлерам **не найдено** — все реальные вызовы в коде статичны, так что риск скрытого вызова низкий. Но это требует вашего подтверждения по каждому: часть явно «задел на будущее», часть может быть нужна для ещё не достроенных экранов (обложки, CRM).

## 4. Категория C — неэффективность (не про удаление)

PDF-экспорт (`lib/pdf-export/`, `lib/export-run/`) и сборка входа альбома (`lib/smart-fill/build-album-input.ts`) уже **хорошо забатчены** (запросы через `.in()` + `Promise.all`/пул параллелизма). Это не общая проблема, а точечные места. По убыванию критичности:

1. **🔴 `app/api/select/route.ts:146-153` — N+1, самый горячий путь (родительский).**
   Цикл `for (const photoId of groupPhotos)` делает по одному последовательному запросу `selections.select('child_id').eq('photo_id', photoId)` на каждое групповое фото. Это финальная проверка «не занято ли фото» — она срабатывает **при каждой отправке фото родителем** (самый частый путь записи в продукте). N фото = N последовательных round-trip'ов; схлопывается в один `.in('photo_id', groupPhotos)`. Самый плохой вариант (последовательно, не параллельно).

2. **🟠 `lib/cover/load-editor.ts:166-170` и `:197-199` — последовательная подпись в загрузке редактора обложки.**
   Два цикла `for` с `await` на элемент (`signDecorPlaceholders` на мастер, потом `getPhotoUrl` на общее фото). Подпись — локальная криптография (без сети), эффект умеренный, но сериализуется; галерея `common_photos` может быть длинной. `Promise.all(...map())` распараллелит. Срабатывает при каждом открытии редактора обложки.

3. **🟠 `app/api/tenant/route.ts:2326-2332` — последовательный `getPhotoUrl` в цикле** (фото персональных разворотов по ребёнку). Та же картина: локальная подпись сериализована в `for` вместо `Promise.all`. Соседние эндпоинты в этом же файле уже используют `Promise.all(...map(async))` — этот (и цикл на `:1040-1043`, подпись портретов обложки) выбиваются. Кабинет, «персональные развороты»; умеренно.

4. **🟡 `app/api/crm/route.ts:343-347` — N+1 UPDATE, редкое админ-действие.**
   `for (const s of stages) await deal_stages.update(...).eq('id', s.id)` — по UPDATE на стадию CRM. Только при переупорядочивании стадий партнёром (несколько строк, редко). Можно батч-upsert, но влияние низкое.

5. **🟡 `lib/cover/upload-covers.ts:178-196` — DELETE+INSERT на мастер обложки, редкое админ-действие.**
   В цикле по мастерам каждый делает свой `delete` (при `force`) и `insert(...).single()`. Путь загрузки IDML обложки (админ). Каждому мастеру нужна своя строка → во многом неизбежно. Для полноты, менять не обязательно.

**Проверено и признано УЖЕ корректным (не находки):**
- `lib/pdf-export/pipeline.ts` — фото грузятся через `prefetchPhotoSources`/`prefetchResampledPhotos` → пул параллелизма (`photo-embed.ts:345-378`). Последовательный `for…await drawPlaceholder` (`pipeline.ts:820`) — **намеренно** (комментарий: «экономим RAM на sharp+fetch буферах»), пиксели уже предзагружены.
- `lib/smart-fill/build-album-input.ts` — параллельные запросы, батч `.in()`, подпись через `Promise.all(...map())`.
- `lib/export-run/typography.ts`, `lib/rule-engine/loaders.ts`, `lib/album-builder/load-template-set.ts` — фиксированное число запросов, в циклах только in-memory преобразования.

## 5. Почему числа knip обманчивы (важно для Этапа 2)

knip выдал большие списки («64 экспорта», «170 типов», «21 файл»), но **бóльшая часть — НЕ мусор**. Не удалять по этим спискам вслепую. Три класса ложных срабатываний (проверены на примерах):

- **Barrel re-export'ы.** Файлы `lib/*/index.ts` намеренно реэкспортируют типы/функции наружу. knip считает строку реэкспорта «неиспользуемой», если barrel не импортируют напрямую. Пример: `filterChildrenByPurchase` помечен мёртвым, хотя сама функция используется тестом — мёртв лишь реэкспорт в `index.ts`.
- **Потребляется воркером/скриптами.** knip не настроен видеть `worker/`/`scripts/` как точки входа. Проверено: `requeueStuckJobs`, `setJobStage`, `loadExportProfileBySlug`, `notifyExportReady` — **используются воркером** (`worker/render-one-job.ts`, `render-worker.ts`), но knip их зовёт мёртвыми.
- **Используется внутри своего файла, но помечено `export` зря.** Проверено: `verifyAccessToken` (`lib/auth.ts`), `twcStorage` (`lib/storage-twc.ts`), `matchStudents`/`OKEYBOOK_TABLE` (`album-structure-okeybook.ts`), `shapeText` (`text-shaping.ts`), `getActiveJob` (`export-queue/index.ts`) — все используются внутри своих модулей. Код живой; «лишнее» — только ключевое слово `export` (микро-уборка, не удаление кода).

Поэтому из всего knip-списка экспортов в Категорию A (реально мёртвое) попали только **4 символа** из §A2. Остальное либо живое, либо требует поштучной проверки.
