# Чек-лист безопасности при переезде на российские сервисы

Составлен 10.06.2026 после закрытия аудита безопасности (см. `security-audit-report.md`).
Цель: при переезде с Vercel+Supabase на Yandex Cloud ничего из защиты не потерять.
Привязан к шагам плана миграции (см. память `project-migration-to-ru`).

> **Главная мысль:** защита, записанная в коде, переедет сама. Перепроверить нужно
> только то, что живёт в инфраструктуре (база, секреты, HTTPS, хранилище).

---

## 🟢 Переезжает само (код) — отдельных действий НЕ требует

Эти фиксы в самом приложении, поедут вместе с репозиторием:
F4 (удалён god-key), F5 (анти-брутфорс — код), F6 (гашение сессий),
H2 (скрытие ошибок), D3 (проверка файлов), C3/B1/B2/C2/C3-select (проверки «своё/чужое»),
G1 (view_as), J1 (версия Next), E2 (.gitignore).

Условие: сайт остаётся на Next.js. Если технологию меняем — заголовки I1 проверить заново (см. ниже).

## 🟡 Требует проверки при переезде

### Шаг «Миграция БД» (pg_dump → Yandex Managed PostgreSQL)

- [ ] **Перенести ВСЕ таблицы**, включая `login_attempts` (защита входа F5) и `sessions`
      (refresh-токены F6). `pg_dump` со схемой переносит их автоматически — убедиться,
      что в дампе они есть.
- [ ] **🔴 Самое важное — RLS-замки (A1).** RLS включён прямо в базе, в коде/миграциях
      его НЕТ. После восстановления базы выполнить в новой БД проверку:
      ```sql
      SELECT relname, relrowsecurity FROM pg_class
      WHERE relkind='r' AND relnamespace='public'::regnamespace
      ORDER BY relrowsecurity, relname;
      ```
      Все таблицы должны быть `relrowsecurity = true`. Если где-то `false` —
      включить: `ALTER TABLE <таблица> ENABLE ROW LEVEL SECURITY;`
      (`pg_dump` обычно переносит RLS, но это надо ПРОВЕРИТЬ глазами — легко потерять).

### Шаг «Миграция Storage»

- [ ] **Фото детей (D1).** Бакет `yearbook-photos` уже в Yandex Cloud и НЕ переезжает —
      приватность и подписанные ссылки остаются. Ничего не трогать.
- [ ] Если для логотипов/фонов заводится **новый** бакет в Yandex — проверить публичность
      осознанно: логотипы/фоны — можно публичные (не ПД); фото детей — только приватный.
- [ ] **Случайно НЕ сделать `yearbook-photos` публичным** при настройке нового проекта.

### Шаг «Развёртка приложения» (VPS + nginx + SSL)

- [ ] **Прописать все секреты-переменные** на новом сервере: `JWT_SECRET` (тот же,
      иначе разлогинит всех), `YC_ACCESS_KEY_ID`, `YC_SECRET_ACCESS_KEY`, `YC_BUCKET_NAME`,
      `DEFAULT_TENANT_ID`, креды новой БД, `ANTHROPIC_API_KEY`. Не коммитить.
- [ ] **HTTPS обязателен** (Let's Encrypt/certbot). От него зависят защищённые cookie
      (`secure`) и заголовок HSTS. Без HTTPS вход будет небезопасным.
- [ ] **Защитные заголовки (I1).** Если остаёмся на Next.js — они в `next.config.js`,
      едут сами. Если перед Next стоит nginx — убедиться, что он их не срезает
      (X-Frame-Options, X-Content-Type-Options, Referrer-Policy, HSTS).
- [ ] `ADMIN_SECRET` на новом сервере НЕ задавать (он больше не нужен; используется только
      одноразовым setup-экшеном, а superadmin уже создан).

### После переключения (cutover)

- [ ] Прямая (неподписанная) ссылка на фото ребёнка → должна давать «доступ запрещён».
- [ ] Рабочие экраны (родительская, кабинет, PDF, ZIP) → фото грузятся через подписанные ссылки.
- [ ] Вход работает; после 5 неверных паролей — блок на 15 минут (F5 жива → таблица перенесена).
- [ ] Смена пароля разлогинивает старые сессии (F6).
- [ ] При ошибке в глубине пользователь видит «Внутренняя ошибка», а не текст базы (H2).
