# Фаза Л — Редактор макета MVP

**Дата создания:** 12.05.2026
**Статус:** спецификация (Л.0)
**Оценка:** 10-14 дней работы, 5 подэтапов (Л.1-Л.5)

---

## 1. Цель и контекст

### Зачем

Сейчас собранный `album_layouts` неизменяем — что собрал `buildAlbum`, то и пойдёт в типографию. Партнёр не может:

- Подправить ФИО ученика в макете (опечатка, изменение фамилии)
- Заменить фото если ребёнку не понравилось как он выбрал
- Поменять состав фото в общем разделе
- Согласовать макет с клиентом перед отправкой в печать

**Это главная функциональная дыра системы перед запуском партнёрки.** Без редактора партнёр вынужден просить OkeyBook пересобирать заново или принимать имеющийся результат.

### Что есть

- `template_sets`/`spread_templates`/`album_layouts` — таблицы готовы (миграции 0.1, 0.3.5)
- `SpreadCanvas` в `app/super/templates/_components/` — Konva-просмотрщик мастеров (фаза 0.8), отрисовывает плейсхолдеры с label/тип/координаты
- `lib/album-builder/` — `buildAlbum()` собирает `BuildResult { spreads: SpreadInstance[] }` и пишет в `album_layouts.spreads`
- PDF-экспорт берёт layout как есть и рендерит — никаких изменений в нём не нужно

### Что строим

Полноэкранный редактор макета как часть `/app`, открывается по кнопке «Открыть редактор» из карточки альбома или вкладки Производство. Партнёр видит развороты, кликает на фото/текст, правит, сохраняет — изменения немедленно отражаются в PDF-экспорте.

## 2. Решения по продукту

Все 4 ключевых вопроса (от Сергея, 12.05.2026):

| Решение | Значение | Обоснование |
|---|---|---|
| **Приоритет** | Баланс: текст+фото + приличный UI | Подтверждено |
| **Drag-and-drop** | @dnd-kit для сайдбара (список разворотов), react-konva native для слотов в canvas | dnd-kit лучше для списков/keyboard a11y; Konva native — для перемещения фото внутри canvas, не нужен переход между системами координат |
| **Undo/Redo** | Базовый Ctrl+Z + Ctrl+Shift+Z (история в памяти, не в БД) | Без undo редактор болезненный — один случайный клик и работа потеряна. Полная история с версиями в БД — это Фаза М/Н, для MVP избыточно |
| **Мобильная** | View-only на мобиле, редактирование desktop ≥768px | Редактировать макет на телефоне физически неудобно. Но партнёр должен *показать* клиенту макет с телефона — это критично для согласования |

### Не входит в Л (отложено)

- Перетаскивание разворотов (изменение порядка) — Фаза М
- Замена шаблона разворота — Фаза М
- Добавление/удаление разворотов — Фаза М
- Полная история версий в БД (rollback к версии Y) — Фаза М
- Согласование клиентом (комментарии, лайки) — Фаза Н
- IDML экспорт — Фаза О

### Точечная замена оригинала (отложенная из К)

Возможность «заменить оригинал одного фото без смены WebP» (для ретуши после согласования) встраивается **в Л.3** как один из вариантов «Заменить фото».

## 3. Архитектура данных

### Что меняем в БД

**Аддитивные изменения в `album_layouts`** (миграция Л.0.1, перед Л.1):

```sql
ALTER TABLE album_layouts
  ADD COLUMN IF NOT EXISTS edited_at timestamptz,
  ADD COLUMN IF NOT EXISTS edited_by uuid REFERENCES users(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS rebuild_warnings jsonb DEFAULT '[]'::jsonb;

CREATE INDEX IF NOT EXISTS idx_album_layouts_edited_at
  ON album_layouts(album_id, edited_at);
```

- `edited_at` — null если layout ни разу не редактировался руками (можно безопасно пересобрать через `buildAlbum`). После первого ручного изменения — заполнен.
- `edited_by` — кто последний раз правил (для audit и UI «правил Иван Иванов, 5 мин назад»).
- `rebuild_warnings` — последние предупреждения от `buildAlbum` (text+code пары). Если партнёр после правок захочет пересобрать заново — увидит что было.

### Что НЕ меняем

- Структура `spreads` (массив `SpreadInstance`) остаётся — редактор работает с тем же форматом что и `buildAlbum`. PDF-экспорт читает то же самое.
- `placeholders` мастера остаются неизменными — редактор НЕ позволяет двигать рамки (это нарушит дизайн), только менять *содержимое* (`data[label] = value`).

### Что хранится в `spreads[i].data`

Текущий формат (фаза 0.10+):

```typescript
type SpreadInstance = {
  spread_index: number;
  template_id: string;        // ID мастера из spread_templates
  template_name: string;       // human-readable, дублирует для отладки
  data: Record<string, string | null>;
  //  ↑ ключ = placeholder.label (lowercase), значение = что вставить:
  //    URL фото (yc:// или storage.yandexcloud.net/...) — для image-полей
  //    Текст (строка) — для text-полей
  //    null — слот пустой
};
```

Редактор меняет только `data[label]`. Тип полей определяется через `placeholder.type` мастера (`image`/`text`).

### Защита от потерь при перестройке

После первой ручной правки `edited_at` ставится в `NOW()`. UI «Пересобрать макет заново» в Обзоре альбома теперь:
- Если `edited_at IS NULL` — пересборка молча, как сейчас
- Если `edited_at` есть — модальное предупреждение «Макет редактировался вручную M дней назад партнёром Иваном Ивановым. Все ваши правки будут потеряны. Продолжить?»

Это нужно реализовать в **Л.4** (сохранение). Сейчас механика пересборки тривиально перезаписывает `spreads`.

### Л.2 — Редактирование текста (2 дня)

**Цель:** партнёр кликает на текст в макете (ФИО, год, заголовок) → правит → сохраняется в `data`.

**Что входит:**
- Клик на placeholder типа `text` → выделение (синяя рамка) + правая панель раскрывается
- Правая панель показывает:
  - Заголовок «Текст: <label placeholder'a>»
  - Текстовое поле `<textarea>` (multiline для случая многострочных текстов вроде стихов)
  - Под полем — preview как это выглядит в макете (мини-canvas)
  - Кнопка «Сбросить» — вернуть значение из `buildAlbum` (берётся из `rebuild_warnings`/исходного билда — потребуется хранить)

- Изменения отражаются в canvas мгновенно (live preview через Konva `Text` компонент)
- На canvas — двойной клик на text-placeholder открывает inline-edit прямо в Konva (без правой панели) — для быстрого редактирования

### Л.3 — Замена фото (3 дня)

**Цель:** партнёр кликает на фото → может его заменить тремя способами.

**Что входит:**

**1. Выделение и правая панель для image-placeholder'ов:**
- Клик на placeholder типа `image` → выделение + правая панель
- Правая панель показывает:
  - Заголовок «Фото: <label placeholder'a>»
  - Превью текущего фото
  - Имя файла + размер + дата загрузки (под превью)
  - 3 кнопки действий (см. ниже)
  - Кнопка «🗑 Убрать фото» — `data[label] = null`

**2. Три варианта замены:**

**a) «📁 Выбрать из загруженных»** — модал с галереей всех фото альбома
- Фильтр по типу placeholder'a (если `placeholder.expected_type='portrait_photo'` — показываем только portraits; если universal — все)
- Поиск по имени файла
- Клик на фото → подтверждение → замена + закрытие модала
- Используем существующий API `/api/tenant?action=album_photos`

**b) «📤 Загрузить новое фото»** — открывается file input
- После выбора файла — стандартный upload pipeline (тот же что в PhotosTab):
  - WebP сжатие на клиенте
  - POST `/api/upload` → создаёт `photos` запись с правильным типом
  - В фоне presigned URL → PUT оригинала → register_original (фаза Б.1.3)
- После завершения загрузки — авто-выбор для текущего слота (`data[label] = photo.storage_path`)
- Прогресс-бар отдельно от глобального

**c) «🎨 Заменить оригинал»** — точечная подмена `photos.original_path` без смены WebP
- Полезно для ретуши *после* согласования с клиентом
- Открывается file input
- Файл загружается через presigned URL в `<album_id>/originals/`
- Обновляется `photos.original_path` через новый action `replace_original` (расширение К.3 `rebind_retouched` — или прямо его)
- WebP и `data[label]` не меняются — макет визуально тот же, но PDF-экспорт возьмёт новый оригинал
- В UI правой панели — небольшой бейджик «оригинал заменён 5 мин назад» если `photos.original_path_updated_at` свежий (не делаем сейчас, оставляем простым)

**3. Drag-and-drop фото между слотами:**

- На canvas можно зажать фото в одном слоте и перетащить в другой
- При drop в пустой слот — `data[targetLabel] = data[sourceLabel]`, `data[sourceLabel] = null`
- При drop в занятый слот — swap (`data[targetLabel] = ...`, `data[sourceLabel] = previous`)
- Используем native react-konva drag (`draggable` на изображении + `onDragEnd`)
- Подсветка целевого слота при наведении

**Backend:**
- `/api/layout-editor` — новые actions (или расширение существующих):
  - Использовать уже существующий action `register_photo` из `/api/tenant` для (b)
  - Использовать новый action в `/api/workflow` `register_retouched_single` (расширение К.3) или прямо `rebind_retouched`

**Файлы:**
- `app/app/layout-editor/_components/PhotoEditPanel.tsx` — правая панель для image
- `app/app/layout-editor/_components/PhotoPickerModal.tsx` — модал выбора из загруженных
- `app/app/layout-editor/_components/PhotoSlotKonva.tsx` — Konva-компонент для image-placeholder с поддержкой drag

**Критерии приёмки Л.3:**
- ☑ Клик на фото → правая панель с превью и 3 кнопками
- ☑ Замена через «Выбрать из загруженных» работает, фильтрация по типу
- ☑ Замена через «Загрузить новое фото» работает, оригинал тоже загружается (проверка CORS!)
- ☑ Замена оригинала отдельно от WebP работает, PDF-export использует новый файл
- ☑ Drag-and-drop фото между слотами с swap работает

### Л.4 — Сохранение и undo/redo (2 дня)

**Цель:** изменения сохраняются в БД, можно откатить через Ctrl+Z.

**Что входит:**

**1. Сохранение:**
- POST `/api/layout-editor?action=save_layout` с body `{album_id, spreads}`
- Дебаунс 1.5 сек — после любого изменения ждём 1.5 сек, потом отправляем
- Заголовок toolbar показывает статус: «Сохранено» / «Сохраняем…» / «Не сохранено» (если был сбой)
- При закрытии редактора (Cmd+W, кнопка ← или browser back) — если есть несохранённые изменения, дополнительный POST и confirm если что-то отвалится
- Обновляет `album_layouts.edited_at = NOW(), edited_by = auth.userId, spreads = новые spreads`
- Проверка прав: только owner/manager (не viewer), не view_as

**2. Undo/Redo:**
- История в памяти: массив `LayoutSnapshot[]`, максимум 50 шагов
- Каждый шаг сохраняется при изменении (после дебаунса 300мс чтобы не плодить шаги при наборе текста)
- Ctrl+Z (Cmd+Z на маке) — undo
- Ctrl+Shift+Z (Cmd+Shift+Z) — redo
- Кнопки «↶ Отменить» и «↷ Повторить» в toolbar (видны на desktop ≥1024px)
- При undo/redo — диспатчим состояние и тригерим сохранение
- При закрытии редактора — история теряется (новая сессия = новая история)

**3. Защита от перезатирки `buildAlbum`:**
- В Обзоре альбома (карточка) кнопка «🔄 Пересобрать макет» теперь:
  - Если `album_layouts.edited_at IS NULL` — пересборка без вопросов
  - Иначе — модальное предупреждение с датой последней правки и кем (если есть `users.email`)
  - На подтверждение — пересборка + сброс `edited_at` обратно в NULL

**Файлы:**
- `app/api/layout-editor/route.ts` — добавляем POST action `save_layout`
- `app/app/layout-editor/_components/useEditorHistory.ts` — хук истории
- `app/app/layout-editor/_components/useDebouncedSave.ts` — хук дебаунс-сохранения
- Обновление в `app/app/page.tsx` (Обзор альбома) — модал предупреждения

**Критерии приёмки Л.4:**
- ☑ Любое изменение → дебаунс 1.5 сек → POST → статус «Сохранено» в toolbar
- ☑ Если сеть отвалилась → статус «Не сохранено», на reconnect retry
- ☑ При закрытии с несохранённым — confirm dialog
- ☑ Ctrl+Z откатывает последнее изменение
- ☑ Ctrl+Shift+Z повторяет
- ☑ После 50 шагов история ротируется (старые забываются)
- ☑ После открытия редактора заново состояние из БД, история чистая
- ☑ Пересборка макета через «Пересобрать макет» предупреждает если `edited_at` есть

### Л.5 — Интеграция и полировка (2 дня)

**Цель:** UI выглядит прилично, минорные баги исправлены, всё связано с остальной системой.

  - Ctrl+S / Cmd+S — принудительное сохранение (intercept default)

- Empty states (например «В альбоме пока нет фото» в PhotoPickerModal)

## 5. Технические решения

### Размер canvas

`SpreadCanvas` принимает `containerWidth` и масштабирует. В редакторе максимальный размер — `min(viewport.width - 200 - 280, viewport.height - 60)`, минимум 400px. Zoom range 0.3x - 3x.

### Производительность

- Миниатюры в сайдбаре — `listening=false`, `pixelRatio=1`, `showLabels=false` (см. оптимизации 0.8.4 в SpreadCanvas.tsx)
- Центральный canvas — full settings
- При количестве разворотов > 30 — виртуализация сайдбара (react-window) — это уже фаза М, не Л

### Сохранение

Дебаунс 1.5 сек выбран как баланс между «не флудить сервер» и «не потерять при закрытии». При unload (beforeunload) сохраняем sync (`navigator.sendBeacon`), на reconnect retry с экспоненциальной задержкой (3, 10, 30 сек).

### Drag-and-drop

- **@dnd-kit** в сайдбаре для перетаскивания миниатюр разворотов (фаза М, в Л.1 только статичный список с переключением кликом)
- **react-konva native** (`draggable` + `onDragMove` + `onDragEnd`) в canvas для фото между слотами (фаза Л.3). Координаты считаем по hit-test с `placeholders` — какой слот под курсором.

Эти системы не конфликтуют потому что разные DOM-зоны.

### Мобильная стратегия

`useEffect` с `window.matchMedia('(min-width: 768px)')` — если false, показываем `<MobileViewOnlyView>`: миниатюра текущего разворота + список разворотов кнопками + кнопка «Открыть на компьютере для редактирования». Прямой доступ к редактору с мобилы дозволен (можно показать клиенту), но никаких click-handler'ов на placeholder'ах нет.

### Защита прав

- viewer не может открыть редактор (404)
- view_as — может (для сотрудника OkeyBook от имени партнёра), но только в режиме read-only (изменения не сохраняются)
- После `workflow_status = 'submitted'` — read-only для всех кроме superadmin

## 6. Риски и подводные камни

### Риск 1: PDF-экспорт может не учесть какие-то новые edge cases

- **Митигация:** в Л.1 сразу прогнать PDF-export на отредактированном вручную layout'е и убедиться что всё ок
- В худшем случае — баг в PDF-export это фаза 3, фиксится отдельно

### Риск 2: Konva может тормозить на больших альбомах (>50 разворотов)

- **Митигация:** не виртуализируем в Л (это М), но миниатюры уже оптимизированы. Если на тесте Сергея тормозит — добавим простой `IntersectionObserver` для отложенной отрисовки скрытых миниатюр
- В крайнем случае — лимит 50 миниатюр одновременно, остальные через «Показать ещё»

### Риск 3: Race condition между save и open другого альбома

- **Митигация:** перед закрытием — `await final save` (с тайм-аутом 5 сек, если зависло — confirm о потере)

### Риск 4: Конфликт с `buildAlbum` если партнёр и менеджер OkeyBook оба правят

- **Митигация:** `edited_at` оптимистический lock — при save проверяем что `edited_at` в БД == последнему загруженному. Если нет — модал «Кто-то ещё внёс изменения. Перезагрузить?» (фактически это фаза М, для MVP — last-write-wins, в комментарии оставить TODO)

### Риск 5: Партнёр случайно сломает дизайн (например удалит все текстовые элементы)

- **Митигация:** undo + автосохранение. Полное восстановление к `buildAlbum` версии — через кнопку «🔄 Пересобрать макет» в Обзоре (с предупреждением).
