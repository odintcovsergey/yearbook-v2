# knowledge/ — база знаний проекта yearbook-v2 (OkeyBook)

Единая точка входа в актуальное состояние проекта. Если открываешь репозиторий
с нуля — читай файлы отсюда в порядке: RECOVERY → PROJECT_CONTEXT → DECISIONS.

## Что внутри

| Файл | Назначение |
|------|------------|
| `README.md` | Этот файл: что это и как обновляется. |
| `PROJECT_CONTEXT.md` | ТЕКУЩЕЕ состояние проекта (что сделано / на паузе / дальше). Снимок последнего `yearbook-context-vN.md`. |
| `DECISIONS.md` | Журнал ключевых решений с датами (почему сделано именно так). |
| `RECOVERY.md` | Как восстановить проект с нуля: клон, env, БД, локальный запуск, деплой. |
| `sessions/` | Лог по датам `YYYY-MM-DD.md` — что делали в конкретный день. |

## Связь с `yearbook-context-vN.md`

В корне репозитория ведётся история контекст-файлов `yearbook-context-vN.md`
(N инкрементируется каждый подэтап, история НЕ переписывается — это летопись).
`knowledge/PROJECT_CONTEXT.md` — это всегда КОПИЯ самого свежего из них, чтобы
не искать максимальный N. Источник истины по истории — сами `vN`-файлы.

## Как обновлять

После каждого подэтапа (когда обновляется `yearbook-context-vN.md`):

1. Скопировать свежий контекст в актуальный снимок:
   ```
   cp "$(ls yearbook-context-v*.md | sort -V | tail -1)" knowledge/PROJECT_CONTEXT.md
   ```
2. Если принято заметное архитектурное/продуктовое решение — добавить строку
   в `DECISIONS.md` (дата, решение, причина).
3. Создать/дополнить лог дня `knowledge/sessions/YYYY-MM-DD.md`.
4. Менялся процесс восстановления (env, миграции, деплой) → обновить `RECOVERY.md`.
5. Коммит отдельно от кода: `docs(knowledge): обновление базы знаний — <что>`.

## Правило

`knowledge/` — для людей и для будущих сессий ассистента. Пиши кратко, по делу,
с датами. Без воды.
