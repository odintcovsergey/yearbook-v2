# DECISIONS — ключевые решения yearbook-v2

Архитектурные и продуктовые решения с датами. Свежие сверху. Подробности — в
`yearbook-context-v*.md` и в памяти Claude Code.

## Архитектура данных и движка

- **section_structure (jsonb пресета) — источник правды о структуре альбома.**
  Движок `buildFromSectionStructure` исполняет список секций; заменил старые
  generic-правила. Per-section конфиг личного раздела (`StudentsSectionConfig`).
- **Семантические теги мастеров** (`page_role`, `slot_capacity`, `applies_to_configs`,
  `default_for_configs`) заполняются ТОЛЬКО из `lib/idml-converter/family-mapping.ts`.
  Меняешь структуру мастера — обновляй mapping.
- **Размеры мастера парсятся из IDML** (`width_mm`/`height_mm`); родной формат
  дизайна = размеры страницы template_set.
- **Миграции — вручную** Сергеем через Supabase Studio. Датированный SQL в
  `migrations/`, дубль в `schema.sql` только для деплоя с нуля. SQL всегда с
  префиксами таблиц в JOIN/WHERE (защита от ambiguous column).

## Печать и формат (16–19.06.2026)

- **Профиль типографии** хранится в `printers.config` (jsonb): forms[], типы листов,
  режим приёма, формат файла, загибы обложки.
- **Корешок — 3 режима** (`lib/printers/spine.ts resolveSpineMm`): ranges (диапазоны),
  formula (base+step×разв/per), fixed. Legacy `spine_ranges` нормализуется к ranges.
- **Адаптация под формат — только uniform-scale** (без stretch), по меньшему
  коэффициенту work-зоны, контент центрируется. **Только ВНУТРИ одного семейства**
  пропорций (vertical_rect/square/horizontal); чужое → не адаптируем + предупреждение.
  Реализация: **адаптируем сам мастер (размеры+слоты) ОДНИМ преобразованием перед
  подачей в холст** (`lib/format-adapt`), как `layoutCover` у обложки — тогда превью,
  редактор и обложка подхватывают формат автоматически, координаты/кроп не ломаются.
- **Корешок обложки физический** — масштабируются только страницы, корешок (толщина
  от числа листов) форматом не масштабируется (`adaptCoverToFormat`).
- **Формат фиксируется на заказе** (типография→формат→тип листа); переключения «на
  лету» в редакторе нет.

## Персональный раздел (19.06.2026)

- Признак `is_personal` у students-секции — только для режимов, где на печатном листе
  ОДИН ребёнок (page/spread/multi_spread), НЕ для grid.
- Движок метит личные страницы (`PageInstance.personal`), вёрстку не меняет.
- Нарезка/нумерация/корешок книг — модель `lib/album-split` (книга 000 общая, 00X
  личные; имена файлов «КНИГА-РАЗВОРОТ»). Рендер файлов — отдельное ТЗ экспорта.

## Экспорт и нагрузка

- **Рендер PDF синхронный** в HTTP-запросе — осознанный временный выбор; упирается в
  таймаут Vercel (300с). Заплатка (поднятие лимита) откатывалась как бесполезная.
  Реальное лечение — ускорение рендера (параллельная загрузка фото) и/или фоновая
  очередь рендера на YC (`docs/tz-load-readiness.md`). Делать к сезону / с переездом.
- **Очередь — на чистом Postgres** (render_jobs + FOR UPDATE SKIP LOCKED), без
  Supabase-специфики — ради переносимости под переезд в РФ.

## Хранение и безопасность

- **Фото детей приватны:** Yandex Cloud бакет закрыт, доступ только через signed URL.
- **Тяжёлый ZIP оригиналов** собирается на клиенте (манифест signed-ссылок + client-zip),
  не на сервере.
- Аудит безопасности пройден: все 🔴/🟠 закрыты (RLS, проверки «своё/чужое», лимиты
  логина, валидация и т.д.).

## Процесс и инфраструктура

- **Двойное зеркало Git:** GitHub (origin) + GitFlic; push идёт в оба одной командой.
  Vercel завязан на GitHub. Бэкап-скрипты `scripts/backup-*.sh`.
- **Тестовая среда сейчас — preview-ссылки Vercel** (по веткам). План: dev-ветка +
  тестовая база, когда появятся реальные партнёры/фотографы.
- **Feature-ветка + preview** для видимых/поведенческих изменений, мелкие правки —
  в main. Перед коммитом 3 зелёные проверки: `vitest run`, `tsc --noEmit`, `next build`.
- **Переезд на российские серверы** (Vercel+Supabase → Yandex Cloud) — запланирован,
  делать когда придёт время (152-ФЗ, школы, масштаб).
