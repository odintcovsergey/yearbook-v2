## Шаг 1 — перенос БД Supabase → Timeweb PostgreSQL (✅ выполнен)

- Цель: Timeweb Managed **PostgreSQL 18.4** (Москва, публичный IP). Источник: Supabase
  **PG 17.6**. Инструменты — клиент **postgresql@18** (brew install, не системный 16).
- SSL: подключение по IP → `sslmode=verify-ca` с `ca.crt` (НЕ `verify-full` — в сертификате
  доменное имя, по IP не сходится). `SELECT version()` → 18.4. ✅
- Дамп: `pg_dump "$SUPABASE_DATABASE_URL" --schema=public --no-owner --no-privileges -Fc`
  (через Shared Pooler `aws-1-eu-west-2.pooler.supabase.com:5432`). Заливка `pg_restore`.
- Грабли:
  - 4 таблицы (children/teachers/invitations/responsible_parents) не создавались — юзают
    `extensions.gen_random_bytes` (pgcrypto). Решение: `CREATE SCHEMA extensions; CREATE
    EXTENSION pgcrypto WITH SCHEMA extensions;` (доверенное, владельцу БД можно) → перелив начисто.
  - 2 RLS-политики на `children` (роли `anon`/`service_role`) НЕ доехали — у пользователя
    Timeweb нет CREATEROLE. Решили «потом» (приложение ходит через сервис-ключ). RLS включён
    на всех 52 таблицах, 8/10 политик создано.
- **Сверка: 52 таблицы, 14060 строк, 0 расхождений** (per-table count). Один альбом со связями
  читается (Тест 30: 30 детей/242 фото/5 учителей).

## Шаг 3 — живая проверка timeweb + фиксы багов (коммит `5ebeeca`)

Сергей включил `STORAGE_BACKEND=timeweb` локально и прокликал. Найдено и починено:
- **№3 Загрузка фона обложки — «Failed to fetch».** Причина: у бакета Timeweb не было
  CORS (`NoSuchCORSConfiguration`); браузерный presigned-PUT упирался в CORS (для показа
  `<img>` CORS не нужен, поэтому всё отображалось). Настроил CORS на бакете
  (`PutBucketCorsCommand`: origins `*`, GET/PUT/POST/HEAD). Проверил preflight → 200.
- **№3b Залитый фон обложки не отображался.** На timeweb новый фон сохраняется коротким
  ключом, а карта `bg_signed` собиралась при загрузке редактора (нового ключа нет).
  Починка: `cover_bg_sign` теперь возвращает `read_url` (signed), клиент держит `bgSigned`
  в state и подмешивает туда новый ключ при загрузке. Прокликано — фон виден.
- **№2 Нет пометок «фото занято/номер разворота».** Опознание фото шло по нестабильной
  signed-URL-строке. Перевёл на стабильный ключ (`storage_path`): `PhotoPalette.findUsage`
  + `matchPhoto` ×4 в layout/page.tsx (бейджи, drag-swap, контекст-меню, замена оригинала).
  Проверено на данных: 41 фото получат бейдж в редакторе РАЗВОРОТА (в обложках номеров
  нет by design — там `spreads=[]`).
- **№1 Нет полупрозрачного превью при drag из палитры** — не переезд (UI dnd-kit, превью
  из `thumb_url` грузится). Отложено в память [[project-timeweb-editor-followups]].
- Пустая плитка портрета — давний осиротевший файл (нет НИ в Yandex, НИ в Timeweb), не переезд.
- Медленная загрузка палитры (~12с на 514 фото) — стоимость подписи >1000 ссылок + удалённая
  БД; не регресс timeweb; ускорится после шага 4; варианты в followups-памяти.
- Грабли процесса: запустил `next build` поверх работающего `next dev` → `ChunkLoadError`
  у Сергея. Урок в память [[feedback-no-build-while-dev]]: глушить dev перед билдом.

## Шаг 4 — АНАЛИЗ + подготовка (выбран Вариант 1: self-host PostgREST)

ТЗ перевести приложение с Supabase API (PostgREST) на Timeweb Postgres напрямую (~34 файла).
Проанализировал 3 варианта (2 разведки по коду). **Замеры:** ~868 `.from()`, `.rpc()`=0,
используется ТОЛЬКО PostgREST (нет Supabase Auth/Realtime/Functions); БД строго server-only
(прямых браузер→БД нет); RLS не рантайм-зависимость (service_role обходит, авторизация в коде);
auth своя (JWT). PostgREST-специфики много: 37 эмбедов, 14 `!inner`/`!fk`, 26 `.or()` сырых,
53 count/head, 239 `.single/.maybeSingle`.

**Решение — Вариант 1 (self-host PostgREST)** рядом с Postgres: приложение остаётся на
`supabase.from()`, меняются endpoint+роль БД, НЕ 868 запросов. Shim = переписать сам PostgREST
(отверг), полный rewrite на Drizzle = недели + ломает «плавно» (отверг; опц. позже strangler).

**Артефакты (коммиты `651e1ef`, `4528582`):**
- `docs/tz-step4-deployment.md` — ТЗ: роли/гранты, конфиг PostgREST, nginx `/rest/v1`,
  проверка на копии, чек-лист cutover, разворачивание приложения на VDS, процедура миграций+reload.
- `scripts/compare-rest-backends.mjs` — сверка A4 (одинаковые запросы на 2 бэкендах + дифф).
- `scripts/db-migrate.mjs` — apply миграции + `NOTIFY pgrst reload`, защита от боевого.
- `scripts/step4-postgrest-roles.sql` — роли authenticator/web_app + гранты (least-privilege).

**✅ A4 ДОКАЗАНА ЛОКАЛЬНО:** поставил PostgREST 14.13 (brew), поднял против КОПИИ Timeweb
(как `gen_user`, db-anon-role=gen_user — CREATEROLE не нужен, владелец обходит RLS), прогнал

   залит scp, локальная копия удалена.
3. **Сборка** `npm run build` от юзера `yearbook`. Приложение под `yearbook.service`

### 🛑 Грабли (важно — занесены в deploy/timeweb/README.md и шаблон nginx)
- **Swap обязателен.** 3.8 ГБ RAM без swap → `npm run build` OOM, рвёт SSH (ловили
  дважды). Добавили 2 ГБ `/swapfile` (в fstab).
- **HTTP-01 из РФ НЕ работает.** Серверы Let's Encrypt не достукиваются до РФ-IP —
  в access-логе nginx запросов LE с токенами НЕТ (мой Мак и Jina/GCP качают на 200,
  валидаторы LE — нет). nginx/MTU/firewall ни при чём (сеть выше сервера). Выпуск —
  ТОЛЬКО DNS-01. reg.ru требует whitelist IP сервера в настройках API.
- **🔴 Authorization-срез (ломало ВЕСЬ доступ к БД, поймано на живом логине).**
  PostgREST без jwt-secret + supabase-js всегда шлёт `Authorization: Bearer` →
  `500 PGRST300 Server lacks JWT secret`. Логин не находил юзера → «неверный пароль».
  Curl без заголовка = 200, поэтому прежний smoke это пропустил. Фикс: в nginx
  `proxy_set_header Authorization ""` (+ apikey ""). Хэш пароля в копии = боевому
  (сверено по md5), verifyPassword=pbkdf2 без env-секрета, NAT-loopback работает —
  дело было только в заголовке.
- `next start` не читает env `HOST` → явный `-H 127.0.0.1`. Сборку от `yearbook`,
  не root. `NEXT_PUBLIC_*` впекаются на build.
