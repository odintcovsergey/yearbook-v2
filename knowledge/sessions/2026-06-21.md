## Шаг 1 — перенос БД Supabase → Timeweb PostgreSQL (✅ выполнен)

- Цель: Timeweb Managed **PostgreSQL 18.4** (Москва, публичный IP). Источник: Supabase
  **PG 17.6**. Инструменты — клиент **postgresql@18** (brew install, не системный 16).
- SSL: подключение по IP → `sslmode=verify-ca` с `ca.crt` (НЕ `verify-full` — в сертификате
  доменное имя, по IP не сходится). `SELECT version()` → 18.4. ✅
- Дамп: `pg_dump "$SUPABASE_DATABASE_URL" --schema=public --no-owner --no-privileges -Fc`
  (через Shared Pooler `aws-1-eu-west-2.pooler.supabase.com:5432`). Заливка `pg_restore`.
- Грабли:
  - 4 таблицы (children/teachers/invitations/responsible_parents) не создавались — юзают
    `extensions.gen_random_bytes` (pgcrypto). Решение: `CREATE SCHEMA extensions; CREATE
    EXTENSION pgcrypto WITH SCHEMA extensions;` (доверенное, владельцу БД можно) → перелив начисто.
  - 2 RLS-политики на `children` (роли `anon`/`service_role`) НЕ доехали — у пользователя
    Timeweb нет CREATEROLE. Решили «потом» (приложение ходит через сервис-ключ). RLS включён
    на всех 52 таблицах, 8/10 политик создано.
- **Сверка: 52 таблицы, 14060 строк, 0 расхождений** (per-table count). Один альбом со связями
  читается (Тест 30: 30 детей/242 фото/5 учителей).

## Шаг 2 — перенос хранилища Supabase Storage + фото → Timeweb S3 (✅ выполнен)

Целевой бакет — один приватный Timeweb S3 (`s3.twcstorage.ru`, `forcePathStyle:true`).

### Код (за переключателем `STORAGE_BACKEND`, по умолчанию `supabase` = поведение как сейчас)
- `lib/storage-twc.ts` — S3-клиент Timeweb (upload/signed-GET/presigned-PUT/getBuffer/list/delete/copy).
- `lib/blob-storage.ts` — диспетчер: `resolveReadUrl` (понимает и старый полный supabase-URL, и
  относительный ключ), `storedValue` (что писать в БД: полный URL в supabase / ключ в timeweb —
  пропущенный читатель ломает только timeweb, не прод), `signForClient`, `signDecorPlaceholders`,
  `serverUpload` (опц. supabase-клиент для тестов), `createUploadTarget`, `removeBlobs`,
  `listBlobs`, `copyBlob`. В Timeweb ключ = `<bucket>/<path>`.
- `lib/blob-upload-client.ts` — клиентская загрузка ветвится по ответу сервера (Supabase token
  vs Timeweb presigned PUT).
- Переписаны: запись/загрузка/удаление в 6 серверных файлах (idml upload, cover upload, covers,
  referral-programs, super backgrounds, tenant) + 4 клиентских места загрузки. Чтение на signed:
  covers-список, реф-картинки (referral-programs/referral/child/tenant — 5 эндпоинтов),
  backgrounds GET, **PDF-экспорт (фон+декор)**.
- ⚠️ ОТЛОЖЕНО на шаг 3 (нужна живая проверка): подпись данных РЕДАКТОРА в `app/api/layout/route.ts`
  (пул фонов/`__bg__`/master-override/default + декор), cover-канвас, `lib/cover/preview-svg.ts`.
  ТОНКОСТЬ: «пикер» фона сохраняет выбор в БД → нельзя класть signed (протухнет за 24ч); хранить
  ПУТЬ, показывать signed — разделить поля и проверить вживую.
- tsc / **1113 тестов** / next build — зелёные. 1 тест декора поправлен под `storedValue`.
