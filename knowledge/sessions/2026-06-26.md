## 1. Диагностика: почему объём Timeweb не растёт после копии заказа

Сергей сделал копию «Тест 30 — копия» (`78cc476a…`) и заметил, что объём хранилища
не вырос. Проверка по БД (только чтение):
- У копии **242 записи `photos`**, но **все 242** `storage_path`/`original_path`
  указывают на префикс **источника** `yc:def23fce…` (album_id «Тест 30»), НЕ на
  собственный префикс копии.
- **Копирование заказа НЕ дублирует файлы в хранилище** — создаются только строки в
  БД, ссылающиеся на те же ключи источника. Поэтому объём не растёт. Это подтверждение
  полного шаринга.
- Own-оригиналов у копии = **0**, shared = 242 (те же файлы держат живые «Тест 30»,
  «Тест 13» + архивный «Тест 24»). Для пробы автоудаления копия НЕ годится — удалять
  нечего (анти-шаринг защитит всё).

## 2. Выбор заказа для пробы

Искали мусорный заказ с СОБСТВЕННЫМИ оригиналами (own>0, не делят живые). Нашли 5
(все в тенанте OkeyBook, демо-данные): Лицей 25 11А (282), **Школа 81 (269, archived)**,
Школа 62 (239), МБОУ Заречье (108), Детский сад 201 (86). Тест-семья 13/24/30/копия —
все own=0 (шарят).

Сергей выбрал **«Школа 81»** (`bb97b2b4…`) осознанно: заказы сданы, исходники есть
локально, удаление не критично.

### 🔑 Урок: `--ttl-days=0` НЕ делает заказ истёкшим без `archived_at`
Первый dry-run `--only=bb97b2b4 --ttl-days=0` вернул **0**. Причина — порядок веток в
`isExpiredAlbum` (core.ts): `if (!a.archived_at) return false` срабатывает РАНЬШЕ проверки
TTL. `--ttl-days=0` схлопывает только ожидание «90 дней», но дата архивации обязана быть.
У всех 5 own-заказов `archived_at` был null (старые архивные до бэкфилла). Сергей
предложил НЕ править боевую БД руками — вместо этого нажал штатную кнопку **«Запустить
отсчёт (90 дней)»** (`extend_archive_ttl`, ставит `archived_at=now`) в карточке заказа.
Это продуктовый флоу, не SQL-хак.

## 3. Проба автоудаления (Фаза 5) — ФИКС ОБНУЛЕНИЯ ПОДТВЕРЖДЁН

Числа «Школы 81»: 269 строк = 269 уникальных оригиналов, 0 дублей, 269 превью (расклад
1:1, удобно для проверки).

- **dry-run** `--only=bb97b2b4 --ttl-days=0`: 269 ключей / 0 анти-шаринг / 1.448 ГБ ✅
- **`--apply`**: `удалено 269 = обнулено 269, ошибок 0` ✅ — скрипт НЕ упал на сверке
  количества. Обнуление прошло 6 пачками по 50 (5×50+19) — то самое место, где раньше
  (баг пробы 25.06) длинный `.in()` молча упирался в лимит URL и обновлял только первую
  пачку (71). Фикс (NULL_CHUNK=50 + обязательная проверка error + сверка `nulledRows ==
  expectedNulled`) работает на боевом объёме.
- **Сверка по БД**: висячих `original_path` = 0, превью 269 на месте (смотрибелен),
  `originals_deleted_at` проставлен, `audit_log` запись `album.originals_autodeleted`
  (keys 269, ~1.55 ГБ), повторный dry-run = 0 (идемпотентно).

## 4. UI-правки — честный текст про отложенное удаление (релиз `6d68e8b`)

Старый UI остался от сломанной логики мгновенного удаления при архивации → два места
врали. Обе правки в `app/app/page.tsx`:

1. **Вкладка «Фото» архивного заказа.** Было: «фотографии удалены из хранилища при
   архивировании». Стало: статус по факту из той же `archiveLifecycleStatus` (core.ts,
   вторую логику не плодим) — countdown «удалятся через N дн.» / not_started / deleted
   «[дата]» / forever. В `PhotosTab` проброшены `archivedAt`/`keepForever`/
   `originalsDeletedAt` (раньше знал только булево `archived`).
2. **Подтверждение архивации.** Было: «удалены с сервера / ссылки для родителей
   перестанут работать». Стало: «через 90 дней оригиналы удалятся автоматически, превью
   останутся». Фразу про ссылки СНЯЛИ: проверка фактом — `/api/child` на `archived` не
   смотрит → после архивации ссылки продолжают работать, claim был ложным.

Гейты: tsc чисто, 1204/1204 vitest, next build зелёный. Прокликано Сергеем на проде.

## 5. Деплой через GitHub SSL-флап (грабля)

Авто-деплой завис ~15 мин: `git fetch origin main` к GitHub с РФ-сервера то падал
(`SSL connection timeout`), то висел весь таймаут → `yearbook-deploy.service` болтался
в `activating`. Распознали, что это ЗАВИСШИЙ ФЕТЧ, а не сборка: `ExecMainStartTimestamp`
пустой, нет процесса `next build`, нет нового `releases/<sha>`.

**Обход (отработал):** доставить коммит git-бандлом с Mac минуя сеть.
1. Mac: `git bundle create yb.bundle ^df30675 main` (инкрементальный, 2.4 КБ; валиден —
   prerequisite `df30675` уже в repo сервера).
2. `scp` → `/srv/yearbook/`, `chown yearbook`.
3. Стоп гонок: `systemctl stop yearbook-deploy.timer yearbook-deploy.service` + мой
   nudge-цикл.
4. Детачем через `systemd-run`: `git -C repo remote set-url origin <bundle>` →
   `env YB_ROOT=/srv/yearbook deploy.sh` (штатный build+health+авто-откат, читает
   `origin/main` из бандла) → ОБЯЗАТЕЛЬНО вернуть `remote set-url origin <github-url>`.
5. `systemctl start yearbook-deploy.timer`, убрать бандл.

Результат: `current → 6d68e8b`, health OK (login/api/health 200), origin назад на GitHub,
таймер авто-деплоя восстановлен.

## 6. Включение таймера чистильщика

1. `cp deploy/timeweb/yearbook-cleanup.{service,timer} /etc/systemd/system/` +
   `daemon-reload`.
2. Сверка `.timer`: `OnCalendar=*-*-* 04:00:00 UTC`, `Persistent=true`, `AccuracySec=5min`.
   ⚠️ ExecStart сервиса = `--apply` (боевое удаление по таймеру — так и задумано).
3. **Холостой прогон** `systemctl start yearbook-cleanup.service` (targets трижды
   подтверждены = 0): журнал `режим APPLY / истёкших 0 / удалено 0 / обнулено 0 / ошибок 0`,
   `Result=success`, `ExecMainStatus=0` — сервис поднялся с правильным окружением под
   юзером yearbook.
4. `systemctl enable --now yearbook-cleanup.timer` → enabled+active. Первое реальное
   срабатывание **Sat 2026-06-27 04:00 UTC**.

Заморозить авточистку = `systemctl disable --now yearbook-cleanup.timer`. 11 старых
архивных с `archived_at=null` чистильщик не трогает, пока не нажать «Запустить отсчёт».
