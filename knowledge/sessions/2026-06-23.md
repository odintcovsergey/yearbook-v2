# Сессия 2026-06-23 — CI/CD авто-деплой на Timeweb (ТЗ №1)

## Итог
Настроен **авто-деплой по push в main** на боевой Timeweb (`app.okeybook.ru`).
Схема — **pull-модель** (сервер сам опрашивает GitHub), потому что входящие
соединения из-за рубежа до РФ-сервера ненадёжны (та же фильтрация, что убивала
HTTP-01 Let's Encrypt и GitHub-релизы). Прод не ломали — всё аддитивно, с откатом.

## Что сделано (по шагам)
1. **Проверка стенда после паузы:** сайт 200, cert LE (до 19.09), `/rest/v1`→401,
   сервисы `yearbook/postgrest/nginx` active. GitHub с сервера тянется
   (`git ls-remote` OK), **GitFlic — НЕ тянется** (FAIL) → primary remote = GitHub.
2. **Мердж `feat/migrate-storage-timeweb` → `main`** (fast-forward, без конфликтов),
   запушен в оба зеркала. Переезд закреплён в main.
3. **Выбор раннера — pull-модель** (Вариант Б). GitHub Actions push (раннер
   SSH-ится внутрь) и self-hosted раннер (control-plane с заблокированного
   `githubusercontent.com`) отвергнуты как хрупкие.
4. **Раскладка `/srv/yearbook`** (capistrano-стиль) рядом со старым `/srv/yearbook-v2`:
   `repo/` (clone), `shared/.env.production` + `shared/node_modules`, `releases/<sha>/`,
   симлинк `current`. sudoers: `yearbook` рестартит сервис без пароля.
5. **Cutover:** `yearbook.service` переключён `WorkingDirectory` на `/srv/yearbook/current`,
   `EnvironmentFile` на `current/.env.production` (секреты те же, `JWT_SECRET` тот же
   → никого не разлогинило). Health прошёл. Бэкап юнита — `yearbook.service.bak-cutover`.
6. **Авто-деплой вооружён:** `yearbook-deploy.{service,timer}` (опрос 2 мин),
   end-to-end проверен на реальном коммите (`726fd3c→ac26e4d` выкатился сам).

## Ключевые находки (грабли)
- **Реестр npm с РФ-сервера НЕДОСТУПЕН** (`registry.npmjs.org` → ETIMEDOUT/000).
  → `npm ci` на деплое невозможен. Решение: общий `shared/node_modules` (засеян из
  рабочего стенда, package-lock идентичен main), сборка `next build` офлайн.
  `deploy.sh` прерывает деплой, если package-lock изменился (нужен ручной rsync
  node_modules с Mac).
- **Порт 3001 = PostgREST** (nginx `/rest/v1/`→`127.0.0.1:3001`). Тест-инстанс
  приложения нельзя поднимать на 3001 — конфликт. Тестировали на 3002.
- **`git fetch` не обновляет рабочее дерево** — `deploy.sh` из `repo/` крутился
  на старом коммите. Фикс: `ExecStartPre` в юните делает `fetch + reset --hard
  origin/main` перед каждым запуском.
- SSH до сервера флапает (обрывы на долгих операциях) → тяжёлые шаги (сборка,
  cutover) запускали детачем через `systemd-run`, опрашивали journal.

## Артефакты (в репо, deploy/timeweb/)
`deploy.sh`, `health-check.sh`, `setup-pull-deploy.sh`, `yearbook-deploy.service`,
`yearbook-deploy.timer`, обновлённый `README.md`; новый роут `app/api/health`.
Старый push-шаблон `deploy-timeweb.yml` помечен заменённым.

## Как выкатить изменение (для Сергея)
Просто `git push` в `main` → через ≤2 мин сервер сам соберёт и выкатит, при поломке
откатится сам. Заморозить: `systemctl disable --now yearbook-deploy.timer` на сервере.
Если коммит менял npm-зависимости — сперва обновить `shared/node_modules` (rsync с Mac).

## Хвосты (не в этой сессии)
- Docs-only коммит сейчас тоже вызывает полную пересборку+рестарт (можно позже
  оптимизировать: пропускать билд, если изменения только в `knowledge/`/`docs/`).
- Без-даунтайма (blue-green на втором порту) — апгрейд на будущее; сейчас рестарт
  моргает ~3-5с.

# Сессия 2026-06-23 (часть 2) — Ночной off-site бэкап БД на актуальный источник

## Итог
Ночной бэкап переключён со **старого мёртвого Supabase** на боевую базу Timeweb.
Главная находка: **GitHub Actions до РФ-базы НЕ доходит** — поэтому бэкап перенесён
**на сам VDS** (второй слой off-site поверх встроенных бэкапов Timeweb). Проверено
вживую end-to-end, включая после смены засветившегося пароля.

## Ключевая находка
- База Timeweb — отдельная управляемая **PG 18.4** на `62.113.37.145:5432` (НЕ тот
  VDS, где сайт). С РФ-машины TCP 5432 открыт (whitelist по IP нет), но с
  заграничного раннера GitHub (Azure) → `psql ... timeout expired`. Та же
  фильтрация, что рубит npm/postgresql.org с сервера. Вывод: GitHub Actions для
  бэкапа РФ-базы непригоден.

## Что сделано
1. **`.github/workflows/backup.yml`** сначала перенацелен на Timeweb (pg-client
   17→18, sslmode, проба связи, S3-опись вместо Supabase-листинга) и закоммичен.
   Ручной прогон показал `timeout 5432` → расписание **отключено** (оставлен только
   `workflow_dispatch`), чтобы не падал каждую ночь.
2. **Бэкап перенесён на VDS** (`/root/yearbook-backup/`, под root):
   - `backup.sh`: `pg_dump` (schema public, `-Fc`) → пуш off-site в приватный
     `github.com/odintcovsergey/yearbook-backups` (`database/`, храним 30 по имени);
     по воскресеньям — опись S3 (node+@aws-sdk из `/srv/yearbook/current`).
   - `yearbook-backup.{service,timer}` — ежедневно **03:00 UTC (06:00 МСК)**.
   - Авторизация в приватный репо — **SSH deploy-key** (`.ssh/id_ed25519`,
     Allow write access). Не PAT — Сергей не хочет истекающие токены.
   - Секрет БД — root-only `backup.env` (600), читается через `sed` (НЕ `source`:
     в пароле спецсимволы шелла).
3. **pg_dump 18** поставлен через **зеркало Яндекса**
   (`mirror.yandex.ru/mirrors/postgresql/`) — apt.postgresql.org с РФ-сервера
   недоступен, Ubuntu-репы дают только client-16 (не дампит сервер 18).
   gpg-dearmor с `--batch --yes`.
4. **Смена пароля gen_user** (засветился при отладке): Сергей сменил в панели
   Timeweb + обновил `.env.local`; новый перенёс на сервер. Проверено — старый
   больше не действует.

## Грабли
- `source backup.env` ломался: в пароле gen_user символы `| ( {` → читаем строку
  через `sed -n 's/^KEY=//p'`.
- Чистка `find -mtime +30` на свежем `git clone` НЕ работает (у всех файлов дата
  клонирования) → заменено на «храним последние 30 по имени» (`sort | head -n -30`).
- gpg `--dearmor` без `--batch --yes` висит на запросе перезаписи через /dev/tty
  (в systemd-run tty нет).
- GitHub отбивал push workflow-файла и dispatch: fine-grained PAT нужны права
  **Workflows: R/W** (push) и **Actions: R/W** (запуск через API). В кеше Keychain
  лежал старый токен без них — пересоздан.

## Проверено вживую
Дамп 1016К / **52 таблицы** с данными; пуш в `yearbook-backups` (commit `78d4f72`);
независимый ре-клон с GitHub подтвердил файл + валидность; в репо ровно 30 дампов.
