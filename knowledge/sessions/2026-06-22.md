## 🔴→✅ Главный блокер cutover закрыт: `/rest/v1` больше не открыт наружу

**Было:** nginx срезал `Authorization`, PostgREST без `jwt-secret` пускал любой
запрос под `gen_user` → снаружи `/rest/v1` отдавал данные на запрос БЕЗ токена
(проверено: 200 + реальные id альбомов). Дыра на чтение+запись всей БД.

**Анализ прав:** `gen_user` — без superuser/createrole/bypassrls (проверено по
`pg_roles`). Боевые роли `authenticator`/`web_app` создать НЕЛЬЗЯ без суперюзера
Timeweb. НО для закрытия двери раздельные роли не нужны.

**Ключевой факт:** `JWT_SECRET` приложения (96 симв., вход пользователей) НЕ
подписывает supabase-ключи (проверено HMAC) — значит слой PostgREST можно сделать
полностью независимым, не трогая вход.

**Разведка по коду (Explore):** браузер в `/rest/v1` НЕ ходит — все DB-запросы
только серверные (API-роуты + `supabaseAdmin`/service role). Anon-ключ
(`NEXT_PUBLIC_*`) используется лишь для загрузки файлов в Storage, не для БД →
**пересборка не нужна**, меняем только серверный токен.

**Что сделал на сервере (бэкапы конфигов: суффикс `20260622-043610`):**
1. PostgREST `postgrest.conf`: задал **`jwt-secret`** (отдельный случайный 64-симв.
   секрет, `openssl`-стиль, живёт только в конфиге), **убрал `db-anon-role`** →
   запрос без валидного JWT = `401 "Anonymous access is disabled"`.
2. Выпустил **служебный токен** HS256 `{"role":"gen_user"}`, подписан тем секретом
   → записал в `.env.production` как `SUPABASE_SERVICE_ROLE_KEY` (server-only).
   `gen_user` подключается, claim role=gen_user → `SET ROLE gen_user` (self, ок).
3. nginx: **убрал срез `Authorization`/`apikey`** → токен доезжает до PostgREST.
4. Перезапустил postgrest + yearbook, reload nginx.

**Проверено вживую:**
- снаружи без токена → 401; фейк-Bearer → 401; старый supabase service-ключ → 401
  (подписан не нашим секретом); локально на :3001 с настоящим токеном → 200.
- приложение читает+пишет базу: POST `/api/auth` login с несуществующим email →
  чистый `{"error":"Неверный email или пароль"}` 401 (не 500) — значит дошло до
  БД (SELECT login_attempts/users + INSERT login_attempts).
- `/login` → 200.

**NB:** least-privilege роли `authenticator`/`web_app` — ОПЦИОНАЛЬНЫЙ хардненинг
на потом (нужен суперюзер Timeweb). Дыру они НЕ закрывали — её закрыл jwt-secret.

## 🔴→🟡 БЛОКЕР доступности из РФ + смена IP

После cutover: `app.okeybook.ru` НЕ открывается из РФ без VPN. Диагностика (сервер исправен,
файрвола нет, DNS верный): пакеты к IP **`5.42.103.168`** режутся в сети ВЫШЕ сервера — TCP
проходит (DPI спуфит SYN/ACK), HTTP/HTTPS-данные НЕ доходят до nginx (в access-логе их нет).
check-host: HTTP-80 таймаутит с 28/30 узлов вкл. РФ. То же, что мешало Let's Encrypt.
**Timeweb подтвердил: это ТСПУ (нац. DPI), причина внешняя, по всей РФ, смена IP — лотерея.**
MTU/MSS ни при чём (на сервере уже `mtu 1400`+MSS 1360, инбаунд режется не из-за пакетов).

**Решение (сработало пока):** доп. публичный IP Timeweb (вкладка «Сеть→Публичные IP», не
«плавающие»; ~180₽/мес) — `5.129.221.197`, привязан к серверу, добавлен на eth0 вручную
(`ip addr add /32`, основной не тронут) + закреплён `systemd floating-ip.service`. Проверка
нового IP ДО смены DNS: check-host HTTP-80 = 34 OK/1 (старый 2/28), HTTPS отвечает со всех
РФ-узлов, **телефон Сергея без VPN ОТКРЫЛ**. Сертификат менять не надо (на домен, валиден на
любом IP — `ssl_verify=0`). Сергей сменил A-запись на reg.ru: → `5.129.221.197`.
